Rate Limiter

Rate Limiter：从需求语义到故障修复

01

从限流语义选择算法，而不是先背 Redis

固定窗口只保存一个计数器，空间很省，却允许用户在窗口交界的两秒内用掉两倍额度；滑动日志逐条保存时间戳，判定最精确，但高频客户会让内存和清理成本随请求数增长；滑动计数器用前后两个窗口加权，代价是近似误差。令牌桶只保留 tokens 与 last_refill，bucket capacity 表达允许的突发，refill rate 表达长期速率，适合 API 的脉冲流量。设计时必须先问清“100 次/分钟”是否容忍边界突发、闲置客户是否满桶启动、一次请求会同时命中用户、IP、端点和全局几条规则。最终允许与否取所有适用规则中最严格的结果，并用 429、remaining、resetTime 和 Retry-After 把服务端判定变成客户端可执行的退避协议。

能迁移的判断：先把突发、公平、内存和误差预算说清，再选择令牌桶或窗口算法。
02

限流器放在哪里，决定了它看得见什么

进程内计数没有网络开销，却只看见落到本实例的流量；五台应用服务器各自允许一百次，就可能把全局上限放大五倍。独立限流服务能读取订阅等级等业务上下文，但给每个请求增加一次 RPC，也成为新的故障点。放在 API Gateway 可在请求进入后端前拒绝滥用，并直接读取 JWT 的 userId、X-API-Key、路径与可信代理写入的客户端 IP，因此是常见选择；代价是高级业务属性必须编码进令牌或本地规则缓存。多层规则的键应明确写成 scope:type:id:endpoint:ruleVersion，不能只按 IP，因为公司 NAT 会把许多正常用户压成一个热点。网关还要防止客户端伪造 X-Forwarded-For，并区分认证失败、限流拒绝和限流依赖故障。

能迁移的判断：部署位置、身份提取和规则作用域必须在同一张请求时序图上解释。
03

用 Lua 封住令牌桶的读—算—写竞态

先 HMGET、在网关计算补充量、再 HSET，即使用 MULTI/EXEC 包住写入也不安全：两个网关可能读到同一枚剩余令牌并都放行。正确边界是一段单键 Lua 脚本，在 Redis 内部读取 tokens 与 last_refill，以 max(0, now-last_refill) 计算补充量，截断到桶容量，再判断是否扣除，最后同时写回并设置空闲 TTL；脚本一次返回 passes、remaining 和 resetTime。规则与桶参数要带版本，避免规则从每分钟一百次改成十次时沿用不兼容状态。Redis Cluster 分片时把 clientId 与 ruleId 放在同一个 hash tag，保证脚本不跨 slot。时间可以由 Redis 端统一取得，或传入网关时间但限制倒退，绝不能让负的时间差凭空增加额度。

能迁移的判断：原子性必须覆盖完整的读取、补充、判定和扣减，而不是只包住两条写命令。
04

把一百万次判定拆成可计算的分片预算

目标是一百万请求每秒，而单个 Redis 实例对复杂脚本的可持续吞吐往往低于简单 GET 的十万到二十万次每秒，所以不能用宣传峰值直接除法。先压测真实 Lua、键大小、连接数和副本复制开销；若安全值是每分片八万次判定，正常流量至少需要十三个主分片，再按一台失效仍不超过百分之七十利用率增加余量。Redis Cluster 的 16384 个 slot 或一致性哈希虚拟节点让增减节点只搬动部分桶，但迁移期间网关必须处理 MOVED/ASK 与短暂重试。普通 clientId 可均匀分散，单个共享 NAT 或遭攻击的 API key 却不能靠加分片解决，因为一个桶必须串行更新。此时应在边缘先封禁明确恶意源，对合法大客户提供批量 API、专属额度或分层的本地小配额。

能迁移的判断：容量计算要同时覆盖总吞吐、单键串行上限和失去一个分片后的余量。
05

Redis 故障时，fail-open 与 fail-closed 是业务决策

主分片不可达时，直接 fail-open 会让洪峰越过保护层，可能把数据库一起压垮；统一 fail-closed 又会让健康后端被限流器拖成全站不可用。应按规则风险分级：登录防爆破、付费写入和正在遭攻击的端点倾向关闭放行；低风险读取可使用网关预留的保守本地令牌并标记 degraded。主从切换还存在异步复制窗口，新主可能缺少刚扣掉的令牌，造成短暂超发，因此要把“允许少量超发”写进一致性预算，而不是声称副本带来无损切换。多地域若追求十毫秒延迟，可把全局额度静态切成各区配额并保留应急池；跨区每次强同步虽然更精确，却会把广域往返放到所有请求路径。故障指标必须区分依赖超时、降级放行、降级拒绝和估算超发量。

能迁移的判断：先确定后端保护与用户可用性的优先级，再为不同规则选择降级模式。
06

动态规则必须带版本、传播界限和回滚路径

生产限额会因促销、攻击或付费等级随时变化。每三十秒轮询配置库实现简单，但紧急收紧规则最多要等一个轮询周期；推送能更快，却必须处理网关断线、重复事件和启动时漏消息。可将规则保存为不可变版本，推送只通知新版本号，网关收到后从权威配置服务拉取完整快照、校验签名，再原子替换本地表；心跳携带 appliedVersion，使控制面能看见仍在旧版的实例。变更发布先到少量网关，观察 429 比例、后端饱和度和判定 p99，再逐步扩散；异常时回退到上一个已知版本，而不是手工改 Redis 桶。若新规则改变桶容量或算法，还要定义旧桶如何截断、重建或按版本并存，防止配置已回滚但状态没有回滚。

能迁移的判断：配置更新不是缓存刷新细节，而是一套可观测、可回滚的控制面协议。

Distributed Cache

Distributed Cache：从需求语义到故障修复

01

先定义缓存 API 的可观察语义

get(key) 不能只返回 value，还应区分命中、键不存在和键已过期；set(key,value,ttl) 要明确覆盖旧值是否重置 TTL，delete 是否幂等，以及最大 key、value 尺寸。缓存不是永久数据库，但一旦 set 已返回成功，客户端仍会期待在约定窗口内读到刚写的版本，因此每条记录至少包含 value、version、expireAt，并把 LRU 访问元数据视作淘汰提示而非业务数据。条件写可暴露 compare-and-set(expectedVersion)，帮助上层避免旧值覆盖新值。TTL 取绝对服务端时间还是相对时长也要固定，否则客户端时钟漂移会制造提前过期。接口语义写清后，才能讨论复制是保证已确认值存活，还是允许节点故障产生缓存未命中。

能迁移的判断：缓存可以丢，但 API 仍必须精确定义成功、过期、覆盖和条件更新。
02

一致性哈希环负责找节点，不负责数据正确

对 key 计算稳定哈希后顺时针找到节点，增减节点只搬动相邻区间；但物理节点数量少时分布会很不均，应该给每台机器放置多个 virtual nodes，并按内存容量分配不同数量。路由可放在客户端库，少一次代理跳转却让成员列表升级困难；也可经无状态 proxy，集中处理拓扑和重试但增加网络开销。成员变更必须有 epoch，客户端若按旧环把 set 发给旧所有者，应得到重定向而不是两个节点都接受写入。迁移采用先复制区间、校验水位、再切所有权，切换后旧节点只转发或拒绝。哈希环解决平均分配，无法解决单个热门键，因为这个键依然只有一个所有者。

能迁移的判断：虚拟节点改善区间均衡；epoch 和所有权切换才防止扩容期间双主写。
03

主副本切换要明确允许丢失多少已确认写

每个分片可设一个主节点和若干副本，get 分散到副本能提高读吞吐，但异步复制意味着副本可能读到旧值，主节点刚确认的 set 也可能在故障提升时丢失。若缓存用途允许把这次故障等价成 miss，就可优先可用性，并让上层回源重填；若上层把缓存当会话或锁存储，语义已不再是普通缓存，需要同步复制、仲裁或改用更合适的事实库。切主由带 epoch 的成员服务决定，新主只接受更高 epoch，旧主恢复后不能继续写。读己之写可以短时间粘到主节点，或让客户端携带 minVersion，副本落后就回主。高可用不是“有副本”四个字，而是故障检测、提升、路由刷新和陈旧窗口的完整协议。

能迁移的判断：先把节点故障解释为 miss、旧读还是已确认写丢失，再选择复制强度。
04

TTL 与 LRU 都需要近似，否则维护成本会吞掉命中收益

每次 get 先比较 expireAt，过期则按 miss 返回并异步清理，这是 lazy expiration；只靠惰性删除会让长期不访问的过期键占内存，因此后台还需分批扫描过期索引或随机采样，避免一次全表清扫造成延迟尖峰。精确 LRU 若每次读取都移动全局双向链表，在高并发下会形成写热点和锁竞争。生产实现常按分片维护近似 LRU，采样若干键的 lastAccess，或用分段队列区分新近与频繁访问。淘汰必须先确认条目版本，防止扫描线程删除刚被覆盖的新值；大 value 的释放也应脱离请求线程。监控 expired、evicted 和 overwritten 三类原因，才能判断是 TTL 设置、内存不足还是写入模式导致低命中。

能迁移的判断：过期保证语义，淘汰管理容量；两者的扫描与元数据更新都要有延迟预算。
05

热读键和热写键是两种完全不同的问题

热门配置被大量读取时，值不变或可短暂陈旧，可把副本扩到更多节点、在 proxy 做 request coalescing，甚至放进进程内 near-cache；版本或短 TTL 控制陈旧，单次回源填充防止缓存击穿。热门计数器被持续 set 时，复制并不会增加写吞吐，因为所有更新仍需在一个顺序上合并。可以检查业务是否允许按来源拆成子键再异步汇总，或把更新改成交换律操作；不允许拆分时只能给该键独立分片、批量合并并施加写入背压。随机复制出多个可写副本会发生覆盖和版本倒退。面试中应先问热点是读、写还是大对象网络带宽，再选方案，而不是统一回答“加副本”。

能迁移的判断：热读靠复制与合并请求；热写必须改变更新模型或承认单键顺序上限。
06

低于十毫秒要优化整条尾延迟，而非只报内存速度

内存查找是微秒级，但客户端建连、proxy 排队、序列化、跨可用区网络和副本超时会把 p99 推到十毫秒之外。客户端使用持久连接池，批量 mget/mset 降低往返；proxy 对大 value 设上限并避免无谓拷贝；分片线程按核隔离，后台过期扫描和再平衡限速。超时重试必须有总预算和同一拓扑 epoch，盲目向多个节点重试会在过载时放大流量。扩容后新节点没有热点数据，命中率骤降会把数据库压垮，所以迁移应预热或限速，并给回源加 single-flight 与并发上限。看板同时展示 get p50/p99、命中率、每分片内存、网络字节、eviction、热点键占比和客户端超时。

能迁移的判断：缓存性能是网络、排队、序列化、后台任务和回源压力共同决定的。

Job Scheduler

Job Scheduler：从需求语义到故障修复

01

把一次性、延时与周期任务统一成明确的数据模型

JobDefinition 保存用户要执行的命令、时区、重试策略和 cron 表达式；JobRun 表示某次具体触发，拥有 runId、scheduledAt、attempt 和状态。一次性任务只产生一个 run，周期任务不能让 worker 执行完后随手算下一次，否则 worker 崩溃会漏排；调度器应根据 definition 的 nextRunAt 事务地产生 JobRun，并推进下一触发时间。cron 必须绑定 IANA 时区，夏令时跳过或重复的本地时间要事先规定“跳过一次、执行一次还是执行两次”。取消定义只阻止未来 run，是否中断已经 running 的实例是另一条 API。把定义与执行分开，才能回答修改计划、补跑、重试和审计时究竟影响哪一个对象。

能迁移的判断：JobDefinition 描述意图，JobRun 记录每次执行；不要用一个 status 字段混合两者。
02

两秒准点目标来自 due 索引、扫描步长和队列延迟

若每秒全表查 scheduledAt 小于现在，数据量一大就无法守住两秒。可按 nextRunAt 建组合索引，并把时间切成秒级或更细的 bucket；多个 scheduler 各负责一组时间分区，提前读取未来一小段 lookahead，把到期 run 放入延时队列。误差预算要拆开：数据库扫描五百毫秒、入队三百毫秒、broker 排队四百毫秒、worker 领取四百毫秒，还需给网络抖动留余量。lookahead 太长会让取消和改期难以及时生效，太短又容易被一次暂停错过。调度器重启后从持久水位回扫最近窗口，按 runId 去重。墙上时钟只用于判定计划时刻，进程内等待应使用单调时钟，避免 NTP 回拨造成重复触发。

能迁移的判断：准点不是一个组件属性，要为扫描、排队、领取和时钟误差分别分配预算。
03

领取任务要靠租约和 fencing token，而不是永久锁

worker 从队列得到 runId 后，以条件更新把状态从 QUEUED 改为 RUNNING，同时写 owner、leaseUntil、attempt 和递增 fencingToken。租约允许 worker 崩溃后由 reaper 重新入队；但旧 worker 可能只是长时间 GC，醒来后仍继续写结果，所以所有受保护的下游更新都必须携带 fencingToken，下游拒绝小于当前值的操作。只用 Redis SETNX 加过期时间无法阻止过期持有者。长任务按心跳续租，续租失败就停止产生副作用；短任务可一次执行完再提交。scheduler leader election 只决定谁扫描某个分区，不能替代每个 JobRun 的领取原子性。租约长度应大于正常心跳抖动，又不能长到故障后数分钟无人接管。

能迁移的判断：租约解决可接管，fencing token 解决旧持有者复活后继续写。
04

至少一次执行不等于业务副作用可以重复

worker 完成付款后、把 JobRun 标成 SUCCEEDED 前宕机，系统只能重新投递，因此调度层最多保证至少一次；所谓 exactly-once 若跨越外部系统通常只是把重复隐藏在幂等协议里。每个 runId 是稳定幂等键，调用支持幂等的下游时直接传递；写自有数据库时，在同一事务中记录 effect(runId) 和业务变更，重复 attempt 读到 effect 就返回原结果。若还要发事件，用 transactional outbox 随状态提交，再由 relay 重发。对不可幂等的邮件或第三方命令，要在产品层接受可能重复、改成人工确认，或引入可查询的供应商请求 ID。重试策略区分瞬时、永久和未知结果，未知不能直接当失败重做。

能迁移的判断：调度器负责重投；业务处理器必须用 runId 把重投收敛为一次可见效果。
05

一万任务每秒需要按时间与租户双重隔离

一万 run/s 不只是队列吞吐，还包含写 JobRun、扫描、入队、状态心跳、完成写和重试，若平均每个 run 有六次持久操作就是六万次写每秒。按 scheduledAt bucket 分区能并行扫描，但某个整点可能聚集大量 cron；可给用户加入可声明的 jitter，把非严格任务散到几十秒窗口。broker 按 tenantId 或 jobType 分区，worker pool 按资源类型隔离，避免一个慢视频任务占满执行线程。每租户设置并发额度与加权公平队列，超过总容量时先拒绝新低优先级计划或延迟非关键 run，而不是让所有任务共同超时。扩容信号看 oldest-due age 与 schedule-to-start p99，队列长度本身无法区分正常长任务和调度已经迟到。

能迁移的判断：把一次 run 的写放大算全，并用时间分桶、资源池和租户配额控制整点洪峰。
06

状态机、重试与补跑共同构成可运营的调度器

JobRun 的合法转换可限定为 SCHEDULED→QUEUED→RUNNING→SUCCEEDED/FAILED，租约过期产生新 attempt 而不是把历史覆盖掉；CANCEL_REQUESTED 需要 worker 协作确认，不能在控制面直接伪造 CANCELLED。重试由 retryAt 索引重新进入调度路径，指数退避带随机抖动，并为永久错误进入 dead-letter 状态。运维页面要显示 scheduledAt、startedAt、attempt、owner、最后心跳和错误分类，使用户能区分“尚未到期、排队、正在运行、结果未知”。补跑按 definition 和时间范围生成新的 runId，并记录 parentRunId，防止和自动重试混淆。核心 SLO 是两秒内开始比例、重复副作用数、租约接管时间、最老到期任务年龄和死信增长率。

能迁移的判断：保留每次 attempt 的证据，才能安全重试、解释延迟并做有边界的补跑。
