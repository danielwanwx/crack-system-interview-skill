接口与状态

输入tenant / user / API key、route、请求成本、policy version。
输出allow 或 429；返回 remaining、reset / Retry-After。
策略每个 tenant + route 的 capacity、refill rate、burst；管理员配置异步下发。
不可省略身份归属、key 维度、429 语义、policy 版本和审计。

算法选择

Fixed Window实现简单；窗口边界可双倍突发。仅用于容忍边界突发的粗粒度配额。
Sliding Window Counter更平滑；近似计算。需要较均匀流量限制时使用。
Token Bucket允许受控 burst；按 refill rate 恢复。默认 API 限流算法。
Leaky Bucket平滑出流；队列与延迟成本。保护下游吞吐，而非简单拒绝请求。

Redis 执行路径

Key

rl:{tenant}:{route}

存 tokens 与 last_refill_ms；TTL 覆盖空闲 bucket。

Atomic

Lua / server-side script

读取状态，按时间 refill，判断成本，扣减并设置 TTL。

Hot Key

不要盲目分片

先识别 tenant 级热点；局部预检查、分配子配额或专用限流器。

Consistency

预算不是账本

本地快速拒绝可接受；全局严格配额必须承担跨地域协调延迟。

故障动作

Redis 超时按 route 风险选择 fail-open 或 fail-closed；记录 bypass 指标。
配置滞后使用带版本的本地缓存；保留最后已知策略和短 TTL。
单 tenant 热点tenant 维度监控；局部配额、隔离池或更严格的 route policy。
多地域先 region-local limit + 全局软配额；只有强业务约束才引入中央协调。
