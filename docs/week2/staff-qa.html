第 2 周 · Staff Q&A

Distributed Rate Limiter

20 个动态追问节点Senior / Staff 压力测试
Round 1

范围与语义

01

请设计一个保护公开 API 的 distributed rate limiter。你先确认什么？

回答必须覆盖
限流对象、key 维度、配额、burst、拒绝语义、区域和风险等级。
继续追问
为什么你选 tenant + route，而不是只按 IP？
02

被拒绝的客户端会收到什么？

回答必须覆盖
429、Retry-After / reset、remaining 的契约；不得泄露内部策略。
继续追问
重试风暴如何避免？
03

哪些请求可以 fail-open，哪些必须 fail-closed？

回答必须覆盖
按支付、写入、读取等业务风险区分，不给统一答案。
继续追问
Redis 不可用时，你的决策依据是什么？
04

配置如何变更？

回答必须覆盖
policy version、审计、异步下发、本地缓存与回滚。
继续追问
旧策略和新策略并存时如何处理？
Round 2

算法与数据

05

为什么选 token bucket？

回答必须覆盖
capacity、refill rate、burst 行为和请求 cost。
继续追问
fixed window 的边界突发在哪里？
06

Redis 中存什么？

回答必须覆盖
tokens、last_refill timestamp、TTL、key cardinality。
继续追问
并发请求如何避免双重消费 token？
07

Lua 脚本完成哪些原子动作？

回答必须覆盖
read-refill-check-decrement-expire 一次执行。
继续追问
脚本超时或 Redis cluster 重定向怎么办？
08

大客户产生 hot key 怎么办？

回答必须覆盖
先测量；局部预检查、子配额、隔离或专用路径。
继续追问
直接把 key 随机分片会改变配额语义吗？
Round 3

高层设计

09

限流器放在什么位置？

回答必须覆盖
gateway / edge 早拒绝；业务服务仍可保留细粒度保护。
继续追问
gateway 执行和服务端执行的 policy 如何一致？
10

读取 policy 的路径是什么？

回答必须覆盖
本地 versioned cache；控制面与数据面分离。
继续追问
控制面挂了，数据面如何继续？
11

Redis 是单点吗？

回答必须覆盖
cluster、replica、failover；明确一致性与可用性影响。
继续追问
replica lag 是否可用于扣减 token？
12

缓存和限流如何一起出现？

回答必须覆盖
缓存降低下游压力；限流保护入口与 cache miss 回源。
继续追问
cache stampede 时只限用户请求够吗？
Round 4

故障与扩展

13

跨 region 的严格全局限制怎么做？

回答必须覆盖
region-local bucket + 全局配额；强一致要说明协调成本。
继续追问
何时值得接受额外延迟？
14

攻击者轮换 API key 或 IP？

回答必须覆盖
多维策略、身份信誉、route cost、异常检测；不只增加阈值。
继续追问
共享 NAT 的误杀如何处理？
15

你会看哪些指标？

回答必须覆盖
allow/deny、Redis latency/error、bypass、hot keys、policy propagation lag。
继续追问
哪一个指标触发 failover 或紧急降级？
16

如何验证新算法没有误杀？

回答必须覆盖
shadow mode、按 tenant rollout、拒绝样本审计、rollback。
继续追问
如何区分真实滥用与配置错误？
mock ≥ 75 / 100 · API、Redis 原子路径、两个 failure deep dives。
