Senior / Staff Drill

Rate Limiter

Q1

两个网关同时看到同一用户只剩一枚令牌，为什么 MULTI/EXEC 仍可能超发，你会怎样实现？

为什么面试官会追：候选人把 HMGET 放在事务外，或只说“Redis 命令原子”却没有覆盖读改写。

详细回答

问题发生在事务开始前：A、B 都读到 tokens=1，各自算出可放行，然后依次写回 0，最终两个请求都通过。我要把取值、按 elapsed×refillRate 补充、截断到 capacity、扣一枚、写 last_refill 和设置 TTL 全部放进同一个 Lua 脚本，由 Redis 对该键串行执行。脚本返回同一次判定的 passes、remaining、resetTime，网关据此生成 429 头；键用 clientId 与 ruleId 组成，并在 Cluster 中用 hash tag 保证不跨 slot。若时钟倒退，elapsed 取零；规则版本变化则按版本选择迁移或重建桶，不能把旧参数悄悄套到新规则。

继续追问：如果一次请求同时受用户、IP 和端点三条规则约束，你怎样避免扣了一条后另一条失败？

常见漏项：把 Redis 的单条 HSET 原子性误当成整个令牌桶判定原子，漏掉外部读取造成的竞态。

现场修复

写出两个并发请求的交错步骤，再把所有会改变允许结果的操作圈进同一原子边界。

证据
  • Rate Limiter · High-Level Design
  • Redis · Redis for Rate Limiting
Q2

全球多地域都要低于十毫秒判定，同时又声明每用户每分钟一百次，你如何解释一致性语义？

为什么面试官会追：候选人既承诺全球精确限额，又让每个地域独立写本地 Redis，没有说明冲突。

详细回答

跨洲强同步一次令牌扣减通常无法守住十毫秒，因此我会先让产品选择“全球精确”还是“低延迟近似”。若接受有界误差，可把一百枚额度分配到各地域，例如按历史流量给 45/35/15，保留五枚中央应急池；每区只扣本地配额，周期性上报消耗并再平衡。用户跨区会暂时受较小额度，极端情况下总放行仍被已分配配额上界限制。若安全接口必须精确，则将该用户固定归属一个 home region，其他地域转发判定并接受更高延迟。无论哪种方案，都要监控配额耗尽率、跨区转发 p99 和切区期间的双写窗口。

继续追问：home region 故障切换时，怎样避免旧区恢复后和新区同时发放令牌？

常见漏项：只说最终一致性可接受，却没有给误差上界、地域额度或用户可观察到的行为。

现场修复

用一个一百枚额度的具体数字演算正常、跨区漫游和单区故障三个场景。

Q3

一个企业 NAT 地址每秒发来五万次合法请求，单个 Redis 键已经打满，你会拆键吗？

为什么面试官会追：候选人看到热点就复制计数器或随机分片，没有考虑多个可写桶会扩大额度。

详细回答

直接把同一 IP 的桶拆成十个独立可写键，会在并发和故障下把可用额度放大，除非事先把总额度严格切成子配额。我会先判断它是攻击还是共享身份：恶意源在 CDN/WAF 边缘封禁，避免流量抵达 Redis；合法企业客户应优先用 API key 或 userId 细分身份，并提供批量接口减少判定次数。若只能按 IP，可给该客户专属分片并把一万次总额度预分给若干网关，每份很小且短租约，到期不续就失效。这样热点被摊开，最大超发受“已发放但未消费的本地额度”限制。还要单独观察该键脚本耗时、额度租借量和封禁命中，而非只看集群平均 CPU。

继续追问：本地小配额所在网关崩溃后，未消费令牌何时可以安全回收？

常见漏项：把热门 IP 一概当成攻击，或用读副本承接需要原子扣减的写热点。

现场修复

先分类合法与恶意流量，再为合法场景写出子配额总和不超过全局额度的约束。

Q4

安全团队要求十秒内把登录接口从每分钟一百次降到十次，怎样保证网关不会长期各用各的规则？

为什么面试官会追：候选人只说把规则放缓存，没定义传播时限、版本观测或失败实例处理。

详细回答

配置库保存带单调版本的不可变规则，控制面推送“版本 42 可用”事件；网关收到事件后拉取完整快照、校验，再一次性把 activeVersion 从 41 切到 42，避免逐条更新产生混合规则。断线网关靠短周期轮询和启动时全量同步补漏，心跳持续上报 appliedVersion；十秒期限到达仍停在 41 的实例从负载均衡摘除，不能继续服务登录流量。规则从一百降到十时，已有桶按新容量截断，且判定日志记录 ruleVersion，方便解释同一时刻的不同结果。发布先灰度并观察误拒率，若配置错误则推送 43 号回滚版本，而不是让版本号倒退。

继续追问：推送事件先到、规则快照后到，或同一版本重复送达时会发生什么？

常见漏项：把消息推送当作可靠广播，没有轮询兜底、版本水位和过期实例隔离。

现场修复

补画控制面状态机：已发布、已拉取、已应用、超时摘除和新版本回滚。

证据

Q5

一百万请求每秒的目标下，你如何计算 Redis 分片数并证明扩容不会破坏同一客户的限额？

为什么面试官会追：候选人直接说部署十台 Redis，没有用真实脚本吞吐、故障余量或重映射说明数字。

详细回答

我先用生产等价 Lua、值大小和副本配置压测，假设单主节点在 p99 预算内只能稳定做八万次判定；一百万除以八万向上取整是十三个主分片。为容忍一台失效且平时不超过七成利用率，实际应再留至少百分之三十到四十余量，而不是恰好十三台。路由键取规范化的 client identity 加 ruleId，使同一桶永远落在一个 slot；Redis Cluster 的 16384 slots 或一致性哈希虚拟节点在加节点时只迁移部分键。迁移期间客户端正确处理 MOVED/ASK，且同一桶不能同时接受两个主写。验收看每分片判定 p99、slot 倾斜、迁移重试和超发估算。

继续追问：如果一个 clientId 自身超过八万次每秒，为什么再加普通分片没有帮助？

常见漏项：用简单 GET 的峰值吞吐估算 Lua 判定，并忽略副本、故障和单热点键的串行瓶颈。

现场修复

把目标 QPS、单分片安全值、利用率上限、N+1 故障和热点例外逐项写入容量表。

Senior / Staff Drill

Distributed Cache

Q1

只有三台物理缓存节点时，一致性哈希为什么仍可能严重偏斜，扩容迁移又怎样避免双写？

为什么面试官会追：候选人只画三个点的哈希环，就声称键会天然均匀且扩容无需协调。

详细回答

三个随机落点会形成长度差异很大的区间，命中键数和内存压力不会自动均匀。我要让每台物理机拥有许多 virtual nodes，并按机器容量分配权重，使区间细化后更接近均摊；扩容时只把新节点接管的虚拟区间搬走。哈希仍只计算“按某份成员表应该去哪里”，所以拓扑要带单调 epoch：先把 A 的目标区间复制到 B，追平水位后切换所有权；B 只在新 epoch 接受写，A 随后只重定向旧客户端。这样虚拟节点处理分布与搬迁粒度，epoch 和切换顺序处理新旧路由并存时的双主风险。

继续追问：客户端长时间离线后带着旧 epoch 重连，服务端返回什么信息才能安全自愈？

常见漏项：把有限重映射误说成原子拓扑变更，遗漏新旧路由并存的现实。

现场修复

画出 copy、catch-up、cutover、redirect 四阶段，并标注每阶段唯一写入者。

Q2

一个键每秒读五十万次，另一个键每秒写五十万次，你会用同一种热点方案吗？

为什么面试官会追：候选人对所有热点都回答加读副本或加机器。

详细回答

不会。热读值若允许几十毫秒陈旧，可复制到多个读副本，并在 proxy 或客户端 near-cache 中按版本缓存；首个 miss 用 single-flight 合并，避免所有请求同时回主。热写键则必须保持版本顺序，增加只读副本不会提高主节点处理 set 的能力。我要先找出更新是否可交换：计数器可拆成多个分片增量后汇总，普通覆盖写可按业务子键拆分；若语义要求严格的最后写入者，只能把该键隔离到专属分片、批量合并并对写方背压。两个场景分别监控副本陈旧度与主键队列长度，不能用集群平均 QPS 掩盖。

继续追问：near-cache 在键被更新时怎样避免几千个客户端同时过期并回源？

常见漏项：用多个独立可写副本分摊热写，却没有冲突合并和版本单调保证。

现场修复

把热点按读、写、value 大小三列分类，再为每列写可接受的陈旧或拒绝语义。

Q3

精确 LRU 为什么可能让缓存越优化越慢？你会如何保留淘汰效果而降低竞争？

为什么面试官会追：候选人声称每次 get 都更新一个全局 LRU 链表，却仍承诺极低尾延迟。

详细回答

精确 LRU 使本来只读的 get 变成元数据写：每次命中都要从链表中摘下节点再移到头部，热门键会让同一锁或同一 cache line 被所有线程争抢，复制也会产生大量访问日志。更实际的做法是每个分片独立维护近似策略，例如只按采样概率更新 lastAccess，从若干候选中淘汰最旧者，或用 segmented LRU 把新对象和高频对象分开。过期键先由 get 惰性判断，再由限速后台扫描清理；淘汰线程删除前比较记录版本，避免误删刚 set 的新值。验收不追求理论顺序完全精确，而看命中率接近基线且 p99、锁等待和元数据写量明显下降。

继续追问：如果 value 很大，淘汰释放内存本身阻塞请求线程，你怎样隔离这段工作？

常见漏项：只背诵 LRU 名称，不估算每次读取引入的写放大与全局锁竞争。

现场修复

把一次 get 的数据访问与元数据维护分开计时，并用采样率设计对照实验。

Q4

主缓存节点在 set 返回成功后立刻宕机，副本没有那条值；你的系统算正确还是丢数据？

为什么面试官会追：候选人说缓存无所谓一致性，同时又把它用于必须读己之写的状态。

详细回答

答案取决于接口承诺，不能事后改口。普通加速缓存可以声明节点故障等价于 miss：set 的成功仅表示当前主已接收，切换后上层从数据库回源并重填；此时短暂丢值是设计内行为。若调用方依赖刚写即读，则 set 至少要等待一个副本确认，客户端在会话内携带 minVersion 或暂时读主，副本落后就不能返回旧值。再高的耐久要求说明这个组件已被当作事实库，应选择带日志和仲裁的存储，而不是继续叫缓存。切主还需提升 epoch，旧主恢复只能丢弃或转交未确认数据，绝不能以旧版本覆盖新主。

继续追问：等待一个副本后，网络分区把主和已确认副本隔开，哪一侧继续接受写？

常见漏项：用“最终一致”掩盖已确认写可能消失，没有说明调用方应回源、重试还是接受旧读。

现场修复

为 set 写一条明确 SLA：确认条件、故障后的读行为和不适合承载的数据类型。

Q5

缓存平时 p99 为三毫秒，扩容后却拖垮数据库，你会怎样解释并修复？

为什么面试官会追：候选人只看缓存节点 CPU 和延迟，忽略迁移造成的冷启动与回源放大。

详细回答

新增节点接管一批哈希区间时内存是空的，原本的命中会变成 miss；若十万客户端同时请求热门键，就形成回源风暴，数据库先饱和，随后缓存请求也因排队超时。扩容流程应限速搬迁并预热已知热点，proxy 对同一键用 single-flight 只允许一个回源，其余等待结果；数据库回源设置全局并发上限，超过时优先返回短期旧值或明确降级，而不是无限重试。监控需要把 hit ratio 按分片和新旧节点拆开，并联看 DB QPS、最老回源等待、客户端超时和重试放大。只有新节点命中率与尾延迟稳定后才继续搬下一区间。

继续追问：如果旧值已经过 TTL，你是否愿意 stale-while-revalidate，边界由谁决定？

常见漏项：认为加节点必然降低负载，没看到数据迁移会暂时把缓存容量变成数据库压力。

现场修复

给扩容 runbook 加上预热、single-flight、回源限额和暂停迁移的阈值。

Senior / Staff Drill

Job Scheduler

Q1

要求任务在计划时间两秒内开始，你会怎样从数据模型推导扫描频率与延迟预算？

为什么面试官会追：候选人只说每秒轮询数据库，没有索引、分区、lookahead 或端到端预算。

详细回答

我把 schedule-to-start 拆成四段：到期发现五百毫秒、持久化领取或入队三百毫秒、broker 排队四百毫秒、worker 领取四百毫秒，余下四百毫秒留给网络和抖动。JobRun 以 (timeBucket, scheduledAt, runId) 建索引，scheduler 分片每二百五十毫秒读取自己负责的近期 bucket，并用一秒 lookahead 提前放入支持 deliverAt 的队列。runId 唯一约束使重叠扫描不会重复创建；进程中断后从持久水位向前回扫。看板直接量 scheduledAt 到 startedAt 的 p99 和 oldest-due age，某一段超预算就定位到扫描、broker 或 worker，而不是笼统加机器。

继续追问：数据库停顿三秒后恢复，如何补扫而不让同一 run 被入队多次？

常见漏项：把两秒理解为调度器轮询间隔，忽略索引查询、排队和 worker 饱和也消耗时间。

现场修复

画一条带毫秒数字的到期时序，并给 runId 唯一约束和回扫水位加故障测试。

证据
  • Job Scheduler · 1) How can we ensure the system executes jobs within 2s of their scheduled time?
  • Data Modeling · Indexing for Access Patterns
Q2

用户在纽约创建“每天凌晨 2:30”任务，夏令时切换那天没有这个时间，你的系统做什么？

为什么面试官会追：候选人只存 cron 字符串或 UTC 时间，没保留时区与歧义策略。

详细回答

JobDefinition 必须保存 cron、IANA zoneId 和明确的 DST policy，不能在创建时永久换算成固定 UTC。春季跳时导致 2:30 不存在时，产品可选择跳过本次或在下一个有效时刻执行；秋季 1:30 出现两次时，要规定执行一次还是两次。我的默认是跳过不存在的本地时刻、重复时刻只执行一次，并把解析出的 scheduledAt UTC 与 localOccurrenceKey 一起写入 JobRun；唯一约束使用 definitionId+localOccurrenceKey，防止重算重复。规则变更要版本化，历史 run 保留当时的 zone 与 policy。测试必须覆盖两次 DST 边界、时区数据库升级和 NTP 回拨。

继续追问：用户把时区从纽约改成洛杉矶，已经生成到队列里的未来 run 如何处理？

常见漏项：认为全部存 UTC 就解决周期语义，却丢失用户表达的本地日历规则。

现场修复

为 cron 模型补 zoneId、DST policy、definitionVersion 与本地 occurrence 唯一键。

Q3

worker 已经向支付服务扣款，尚未确认任务就崩溃；至少一次执行怎样避免再次扣款？

为什么面试官会追：候选人把消息 ack 和业务提交当成一个原子动作，或承诺跨系统 exactly-once。

详细回答

恢复后的 worker 必然可能再次收到同一 run，所以 runId 要作为下游幂等键。支付服务首次处理时把 runId 与 chargeId 建唯一映射并返回结果，重复调用只返回原 chargeId；调度器拿到结果后把 attempt 完成。若支付请求超时，worker先按 runId 查询支付状态，确认不存在才重试，不能把超时解释为未扣款。对于自有数据库，业务变更与 processed_run(runId) 在同一事务提交；随后要发事件则用 outbox。这样 broker 的重复投递、worker 崩溃和 ack 丢失都只产生重复尝试，不产生第二个可见副作用。无法提供幂等接口的外部动作必须暴露人工核对或可能重复的契约。

继续追问：幂等记录保留多久，周期任务多年后 runId 是否可能碰撞或被清理？

常见漏项：回答“先执行再 ack”就结束，没有处理执行成功但 ack 丢失的结果未知窗口。

现场修复

用扣款实例写出首次调用、超时查询、重复投递和返回原结果四条路径。

Q4

任务租约到期后新 worker 已接管，旧 worker 从长时间 GC 中醒来，如何阻止它继续写结果？

为什么面试官会追：候选人只给分布式锁设置 TTL，默认过期持有者会自动停止。

详细回答

锁过期不会终止旧进程，所以每次领取 JobRun 都要原子递增 fencingToken。新 worker 拿到 token 58 后，下游受保护资源记住见过的最大 token；旧 worker 带 57 写入时即使网络已恢复也被拒绝。worker 心跳续租失败后应主动停止，但 fencing 是最后防线。reaper 只有在 leaseUntil 超时并确认 attempt 仍为 RUNNING 时才重新入队，接管产生新的 attempt 记录，历史错误不被覆盖。ZooKeeper 选 leader 可避免多个 scheduler 扫同一分区，却不能替代 run 级 token，因为 leader 或 worker 都可能在网络分区中成为僵尸持有者。

继续追问：第三方 HTTP 服务不理解 fencing token 时，你还能提供什么保护？

常见漏项：把租约当作撤销旧 worker 的远程开关，漏掉暂停、GC 和网络分区后的复活。

现场修复

在故障时序中加入旧 worker 恢复，并要求下游用 token 单调比较拒绝旧写。

Q5

每天整点有十万条 cron 同时到期，但平均只有一万任务每秒，你怎样避免准点 SLO 全线失守？

为什么面试官会追：候选人用日均或分钟平均容量估算，没有整点突发、租户公平和拒绝策略。

详细回答

十万条在两秒内开始意味着瞬时需要五万 run/s，不能拿一万的长期平均掩盖。我会先允许非严格任务在创建时声明例如三十秒 jitter，把备份、报表等工作散开；严格任务保留原时刻并按优先级占用预留 worker。到期 bucket 内读取后进入按 tenant 加权的队列，每租户有并发额度，防止一个客户占满所有执行槽。worker pool 按 CPU、IO 和长任务类型隔离，broker 积压超过阈值时延迟低优先级任务并拒绝新的非关键计划。容量测试要把写 JobRun、入队、心跳和完成更新的放大都算入，并以 schedule-to-start p99 与 oldest-due age 触发扩容或负载卸载。

继续追问：如果所有十万条都声明为严格高优先级，你对产品给出什么容量或拒绝契约？

常见漏项：只增加队列分区，未解决 worker 执行能力、数据库写放大和租户争用。

现场修复

重做整点两秒窗口的峰值计算，并明确哪些任务可抖动、降级、延迟或拒绝。
