Day 1

读流量打来了
先加缓存

系统在读负载下先崩，通常不是因为数据总量，而是热点数据被反复打到磁盘。缓存把慢查询变成内存查找，是读扩展的第一步——但缓存放哪里、怎么写，决定了它能不能真正帮到你。
本节建立缓存策略的完整认知，为后续的 Consistent Hashing 和 Redis 操作做铺垫。

参考原文 缓存位置选型 Cache-Aside Write-Through Write-Back DDIA Ch6 · 复制
缓存可以放在哪里
External Cache（推荐）
独立服务（Redis / Memcached），多台应用服务器共享。LRU 驱逐 + TTL 过期，跨进程可见。

适用：所有多实例服务的共享数据缓存。
CDN Cache
全球边缘节点缓存，用户就近访问。将延迟从 250-300ms 降至 20-40ms。

适用：静态资源、不频繁变化的 API 响应。
In-Process Cache
进程内内存（如 HashMap），极快但只对当前实例可见。

适用：配置信息、Feature Flag、本地频繁读取且低一致性要求的数据。
Client-Side Cache
浏览器缓存 / localStorage。后端控制力有限，难以主动失效。

适用：用户个人数据、可接受短暂过时的内容。

写数据时：先写库还是先写缓存
策略写入流程优势劣势Bitly/缓存适用
Cache-Aside 读：先查 Cache → Miss → 查 DB → 写入 Cache 最常见，灵活 Cache miss 有延迟；可能 stale read ✓ Bitly 跳转路径
Write-Through 写：同步写 Cache + DB 强一致性 写入慢（两次写） ✓ URL 创建
Write-Back 写：先写 Cache，异步刷 DB 写入极快 Cache 宕机丢数据 ✗ URL 不能丢失
Write-Around 写：绕过 Cache 直写 DB；读 miss 才加载 写快；避免冷数据污染缓存 首次读必须回 DB ◐ 低频访问链接
📖

DDIA Ch6 · 复制延迟（Replication Lag）：
主从复制下，写入主库后副本库有延迟才能同步。如果读请求路由到副本，用户可能「读到旧数据」——这就是 Cache-Aside 必须设 TTL 的根本原因：缓存的 stale read 与副本的 replication lag 是同一类问题的不同体现。

Day 2

缓存集群
如何水平扩展

上节选好了缓存策略，但单台 Redis 终究有容量上限。加节点时，最直觉的 Modulo Hashing 会让几乎所有 key 全部失效——这是分布式缓存扩展的核心难题。
Consistent Hashing 用哈希环把节点增减的影响控制在 1/N，是面试必须能画出来、说清楚的算法。

参考原文 Modulo Hashing 缺陷 Hash Ring 原理 Virtual Nodes Real World 应用
问题起源：Modulo Hashing 的致命缺陷
Modulo Hashing
节点变更灾难
node = hash(key) % N

N 台节点时工作正常。但当 N 变成 N+1（新增节点）或 N-1（节点宕机）时，几乎所有 key 的映射都会改变。

后果：大规模缓存失效 → 所有请求同时打到数据库 → 级联崩溃。
Consistent Hashing
推荐
节点和 key 都映射到同一个哈希环（0 → 2³² - 1）。查找时顺时针找最近节点。

新增节点：只需迁移该节点与其顺时针邻居之间的 ~1/N 数据。
节点下线：只有该节点的数据迁移到下一个顺时针节点。

99% 的 key 不受影响。
Hash Ring 示意

      N0(("0°"))
      NA["Node A
hash=90"]
      NB["Node B
hash=180"]
      NC["Node C
hash=270"]
      N0 --> NA --> NB --> NC --> N0
    end
    K1["Key user:123
hash=100"] -->|顺时针| NB
    K2["Key post:456
hash=200"] -->|顺时针| NC
Virtual Nodes — 解决负载不均问题
问题：单节点区间过大
3 台节点物理分布不均时，某台节点可能承担 60% 的 key，另一台只有 10%。节点故障时负载会全部压向相邻的单台节点。
解决：每台节点 → 多个虚拟节点
为每台物理节点创建多个虚拟位置（如 Node-A-v1, Node-A-v2, ..., Node-A-v150），分散到环的各处。

故障时该节点的流量分散到多个幸存节点，而不是全部压到一台。
现实应用
Apache Cassandra
使用 consistent hashing 分配数据到节点，支持 Replication Factor 自动复制到 N 个顺时针节点。
Amazon DynamoDB
Dynamo 论文（2007）奠定了 consistent hashing 在生产系统中的实践标准，包含 virtual nodes。
Redis Cluster
使用固定 16,384 个 hash slots（而不是传统 ring），简化实现但本质思路相同——每台节点负责一段 slot 范围。

Day 3

Redis 能做什么
不能做什么

哈希环解决了数据分布，而每个节点内部的存储引擎就是 Redis。把 Redis 当「黑盒缓存」只会在追问时露馅——面试官问的是：你选 String 还是 Hash？原子操作怎么保证？节点 lag 了怎么办？
五种数据结构 + 原子性 + 复制风险，是本节要建立的 Redis 认知骨架。

参考原文 Redis · 缓存 Redis · 命令 Redis · 限流 DDIA Ch6 · 复制
五种数据结构，各解决一类问题
数据结构适用场景Rate Limiter 应用
String简单 key-value、计数器固定窗口计数：INCR user:123:minute:1720

分布式场景下的正确性问题
为什么需要原子性？
非原子操作下，两个请求同时「读取计数 → 判断 → 写入」，可能都读到旧值，导致都认为「未超限」，实际允许了两倍请求。
解决方案
  • INCR：原子递增，返回递增后的值
  • EXPIRE：原子设置 TTL
  • MULTI/EXEC：事务块，所有命令原子执行
  • Lua Script：服务端原子执行复杂逻辑
Token Bucket — Redis Lua 实现
-- 原子执行：读 tokens → 判断 → 写回
local tokens = redis.call('HGET', key, 'tokens')
local last    = redis.call('HGET', key, 'last_refill')
local now     = tonumber(ARGV[1])
local refill  = (now - last) * rate
tokens = math.min(capacity, tokens + refill)
if tokens >= 1 then
  tokens = tokens - 1
  -- allow request
end
redis.call('HMSET', key, 'tokens', tokens, ...)
使用 Redis 时会踩的两个坑
⚠️ Replica Lag（读一致性风险）
Redis 主从复制是异步的。写入主节点后，从节点可能延迟数毫秒到数秒才同步。

如果 Rate Limiter 读从库计数，可能看到过时数据，导致超限请求被放行。

结论：Rate Limiter 的写和读都必须走主节点。
⚠️ Hot Key（单节点压力）
某些 key（如 Twitter 名人的 user_id）被数十万 QPS 读取，全部命中同一个 Redis 节点。

解决方案：
  • 本地 in-process 缓存做第一层
  • 对 hot key 做 key sharding（user:123:shard_1 ~ shard_N）
  • Read Replicas 分散读压力

Day 4

流量来了
谁来说不

缓存解决了读路径的性能问题，但恶意流量或突发峰值仍然会把系统打垮。限流是系统的第一道防线，决定哪些请求通行、哪些被拒绝。
五种算法（Fixed Window → Sliding Window → Token Bucket → Leaky Bucket）各有适用场景，面试里要能说出「为什么选这个，不选那个」。

参考原文 Token Bucket Sliding Window Fixed Window 放置位置 Fail-Open / Closed CAP Theorem
五种算法，各有适用场景
算法原理优势劣势推荐
Fixed Window 每分钟重置计数器。如：每分钟最多 100 次 实现最简单；O(1) 空间 边界攻击：59秒100次 + 61秒100次 = 1秒内200次 ✗
Sliding Window Log 记录每次请求的时间戳，统计滑动窗口内的数量 精确无边界问题 内存占用高：每次请求都存时间戳 ✗ 大规模下
Sliding Window Counter 混合：当前窗口计数 + 上一窗口计数 × 剩余比例 内存低 + 近似精确 略有误差（可接受） ✓ 通用
Leaky Bucket 请求以固定速率流出；桶满则拒绝 输出速率平滑 无法处理突发；实现稍复杂 ◐ 平滑出站流量
Token Bucket 令牌以固定速率填充；请求消耗令牌；无令牌则拒绝 允许短暂突发；更接近真实流量 需存储 tokens + last_refill 两个字段 ✓ 推荐

Rate Limiter 放在哪里？
In-Process
最快，但每个服务实例独立计数，无全局视图。
问题：10 个实例各自允许 100 次 = 实际 1000 次
Dedicated Service
全局可见，但每次请求都多一次网络调用。
适合需要精确跨实例限流的场景。
✓ API Gateway — 推荐
在请求到达后端前就拦截。全局可见，不增加业务逻辑复杂度，是大多数系统的最佳选择。
CAP 在 Rate Limiter 中的体现
Fail-Open（AP 取向）
可用性优先
Redis 不可用时，放行所有请求。
风险：恰恰在流量洪峰导致 Redis 过载时，反而取消了限流保护，加速系统崩溃。
Fail-Closed（CP 取向）
推荐
Redis 不可用时，拒绝所有请求（或用本地限流降级）。
理由：Rate Limiter 最需要在流量冲击时发挥作用——恰好是 Redis 最可能过载的时刻。保护优先。

面试官会追问的五个方向
Key Distribution
Consistent Hashing + Virtual Nodes。节点变更时只迁移 ~1/N 的数据，Virtual Nodes 防止单节点过载。
Replication
每个 key 写入到顺时针方向的 N 个节点（Replication Factor=3）。节点故障时自动从副本恢复读取。
Eviction Policy
LRU 是默认选择，适合大多数工作负载。LFU 适合内容始终热门的场景（如 YouTube 热门视频）。
Consistency
接受最终一致性。短 TTL（秒级）控制 stale read 窗口。写时同步更新 Cache 减少不一致窗口。

出错了怎么办

Day 5 画出了理想路径，Day 6 问的是：当缓存失效、节点崩溃、热点 key 打垮单节点时，系统靠什么撑住？
主动说出故障场景并给出应对方案，是 Senior 候选人在深挖环节拉开差距的关键时刻。

参考原文 Cache Stampede Hot Keys Hot Key in Rate Limiter Cache Consistency
故障场景 1 — Cache Stampede（雪崩 / 惊群）
触发过程
1
热门 key 的 TTL 到期 （如某商品缓存过期）
2
数千请求同时 Cache Miss 全部打向数据库
3
数据库过载 响应变慢 → 更多请求积压
4
级联崩溃 数据库宕机，整个服务不可用
解决方案
  • Request Coalescing（请求合并）：第一个 Miss 去查 DB，其他请求等待结果，不重复查询
  • Cache Warming（预热）：系统启动或 key 即将过期前，主动刷新缓存
  • TTL Jitter（抖动）：不给所有 key 相同 TTL，错开过期时间：TTL = base + random(0, 60s)
  • Probabilistic Early Expiration：在 TTL 到期前一小段时间，以一定概率提前刷新
故障场景 2 — Hot Key（热点 Key）
问题：单节点瓶颈
不可用风险
Katy Perry 发推文 → 数百万人同时刷新 → user:123 这个 key 全部命中同一个 Redis 节点 → 单节点 CPU 100% → 整个节点不可用。
解决：多层缓存 + Key Sharding
推荐
① In-process 缓存：每个应用实例本地缓存 hot key，大部分请求不进 Redis
② Key Sharding：user:123:shard_0 ~ user:123:shard_N，分散到 N 个节点，读时随机选一个
③ Read Replica：多台 Redis 只读副本分担读压力
故障场景 3 — Redis 节点失联
Sentinel 模式
监控主节点，主节点宕机时自动提升从节点为新主节点。适合单主架构，自动故障转移。
Cluster 模式（推荐）
多主多从，每个主节点负责部分 hash slots。某主宕机时，其从节点自动接管。无单点故障。
降级策略
Redis 完全不可用时：降级到直接读 DB + 限流（保护 DB）。告警通知，快速恢复。

Two Pointers
双指针模式

双指针是处理数组/字符串的核心模式：用两个索引协同移动，将 O(n²) 暴力解降到 O(n)。
识别信号：题目涉及「有序数组」「对称判断」「区间收缩」「快慢指针」。

题目链接 Valid Palindrome Two Sum II 3Sum Container With Most Water Trapping Rain Water
核心不变量 — 双指针的思维模型
对撞指针（Inward）
left 从左，right 从右，向中间逼近。
触发条件：数组有序，或需要对称判断 
left, right = 0, len(arr)-1
while left < right:
    if condition_met:
        return result
    elif need_larger:
        left += 1
    else:
        right -= 1
快慢指针（Fast/Slow）
slow 每次走一步，fast 每次走两步。
触发条件：找中点、去重、检测环 
slow, fast = 0, 0
while fast < len(arr):
    arr[slow] = arr[fast]
    fast += 1
    if valid(arr[fast]):
        slow += 1

Valid Palindrome Easy 对撞指针 left/right 同时跳过非字母数字，比较字符
Two Sum II Medium 对撞指针 有序数组：和小了移 left，和大了移 right
3Sum Medium 固定 + 对撞 排序后固定一个数，剩余两数用双指针。注意跳过重复
Sort Colors (荷兰国旗) Medium 三指针 low/mid/high 三指针原地分区，经典面试题
Container With Most Water Medium 对撞 + 贪心 移动较短的那侧指针，因为移动较长侧不可能增大面积
Trapping Rain Water Hard 对撞 + 前缀最大 维护左右最大高度；当前格积水 = min(leftMax, rightMax) - height[i]
常见错误 — 双指针陷阱
去重遗漏
3Sum 中固定的数和移动的指针都要跳过重复值，否则结果里有重复三元组
边界条件
while left < right，不是 ≤。两指针相遇时停止，否则可能越界
误用 Two Pointers
无序数组不能用对撞指针。先排序（O(n log n)）才能保证收缩方向正确
