Week 7｜Robinhood 与 Uber 两分钟预听

不自动生成

这一周把两种高频实时系统放在一起比较。Robinhood 的行情流允许短暂陈旧，却必须让订单状态和经纪商回报可追溯；Uber 的司机位置可以近似，却不能把同一司机同时承诺给两位乘客。学习时先为每个系统画两条不同语义的路径：Robinhood 的 market-data fanout 与 order lifecycle，Uber 的 location ingestion 与 ride matching。行情和位置都要面对热点、背压、断线重连和客户端续传，但正确性边界不同。订单应由幂等 command、单调状态机和外部回报驱动，任何超时都先查询已有状态，不能盲目重发；派单则以有 TTL 的 reservation 或条件更新锁住司机，再由可持久化工作流处理接受、拒绝、超时和重新匹配。容量扩展也不从“上 Kafka、做分片”开始，而从符号热度、地理密度、更新频率、消费者速度和过期窗口开始。最终 mock 要能主动回答三个问题：哪些数据允许旧、哪些动作绝不能重复、哪个指标证明系统正在失控。把这三个答案讲清楚，实时系统才从组件图变成可恢复的产品协议。
