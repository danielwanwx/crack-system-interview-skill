Metrics Monitoring

Metrics Monitoring：从需求语义到故障修复

01

Metric identity 决定基数与成本

一条 sample 包含 metric name、label set、timestamp 和 value；name+规范化 labels 定义 series ID。label 必须限制长度、数量和值域，禁止 request_id、user_id 等无界维度，否则每个请求都会创建新 series。接入层认证租户、校验时间、去重 batch 并计算 cardinality budget，超限时拒绝或丢弃低优先级标签，而不是等 TSDB 崩溃。metadata catalog 记录 series 生命周期和 owner，帮助查询、配额与清理。系统按 active series、new series rate、bytes/sample 和 rejected labels 计费与报警。容量评审还要按租户估算高峰 active series、采样频率和保留期，确认预算变化有审批、灰度和回滚记录。预算告警必须早于硬拒绝。

能迁移的判断：标签设计是容量协议；基数在入口治理，不能只靠扩容。
02

写路径按时间与 series 分布

agent 批量 remote-write，携带 batch/request ID；ingest gateway 做限流、压缩解码和时间范围校验，再按 tenant+series hash 路由到 owner。内存 buffer/WAL 先耐久接收，随后形成按时间排序的 block 写入对象或本地存储，catalog 更新 block 元数据。相同 timestamp 的重复样本按明确规则去重，迟到样本进入仍可写时间窗或 side path。队列积压时先保护最近高优先级指标，降低 scrape 频率或拒绝低价值租户，不能无限缓存。故障演练应覆盖确认响应丢失、owner 转移和 WAL 重放，逐条验证 batch 去重、block 元数据提交与采样缺口。

能迁移的判断：批量、WAL、稳定分区和背压共同支撑高吞吐写入。
03

长范围查询需要 rollup 与 partition pruning

dashboard 查询先解析 label matcher，从 catalog 找 series 和覆盖时间段的 blocks；按时间裁剪分区并行读取。短范围选择原始粒度，数天或数周范围自动选择 1m、5m 等 rollup，避免返回超过屏幕像素的无意义点。rollup 保存 count、sum、min、max 及必要分位结构，不能只存 average 后丢掉可组合性。query coordinator 限制 fanout、合并结果并缓存规范化 query+resolution；部分 shard 超时可返回明确 partial 标记，而不是静默少数据。rollup 延迟或纠错时要按数据 watermark 失效缓存，并以抽样原始查询对比聚合误差和缺失分区。

能迁移的判断：查询计划根据时间范围和展示分辨率选择最小必要数据。
04

告警是一台持久化状态机

rule evaluator 按固定 schedule 或样本流触发，读取窗口聚合后维护 inactive、pending、firing、resolved 状态。for:5m 之类规则不能因一次采样缺失立即 firing，也不能在 evaluator 重启后忘记 pending 时长；状态和 last_evaluated watermark 要 checkpoint。通知以 alert fingerprint 去重，支持 silence、grouping、retry 与 resolved event。多副本 evaluator 需要明确 leader/partition ownership 或幂等 sink，避免双重通知。指标包括 evaluation lag、state recovery、notification latency 和 duplicate alert。规则版本、时钟偏差和缺失数据策略也必须入状态；用合成样本验证 pending 时长、恢复后续算和 resolved 通知顺序。配置发布采用影子评估，先比较新旧规则触发差异，再切换 owner 读取的活动版本，避免一次语法或阈值错误制造通知风暴。

能迁移的判断：低延迟告警不仅是快速查询，还要有持续状态、去重和恢复。
05

高可用先隔离租户和工作负载

监控系统在事故时负载反而最高，因此写入、dashboard 和 alert 不能共享无界资源。按租户和优先级设置 queue、CPU、series 与 query fanout 配额；alert evaluation 保留独立容量，重型 ad-hoc 查询可排队或降采样。ingest owner 使用多副本和 WAL，region 故障时客户端缓冲并重试；查询可读取对象存储和 surviving replica，标记 partial。恢复时限制 replay 与 cache rebuild，避免二次雪崩。关键指标按租户观察 oldest ingest、query queue、rule lag 和 dropped samples。区域切换演练需记录允许的数据缺口、告警接管时间与回放速率，确保恢复流量不会挤占新的实时样本。

能迁移的判断：事故负载下优先保护写入和告警，探索查询必须有资源边界。
06

Cardinality explosion 要有产品化修复

发现 active series 或 new-series rate 异常时，先按 tenant、metric、label key 找 top contributors，并阻止继续创建无界 series；已有数据仍可查询和按 retention 清理。系统可提供 label lint、预算预警和采样预览，让团队在上线前发现 request_id 一类错误。紧急降级可以删除非 allowlist 标签、降低 scrape 或限制新 series，但要记录被变换的数据语义，不能悄悄合并导致错误告警。修复后回填 dashboard/rule mapping，评估存储、compaction 和查询缓存恢复时间。验收要对比变换前后的查询与告警集合，并确认预算恢复后不会一次性重建所有被拒绝的 series。

能迁移的判断：基数治理包含预防、定位、限流和语义可见的降级。

Ad Click Aggregator

Ad Click Aggregator：从需求语义到故障修复

01

先完成跳转，再异步形成报表

点击入口验证 ad/campaign 状态，生成或接收 impression_id 和 click event_id，将事件耐久追加到多副本日志后返回 302。不能在用户跳转热路径同步更新聚合数据库，否则查询存储抖动会直接增加广告落地延迟。若日志短暂不可用，产品必须选择 fail-closed 保住计费完整性，或有限本地 spool 后 fail-open 并标记风险，不能静默丢数。redirect p99、append latency、spool bytes 和 lost-risk counter 分开监控。目标页面 URL 也要防开放重定向和篡改。容量测试要分别压测正常追加、日志降副本和本地缓冲耗尽，确认用户体验与可计费数据损失都符合已声明边界。

能迁移的判断：热路径只做验证与耐久记账，报表计算从日志异步派生。
02

事件身份定义一次业务计数

click event 至少包含 event_id、impression_id、ad_id、campaign_id、event_time、received_at、device/session 风险信号和 schema version。producer retry 必须复用 event_id；stream job 按 event_id 去重，聚合 sink 以 campaign+minute+window_version 幂等 upsert。impression_id 由广告展示服务签名生成，绑定广告、时间与可用上下文，点击入口校验后再记账。相同用户可能合法多次曝光，所以不能仅按 user/IP 永久去重；业务规则应表达每个 impression 是否最多计一次以及允许窗口。去重状态的保留期必须覆盖生产者重试和日志回放，否则超期重放仍会悄悄形成第二次计费。保留期变更也要先检查最长审计回放范围。

能迁移的判断：业务一次性来自 event/impression 身份和规则，不来自传输恰好一次。
03

窗口必须声明 event time 与迟到策略

广告主按一分钟查看点击，窗口应以 click event_time 分桶，而不是 worker 收到时间，否则网络延迟会把事件放错分钟。watermark 表示系统认为大部分事件已经到达，窗口先输出 provisional 结果；在 allowed lateness 内到达的事件生成修正 version，超过窗口可进入 late side output 和离线重算。API 可返回 data_through 和 finalized 标记，让广告主知道最近分钟仍会变化。监控 event-time lag、late ratio、watermark idle partition 和 correction volume，避免为了低延迟过早关闭窗口。结算截止点之后的超晚事件应生成可追踪 adjustment，而非改写已经出具的历史账单。

能迁移的判断：窗口结果是否最终，是对迟到数据的产品契约。
04

流式容错要延伸到查询 sink

Kafka 副本只保证事件日志耐久，Flink checkpoint 只保存算子状态和 source offset；若 sink 在 checkpoint 前后重复提交，报表仍可能双计。聚合结果应写稳定主键和单调 window_version，条件 upsert 拒绝旧版本；或使用 sink 支持的事务提交协议。raw events 保留到足以重算，batch job 定期按 campaign/minute 重聚合并与流结果比较。checkpoint age、recovery time、duplicate events、sink version conflict 和 reconciliation drift 是必须指标。恢复演练要在 sink 提交前后分别终止作业，检查同一窗口最终只保留一个可解释版本且能够从原始事件复算。差异超过阈值必须阻断结算。

能迁移的判断：端到端正确性包括 source、state、sink 和重算，不止消息队列。
05

防刷以 Impression 为起点

仅按 IP 或 user_id 去重会误伤共享网络，也容易被攻击者绕过。展示广告时生成签名 impression_id，记录 ad/campaign、placement、粗粒度主体和过期时间；点击必须携带有效 impression，规则可规定每 impression 只计一次，但仍保留全部原始事件用于风控。设备、速度、重复模式和异常来源进入风险评分，可把可疑点击标为 billable=false，而不是删除证据。策略版本写进事件，广告主报表区分 raw、valid 和 adjusted，支持申诉与重算。规则上线先做影子评估，比较误杀、漏判和收入影响，再按广告主或流量比例逐步启用并保留回滚版本。

能迁移的判断：去重与反欺诈是可审计策略判断，不是粗暴丢弃相同 IP。
06

查询低延迟来自预聚合与分区

广告主通常按 campaign/ad 和时间范围查询分钟序列，写入时就维护 campaign_id+day/time bucket 分区和 1m aggregates；长范围可进一步 rollup 到小时/天。查询服务做 partition pruning、批量读取和有限 cache，不扫描 raw click。最近窗口返回 provisional/version，历史 finalized 可长缓存。raw archive 存对象存储供重算和审计，不走交互查询。租户配额限制超大范围和高并发，指标观察 scanned partitions、rows、cache hit、freshness 和 correction rate。查询响应还应暴露口径版本与调整状态，使广告主能把页面数字和下载账目定位到同一份投影。

能迁移的判断：为真实报表访问模式建立可重算聚合，而不是把原始事件当查询表。

YouTube Top K

YouTube Top K：从需求语义到故障修复

01

先定义榜单窗口和精确性

Top K 不是单一需求。all-time 只累加，小时/日/月 tumbling window 在固定边界归零，sliding 24h 则每时刻都要减去过期事件；全球榜单、地区榜单和类别榜单还会倍增状态。产品需说明 view 去重、event time、更新频率、K 上限和允许误差。精确账目可能要求 raw event 可重算，推荐榜单则可接受近似。只有先写清这些语义，才能判断用普通计数、时间 bucket、stream state、Redis ZSET、OLAP 还是 sketch。还要定义并列排名、删除视频和隐私下架的处理，否则相同计数与撤回事件会让各客户端看到不同榜单。

能迁移的判断：窗口、维度、更新频率和误差预算共同定义 Top K 问题。
02

高频写入采用局部聚合与分层合并

每个 view 直接更新 video 全局计数会形成热门视频单 key 热点。入口把事件耐久写日志，consumer 按 video_id 分区；每个 partition 在短窗口本地累加，再批量写 partial counts。第二层按时间和维度 merge partial，维护候选 Top M 和最终 Top K。热门 key 可加随机 salt 做多 shard counter，读取时合并；event_id 或业务去重规则防重放双计。checkpoint 与幂等 sink 让恢复安全，raw log/archive 支持离线对账。指标看 ingest lag、hot key、merge age 和 drift。压测应包含单视频突发和分区扩缩容，确认 salted partial 在重放与迁移后仍只被合并一次。合并结果需记录输入版本。

能迁移的判断：把逐事件随机写变成可重放日志和层级批量聚合。
03

查询应读取物化排名而非扫描

每个常用 window/dimension 周期性生成有序 Top K snapshot，主键包含 window_type、window_start、dimension、generation；API 读取一条小结果并缓存。更新可用 copy-on-write 写新 generation 后原子切换，避免用户读到半个榜单。最近窗口带 data_through 和 provisional，历史完成窗口长缓存。若请求 K 小于存储 Top M 直接截断；临时维度或超长范围走受限 OLAP job，而不是拖垮在线库。监控 materialization lag、cache hit、snapshot size 和 query fallback。生成失败时继续服务上一代并标出陈旧时间，成功切换后异步回收旧代，避免空榜或缓存读到不存在的对象。

能迁移的判断：预计算把昂贵全局排序移出请求路径，并用 generation 保证一致快照。
04

Sliding window 需要显式减去过去

精确滑窗可把时间分成分钟或小时 buckets，保存每 video 每 bucket count；当前 24h 结果是最近 buckets 的和，窗口推进时减去最老 bucket。bucket 越细误差越小但状态越大，越粗则边界近似更明显。stream engine 可用 keyed state 和 timers 管理过期，checkpoint 恢复；迟到事件更新仍在保留的 bucket，并触发新 generation。不能只在 all-time counter 上加数据后声称支持滑窗。状态 TTL、watermark、late policy 和重算都要明确。用窗口边界前后的一组事件手算增减，并验证重启后 timer 不会重复扣除或遗忘旧 bucket。对账还要覆盖跨日时区边界。

能迁移的判断：滑窗的核心是可定位、可删除的时间贡献。
05

近似结构用于候选，不替代产品判断

Count-Min Sketch 用固定内存估计频率且只会高估，适合在巨大 key 空间中筛出可能热门的视频；配合小型 heap 保存候选，再从精确 partial counts 对候选重算 Top K，可大幅降低全量排序成本。误差由宽度、深度和总计数决定，碰撞可能让某视频虚高。若产品允许近似趋势榜，可直接标注 freshness/approximate；若涉及奖金或结算，必须精确。滚动窗口需要每 bucket sketch 或可衰减结构，不能从单个 sketch 删除任意旧事件。上线参数要用真实长尾分布回放，测候选召回率、排名偏差与内存，并在流量增长后重新校准。

能迁移的判断：sketch 缩小候选集合，最终是否精排取决于业务误差预算。
06

专用数据库要由查询与运维触发

Redis ZSET 适合小规模实时有序集合，但热门写、持久恢复、多窗口和跨维度会迅速增成本；TimescaleDB 适合时间 bucket 与连续聚合，Druid/Pinot/ClickHouse 适合高吞吐事件和多维 OLAP，Flink 擅长持续窗口状态。选型应比较 ingest、window、dimension、exactness、query p95、retention、rebuild 和团队运维，而不是列工具清单。简单阶段可用日志+批处理+数据库 materialized table，只有容量和延迟证据触顶才引入专用引擎。评审时必须演示从原始事件重建榜单的时长与步骤，因为恢复成本常比稳态查询延迟更能决定方案。

能迁移的判断：技术选择服务已定义的窗口与查询，不反过来塑造模糊需求。
