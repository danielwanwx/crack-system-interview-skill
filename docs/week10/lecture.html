答题主线

01 · 核心判断

先定义观察事实

价格跟踪的真相是带采集时间、来源、parser 版本和置信度的 price observation，而不是页面当前字符串。用户订阅的是对某个商品/商家关系的规则，抓取 URL 只是取得观察的手段。

02 · 核心判断

数据模型

tracked_item、merchant_page、crawl_task、snapshot、observation 和 alert_rule 分开建模。item 与 merchant_page 是多对多，才能让多个用户共享一次抓取，同时保留不同商家的价格历史。

03 · 核心判断

URL 与 frontier

URL canonicalization 在入队前去掉 tracking 参数、统一 host/path 并处理 redirect，防止同一页面无限重复。frontier 按 domain 分区并维护下一次允许抓取时间，不能用全局 FIFO 破坏礼貌与公平。

04 · 核心判断

抓取管线

fetch、parse、normalize、change detect 和 alert 是可独立重试的阶段。错误要区分网络、robots、反爬、页面变更和价格无变化；把它们都当 retry 会造成同域雪崩。

05 · 核心判断

变化检测

价格比较使用货币最小单位、解析置信度和去噪规则，而不是浮点字符串比较。只有在新观察通过质量阈值且满足 alert rule 时才生成通知事件，并用事实键去重。

06 · 核心判断

历史与成本

原始快照可短期保留用于 parser 回放，长期价格曲线可下采样但不能丢失用户看到的关键转折点。保留策略要同时服务调试、合规和成本，不能把所有 HTML 永久存储。

07 · 核心判断

迁移到通用 crawler

通用 crawler 关注 URL frontier、visited set、robots 和链接发现；价格跟踪额外关注解析质量、变化语义和订阅触发。两者共用 domain-aware scheduling，但后者需要更强的内容 schema 与 alert contract。

08 · 核心判断

SLO 与伦理边界

freshness 是按商家与订阅重要性定义的目标，不是盲目高频抓取。系统必须遵守 robots、速率限制和站点条款；遇到验证码或拒绝时降级而非升级对抗。
