Senior / Staff Drill

Metrics Monitoring

Q1

查询数周数据如何保持低延迟，同时不返回错误平均值？

为什么面试官会追：候选人只说 downsample，没有可组合统计量。

详细回答

query planner 根据时间范围和目标像素选择原始、1m 或 5m rollup，并按时间 partition pruning。每个 rollup bucket 保存 count、sum、min、max，必要时保存可合并的 histogram/sketch；跨 bucket 平均用总 sum/总 count，不能平均各 bucket 的 average。label matcher 先从 catalog 缩小 series，block 并行读但限制 fanout，结果缓存绑定 query、resolution 和 data watermark。缺 shard 时返回 partial/error。监控 scanned bytes、series count、resolution、cache hit 和 partial rate，才能验证低延迟没有牺牲正确性。再用固定金丝雀查询同时读取原始与 rollup，持续校验聚合版本、桶边界和纠错后的缓存失效。

继续追问：p99 latency 的 rollup 为什么不能简单取每分钟 p99 的最大或平均？

常见漏项：平均的平均、分位数的平均造成统计错误。

现场修复

用两个 count 不同的 bucket 手算 average，并说明 histogram 合并。

证据
  • Metrics Monitoring · 1) How do we serve low-latency dashboard queries over weeks of data?
  • Time Series Databases · Downsampling and Rollups
Q2

怎样把 alert latency 降到一分钟以内并证明达标？

为什么面试官会追：候选人只缩短 cron interval，没有端到端预算。

详细回答

从 sample event_time 开始分解：agent batch/网络、ingest queue、WAL/stream、rule evaluation、state commit 和 notification delivery。给各段设预算与 timestamp，按 p95/p99 观测。关键规则可由流式增量状态更新，普通规则周期查询；evaluator 按 tenant/rule 分区并 checkpoint，lag 超标时扩容或减少低优先级规则。通知 sink 以 alert fingerprint 幂等，并监控 provider retry。事件晚到时依据 watermark 决定修正或记录 late sample，不能为了“快”永久漏掉真实异常。上线前持续注入带已知 event_time 的合成异常，端到端量出发现、触发和送达分位数，并在积压演练中确认降级顺序。

继续追问：agent 离线两分钟后补发样本，是否应触发过去的告警？

常见漏项：把 evaluation interval 当成完整告警延迟。

现场修复

为一条样本填写六个阶段时间戳，找出最慢分位数。

Q3

监控系统在事故流量暴涨时如何保持可用？

为什么面试官会追：候选人只说多副本，没有工作负载优先级。

详细回答

把 ingest、alert evaluation、dashboard 和 ad-hoc query 放入独立队列与资源池；按租户配额和重要级别限流。先保护 WAL 接收与关键 alert，重型历史查询排队、限制 fanout 或强制更粗 rollup。agent 本地有界缓冲并带 batch ID 重试，服务端多副本 owner/对象存储保证恢复；不可接收时明确返回 backpressure，不能 OOM。region 故障时 query 可 partial，alert ownership 安全接管。监控 oldest ingest、dropped samples、rule lag、query queue 与 recovery replay，恢复阶段也限速。还要预先定义各优先级的丢弃预算与恢复目标，并把实际缺口暴露给查询和事故复盘。

继续追问：本地 agent buffer 满了，哪些样本可丢，怎样让用户知道？

常见漏项：所有请求同等优先，查询风暴把告警和写入挤死。

现场修复

给四种 workload 排优先级，并写每种 overload 动作。

  • Scaling Writes · Handling Bursts with Queues and Load Shedding
Q4

某团队把 request_id 放进 label，系统如何在拖垮前阻止？

为什么面试官会追：候选人只在存储层扩容或事后清理。

详细回答

接入层为 tenant、metric 和 label key 维护 active/new series 预算与近似计数，label allow/deny 规则在写入前执行。new-series rate 突升时拒绝新 series、删除明确非语义 label 或降采样，并返回可诊断错误；已有合法 series 继续写。后台 top-contributor 报告指出具体 metric/label/value 样本，预算在 CI 或配置发布前也可 lint。紧急变换必须记录规则版本，避免多个 request 被合并后告警含义悄然改变。监控 reject、series churn、compaction backlog 和恢复 ETA。

继续追问：如何低成本估计每个 label key 的 distinct values？

常见漏项：依赖精确全量 COUNT DISTINCT，本身就成为高基数负担。

现场修复

定义 tenant/metric/label 三层预算、告警与拒绝响应。

  • Data Structures for Big Data · HyperLogLog
Q5

两个 evaluator 副本同时运行，怎样避免重复告警？

为什么面试官会追：候选人认为多副本天然提高可用性而无副作用。

详细回答

按 rule group 分区并用带 epoch 的 owner lease，只有当前 owner 推进 pending/firing 状态；旧 owner 的 state commit 被 fencing。即使 failover 短暂双执行，notification event 使用稳定 alert fingerprint、state transition version 和 time bucket，sink 做幂等 upsert；provider retry 也复用同一 key。状态 checkpoint 保存 last evaluated watermark，接管者从该点继续而不是重放无限历史。silence 与 resolved 同样版本化。监控 ownership conflict、duplicate suppressed、evaluation gap 和 failover time，验证 HA 没有变成通知风暴。用旧 owner 暂停后复活的故障注入验证 fencing，并对供应商侧重复发送做周期性对账。

继续追问：如果通知供应商不支持幂等 key，如何限制重复短信？

常见漏项：只在 evaluator 内存里记录已发送，重启即丢。

现场修复

模拟 leader 暂停、lease 过期、新 leader 发送、旧 leader 恢复的时序。

Ad Click Aggregator

Q1

10k clicks/s 为什么不需要每次直接写分析数据库？

为什么面试官会追：候选人同步 insert 并聚合，放大 redirect 延迟。

详细回答

redirect 热路径只需验证并把事件耐久 append 到分区日志，批量 producer 可轻松摊薄网络和副本成本；stream consumers 按 campaign/ad 分区聚合，周期性或达到阈值后批量 upsert 分钟结果。分析数据库因此承受 aggregates 而非所有点击的随机写，查询也不与入口争资源。先算 10k/s×event bytes、partition 吞吐、保留和峰值，再选择分区数。若 consumer 落后，redirect 仍可继续到日志容量上限，并以 lag 报警，而不是让报表存储故障阻断用户跳转。

继续追问：单个超大 campaign 成为分区热点时怎样拆分且保持聚合正确？

常见漏项：看到 10k/s 就分库，却没有批量和派生视图。

现场修复

画 append、local aggregate、global merge、query store 四级写路径。

  • Scaling Writes · Batching and Hierarchical Aggregation
Q2

如何保证不丢点击，又不虚假承诺 exactly once？

为什么面试官会追：候选人说 Kafka exactly once 就结束。

详细回答

入口在返回 redirect 前等待多副本日志确认，producer retry 复用 event_id；若 append 结果未知则重试同一事件。consumer 至少一次处理，按 event_id 去重，checkpoint 保存 offset 与窗口 state。sink 用 campaign+minute+window_version 条件 upsert，重复 checkpoint 不双计。raw log/archive 保留并定期 batch 重算，对比流聚合发现 drift。若日志完全不可用，产品明确 fail-closed 或本地有界 spool 的风险。这样每个故障点都有恢复与证据，而不是依赖单个组件口号。

继续追问：日志确认成功但 302 响应丢失，浏览器重试会发生什么？

常见漏项：忽略 producer 和 sink 的重复，只看 consumer。

现场修复

为 producer、log、state、sink、reconcile 五层写唯一 ID 和 checkpoint。

Q3

如何阻止同一用户反复点击广告刷量而不误伤共享 IP？

为什么面试官会追：候选人按 IP 永久去重。

详细回答

广告展示时生成签名 impression_id，绑定 ad/campaign、placement、时间窗口和必要上下文；点击入口验证签名和过期，每 impression 按规则最多计一次 billable click。所有重复原始点击仍写日志，标记 dedupe/risk reason，便于风控和申诉。IP、device、velocity、referrer 只作为风险特征，不作为唯一身份；策略版本进入事件，后续调整可重算 valid/billable projection。匿名场景也能限制单 impression，而共享网络中的不同 impression 不会被合并。

继续追问：攻击者伪造大量 impression 再点击时，信任边界在哪里？

常见漏项：删除可疑事件导致无法审计，或将 NAT 用户视为同一人。

现场修复

区分 raw click、valid impression、deduped click、billable click 四层。

Q4

广告主查询一个月分钟数据，怎样保证低延迟和可修正？

为什么面试官会追：候选人从 raw clicks 做 GROUP BY。

详细回答

流处理预先按 advertiser/campaign/ad+minute 聚合，查询存储按 tenant 和时间分区；一个月范围可读取分钟或根据图表分辨率选择小时 rollup。最近窗口带 provisional、data_through 和 version，迟到事件在允许窗口内写更高 version；历史 finalized 结果可长缓存。查询做 partition pruning、范围上限和租户配额，raw archive 只供重算审计。若重算发现差异，以新 projection version 发布并记录 adjustment，不能静默覆盖计费账目。监控 scanned rows、p95、freshness 和 correction。

继续追问：计费已结算后又发现迟到点击，如何处理账务调整？

常见漏项：预聚合后认为结果永远不变，没有版本或迟到契约。

现场修复

设计 minute aggregate 的主键、version、finalized_at 与 correction API。

Q5

event time 窗口何时关闭，迟到点击算到哪里？

为什么面试官会追：候选人只按服务器接收时间分桶。

详细回答

事件按客户端/边缘记录的 click event_time 分桶，但先校验合理偏差并保留 received_at；watermark 由各 partition 已观察进度和允许乱序生成。窗口在 watermark 越过 end 后先输出，allowed lateness 内到达的事件更新同一分钟并递增 version，超过后进入 late side output，由离线重算或账务调整处理。空闲 partition 要标记 idle，避免 watermark 永久卡住。API 公开最近数据尚未 finalized 的状态。延迟与正确性取舍通过 late ratio、correction volume 和业务结算期限调节。

继续追问：恶意客户端伪造未来 event_time 会怎样影响 watermark？

常见漏项：忽略事件时间不可信和 idle partition。

现场修复

画 on-time、允许迟到、超晚三条事件路径及输出版本。

YouTube Top K

Q1

怎样减少 Top K 查询次数和数据库扫描？

为什么面试官会追：候选人每次请求 GROUP BY/ORDER BY 全量数据。

详细回答

对固定 window/dimension 周期性物化 Top M snapshot，写入带 generation 的小表或对象并在区域缓存；请求只读取对应 snapshot、截取 K。新 generation 完整写好后原子切换，避免半更新。最近窗口按秒/分钟刷新并标 provisional，历史完成窗口长缓存。cache miss 用 singleflight 回源，不触发临时全量计算；不常见维度进入有配额的 OLAP job。监控 materialization lag、snapshot build time、cache hit 和 fallback，确保查询低延迟来自准备好的数据而非隐藏昂贵工作。

继续追问：榜单更新时如何让全球 cache 同时看到同一 generation？

常见漏项：只缓存昂贵查询，却仍在每次过期时扫描全库。

现场修复

设计 snapshot key、generation pointer、TTL 和原子发布步骤。

Q2

热门视频的每次播放都更新一个 counter，会发生什么？

为什么面试官会追：候选人没有 hot key 或写放大分析。

详细回答

单 counter 会受一个 partition、锁或网络 owner 限制，重试还可能双计。view events 先进入按 video hash 分区日志，consumer 本地累加并批量写 partial；超热点 video 可按 salt 拆多个 counter，下一层周期 merge。去重依据 event_id 或明确 view 规则，sink 用 bucket+video+partition/version 幂等。写延迟与榜单 freshness 解耦，lag 超标再扩 consumer 或缩短 batch。raw events 保留以对账。监控 top-key rate、partition skew、batch size、merge age 和 duplicate ratio。扩分区时以 epoch 隔离旧新 partial，并通过故障回放确认同一批次不会在两个 owner 上被重复合并。

继续追问：salt 数量变化时如何迁移而不重复或漏计？

常见漏项：只加数据库 shard，却仍让一个 video 落在一个 shard。

现场修复

画 event log、salted partial、global merge、snapshot 四层。

Q3

精确 sliding 24h Top K 如何在窗口推进时删除旧贡献？

为什么面试官会追：候选人只有 all-time counter 或 tumbling day。

详细回答

把 event time 切成固定小 bucket，例如一分钟，保存 video/bucket count；当前 24h count 是最近 1440 个 buckets 的和。stream state 在新 bucket 到来时加贡献，在 timer 到期时减最老 bucket，并更新候选 ranking；checkpoint 保存 state/timer。bucket 越粗边界误差越大，越细状态与写入越多。迟到事件若 bucket 仍在保留期内就修正并发布新 generation，超晚进入重算。也可由 OLAP 按时间过滤查询，但在线 p95 与成本要证明可接受。

继续追问：不能为每个 video 保存 1440 个 bucket 时，怎样压缩状态？

常见漏项：把 tumbling 24h 误称为任意时刻 sliding 24h。

现场修复

用 4 个小时 bucket 手算窗口前进和过期减法。

Q4

Count-Min Sketch 会把冷门视频误判成热门，为什么还能用？

为什么面试官会追：候选人说近似但没有误差方向和 refine。

详细回答

Count-Min Sketch 因 hash 碰撞只高估频率，宽度/深度决定误差概率。它适合从巨大 key 空间持续筛出候选 Top M，而不是直接决定高风险精确榜单；候选进入小 heap 后，再从精确 partial counts 读取并重排 Top K，可消除大部分碰撞误判。趋势推荐若接受近似，可公开 freshness/误差预算；结算、奖励不能直接使用。滑窗需要每时间 bucket sketch 或衰减方案，因为普通 sketch 不能删除单个旧贡献。监控候选召回与 exact sample error 验证参数。

继续追问：Sketch 参数如何随总事件数增长调整并平滑升级？

常见漏项：只说省内存，不说明 false positive/high estimate 和业务影响。

现场修复

写 candidate generation → exact refine 两阶段，并标可接受误差。

Q5

何时选择 Redis、Flink、TimescaleDB 或实时 OLAP？

为什么面试官会追：候选人罗列产品，不把能力映射到需求。

详细回答

先用访问模式决策：Redis ZSET 适合有限维度、小状态、低延迟排名，但多窗口持久重建和热点写成本高；Flink 适合持续 event-time 窗口与 checkpoint state，仍需要外部 query sink；TimescaleDB 适合时间 bucket、连续聚合和 SQL 范围查询；Druid/Pinot/ClickHouse 适合高吞吐、多维过滤和大范围 OLAP。若初始 QPS 可由日志+批处理+关系表满足，应选更简单基线。比较 ingest、窗口、exactness、p95、retention、rebuild、成本和团队能力后再选，而不是默认组合全部。

继续追问：Flink state 已算好 Top K，为什么还需要查询存储？

常见漏项：工具驱动设计，忽略 source of truth、恢复与运营。

现场修复

为四种技术填 capability/cost/failure/rebuild 四列表。
