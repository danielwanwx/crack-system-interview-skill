先独立作答，再展开下一个追问。

01

item 和 merchant URL 如何建模？

参考回答
一个商品可在多个商家页面出现，同一商家页面也可能被多个用户关注，因此不能把 URL 塞进 item 一列。用关系表共享 crawl 与 observation，再让用户 subscription 保存目标价和通知偏好，才能避免重复抓取。
常见失分
每个用户复制一份 URL 和历史。
继续追问
商家页面对应多个 SKU 怎么处理？
02

canonicalization 要处理什么？

参考回答
抓取前要统一协议、大小写、默认端口和无关 tracking 参数，并尊重页面 canonical/redirect。否则同一内容会进入多次 frontier、浪费配额并制造重复提醒；但不能盲删会改变商品变体的参数。
常见失分
把 query string 全部删掉。
继续追问
颜色或尺寸参数如何判定业务相关？
03

为什么 frontier 按 domain？

参考回答
同一个域名的请求必须受共同速率和并发限制，全球 FIFO 会让一个热门站点占满队列或被突发重试压垮。按 domain 分区后可独立 token bucket、退避和优先级；代价是调度器需要维护多个队列。
常见失分
只按用户或 URL 哈希分区。
继续追问
子域名能否共享同一预算？
04

robots、429、验证码怎么处理？

参考回答
robots 禁止或站点明确拒绝时不应继续抓取；429 触发指数退避并降低域名预算。验证码说明自动化路径不可靠或不被允许，应标记降级/人工处理，而不是把对抗反爬当系统设计能力。
常见失分
所有错误无限重试。
继续追问
用户如何知道某商家已无法更新？
05

如何避免页面噪声误报？

参考回答
parser 输出结构化金额、货币、库存和置信度，变化检测只比较同一 SKU 的规范化值。对异常大幅变化可要求二次观察或回退到上一 parser 版本；这增加一点延迟，但避免一页促销文案误触发千万通知。
常见失分
直接比较 HTML 文本。
继续追问
价格含会员价时如何建模？
06

多个用户订阅同一页面如何合并？

参考回答
抓取调度以 merchant_page 为单位，一次 fetch/parse 产出一条 observation。随后按关联 subscription 评估不同阈值和渠道；这样把昂贵的网络请求共享，而不是让用户数线性放大爬虫负载。
常见失分
每个订阅创建一个 crawl job。
继续追问
不同用户要求不同 freshness 怎么协调？
07

retry 如何不压垮同域？

参考回答
retry 预算属于 domain，不属于单个 task；429/5xx 共享退避，parser bug 则停止重试并进入修复队列。这样不会因一万条 URL 同时失败而立即再打一万次请求；恢复后再渐进放量。
常见失分
每条 task 独立指数退避。
继续追问
如何检测某域名整体失效？
08

价格历史如何保留？

参考回答
原始 HTML/响应主要用于调 parser 和争议复现，可设较短 TTL 与访问控制。结构化 observation 则保留更久，并按时间窗口下采样；关键低点和触发 alert 的点必须保留以解释通知。
常见失分
永久保存所有网页或只存最新价。
继续追问
如何支持一年前的价格图？
09

parser 升级如何重放？

参考回答
保留有限的原始 snapshot 和 parser_version，让新 parser 在隔离环境回放样本并比较差异。确认后再逐步启用；不能覆盖旧 observation，否则无法分辨是真价格变动还是解析规则变化。
常见失分
直接全量重跑线上抓取。
继续追问
没有原始 snapshot 时怎么修复？
10

alert 怎样幂等？

参考回答
一次观察命中规则后，生成带稳定事实键的 alert event；通知 worker 重试也只会更新同一 delivery record。规则改版要进入键或版本，否则同一价格会被错误地当成新触发或被永远抑制。
常见失分
按发送时间去重。
继续追问
价格反复上下波动如何设置 cooldown？
11

商家页面消失怎么办？

参考回答
404 可能是下架，redirect 可能是迁移，HTML 变化也可能只是 parser 失效，因此不能一概删除商品。记录 page 状态和最后成功时间，必要时停止提醒并提示用户；修复 parser 后再恢复。
常见失分
一次 404 就删除历史。
继续追问
商家临时返回软 404 怎么识别？
12

价格与库存时间不一致怎么办？

参考回答
价格和库存可能来自不同组件或不同采集时刻，必须分别带 timestamp/置信度。提醒依据明确的价格合同，UI 则展示最后观察时间；假装两者同时实时会误导购买决策。
常见失分
只存一个 updated_at。
继续追问
缺货时是否还发降价提醒？
13

大站点如何隔离？

参考回答
将大站点的 frontier、速率限制和 parser worker 隔离，使其流量、反爬或页面改版不会拖累其它商家。共享基础设施可以复用，但容量和告警要按 domain 维度拆开。
常见失分
仅给大站点更多全局优先级。
继续追问
站点恢复时怎么缓慢加速？
14

需要强一致价格吗？

参考回答
价格跟踪是信息服务，不执行交易，因此更重要的是明确观测时间和错误边界，而不是对每次读做强一致。用户点击购买时必须跳转商家确认最终价；把旧观察当保证价格会制造错误产品承诺。
常见失分
为抓取结果做跨区强读。
继续追问
哪些业务会真的需要强价格锁？
15

visited set 如何分片和过期？

参考回答
visited set 防止重复发现循环链接，可按 canonical URL 哈希分片并用布隆过滤器做快速近似预检。它不能替代 last_fetch 时间：页面需要周期性重抓，所以永久 visited 与可调度 frontier 是两种不同状态。
常见失分
访问过一次就永不再爬。
继续追问
布隆过滤器 false positive 的代价？
16

核心 SLO 是什么？

参考回答
订阅页面的 freshness 和从有效观察到通知的延迟最接近用户价值；parser 成功率则揭示页面改版。必须按 domain 监控 429、错误和队列年龄，整体平均会掩盖一个重要商家完全失效。
常见失分
只看总抓取 QPS。
继续追问
哪项指标触发暂停一个 parser？
