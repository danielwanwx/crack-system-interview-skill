Week 12｜ChatGPT 服务系统两分钟预听

不自动生成

设计 ChatGPT 时，把大模型当作一个昂贵、流式、可取消的黑盒服务，不讨论训练模型本身。用户发送消息后，API 先把 user message 以幂等方式写进 conversation，再创建 generation run；调度器根据模型、上下文长度、付费层级和 GPU 容量排队，worker 开始生成后通过 Server-Sent Events 把 token 流给客户端。客户端断线不代表 run 必须继续：要有 cancel、timeout 和 reclaim GPU 的协议。真正稀缺的不是普通 API QPS，而是 token-seconds、KV cache 和 GPU memory，所以公平调度要按预计成本而不是请求数限流。长对话每轮都重发全部历史会让首 token 延迟和成本持续增长，可以截断、摘要或缓存稳定前缀，但摘要必须带版本并保留原消息作为权威记录。生产级答案还要讲 overload shedding、模型故障回退、内容安全、租户隔离、trace 与 eval。最后记住：agent 或 tool calling 只是可选的应用层扩展；只有当任务需要动态决策时才引入，并且所有工具权限、参数校验和副作用确认都必须留在可信运行时。试听后请关掉页面，从 admission、queue、GPU、stream 到 conversation persistence 画一次完整时序，并为断线、取消、超时和重复计费分别标出唯一的修复责任方。
