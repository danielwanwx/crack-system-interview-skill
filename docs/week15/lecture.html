答题主线

01 · 核心判断

候选不是匹配

geo index 只找附近候选，最终是否展示还要经过偏好、年龄、block、可见性和 ranking 规则。将候选召回与双边 match 分开，才能保护隐私并允许各自扩展。

02 · 核心判断

状态模型

profile、location_cell、swipe、match、block 各自有不同寿命；location 是带 TTL 的软状态，match 是 durable 事实。swipe 写入需要稳定 id 和方向，双向条件满足时只创建一次 match。

03 · 核心判断

地理与新鲜度

位置上报不要求每次全局强一致，但必须携带 timestamp、精度和 TTL，过期位置绝不能进入候选。cell 边界要查邻格并在最终层算距离，GPS 抖动则用平滑和最小移动阈值处理。

04 · 核心判断

双边写入

right swipe 到达后检查反向 swipe，并用唯一约束或条件插入创建 match。通知在 match 提交后通过 outbox 触发，重复 swipe 不得产生重复 match 或重复通知。

05 · 核心判断

隐私与撤销

block、unmatch、隐身和删除必须比候选缓存更快收敛：读取时还要做权威过滤。历史数据可以为安全/合规保留受控审计，但不得继续参与推荐或通知。

06 · 核心判断

热点城市

大城市和热门时段会让少数 geo cell 过热，需分层 cell、复制读取和按地区隔离。仅按 user_id 分片会使附近查询跨大量 shard，无法控制延迟。

07 · 核心判断

迁移到 Uber

Uber 的 driver location 也是软状态，但 dispatch 需要对司机/订单建立短暂排他 reservation，不能只返回推荐列表。Tinder 的错误代价是较差体验，Uber 双派或错派直接影响履约，因此强约束边界更大。

08 · 核心判断

实时系统运营

candidate freshness、match success、stale location 和 dispatch latency 是关键用户信号。socket 连接数并非全部；定位上报和通知必须能降采样或回退到 pull，避免高峰拖垮权威写路径。
