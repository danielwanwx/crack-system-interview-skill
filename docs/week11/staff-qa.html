01

URL 相同、标题相近、转载如何判重？

参考回答
完全相同内容可用 canonical URL 和 content hash 去重，转载或同事件报道则进入相似度聚类。两者不能混为一谈：前者减少副本，后者组织多视角；原始 article 仍要保留，便于重新聚类。
常见失分
只用标题 hash。
继续追问
突发事件标题随后更新怎么办？
02

canonical article 怎样选？

参考回答
cluster 选择代表文章可考虑来源可信度、信息完整度、发布时间和可访问性，而不是第一次抓到的文章。代表改变不应改变 cluster 身份；用户链接和历史需稳定，因此选择是一个可更新投影。
常见失分
永久固定第一篇。
继续追问
代表文章被删除时怎么办？
03

为什么 index document 不是 DB row 拷贝？

参考回答
业务库以写入和事实一致性组织，搜索文档以 token、filter、排序和分片组织，两者优化目标不同。把索引视为可重建投影，允许 schema 双写和 backfill；直接镜像会让每次搜索需求都污染主数据模型。
常见失分
数据库加全文索引即可。
继续追问
索引 schema 改变如何迁移？
04

refresh 延迟如何解释？

参考回答
新文章写入成功后，搜索索引可能在数秒内不可见，这应被明确为 freshness SLO。详情页可从权威 article 读到最新状态，搜索页接受短暂延迟；如果延迟过长则以 index lag 告警而不是假称强一致。
常见失分
要求同步更新所有 shard。
继续追问
删除为何通常要更高优先级？
05

如何过滤并排序？

参考回答
先以倒排索引召回相关候选，再应用语言、地区、时间和权限 filter，最后进行更昂贵的 ranking。顺序可随选择性调整，但必须能解释每一层；把 filter 混进最后排序会浪费计算且有泄露风险。
常见失分
回答“交给 Elasticsearch”。
继续追问
filter 很宽时怎样保护 P99？
06

ranking 如何安全回滚？

参考回答
每个结果记录 ranking version 和主要 feature，先用 shadow traffic 或小流量实验验证质量与延迟。出现偏差可把流量切回旧版本，而无需重建全部 index；特征数据也要有 freshness 监控。
常见失分
直接上线新权重。
继续追问
怎样防止 CTR 目标压低来源多样性？
07

热点事件 query skew 怎么办？

参考回答
突发新闻会让少数 query 和时间段成为热点，先用结果缓存、请求合并和热点 key 复制保护 shard。必要时将热 topic 的索引或查询路由隔离；不能只靠均匀哈希，因为访问模式本身不均匀。
常见失分
只增加所有 shard。
继续追问
热点事件更新频繁时缓存如何失效？
08

shard key 如何避免 topic 热点？

参考回答
只按 topic 分片会让一个重大事件集中到单 shard，通常按 document 或索引段均衡分布，再通过副本和 routing 优化查询。热点仍需单独观测和临时复制；分片设计不能假设 topic 平均。
常见失分
按热门关键词 hash 即可。
继续追问
按用户权限过滤会改变分片选择吗？
09

删除如何从 cache/index 清理？

参考回答
权威 article 标记删除后，读路径应立即检查可见性或使用撤销版本，随后由事件清理 cluster、索引和缓存。删除事件需要可重放和版本比较，避免旧索引更新把已删内容重新写回。
常见失分
只删主库行。
继续追问
缓存仍展示时如何缩短窗口？
10

为什么召回、排序、摘要拆层？

参考回答
召回追求高覆盖和低成本，排序追求精确质量，摘要可只对少量 top results 运行。拆层能在 P99、成本和效果之间做明确取舍，也能定位是没召回还是排错了。
常见失分
对全库运行最贵模型。
继续追问
召回率下降如何排查？
11

Post Search 的 ACL 放在哪里？

参考回答
权限约束要作为检索条件进入候选集，或以安全的分区/bitmap filter 实现，确保不可见帖子从不参与 ranking。仅在响应组装时隐藏会泄露计数、摘要或相关性，也浪费大量计算。
常见失分
最后一层做 UI filter。
继续追问
好友关系改变怎样让旧索引失效？
12

ACL 改变如何收敛？

参考回答
ACL 变更产生高优先级事件更新索引字段和缓存，敏感场景在读路径再做权威校验。这样在异步传播窗口宁可少返回，也不返回已撤销内容；设计上要接受短暂 false negative。
常见失分
等待全量重建索引。
继续追问
百万帖子共享范围变化如何处理？
13

synonym/typo correction 的代价？

参考回答
同义词和拼写纠正能提高 recall，但也会引入歧义和意外匹配，尤其跨语言时。它们应是可版本化的 query rewrite，并通过离线样本和在线 zero-result/abandon 指标验证。
常见失分
把所有近似词都扩展。
继续追问
品牌名被纠正错时怎么回滚？
14

如何监控来源失真？

参考回答
要看每个来源的抓取 freshness、解析成功、被聚类比例和在结果中的展示份额，避免单一来源悄悄失效或垄断。整体文章数正常并不代表覆盖健康；按语言/地区分层更重要。
常见失分
只看总入库量。
继续追问
如何检测低质量来源刷量？
15

index backfill 怎样不影响线上？

参考回答
创建新 schema/index，按 checkpoint 限速回放历史并持续双写新事件，验证覆盖和质量后原子切换 alias。旧索引保留回滚窗口；直接原地重建会同时伤害搜索可用性和恢复能力。
常见失分
停机全量重建。
继续追问
backfill 与实时写竞争怎么办？
16

核心 SLO 有哪些？

参考回答
在线上看 index lag、query P99、zero-result 和重复结果率，离线/实验看 NDCG、来源多样性和用户满意度。速度与相关性必须一起守，否则系统会稳定地产出错误结果。
常见失分
只报 QPS 与延迟。
继续追问
哪个指标会触发 ranking rollback？
