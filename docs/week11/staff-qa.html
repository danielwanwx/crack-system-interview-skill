第 11 周 Staff 追问与详细答案

Senior / Staff Drill

Price Tracking Service

Q1

500M 商品怎样在合理时间内追踪，为什么不能每天全抓？

为什么面试官会追：候选人没有 pages/s、来源限速或分层频率。

详细回答

每天全抓 500M 约需 5.8K fetch/s 平均吞吐，乘峰值、失败和多 marketplace 后更高，还可能违反来源 politeness。应让每个商品保存 next_due，按订阅人数、变化率、价格临近阈值和来源更新特征分层：热商品分钟/小时，冷商品天/周；RSS/feed 或扩展信号可提前触发验证。scheduler 扫到期时间桶并按 domain 配额发 lease，失败退避。监控 due lag、coverage、429 和每层 freshness，用业务价值分配有限抓取预算，而不是追求无差别频率。

继续追问：新商品没有历史变化率，初始抓取频率如何设定？

常见漏项：只横向扩容，不考虑外部来源允许的总速率。

现场修复

给 hot/warm/cold 三层写规模、周期和最低 fetch/s。

Q2

恶意扩展用户上报一分钱价格，如何防止触发全量提醒？

为什么面试官会追：候选人信任客户端或只按 user_id 去重。

详细回答

扩展 observation 先进入低信任 raw store，不直接更新 canonical price。入口验证签名版本、product URL、时间与 schema，按 installation/IP/device 限流；风险层比较近期价格、多个独立来源、服务器抓取和 DOM fingerprint。异常值标记 quarantine，并触发受控服务器验证；只有达到策略置信度的 observation 才发布 canonical price version。通知只消费 canonical event，因此恶意 raw signal 不会直接触发。所有 accepted/rejected reason 与策略版本保留，可重放审计；异常 campaign 按来源和扩展版本快速熔断。

继续追问：真实 flash sale 也会价格暴跌，怎样减少误判？

常见漏项：粗暴丢弃所有异常值，既不可审计又漏掉真实促销。

现场修复

画 raw observation → risk score → corroboration → canonical update 四层。

Q3

价格在阈值附近上下抖动，如何避免重复通知？

为什么面试官会追：候选人每次 price<threshold 都发送。

详细回答

Subscription 是有状态规则：armed 时只有从阈值上方跨到下方才创建 crossing_version；发送后进入 triggered/cooldown，价格持续低于阈值不再发。只有回升到 threshold+hysteresis 或用户设定的 rearm 条件才重新 armed，避免小幅抖动。通知唯一键 subscription_id+crossing_version，provider retry 复用 notification_id；结果未知先查询或幂等重试。迟到价格修正是否补发要有时间窗口。状态 checkpoint 与价格 version 保证 consumer 重放不会再触发旧 crossing。

继续追问：用户修改 threshold 时，当前低价是否立即触发？

常见漏项：只有消息去重，没有业务状态边沿和 re-arm。

现场修复

用 100→79→81→78 的价格序列演算 armed、triggered、rearmed。

证据
  • Price Tracking Service · 3) How do we efficiently process price changes and notify subscribed users?
  • Kafka · Handling Retries and Errors
Q4

十年价格历史图如何在 200ms 内返回？

为什么面试官会追：候选人扫描所有 raw observations 或只说加 cache。

详细回答

先把重复抓取转换为 canonical price-change series，再按 product/time 分区；query planner 根据图表像素和范围选择原始、日或周 rollup，做 partition pruning。rollup 保存 open/high/low/close、代表值和 count，版本关联 normalization/currency。热门 product/range 缓存绑定 data watermark，历史 finalized 分区可长 TTL，最近分区短 TTL。响应限制最大点数并返回 resolution、data_through。raw observation 留作审计重算，不走交互查询。监控 scanned partitions、points、cache hit 和 rollup lag。用十年冷数据与刚纠错的热分区分别压测，确保长缓存不返回旧币种或旧聚合版本。

继续追问：历史价格被纠错后，旧 rollup 和 cache 如何更新？

常见漏项：缓存掩盖全表扫描，过期时仍产生雪崩。

现场修复

为 1天、1月、10年查询分别选择 resolution 和最大点数。

证据

  • Time Series Databases · Downsampling and Rollups
Q5

抓取成功、价格已写入，但 scheduler ack 丢失会怎样？

为什么面试官会追：候选人认为 durable queue 会阻止重复抓取。

详细回答

lease 到期后任务会重做，这是预期的至少一次行为。fetch attempt 使用稳定 schedule_run_id，PriceObservation 以 product+source+observed_at/content hash 或 event_id 条件写；重复结果返回已有 version，不重复生成 canonical price event。下一次 next_due 只接受匹配 schedule version 的条件更新，旧 worker 不能覆盖新计划。若 body 已保存但 observation 未提交，重试可复用原始响应或 orphan scanner 清理。指标记录 expired lease、duplicate observation、schedule conflict 和 pending age。正确性来自幂等提交，而非无重复执行。故障测试要在写入、事件发布和调度确认之间逐点中断，核对 observation、canonical version 与 next_due 最终收敛。

继续追问：旧 worker 长时间暂停后恢复，如何避免把 next_due 推回过去？

常见漏项：忽略 worker pause 和旧 lease 持有者复活。

现场修复

模拟 lease expiry、新 worker 完成、旧 worker 恢复的 version 条件。

Senior / Staff Drill

Strava

Q1

手机离线三小时并重启，如何保证活动不丢且不重复？

为什么面试官会追：候选人依赖内存 buffer 或重连后重新创建 activity。

详细回答

start 时生成稳定 activity_id，GPS 与 pause/stop 事件先顺序写本地 durable log；重启从最后 fsynced sequence 恢复。上传按 activity_id+chunk_index 幂等，chunk 带 first/last sequence 与 checksum，服务器可乱序接收并记录 bitmap。响应丢失就重发同一 chunk，不产生新活动。stop 后 finalize 声明 last sequence/expected chunks，只有无缺口且 checksum 合法才 commit；重复 finalize 返回同一 version。缺口请求客户端补发，无法恢复时明确 partial/rejected，而不是静默截断。再通过杀进程、断网和重复 finalize 的设备测试核对本地日志、服务端 bitmap 与最终点数一致。

继续追问：设备存储即将满时，本地 log 如何降级？

常见漏项：把网络重试当同步协议，没有稳定 ID 和完整性检查。

现场修复

画 device log、chunk bitmap、finalize 三层，模拟 chunk 2 丢失与重复。

Q2

10M concurrent activities 的瓶颈如何估算并拆解？

为什么面试官会追：候选人直接说 Kafka 分区或横向扩容。

详细回答

先假设每设备采样频率、压缩后 bytes/point 和 batch interval。例如 1Hz、约几十字节、每 30 秒一批，则 points/s 是 10M，upload QPS 约 333K，再计算入口带宽、对象写和 metadata QPS。客户端批量压缩，本地断线缓冲；gateway 按 activity hash 分区，raw chunks 直写对象存储，metadata 小事务提交，processing 从队列异步消费。大型赛事会造成区域和热门 segment 偏斜，需区域入口、配额和独立 downstream。监控 batch size、ingest lag、missing chunk 和 backlog 验证假设。容量表还要加入断网后集中补传的峰值系数，以及对象存储重试和处理追赶所需的独立余量。

继续追问：若每台设备改为 5Hz 采样，哪一层先需调整？

常见漏项：只算用户数，不算点、批次、字节和派生 CPU。

现场修复

完成 points/s、uploads/s、bytes/s、processing events/s 四行容量表。

Q3

朋友实时看活动，连接断开后如何恢复且不泄漏位置？

为什么面试官会追：候选人只说 WebSocket 自动重连。

详细回答

每个 live activity 有 share_version、允许 viewers、expires_at 和单调 location sequence。viewer 连接时授权，重连携带 cursor；短期 replay buffer 补缺口，超窗返回 latest snapshot，并显示 location age。每次推送或批次都验证 share_version/deny cache，athlete 撤销或结束时立即 bump version、断开连接并清理短 TTL state。旧 cursor 不能绕过新权限。位置过旧显示 stale，不插值成精确轨迹。名人热点复制 fanout，但原始精确点不长期缓存。监控 unauthorized attempt、revocation lag、replay miss 和 stale viewers。通过撤权与边缘分区并发演练，验证缓存失联时默认拒绝而不是继续广播。

继续追问：权限服务故障时，已有连接是否继续推送？

常见漏项：只考虑数据连续，没有 consent、撤销和保留。

现场修复

画 authorize → push → revoke → reconnect 四步及 share_version。

Q4

热门 segment 排行榜怎样处理迟到上传、作弊和算法升级？

为什么面试官会追：候选人只维护 Redis Sorted Set。

详细回答

权威 Effort 保存 activity_id、segment_version、elapsed、confidence、eligibility、processor_version 和 moderation state；排行榜只是 segment/category/period 的投影。迟到 activity 或重新处理产生版本化 effort，consumer 以 effort_id/version 幂等 upsert ranking；作弊检测可把项标 ineligible 并发布新 generation，不删除审计。算法升级从 raw GPS 重跑影子版本，对比 drift 后切换。热门 segment 由分区局部 Top M 合并，完整列表在持久存储分页。响应返回 data_through/rules_version，监控 detection lag、rank correction 和 moderation backlog。切换前按热门与长尾 segment 抽样比较名次变化，并保留一键回退到旧 generation 的能力。

继续追问：两名运动员相同时间时，稳定 tie-breaker 是什么？

常见漏项：把排序结构当 source of truth，无法解释修正和重建。

现场修复

区分 Effort 事实、eligibility 判断、Leaderboard generation 三层。

Q5

项目页没有 Senior 小节，如何定义这题的 Senior 深度？

为什么面试官会追：候选人猜测不存在的项目 anchor，或只复述 HLD。

详细回答

我会显式记录 source gap，等级依据引用 Hello Interview 通用 Senior 说明，不伪造 Strava 项目链接。把通用 breadth、depth、proactiveness 落到本题：breadth 要覆盖 offline sync、ingest、派生处理、live/privacy 和 leaderboard；depth 至少能证明 chunk 幂等或排行榜版本修正；proactiveness 要主动提出位置隐私、设备失败、热门赛事偏斜和重算证据。验收使用本项目 deliverable、指标、Q&A 与 repair。这样既保留来源准确性，也让等级要求成为可观察行为。
