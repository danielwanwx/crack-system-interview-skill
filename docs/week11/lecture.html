Price Tracking Service

Price Tracking Service：从需求语义到故障修复

01

五亿商品需要分层调度而非全量 cron

每个 ProductSchedule 保存 next_due、last_success、change_rate、priority、source policy、attempt 和 lease。scheduler 只扫描到期时间桶，按商品热度、价格波动、订阅人数和来源限速调整周期；长期不变的冷商品降低频率，新订阅或剧烈变化商品提高。worker 以短 lease 领取，成功后原子写 observation 并计算下次 due，失败按错误类型退避，永久失败隔离。全局队列按 marketplace/domain 公平调度，避免热门来源封禁拖累全部。关键指标是 due lag、fetch success、429、attempt 和 coverage。调度扩缩容与时钟漂移演练要确认同一任务可重复执行但结果幂等，并且旧 lease 无法把较新的 next_due 覆盖。

能迁移的判断：周期任务是持久状态和优先级调度，不是进程内定时器。
02

价格是带来源与置信度的观察

PriceObservation 至少包含 product_id、amount、currency、source、observed_at、received_at、event_id、parser_version 和 confidence。网页抓取、官方 feed 与浏览器扩展可信度不同，不能直接 last-write-wins。验证层检查币种、数量级、与近期趋势、多个来源和页面签名的一致性；异常 observation 保留但不更新 canonical current_price，进入复核。解析规则升级可从保存原始响应重跑。当前价格记录指向被采用 observation 和 version，历史修正有审计轨迹，避免恶意或错误值悄悄污染提醒。每天抽样回查页面证据与采用结果，按来源统计误报和陈旧度，才能校准 confidence 而不是把它当静态常数。

能迁移的判断：外部价格不是事实写入，而是需要验证、可追溯的观测。
03

Chrome 扩展是低信任数据源

扩展事件由已发布版本签名，携带 installation/session 粗身份、页面 URL、DOM fingerprint 和 observation_id；入口做认证、限流、schema 与时间校验。单个用户或单个 IP 的价格不能直接生效，系统按独立来源数、与服务器抓取差异、速度和设备信誉聚合风险。可疑事件仍保存为 raw signal，标记 accepted/rejected reason；策略版本进入判断以支持重放。隐私上不收集不必要浏览历史，只提交用户主动访问的目标商品和最小字段，并提供删除与保留边界。

能迁移的判断：客户端众包提升覆盖，但必须是可审计信号而非权威写入口。
04

阈值通知是一台可重入状态机

Subscription 保存 product、threshold、direction、channel、last_observed_version、armed_state 与 cooldown。canonical price 更新生成 versioned event；处理器比较前后状态，只有从 above 到 below 的 crossing 创建 notification，唯一键为 subscription_id+crossing_version。outbox 异步发送，provider timeout 重试同一 notification_id，成功回执记录。价格继续低于阈值不重复提醒，回升到 rearm threshold 或用户配置后才重新 armed；滞后/修正 observation 是否补发由明确策略决定。指标看 crossing lag、duplicate suppressed 和 provider retry。用抖动价格序列、消费者回放和供应商超时三类测试，证明业务边沿与发送副作用都只发生一次。用户修改阈值、币种或通知渠道时也要创建配置版本，并明确是否对当前价格重新评估，避免旧 crossing 被新规则再次发送。

能迁移的判断：提醒由状态边沿触发，不由每条低价事件触发。
05

价格图表读取时序投影

raw observations 按 product_id 和时间分区，保留来源与修正；canonical price change 形成较小序列，仅当有效价格变化时新增点。短范围查询读原始变更，年度图选择日/周 rollup，bucket 保存 open/high/low/close 或代表值与 sample count。query service 做 partition pruning、resolution 选择和 cache，响应带 data_through 与 currency/normalization version。不能只保存当前价格，否则无法重算提醒和解释异常；也不应为图表扫描所有重复抓取。监控 scanned points、rollup lag 与 correction。历史纠错要重建受影响的 bucket 并按 watermark 失效缓存，再以原始序列抽样核对图表点和极值没有被吞掉。

能迁移的判断：原始观测用于审计，规范化变更与 rollup 用于交互查询。
06

失败、反爬与恢复必须可运营

来源可能返回 429、验证码、布局变化或错误货币，notification provider 也可能接受后超时。不同错误分开处理：429 遵守 Retry-After，解析失败隔离 parser/version，页面不存在进入终态，未知通知查询或幂等重试。队列积压时保护高订阅商品并拉长冷商品周期，不增加并发继续撞来源。dashboard 展示 schedule lag、source coverage、parser error、anomaly rate、price freshness、notification pending age 和 reconciliation drift。修复 parser 后从原始响应重跑，不必重新抓取全部页面。恢复计划还要限制回放速率并标记 parser 版本，防止一次修复同时触发抓取洪峰、价格震荡和通知风暴。

能迁移的判断：长期服务质量来自分类失败、限速恢复和可重处理原始证据。

Strava

Strava：从需求语义到故障修复

01

离线优先从设备状态机开始

手机在 start 时生成 activity_id，并把 GPS point 追加到本地 durable log；每点有 sequence、event_time、accuracy 和传感器元数据。pause/stop 也是日志事件，应用被杀后可从日志恢复。网络可用时按连续 sequence 分 chunk，携带 checksum 和稳定 upload_session 重试；服务端按 activity_id+chunk_index 幂等保存，乱序可以接受但记录缺口。stop 后 finalize 请求声明 expected chunks/last sequence，只有完整校验通过才把 Activity 从 uploading 推到 committed，避免半条路线进入 feed。本地空间耗尽时要优先保留边界与最近未确认 chunk，并明确提示用户数据风险，不能静默覆盖旧轨迹。

能迁移的判断：设备先可靠记录，上传协议再保证服务器最终获得同一活动。
02

原始轨迹与派生 Activity 分离

原始 GPS points 是可审计输入，保存在对象或时序存储；Activity 表保存 owner、状态、时间范围、版本和处理 watermark。后台 job 对 committed 轨迹做去噪、map matching、距离/海拔计算、隐私裁剪和 segment detection，结果以 activity_id+processor_version 幂等写入。算法升级时从 raw points 重算新 version，再原子切换展示。feed、统计和 leaderboard 只消费 published Activity event。若 processing 失败，用户看到处理中并可重试，不重复上传或创建第二条活动。版本切换前需比较距离、海拔与 segment 命中漂移，异常活动留在旧版并进入人工或批量修复队列。

能迁移的判断：保留原始事实，所有计算结果都带版本并可重建。
03

10M 并发活动靠批量与稳定分区

客户端不应每秒发单点请求，而是压缩数十到数百点批量上传，断线时本地有界缓冲。gateway 按 athlete_id 或 activity_id hash 路由，单活动顺序由 chunk sequence 保证；对象存储承担大字节，metadata/commit 服务只写小状态。大型赛事会让某地区入口和热门 segment 处理过热，因此 ingest 与 downstream processing 分离，队列按租户/区域限流。容量从 points/s、bytes/point、batch size、upload QPS 和 retention 推导，监控 missing chunk、ingest lag 和 processing backlog。压测要同时注入区域突发、超长活动和大量重试，确认元数据分区、对象存储前缀与下游队列都不会形成新热点。

能迁移的判断：批量压缩大幅降低请求数，日志与异步处理隔离实时上传峰值。
04

实时分享是短期、受控的位置流

athlete 明确开启 live sharing 并选择 viewers/expiration 后，设备发送低频位置点；服务端验证 activity、权限和 sequence，保存短 TTL latest state 与短期 replay buffer，通过 SSE/WebSocket 推送。viewer 重连提交 cursor，缺口可 replay，超窗返回 snapshot；位置 age 超阈值显示 stale，不外推猜测。撤销分享或活动结束写 deny/version，连接层每次推送检查当前 permission，旧 cache 不能继续泄漏。名人热点通过区域 fanout/edge 复制，但原始精确位置和保留最小化。安全测试应测权限撤销到所有边缘断连的最坏延迟，并确认过期 replay 与日志不再泄露精确路线。

能迁移的判断：live location 的核心是 consent、freshness、撤销和短保留，不只是连接协议。
05

Feed 是可重算的社交投影

Activity 发布后生成事件，fanout 服务依据 follower graph 物化活跃用户 inbox，长尾可在读取时合并；名人采用 pull 或混合策略避免写爆。feed item 只保存 activity_id、actor、rank fields 和 version，详情回 Activity 服务；block、privacy change 和删除通过快速 deny/final check 覆盖陈旧 inbox。点赞与评论独立写入并异步更新计数，不阻塞活动发布。缓存按 viewer/feed generation 分隔，指标观察 fanout lag、stale denied、celebrity skew 和 first-page p95。投影丢失时可从 published event 重建，重建期间读取必须继续做最终隐私校验并标记 feed 陈旧时间。删除和拉黑事件走高优先级旁路，保证隐私收敛不被普通 fanout 积压阻塞。

能迁移的判断：社交 feed 可以最终一致，但权限撤销和详情事实必须回权威服务。
06

排行榜需要资格、版本和修正

segment 定义为版本化路线 geometry，processing 从 GPS 轨迹识别 effort 并计算 elapsed time、athlete、activity 和 confidence。Leaderboard 以 segment+category+period 物化 Top N，但只有满足隐私、运动类型、GPS 质量和反作弊规则的 effort 才 eligible。迟到上传或算法修正生成更高 version，ranking projection 幂等重排；被举报结果可先隐藏并保留审计。热门 segment 使用局部 Top M 合并，完整 efforts 在分区存储供分页和重算。指标包括 detection lag、rank drift、moderation backlog 和 rebuild time。榜单 generation 切换应原子完成，旧规则版本可追溯，用户申诉后能说明某次排名为何出现、消失或改变。重建完成后抽样核对边界名次。

能迁移的判断：排行榜不是一个 sorted set，而是规则驱动、可修正的派生视图。
