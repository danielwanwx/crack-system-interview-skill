第 11 周讲义 · 主项目 News Aggregator · 迁移 FB Post Search

搜索、索引与排序

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

从事实流开始

新闻聚合先是 source article 的采集、规范化和聚类问题，搜索索引只是派生读模型。article、source、cluster 和 index_document 要分开，以便保留原文事实、去重决定和搜索 schema 的独立演化。

02 · 核心判断

去重与聚类

完全相同 URL、转载文章和同一事件的不同报道应分别处理：URL/hash 去重解决复制，语义聚类服务事件页。聚类要允许合并和拆分，并选择 canonical article 作为展示代表，不能把标题相似就永久合并。

03 · 核心判断

索引是投影

index_document 为检索访问模式而建，包含规范字段、token、时间、语言、权限和排名特征，不是业务表的镜像。写入后的 refresh 延迟是产品可见的最终一致性，需要在 UI 和指标中承认。

04 · 核心判断

查询三段式

先召回候选，再做 filter/ACL，最后根据文本相关性、freshness、质量和多样性排序。把所有逻辑塞进一个搜索查询难以解释和回滚；分层后可分别控制成本和效果。

05 · 核心判断

排名可演化

ranking feature 需要版本化、离线评估和可回滚，不能直接把新模型全量推给用户。新闻尤其要平衡 freshness、来源多样性和相关性，单纯点击率会放大低质量热点。

06 · 核心判断

删除与更新

原文撤回、权限变化和事实更正必须从 source of truth 通过事件传播到 cluster、index 和 cache。删除是保守问题：宁可短暂少召回，也不能长期展示已无权或已撤回内容。

07 · 核心判断

迁移到 FB Post Search

Post Search 的关键差异是可见性：每次检索都必须只从调用者可访问的内容召回。权限不能在生成摘要后再遮盖，ACL filter 必须进入索引或候选阶段，否则已泄露召回事实。

08 · 核心判断

验证效果

search 系统同时看 correctness、quality 和 runtime：index lag、zero-result、duplicate rate、NDCG/CTR 与 P99 缺一不可。只优化吞吐会让用户搜到快但不对的结果。
