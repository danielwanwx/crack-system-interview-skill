维度分值验收标准
Problem framing10能收敛范围，明确 create path / redirect path，说明 out of scope。
Requirements / NFR15FR 清楚，NFR 有数字，且每个 NFR 对应设计影响。

HLD15create flow 和 redirect flow 分开，cache/DB/LB/CDN 位置合理。
Deep dives / failure15能讲短码生成、fast redirect、cache miss、DB down、hot key、counter outage。

Critical misses

出现下面任一情况，即使总分看似够，也要重做一次 15 分钟压缩版。

  • 没有说明 short_code 唯一性。
  • 没有 302 / 410 / 404 语义。
  • 把 cache 当成 source of truth。
  • 没有估算就直接 sharding。
  • redirect path 依赖同步 analytics 或 AI 调用。
  • 没有 cache miss / DB down failure path。
  • API 没有错误码和重试边界。
  • Staff Q&A 少于 8 题能答清。

Repair loop

低分维度不要只写“复习”。必须生成一个可执行修复动作。

低分维度修复动作复测
API重写 API contract table，补 status code 和 idempotency。用 Day 3 口述 5 分钟。
Data model从 redirect lookup 反推 schema 和 index，不保留无解释字段。用 Day 4 口述 5 分钟。
Capacity重新估算 storage / QPS / cache hit / DB lookup。用 Day 5 口述 5 分钟。
Failure补 cache miss、DB down、hot key、counter outage、expired link。用 Staff Q&A 抽 4 题。
