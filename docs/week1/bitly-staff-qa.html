四个维度里，自己最弱的是哪个？如何避免“凭感觉自评”？

结论先行

不要回答一个抽象弱点，而要拿最近一次 45 分钟 mock 的证据：在哪一分钟卡住、漏了哪个产物、哪个追问只能说“看情况”。最低分维度就是下一轮训练目标。

Staff-level 展开

把失败映射到可观察产物：需求缺失＝没有明确 scope；HLD 失败＝组件无法对应需求；Deep Dive 失败＝没有候选方案、决策规则和 failure mode；沟通失败＝长时间沉默或没有时间检查点。

追问：怎么修？只重练最低维度 15–20 分钟，然后立刻做一次完整 mock，验证局部修复没有破坏全局节奏。

哪一阶段最容易超时？面试官在 HLD 阶段持续追问怎么办？

结论先行

常见超时点是需求发散和过早深挖。先回答足够支撑主路径的最小问题；遇到细节追问，用 30–60 秒给方向并确认：“我先完成端到端设计，稍后把这个作为第一个 deep dive，可以吗？”

Staff-level 展开

时间盒不是机械计时，而是风险预算。0–5 分钟冻结 scope，5–12 分钟产出 entities/API，12–27 分钟完成读写主路径，27–45 分钟用于最高信息量的 tradeoffs。每个阶段结束时明确说出已完成产物和下一步。

失败模式：把所有追问都推迟会显得回避；立即深入又可能交付不完整。判断标准是：该问题是否阻塞当前 HLD 的正确性。

Bitly 的读写比对架构有什么影响？10:1 与 100:1 到底用哪个？

结论先行

先声明假设，再从假设推导。原讲义同时出现 10:1 和 100:1，这是不一致，不应混用。面试中可以选 10:1 作为保守基线，再用 100:1 做敏感性分析；两者都说明 redirect path 比 create path 更值得优化。

Staff-level 展开

假设 100M DAU，每人每天创建 1 条、点击 10 次，则平均约 1.2K write QPS 与 11.6K read QPS。峰值再乘 5–10 倍，而不是把平均值当容量。读多驱动 cache-aside、热点指标和读降级；写少不代表唯一 ID、幂等和条件写可以忽略。

决策规则：数据库在峰值下仍有足够余量且热度低，可以暂缓缓存；当延迟、成本或热点成为瓶颈，再引入 Redis。Hit rate 不是固定及格线：DB miss QPS = read QPS × (1 − hit rate)，必须低于数据库安全容量。

如果链接过期进入范围，需要增加哪些字段和逻辑？

结论先行

增加 expires_at，并在每次 redirect 的业务路径上同步判断；数据库 TTL 只负责异步回收，不负责准时阻止跳转。缓存值也要带过期时间，缓存 TTL 不得越过业务到期时间。

Staff-level 展开

NULL 表示永不过期，未来时间表示业务截止点。写入时校验范围；读取时以服务端时间判断；到期返回 410（如果系统知道它曾存在且明确失效），并删除/覆盖缓存。DynamoDB TTL 后台删除可能延迟，因此不能把“记录还在”解释为“仍有效”。

故障模式：只依赖 DynamoDB TTL 会在后台删除前继续跳转；只依赖 Redis TTL 会在 cache miss 后从 DB 重新加载已过期记录。

证据：AWS · Working with expired items and TTL

功能需求和非功能需求谁先问？哪些功能应暂缓？

结论先行

先用一两句话确认核心功能，再问会改变架构的 NFR。核心是 create 与 redirect；custom alias、expiration 需确认；analytics、二维码、A/B testing 默认暂缓。

Staff-level 展开

功能需求定义“系统必须完成什么”，NFR 定义“在什么规模和质量下完成”。没有功能范围，容量数字没有对象；没有 NFR，组件选择没有依据。Staff 级表现是主动说明暂缓项不是永远不做，而是为了先交付闭合主路径。

GET /{short_code} 的 302 和 Location header 应该是什么？如何用 curl 演示？

结论先行

Location 是完整的原始目标 URL，例如 https://example.com/article。响应没有必要返回 JSON；客户端根据 3xx 与 Location 继续导航。

演示

curl -i https://sho.rt/abc123 查看原始 302；curl -IL https://sho.rt/abc123 跟随跳转并显示每一跳。生产环境还应检查 Cache-Control、追踪 ID 和安全过滤。

302 vs 301：302 保留服务端控制，适合可修改、可过期或需统计的短链；301 允许更积极的永久缓存，降低服务负载，但撤销和修正更困难。若必须保持请求方法语义，进一步讨论 307/308。

证据：RFC 9110 §15.4.3

链接不存在或已过期，返回 404 还是 410 Gone？

结论先行

未知、从未存在、或不愿暴露存在性的 code 返回 404；系统确认资源曾存在且被永久删除/过期时可返回 410。410 是更强的永久移除信号，但不是安全边界。

Staff-level 展开

API 语义应与产品隐私策略一致。对公开短链，410 有助于客户端和爬虫理解永久失效；对敏感租户，统一 404 可减少枚举信号。负缓存这两类结果时使用短 TTL，避免刚创建或恢复的链接长时间不可见。

证据：RFC 9110 · 404 · 410

short_code 是主键时，如何查询某用户创建的所有短链？

结论先行

新增 GSI：partition key = created_by，sort key = created_at#short_code，支持按用户分页和时间排序。不要对主表做 Scan。

Staff-level 展开

主表继续为 redirect 点查优化；GSI 为第二访问模式付出额外写放大、存储和最终一致性代价。分页使用最后返回的 key，而不是 offset。若单一大客户产生极端写热点，可在 GSI key 中加入桶并在读取时有限 fan-out。

故障模式：刚创建后立即列举，GSI 可能暂时看不到；产品需要 read-your-writes 时，可在响应中回显新记录或短时从主键补读。

证据：AWS · Global Secondary Indexes

为什么 short_code 做主键？NoSQL 一定优于关系型吗？

结论先行

short_code 是 redirect 唯一已知的查询键，因此它应直接定位记录。NoSQL 是合理选择，但不是必然答案；关键是访问模式、容量、团队能力与一致性需求。

Staff-level 展开

最小记录包含 short_code、long_url、created_by、created_at、expires_at 和必要的状态字段。DynamoDB 提供按 key 水平分布和托管扩缩；PostgreSQL 在规模尚小、需要强约束或复杂管理查询时更简单。Staff 决策应给出切换触发器，而不是用“互联网规模”替代证据。

缓存放在哪里？Hit rate 多高才够？

结论先行

共享 Redis 放在 App Server 与 source-of-truth DB 之间，由应用执行 cache-aside。LB 不理解业务 key 和过期语义，不是该缓存的所有者。Hit rate 由数据库安全容量反推，不使用拍脑袋的 80%。

Staff-level 展开

若峰值 read QPS 为 60K，而 DB 安全承载 12K miss QPS，则需要至少 80% hit rate；还要给故障和增长留余量。指标必须分层：overall hit rate、top-key hit rate、miss QPS、DB throttling、P99、eviction 和 cache error rate。

选择：共享 Redis 减少跨实例重复并支持统一失效；本地 L1 更快且可在 Redis 故障时缓冲，但带来一致性和冷启动问题。可组合 L1 + Redis，但只有测量证明需要时才加复杂度。

证据：Redis · Cache-aside · Amazon Builders' Library · Caching challenges

Redis 宕机后，短链访问如何降级而不把数据库打死？

结论先行

缓存是优化，不应成为正确性来源；但“所有请求直接打 DB”也不是安全降级。使用短超时、熔断、请求合并、DB 并发上限、负载丢弃，并可对已知热链接短时 serve stale。

Staff-level 展开

先检测 cache error rate 与连接池耗尽，快速 bypass；对同一 code 的 miss 做 single-flight；限制每实例到 DB 的并发和全局 QPS；优先保护已有热门链接，非关键管理 API 可降级。恢复时渐进预热并给 TTL 加 jitter，避免集体过期。

最危险的链式故障：Redis outage → 全量 miss → DB throttle → App thread/connection pool 堵塞 → 健康检查失败 → 全 fleet 抖动。必须定期做 cache-disabled load test。

证据：Amazon Builders' Library

如何保证 App Server 无状态？Session 放哪里？

结论先行

任何实例都能处理任意请求：URL mapping、ID 状态、限流计数和任务状态不保存在单机内存。若 create API 需要认证，优先使用可验证 token；确需 server-side session 时放共享 session store，并接受它成为新依赖。

Staff-level 展开

无状态使 LB 可自由路由、滚动发布和 autoscaling。进程内只保留可丢弃缓存和连接池。不要用 sticky sessions 掩盖状态耦合；它会制造热点并使故障转移变差。

90TB 如何分片？short_code 会不会产生热点？

结论先行

用高基数、分布均匀的 short_code 作为 DynamoDB partition key，让服务自动分区。存储量决定必须横向扩展，但请求热点来自访问分布，不来自总字节数。

Staff-level 展开

随机/打散后的 code 通常能均匀分布不同逻辑 key；viral link 是单 key 读热点，应优先由 CDN/Redis/L1 吸收。不要默认给 mapping 表做随机前缀再 fan-out——点查本来只有一个 key，盲目 salting 会把一次读变多次读。只有单 key 超过底层限制且缓存仍不足时才考虑复制/分片该热点。

证据：AWS · Partition key best practices

为什么不用 MD5 截断？Snowflake 如何解决多节点唯一性？

结论先行

截断 hash 只有概率唯一，必须做条件写和碰撞重试；Base62 只是编码，不产生唯一性。Snowflake 类 ID 通过时间戳、节点 ID 和同毫秒序列组合，在没有中心化逐次协调的情况下生成唯一整数，再 Base62 编码。

Staff-level 展开

经典 Snowflake 布局通常是 1 个符号位、41 位时间、10 位 worker、12 位序列，但布局可以按寿命、节点数和单节点吞吐调整。真正难点是 worker ID 分配、时钟回拨、纪元、跨区策略和容量耗尽。

替代方案：数据库 sequence 简单且强唯一，但可能形成集中依赖；预分配 ID 区间减少协调；随机 7–10 字符配合 conditional put 最简单、不可预测，但需要碰撞重试。选择取决于可预测性、安全性、写吞吐和运维成熟度。

故障模式：时钟回拨导致重复或停发；worker ID 重复导致同毫秒碰撞。必须监控 clock skew，并让协调系统租约化分配 worker ID。

参考实现：Twitter Snowflake archive

短链目标更新后，主动失效还是等 24h TTL？TTL 应该多长？

结论先行

写 source of truth 成功后主动删除缓存，再由下一次读取回填；TTL 是失效消息丢失时的安全上界，不是主要一致性机制。TTL 应由更新频率、可接受陈旧窗口、内存预算和数据库回源能力共同决定。

Staff-level 展开

cache-aside 写路径采用 DB update → cache delete。若先删 cache 再写 DB，期间读可能把旧值重新填入。即便采用正确顺序，删除失败仍会陈旧，因此加重试/事件驱动失效和版本字段；高价值链接可使用 versioned cache key。

24 小时不是答案：不可修改的链接可以更长；可修改/撤销的链接应短得多或主动失效。TTL 加随机 jitter，热 key 过期前异步刷新，并对 miss 做请求合并，避免 stampede。

证据：Redis · Cache-aside and invalidation

整个设计最脆弱的单点在哪里？

结论先行

没有上下文时不能只点名一个组件。优先审查“故障会同时阻断 create 与 redirect、且没有安全降级”的依赖：DNS/LB、ID allocator、主存储、缓存故障后的 DB 容量，以及错误配置的全局发布。

Staff-level 展开

ID 服务故障通常只影响 create，可让 redirect 继续；数据库或区域故障会威胁 redirect 正确性；Redis 故障本应可降级，但可能通过流量放大拖垮 DB。Staff 级答案要区分 blast radius、MTTR、RPO/RTO、检测信号和用户体验，而不是只说“多副本”。

优先级：先保护 redirect，因为已发出的链接是外部承诺；create 可以明确返回 503 + Retry-After，而不是生成无法保证唯一性的 code。

什么链接会成为 hot key？如何提前检测和处理？

结论先行

新闻、活动、网红或突发事件中的单一短链会造成高度偏斜。检测 top-K code QPS、partition throttling、cache miss amplification、key latency 与 eviction；基线用分位数和同比增长，而不是只看全局平均。

Staff-level 展开

处理顺序：延长该 key TTL/提前刷新 → 多级缓存或边缘缓存 → request coalescing → 复制热点值。写 salting 适合写热点；对单记录点查盲目 salting 会引入读 fan-out。热点消退后要回收特殊策略，避免永久复杂度。

7 位 Base62、90TB 和 QPS 的计算应该精确到什么程度？

结论先行

容量规划要数量级正确、假设透明、能驱动决策，不需要个位数精确。7 位 Base62 有约 3.52T 种组合；若每天 100M 创建、持续 5 年约 182.5B 条，名义空间约 19 倍，但还需考虑保留字、随机碰撞策略和增长。

Staff-level 展开

如果使用唯一整数再编码，空间利用接近确定；若随机抽取 code，不能把“空间大于记录数”当成无碰撞保证，必须用条件写和重试。存储估算应拆成 record bytes、索引、复制、备份和增长余量；90TB 只是一个假设结果，不是产品事实。
