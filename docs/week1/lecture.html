第 1 周讲义 · URL Shortener / Bitly

把短链接系统讲到可辩护

核心讲义 60m白板复述 30m追问修复 30m
维护 short_code → long_url 映射；create path 保证唯一性和持久化，redirect path 保证 p99、可用性和过期语义。

需求边界

类别本周答案设计影响
Functional Requirements创建短链接；通过短码跳转；可选 expiration；可选 custom alias。需要 write path、read path、unique constraint、过期判断。
Non-Functional Requirementsredirect p99 低延迟；高可用；读多写少；数据持久；合理防滥用。cache/read replica、DB source of truth、rate limit、监控。
Out of scope完整 analytics、campaign dashboard、custom domain、多地域双活、机器学习风控。可以提 optional queue，但不要让主线失焦。

API 契约

Endpoint语义关键点
POST /urls创建短链接，body 包含 long_url、optional custom_alias、optional expires_at。返回 short_code、short_url、expires_at；身份来自 auth context；custom_alias 冲突返回 409。
GET /{short_code}跳转到 long_url。成功返回 302 Location；过期返回 410；不存在返回 404。
GET /urls/{id}可选管理接口。不是 redirect hot path，权限和 pagination 可以简化。
301 vs 302：301 更容易被浏览器和中间层长期缓存，后续变更目标 URL 或过期处理会变困难。面试默认用 302 更稳，除非业务明确需要永久跳转。

数据模型

CREATE TABLE url_mappings (
  id BIGSERIAL PRIMARY KEY,
  short_code VARCHAR(16) NOT NULL UNIQUE,
  long_url TEXT NOT NULL,
  owner_user_id BIGINT NULL,
  created_at TIMESTAMP NOT NULL,
  expires_at TIMESTAMP NULL,
  is_disabled BOOLEAN NOT NULL DEFAULT FALSE
);

CREATE INDEX idx_url_mappings_owner_created
  ON url_mappings(owner_user_id, created_at DESC);

CREATE INDEX idx_url_mappings_expires_at
  ON url_mappings(expires_at);

为什么不是一开始就 NoSQL

Bitly 的核心关系很简单，唯一约束和索引纪律比换数据库更重要。关系型数据库足够表达第一版，并方便讲事务、唯一性和 source of truth。

什么时候考虑分片

当单库写入、索引大小、读 replica、运维窗口或多地域要求真的成为瓶颈，再讨论按 short_code hash 或 tenant/region 分片。

容量判断

问题面试级估算结论
1B URLs 存储如果每条映射粗估 500B，约 500GB，不含索引和冗余。这本身不足以证明必须分片。
100M DAU redirect如果每人每天 10 次 redirect，约 1B/day，平均约 12K QPS，峰值可按 5-10 倍估。读路径需要 cache/read replica，p99 是重点。
创建短链通常远小于 redirect。可以接受更高写延迟，重点是唯一性和持久化。

高层设计

create path / redirect path

路径步骤风险
Create pathClient -> API Gateway -> URL Service -> Code Generator -> DB -> optional event queue。short_code collision、counter service outage、custom alias 冲突、恶意 URL。
Redirect pathClient -> DNS/LB/CDN -> Redirect Service -> Cache -> DB fallback -> 302。cache miss storm、hot key、DB failover、过期链接缓存不一致。
核心表达：cache 是加速层，不是真实来源。DB 决定 long_url、expiration、disabled 状态；cache TTL 必须尊重 expires_at 和禁用逻辑。

短码生成

Hash long_url

简单但有冲突

同一 long_url 可稳定生成，但 collision、可枚举、custom alias 和安全语义要处理。

Counter + base62

可控且高效

递增 ID 转 base62，唯一性清楚。需要避免可猜测，可加 salt/shuffle 或分段发号。

Random code

实现简单

生成后查唯一索引，冲突重试。高规模下要讨论码长、冲突概率和重试成本。

Applied AI 叠加

如果加 AI 功能正确位置不能做什么
自动生成 link title / campaign summary创建后异步任务，结果写 metadata。不能阻塞 POST /urls 的核心创建成功。
