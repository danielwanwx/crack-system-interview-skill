Bitly

Bitly：从需求语义到故障修复

01

先定义重定向语义，再谈架构

Bitly 的用户故事只有创建短链和按短码跳转，但 API 语义会直接决定缓存与一致性。创建接口应接收 long_url、可选自定义 alias 和过期时间，并使用 idempotency key 防止客户端超时重试时生成多个短链。跳转接口通常返回 302，使浏览器仍回到 Bitly，便于更新目标和记录访问；若产品选择永久不可变的链接才考虑 301。不存在、已过期与被禁用最好分别落到 404、410 或受控错误页，否则缓存无法表达真实状态，运维也无法区分数据缺失和策略封禁。

能迁移的判断：状态码、幂等键和过期语义都是数据模型与缓存策略的一部分。
02

从访问路径反推最小数据模型

redirect path 的核心查询是按 short_code 精确读出 long_url、status、expires_at，因此 short_code 必须是主键或唯一索引；owner_id + created_at 只在产品要求用户查看自己的链接时才需要二级索引。数据库是 expiry、disabled 与目标 URL 的 source of truth，cache 只存可丢失的投影。使用 Postgres 能用唯一约束和事务快速得到清晰基线；DynamoDB 也能以 short_code 做 partition key，但应先说明条件写、热点 key、二级索引和多地域一致性成本，不能因为题目说“大规模”就直接换库。还要为自定义 alias 的抢占、过期记录的再利用和 owner 查询写出明确约束，否则同一张表在边界请求下仍会产生歧义。

能迁移的判断：先列查询，再选 key、index 和数据库；技术名词不能替代访问模式。
03

短码唯一性是一条写协议

随机 Base62 码的优点是分散且不容易枚举，但必须依赖数据库唯一约束检测碰撞，并在有限次数内重试；随着码空间利用率增长，碰撞成本会上升。单一全局计数器可保证唯一，却会成为可用性和吞吐瓶颈，也暴露业务规模。更成熟的方案是让 allocator 预分配号段，每个实例在本地单调消费，再做可逆混淆；号段租约、实例崩溃后未用 ID 和重复领取必须有明确状态。面试中不需要一开始实现最复杂方案，而要展示从简单随机重试到号段分配的演化条件。

能迁移的判断：唯一性由数据库约束或分配协议保证，不由“随机概率很低”保证。
04

读路径优化必须保留修复路径

redirect 是极端读多写少，适合 cache-aside：先查 Redis，miss 再读权威 DB，并写入带 TTL 的 cache。热门 key 过期会造成 stampede，可用 singleflight、提前刷新和 jitter 分散过期；不存在的 short code 可短 TTL negative cache，防止恶意扫描打穿数据库。更新目标或禁用链接时要先改权威状态，再删除或版本化 cache；即使失效消息丢失，TTL 仍应让系统最终收敛。CDN 可以吸收全球热点，但其缓存 key、302 响应、撤销窗口和 abuse 风险必须比普通静态内容更谨慎。设计时要逐层写出最大陈旧窗口、回源并发上限与失效失败指标，才能证明这条读路径可控。

能迁移的判断：缓存优化延迟，TTL 与权威回源保证可恢复性；二者职责不同。
05

容量数字用来拒绝过早分片

假设 1B 条 URL mapping，每条连同索引约数百字节，数据量仍可能落在单个成熟数据库集群可处理的范围。真正可能先触顶的是 redirect 峰值 QPS、热点分布和跨地域延迟，而不是总行数本身。应分别估算 create QPS、redirect average/peak QPS、cache hit ratio、回源 QPS、存储与带宽，再决定读副本、缓存或分片。分片触发器应可观测，例如主库写吞吐、索引尺寸、checkpoint 或复制延迟持续越界，而不是口头说“用户多就 sharding”。估算还要保留峰均比与增长假设，说明哪个数字变化时会推翻当前单库结论。

能迁移的判断：规模题的正确动作是给出负载模型和触发器，不是背诵分布式组件。
06

Senior 答案要主动讲故障和运营

Senior 候选人不能停在 happy path。cache 全失效时要限速回源并保护数据库；allocator 不可用时可以暂时只读而不是生成重复码；数据库 failover 时创建接口可重试，但 redirect 是否允许短暂读取旧 cache 要由链接撤销和 availability 优先级决定。还要主动指出 abuse：短码空间可被枚举、恶意 URL 可能借平台传播、热点链接可能成为 DDoS 放大点。可观测指标至少包括 redirect p99、cache hit、DB fallback、allocator collision/retry、410/404 比例和热点 key 分布，这些信号决定下一轮扩展。每个降级还应写清退出条件和数据回补方式，避免临时策略在故障结束后变成永久不一致。

能迁移的判断：Senior 深度来自主动识别风险、选择降级并给出恢复证据。

Dropbox

Dropbox：从需求语义到故障修复

01

把 Blob 与 Metadata 分开建模

Dropbox 的大字节内容适合不可变对象存储，而文件名、namespace、owner、ACL、当前 version、content hash 和同步 sequence 属于 metadata store。二者优化目标不同：对象存储提供高耐久和大吞吐，metadata 需要小记录事务、索引和权限判断。应用服务负责控制面：鉴权、创建 upload session、签发短期 URL、提交版本和发布 change event；客户端直接和对象存储交换字节。这样 50GB 文件不会占满应用服务器带宽，同时也能把“对象存在”与“文件对用户可见”明确分开。白板上必须用不同边界标出控制面和数据面，避免把上传成功误说成版本已经提交。

能迁移的判断：对象存储拥有字节，metadata store 拥有文件语义；只有后者能宣布版本可见。
02

上传是两阶段可恢复协议

客户端先创建 upload session，服务端记录 file_id、base_version、part size、预期 hash 与 idempotency key。每个 part 独立上传、校验并可重试，完成后客户端发 commit；服务端验证对象完整性和权限，再用条件更新把 current_version 原子前进并追加 change log。对象上传成功但 commit 超时，不应重新上传全部字节，而应查询 session 并用同一幂等键继续。最终未被 metadata 引用的 orphan blob 由延迟 GC 清理，不需要跨数据库与对象存储的分布式事务。服务还要区分 part 重传、完整对象校验失败和 metadata 条件写冲突，让客户端知道该续传、重算还是创建冲突版本。

能迁移的判断：先保存不可变字节，再原子提交引用；重试和 GC 取代跨存储事务。
03

同步正确性来自 Change Log 与 Cursor

每个 namespace 维护单调 change sequence，设备保存已经成功应用的 cursor。新版本提交后写入 change log，push/WebSocket 只负责唤醒在线设备；设备随后从 cursor 拉取增量，在本地事务成功后才推进 cursor。通知丢失、合并或乱序只会增加 freshness，因为周期 pull 与 replay 仍能补齐。服务端按 event_id/version 幂等，客户端检测 sequence gap 并请求缺口。把同步建立在可重放日志上，才能解释离线数天、连接切换和进程崩溃后的最终收敛。日志保留期之外则返回带基准 sequence 的完整 snapshot，完成后再恢复增量读取，不能假装旧 cursor 永远有效。

能迁移的判断：通知是加速器，cursor pull 是证明不丢变更的 repair path。
04

并发修改必须有用户可见语义

每次提交携带 base_version。若服务端 current_version 已经前进，说明两台设备从同一旧版本分叉。对一般二进制文件，安全做法是保留两个版本、生成 conflict copy 或要求用户选择；静默 last-write-wins 会不可察觉地丢数据。对可合并文本可另建 merge 语义，但这已经接近 Google Docs 的 operation ordering 题。rename、delete 与 restore 也应成为带版本的 metadata event，删除用 tombstone 保留足够时间，使离线设备重连时不会把已删文件再次上传为当前状态。冲突策略必须成为用户可见契约，并通过 parent version 保留审计和回滚路径。

能迁移的判断：冲突不是异常日志，而是必须设计的产品状态和恢复流程。
05

下载性能不能绕过权限

下载请求先在 metadata 路径验证 ACL、版本和可见性，再签发短时效 presigned URL，让客户端直接从对象存储或 CDN 拉取。热门公共文件适合 CDN，但私有共享文件的 cache key、签名过期和权限撤销必须限制暴露窗口。撤销权限后应立即更新权威 ACL，并使新签名失败；已发出的短期 token 和 CDN 副本只能在承诺窗口内收敛。已经下载到离线设备的字节无法远程魔法删除，应把这一现实边界明确告诉面试官。

能迁移的判断：CDN 降低延迟，短期签名和权威 ACL 限制它带来的撤销风险。
06

Senior 答案要覆盖局部失败与运营信号

metadata 主库不可用时，应停止产生新 current_version，避免同一文件出现多个权威头；已经 commit 的文件可在 ACL 允许下从缓存或对象存储降级只读。对象存储慢时保留 upload session 并退避重试，不能让客户端无限并发。通知服务故障不影响正确性，但会拉高 sync freshness。需要观测 upload-to-commit completion、part retry、orphan blob、cursor lag、conflict rate、ACL deny、签名 URL 失败和 CDN stale window；这些指标直接回答用户的文件是否真的保存、同步和受保护。Senior 回答还要把告警阈值连到具体 runbook：先保护 metadata，再暂停回收，最后按 cursor 与引用表完成对账。

能迁移的判断：每个依赖都要定义检测、降级、恢复和用户可见状态。
