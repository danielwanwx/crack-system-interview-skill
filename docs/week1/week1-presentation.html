Day 1

45 分钟内
如何完整交付

上一节说面试考的是"导航能力"——导航的第一步就是知道自己在地图上的哪个位置。
45 分钟不是很长，没有时间框架就是在赌运气。

参考原文 HI · Delivery Framework
大多数人准备了几周，但45分钟到了却没讲完——为什么？
大多数人的第一直觉
"没讲完说明知识不够，多背几个组件就行了。" 于是继续刷文章。
但事实是：没讲完的人通常是在需求阶段或高层设计阶段花了太多时间， 导致Deep Dives——整场分值最重的环节——根本没时间讲。
核心认知
时间分配是答题质量的乘数。正确框架把 45 分钟分成五段：

需求收集（5 分钟）→ 核心实体（2 分钟）→ API 定义（5 分钟）→ 高层架构（15 分钟）→ 深挖 Tradeoffs（15 分钟）

前两段是约束，后三段才是得分。超了前两段，你就在从最重的科目里偷时间。
读完后你应该能回答
  • 五个阶段各叫什么？各自的目标时间是多少？
  • 为什么需求阶段超过 5 分钟是危险信号？
  • Deep Dives 阶段占 45 分钟的多大比例？为什么是最后讲的反而是分最重的？
  • 如果面试官在高层设计阶段一直追问细节，你该怎么处理？

"我打算用 5 分钟确认需求，5 分钟定 API，15 分钟搭架构，剩下时间深挖——你觉得这个节奏合适吗？"（把主动权交给面试官，展示你在导航。）

Day 2

面试官说
「设计 Bitly」

时间框架建好了，下一步是把"设计 Bitly"这句话变成可以动手的需求。
你的第一句话是什么？这一句话决定了接下来整场面试的走向。

参考原文 Bitly · 功能需求 Bitly · 非功能需求
面试官说「设计 Bitly」，你的第一句话是什么？
大多数人的第一直觉
直接开始画架构图，或者开始说"我们需要一个数据库来存 URL……"
这是最常见也最致命的失误：跳过需求直接设计，等于在没有目的地的情况下开始导航。你解决的可能是一个完全错误的问题。
核心认知
需求分两类，必须都问：

功能需求——用户能做什么？Bitly 的核心是：提交长 URL 得到短码，用短码跳转回原始 URL。其余（自定义短码、过期时间、分析统计）需要问清楚是否在范围内。

非功能需求——系统要多快、多可靠、多大规模？Bitly 的关键约束：读多写少（100M DAU）、重定向延迟低（<100ms P99）、链接不能死（高可用）、短码不能重复（唯一性）。
读完后你应该能回答
  • Bitly 的两个核心功能需求是什么？（用一句话描述每个）
  • 哪些功能是明确在范围之外的？（列举两个）
  • 为什么"重定向延迟"是关键非功能需求？它驱动了哪个架构决策？
  • "100M DAU"这个数字后面，你能推算出每秒大约多少次写入、多少次读取吗？
  • 功能需求和非功能需求，哪个先问？为什么？
面试里一句话
"在开始设计之前，我想先确认几个需求。核心功能我理解是：用户提交长 URL 收到短链，然后短链可以跳转回去——这个理解对吗？除此之外，比如自定义短码和数据统计，这次需要考虑吗？"
核心需求清单
功能需求
  • ✅ 创建短链：提交长 URL，系统返回唯一短码
  • ✅ 跳转还原：访问短链，系统重定向到原始 URL
  • 可选：自定义短码（需确认是否在范围内）
  • 可选：链接过期（需确认时间 TTL 策略）
  • ❌ 超出范围：分析统计、A/B 测试、二维码生成
非功能需求
  • 高可用：短链不能失效，99.9%+ uptime
  • 低延迟：重定向 <100ms P99（读路径优先）
  • 大规模：100M DAU，读写比约 100:1
  • 唯一性：不同长 URL 的短码不能碰撞
  • 最终一致性可接受：不需要强一致性

"Bitly 读写比约 100:1——这个数字对架构决策有什么影响？你会因此做哪些不同的选择？"

"如果面试官说'链接过期'也在范围内，你需要增加哪些字段和逻辑？"

Day 2

API 是
需求的验证机

需求确认了，下一步不是画架构图——是定 API。
API 把你的功能需求翻译成可以验证的契约，每一个端点都对应一个功能需求。

参考原文 Bitly · The API MDN · 302 Found MDN · 301 Moved Permanently
如果跳过 API 定义直接画架构图，会发生什么？
大多数人的第一直觉
"API 是实现细节，到时候写代码再定——先把大图画出来更重要。"
结果：架构图里有一个"App Server"，但面试官问"用户怎么创建短链"时，你才发现自己没想清楚请求格式、响应格式、状态码，讲起来含含糊糊。
核心认知
API 有三个价值：

① 验证功能需求：每个功能需求对应一个端点，没有 API 就没有验证。
② 约束架构设计：Bitly 的核心 API 只有两个—— POST /api/v1/urls（创建）和 GET /{code}（跳转）。这直接决定了读写路径的设计。
③ 暴露关键决策点：跳转用 302（临时重定向）还是 301（永久重定向）？这是个真实的工程权衡，不是随便写一个数字。
302 vs 301：一个真实的工程决策
301 Moved Permanently：浏览器会缓存这个重定向，后续访问不经过服务器。
→ 好处：减少服务器负载。坏处：如果链接需要更新或过期，浏览器缓存永远不会失效。

302 Found：每次访问都回到服务器，服务器控制重定向目标。
→ 好处：可以统计点击数、支持链接过期。坏处：服务器压力更大。

Bitly 选 302：因为他们需要统计每次点击，且支持链接过期/更新。
读完后你应该能回答
  • Bitly 的两个核心 API 端点是什么？（HTTP 方法 + 路径）
  • 创建短链的响应体里应该包含什么字段？
  • 跳转端点返回的不是 200，而是什么状态码？为什么？
  • 为什么 Bitly 选 302 而不是 301？如果不需要统计点击，应该选哪个？
  • 链接不存在时应该返回什么状态码？404 和 410 Gone 哪个更准确？
面试里一句话
"跳转我用 302，因为这样每次点击都经过服务器，我可以统计数据、支持链接过期。如果不需要这些，301 会更省资源。"
核心端点定义
POST
/api/v1/urls
提交长 URL，返回唯一短码。请求体：{ "long_url": "https://..." }。响应体：{ "short_code": "abc123", "short_url": "https://bit.ly/abc123" }
201 Created400 Invalid URL429 Rate Limited
GET
/{short_code}
用短码跳转到原始 URL。响应是重定向，不是返回 URL 字符串。Location header 指向原始 URL。
302 Found 404 Not Found 410 Gone (已过期)
301 Permanent
不推荐
浏览器缓存重定向结果，不再回服务器。链接过期/更新时，旧用户永远看到旧目标。无法统计真实点击数。
302 Temporary
Bitly 选择
每次点击都回到服务器。支持统计、过期、目标 URL 更新。代价是服务器压力更高——用缓存解决。

"GET /{short_code} 这个端点返回的 302，Location header 里的值是什么？用 curl 能演示一下吗？"

"如果一个短链已经过期，返回 404 和 410 Gone 哪个更合适？区别在哪里？"

Day 3

Schema 从
查询路径推导

API 定好了，现在问：这些数据存在哪里，表结构长什么样？
不要凭直觉拍字段，从访问路径反推 Schema——每个字段都需要理由。

参考原文 Bitly · Core Entities HI · 关系型数据库 HI · NoSQL 数据库
你的 url_mappings 表需要什么字段？每个字段你能说清楚为什么需要它吗？
大多数人的第一直觉
"id, long_url, short_code，完事。"
这个 Schema 能用，但面试官会追问：谁创建的？什么时候过期？怎么查询？没有 created_by、expires_at、索引策略，你的设计是不完整的。
核心认知：从读路径推 Schema
先问：最高频的查询是什么？

读路径：GET /{short_code} → 用 short_code 查 long_url，然后重定向。
→ 所以 short_code 必须是主键或有唯一索引，不然每次查询都是全表扫描。

写路径：POST /api/v1/urls → 生成 short_code，插入一条记录。
→ 需要保证 short_code 全局唯一，这是深挖话题（见 S7）。

Bitly 的核心 Schema 应该包含：short_code（主键）、long_url、created_by（user_id）、created_at、expires_at（可选）。
存储选型：关系型 vs NoSQL
关系型（PostgreSQL/MySQL）：事务支持、Schema 约束、JOIN 能力。写入是行级锁，高并发写时可能成瓶颈。

Key-Value NoSQL（DynamoDB/Cassandra）：以 short_code 为 key，long_url 为 value，查询是 O(1)，天然适合 Bitly 的读路径。水平扩展容易。

Bitly 推荐 NoSQL：读路径极简（点查询）、无需 JOIN、需要支撑亿级别读写量。
读完后你应该能回答
  • url_mappings 表最少需要哪几个字段？每个字段的用途是什么？
  • 为什么 short_code 要作为主键而不是用自增 id？
  • expires_at 字段存 NULL 和存一个未来时间，在查询时有什么区别？
  • 为什么 Bitly 这个场景更适合 NoSQL 而不是关系型数据库？
  • 如果用 DynamoDB，partition key 应该是什么？为什么？
面试里一句话
"读路径是按 short_code 点查，所以 short_code 就是主键。我会用 NoSQL，因为无需 JOIN、水平扩展容易、读路径是 O(1) 查询。"
核心数据模型
字段类型说明索引
short_code VARCHAR(8) 主键，唯一短码（Base62 编码） PRIMARY KEY
long_url TEXT 原始长 URL，最长 2048 字符 —
created_by UUID 创建者用户 ID，外键关联 users 表 INDEX
created_at TIMESTAMP 创建时间，用于排序/审计 —
expires_at TIMESTAMP 过期时间，NULL 表示永不过期 INDEX（清理任务用）
关系型数据库
不推荐
支持事务和复杂查询。但 Bitly 的读路径根本不需要 JOIN，垂直扩展有上限，亿级读写时需要手动分片。
Key-Value NoSQL
推荐
short_code → record 的点查是天然 K-V 访问模式。DynamoDB/Cassandra 都能水平扩展、毫秒级读取，完美匹配 Bitly 的读路径。
主持提问

"如果要查'某个用户创建的所有短链'，用 short_code 做主键的 NoSQL 怎么支持？需要什么额外的设计？"

"expires_at 字段超时了但用户还是访问了短链，系统应该怎么处理？"

Day 3

用数字
驱动架构决策

数据模型定了，现在要问：这个系统要处理多大的量？
没有数字的架构决策是猜测，面试官要的是推算过程，不只是结论。

参考原文 Bitly · 容量规划 HI · Numbers to Know
面试官问「你的系统需要分片吗？」——你怎么回答？
大多数人的第一直觉
"肯定需要，Bitly 是大公司，系统规模肯定很大，所以要分布式……"
这是感觉驱动的答案，不是数据驱动的答案。面试官真正想看的是你能不能用数字说话，推算出系统的实际规模，再决定是否需要分片。
核心认知：推算流程
从已知的非功能需求出发：100M DAU，开始推算：

写入（创建短链）：假设每个用户每天创建 1 条短链 → 100M 条/天 ÷ 86,400 秒 ≈ 1,200 QPS 写入

读取（跳转）：假设每个用户每天点击 10 次短链 → 1B 次/天 ÷ 86,400 ≈ 11,500 QPS 读取

读写比约 10:1（甚至更高）→ 这驱动了缓存优先的架构。

存储：100M 条/天 × 365 天 × 5 年 ≈ 180B 条记录，每条 ~500 字节 → 约 90 TB → 单机放不下，需要分片或分布式存储。
读完后你应该能回答
  • 按 100M DAU 估算，每秒大约多少次写入？多少次读取？
  • 读写比是多少？这个比例对缓存策略有什么影响？
  • 5 年数据量大约多少 TB？这个数字说明需要什么存储方案？
  • 短码最短几位可以支持 100M 条/天、存 5 年不重复？（Base62 计算）
  • 面试里，容量规划花多少时间合适？需要精确到个位数吗？
面试里一句话
"100M DAU，假设每人每天创建 1 次、点击 10 次——写入约 1K QPS，读取约 10K QPS，读写比 10:1。存储 5 年大约 90 TB，单机放不下，需要 NoSQL 水平扩展。"
关键数字速查
~1K
写入 QPS
（创建短链）
~10K
读取 QPS
（重定向跳转）
90 TB
5 年存储量
（需要分布式存储）
短码位数推算
Base62 字符集：[0-9a-zA-Z] = 62 个字符
需要覆盖的 URL 数量：100M/天 × 365 × 5 年 ≈ 182B 条

6 位 Base62 = 62⁶ ≈ 56B → 不够
7 位 Base62 = 62⁷ ≈ 3.5T → 足够（覆盖 5 年 20 倍余量）

∴ 短码选 7 位，既够用又不太长。
主持提问

"如果读写比是 10:1，你会把缓存放在哪里？缓存的 hit rate 需要达到多少才能让 DB 压力可接受？"

"90TB 数据怎么分片？按什么维度分？有没有热点风险？"

Day 4

从请求路径
出发画架构

数字算好了，现在可以画架构图了。
不要列组件清单——跟着请求走，从用户到数据库，每一跳都需要理由。

参考原文 Bitly · High-Level Design Bitly · Key Technologies
你的架构图里有哪些组件？每个组件的必要性你能用一句话说清楚吗？
大多数人的第一直觉
"加一个 Load Balancer，加一个 Cache，加一个 Message Queue，加一个 CDN……"
但当面试官问"为什么需要 Message Queue？"时，你说不出来。组件不是越多越好，每个组件的必要性都需要用数字或功能需求来支撑。
核心认知：跟着请求走
写路径（POST /api/v1/urls）：
用户 → Load Balancer → App Server → 生成短码 → 写入 DB

读路径（GET /{code}）：
用户 → Load Balancer → App Server → 查 Cache（Redis） → （miss 时）查 DB → 302 重定向

每个组件的理由：
→ Load Balancer：10K QPS 单机扛不住，需要水平扩展
→ Redis Cache：读写比 10:1，绝大多数请求是重复的热链接，缓存命中率可达 80%+
→ NoSQL DB：90TB 数据，需要水平分片能力
→ 无 Message Queue：写入路径简单，不需要异步解耦（除非加统计功能）
读完后你应该能回答
  • 读路径经过哪几个组件？每个组件做什么？
  • 写路径和读路径的主要区别是什么？
  • 为什么 Cache 放在 App Server 和 DB 之间，而不是放在 LB 之前？
  • 缓存未命中时的"回填"流程是什么？（Cache-Aside 模式）
  • 如何保证 App Server 是无状态的？为什么无状态很重要？

Day 5–6

深挖三大
工程决策

架构图搭起来了，接下来是整场分值最重的环节：深挖 Tradeoffs。
面试官不是要你背答案，而是要看你能不能展开权衡——为什么选这个，不选那个会怎样。

参考原文 深挖①：唯一性 深挖②：重定向速度 深挖③：扩展性
面试官说「聊聊如何保证短链全局唯一」——你的第一反应是什么？
大多数人的第一直觉
"用 MD5 hash 长 URL，截取前 7 位作为短码。"
这个方案有两个根本问题：碰撞（不同 URL 的 hash 前缀可能相同）和无法保证全局唯一（多个 App Server 并发生成时没有协调机制）。
深挖① — 短码唯一性
方案：自增 ID + Base62 编码

步骤：DB 或专用 ID 生成服务生成全局自增 ID（如 1234567），然后用 Base62 编码：1234567 → "5lj"（3 位）或补齐到 7 位。

优点：无碰撞、长度可预测、ID 有序（可以用于分析）。
缺点：暴露业务量（从 ID 可以推算每天有多少链接被创建）；需要全局唯一 ID 生成器，单点风险。

面试追问：如果用 Twitter Snowflake 方案（时间戳 + 机器 ID + 序列号）怎么解决单点问题？
深挖② — 重定向速度
三层缓存策略：

第一层：浏览器缓存——如果用 301，浏览器自己缓存（但无法统计/更新）
第二层：Redis 缓存——80%+ 热链接命中，P99 延迟降至 <5ms
第三层：CDN 边缘节点——把 302 响应推到离用户最近的节点

Bitly 实际：选 302（每次回服务器）+ Redis Cache（大量热链接命中）。
深挖③ — 扩展到 1B 链接
DynamoDB 水平分片：用 short_code 作为 partition key，DynamoDB 自动分布到多个分区。

热点风险：如果某个短链突然爆红（viral link），单个分区可能过热。解决方案：在热 key 前加随机前缀（读时并发查询多个前缀，聚合结果）。

App Server 扩展：无状态 + Auto Scaling Group，配合 LB 水平扩展，无状态是前提。
读完后你应该能回答
  • 为什么 MD5 hash 截断不适合作为短码生成方案？
  • 自增 ID + Base62 的主要优点和缺点各是什么？
  • 如果同时有 100 台 App Server，用自增 ID 需要什么额外组件保证全局唯一？
  • Redis 缓存的 TTL 设多长合适？TTL 太短和太长各有什么问题？
  • DynamoDB 的 partition key 用 short_code 会有热点问题吗？如何缓解？
  • 什么样的短链最容易成为"热 key"？你怎么提前检测？
面试里一句话
"短码用自增 ID + Base62——无碰撞、可预测长度。代价是暴露业务量，可以接受。速度靠 Redis Cache，命中率 80%+ 时 P99 < 10ms。扩展靠 DynamoDB 自动分片 + App Server 无状态水平扩展。"
深挖① — 短码生成策略对比
MD5 Hash 截断
有风险
MD5(long_url) → 取前 7 位。问题：
• 碰撞概率：7 位 = 62⁷ ≈ 3.5T，生日问题下 1B 条时碰撞概率 ~0.01%，不可接受
• 碰撞检测需要重试，增加写入延迟
• 多节点并发无法保证全局唯一
自增 ID + Base62
推荐
全局自增 ID（1, 2, 3…） → Base62 编码 → 短码。
• 零碰撞概率（数学保证唯一）
• 长度可预测（7 位覆盖 3.5T）
• 代价：需要全局 ID 生成器（可用 Snowflake 或 DB 序列）
深挖② — 缓存 TTL 权衡
TTL 太短（如 1 分钟）
  • 缓存命中率低，DB 压力大
  • 热链接每次重新从 DB 加载
  • 成本高（大量 DB 查询）
TTL 合理（如 24 小时）
  • 热链接全天命中，DB 压力低
  • 支持 24 小时内的链接更新窗口
  • 内存占用可控（7 字节 × 命中 key 数量）
