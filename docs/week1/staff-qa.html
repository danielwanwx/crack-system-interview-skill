Bitly

Q1

为什么不直接对 long URL 做 hash 截断成 short code？

为什么面试官会追：候选人说“hash 一下就唯一”，没有讨论碰撞或相同 URL 的产品语义。

详细回答

截断 hash 只能提供概率分散，不能提供数学唯一性；两个不同 URL 仍可能映射到同一 short code，所以写入必须使用唯一约束并在冲突时加盐或重新生成。相同 long URL 是否复用同一短链也是产品选择：若要按用户、过期时间或 campaign 分开统计，就不能只按内容去重。我会先用随机 Base62 + conditional insert 的简单方案，监控 collision rate；当写吞吐或重试成本成为瓶颈，再迁移到号段 allocator。这样唯一性由可验证的写协议保证，而不是由“碰撞很少”保证。

继续追问：如果数据库在 conditional insert 后响应丢失，客户端重试如何避免生成第二个短链？

常见漏项：把 hash 冲突概率低等同于不存在；忽略同一 long URL 的多租户语义。

现场修复

画出 generate → conditional insert → conflict/retry → response 丢失后的完整状态机。

  • PostgreSQL · Transactions
Q2

redirect path 为什么选择 302，而不是 301？

为什么面试官会追：候选人只说两者都是重定向，或认为 301 一定更快。

详细回答

301 会鼓励客户端和中间缓存长期记住目标，后续请求可能绕过 Bitly；这能降低服务端流量，却会削弱点击统计、目标更新、撤销和恶意链接封禁。302 让每次访问仍回到服务端，可执行状态检查并保留运营控制，代价是多一次网络和缓存成本。我会根据产品是否允许目标改变、是否需要实时撤销和分析来选择；若提供不可变链接层级，可让那一类明确使用长期缓存，而不是对所有链接一刀切。

继续追问：如果必须在一分钟内撤销恶意热点链接，CDN 和 browser cache 如何配合？

常见漏项：只从 HTTP 定义回答，没有连接到 analytics、撤销与缓存。

现场修复

列出 301/302 对控制权、延迟、统计和失效窗口的四列表。

证据
  • Bitly · 2) How can we ensure that redirects are fast?
  • Caching · CDN (Content Delivery Network)
Q3

一个明星短链突然成为热点，缓存已经有值，为什么还可能出问题？

为什么面试官会追：候选人认为 cache hit 就意味着热点已经解决。

详细回答

热点仍会集中到单个 Redis shard、单个网络连接池或同一 CDN origin；如果 key 过期，所有请求还会同时回源形成 stampede。解决顺序是先测量热点位置，再做本地 in-process cache、CDN/edge cache、热点 key replication 或 request coalescing，并为 TTL 加 jitter 与提前刷新。失效路径同样重要：复制越多，撤销窗口越难收敛，所以恶意链接状态可用短 TTL 或版本号快速覆盖。优化目标不是“更多 cache”，而是让热点读量和失效成本都可控。

继续追问：若热点 key 已被禁用，但某个 region 的失效事件丢失，如何限制错误跳转窗口？

常见漏项：只回答加机器或增加 TTL，没有故障与撤销语义。

现场修复

画出热点 key 在 edge、process cache、Redis 和 DB 四层的命中与失效时序。

证据
  • Caching · Hot Keys
  • Caching · Cache Stampede (Thundering Herd)
Q4

1B URLs 是否意味着现在必须分片？

为什么面试官会追：候选人看到十亿就直接画 shard router。

详细回答

不能只用行数下结论。我会先估每条记录和索引的字节数，得到存储规模；再估 create 和 redirect 的平均/峰值 QPS、读写比、cache hit 与回源 QPS。URL mapping 很小且读多写少，单个 Postgres 集群配读缓存可能已经足够，过早分片会增加全局唯一、再平衡、跨 shard 运营查询和恢复复杂度。只有当主库写吞吐、索引/备份窗口、复制延迟或单节点容量持续触顶时才引入分片，并选择能保持 short_code lookup 单分区的 key。

继续追问：如果要支持 owner 列表查询，按 short_code 分片会带来什么新问题？

常见漏项：没有任何数字，也没有说明分片的代价。

现场修复

写一张 storage/QPS/cache-hit 假设表，并给出三条可观测分片 trigger。

证据

  • 2. 定义非功能性需求 · 可伸缩性
Q5

数据库故障时，redirect 应该 fail-open 读旧缓存还是 fail-closed？

为什么面试官会追：候选人笼统回答“为了高可用继续服务”或“为了一致性全部失败”。

详细回答

这是业务风险选择，不是 CAP 口号。普通不可变短链在 DB 暂时不可用时，可以从带版本和有限 TTL 的旧缓存继续重定向，以保护 availability；但刚被禁用、过期或涉嫌恶意的链接不能无限 fail-open。可以把 revoke 状态放在更短 TTL 的 deny cache，或让高风险域名强制回权威路径。恢复后要限制回源并监控旧版本占比。答案必须明确哪些状态允许陈旧、最大窗口多长、如何发现和结束降级，而不是对整个系统统一选 C 或 A。

继续追问：网络分区只影响一个 region 时，如何防止该 region 永久读取旧 cache？

常见漏项：没有区分普通链接与撤销链接，也没有恢复条件。

现场修复

为 active、expired、disabled、malicious 四种状态分别写 fail-open/fail-closed 策略。

证据

  • Bitly · Senior

Senior / Staff Drill

Dropbox

Q1

为什么不让文件先经过应用服务器再写对象存储？

为什么面试官会追：候选人画了 client → API server → S3 的整文件数据路径。

详细回答

应用服务器适合鉴权和 metadata，不适合承担 50GB 文件的双倍网络带宽、连接时长和重试状态。更稳妥的方案是先创建受 ACL 约束的 upload session，再给客户端短期预签名 URL 直接上传对象存储；应用只在 commit 时验证 content hash、size、base version 和权限。这样数据面可以独立扩展，应用故障也不会迫使已上传分片重传。代价是要显式设计 session、part 校验、过期、滥用限制和 metadata commit，不能把 presigned URL 当成跳过授权的捷径。

继续追问：客户端拿到 URL 后上传了超大或错误 MIME 文件，服务端如何阻止它成为可见版本？

常见漏项：只说“省带宽”，没有提交协议、权限和校验。

现场修复

画 control plane 与 data plane，并标出每一步授权、hash 和状态。

证据

  • Dropbox · 1) How can you support large files?
Q2

对象已上传成功，但 metadata commit 超时，系统现在是什么状态？

为什么面试官会追：候选人说用分布式事务，或直接重新上传。

详细回答

对象此时只是不可见的候选 blob，文件的 current_version 仍由 metadata 决定。客户端用同一 idempotency key 查询或重试 commit；服务端若已提交就返回同一 version，若未提交则验证 blob/hash 后执行条件更新。如果最终不再重试，session 到期后 GC 扫描没有 metadata 引用的 blob。这样不需要让对象存储参与数据库事务，也不会把半完成上传展示给其他设备。用户界面应显示 pending/retry，而不是错误地显示“已同步”。

继续追问：GC 如何避免删除一个 commit 刚成功但索引尚未可见的对象？

常见漏项：把对象存在等同于文件版本已生效。

现场修复

写出 uploading、uploaded、committed、expired 四状态和幂等转移。

证据

Q3

如果 WebSocket 通知丢了，为什么你仍能保证设备最终同步？

为什么面试官会追：候选人把 WebSocket/Kafka 说成唯一同步来源。

详细回答

正确性依赖可重放 change log，而不是瞬时通知。每个 namespace 有单调 sequence，设备只在本地成功应用事件后推进 cursor；WebSocket 只告诉设备“有新版本”，随后设备仍按 cursor 拉取。通知丢失时，周期 pull、应用重新前台或重连都会请求 cursor 之后的事件；发现 gap 时可以补洞。事件带稳定 ID/version，重放是幂等的。因此丢通知影响 freshness，不会永久丢文件。需要监控 cursor lag 和 gap repair，而不是只看连接数。

继续追问：change log 保留期结束，而某设备离线更久怎么办？

常见漏项：回答自动重连，但没有 cursor、gap 和 retention。

现场修复

画通知丢失后的 pull replay 时序，并定义 full resync 触发条件。

证据
  • Dropbox · 2) How can we make uploads, downloads, and syncing as fast as possible?
  • Real-time Updates · "How do you handle connection failures and reconnection?"
Q4

两台离线设备同时修改同一文件，为什么不用 last-write-wins？

为什么面试官会追：候选人用时间戳选最新版本，未讨论时钟与数据丢失。

详细回答

last-write-wins 会把一个真实用户版本静默覆盖，而且客户端时钟不能作为可靠全局顺序。提交应携带 base_version；服务端发现 current_version 已变化时把它识别为并发分叉。对二进制文件可保存 conflict copy 或让用户选择，对可合并文本另走 merge 流程。两个版本都保留可审计 parent，后续清理有明确策略。这样用户至少看得到冲突并能恢复。Dropbox 与 Google Docs 的差别也在这里：前者以文件版本为原子单位，后者需要 operation 级收敛。

继续追问：rename 与 delete 并发时，冲突副本策略如何调整？

常见漏项：假设服务端收到顺序就是用户因果顺序。

现场修复

用 base_version 画出两个客户端从 v7 分叉、服务端收到 v8/v9 的处理。

Q5

共享权限刚被撤销，但 CDN 仍有文件副本，如何定义安全边界？

为什么面试官会追：候选人说 purge CDN 即可，或忽略已签发 URL。

详细回答

先承认撤销不是瞬时魔法。权威 ACL 更新后，服务端不再签发新 URL；现有 token 必须短时效、绑定对象版本和权限范围，CDN cache key 不能把私有内容当公共对象共享。高敏感文件可要求每次经鉴权 edge 或完全不进入长 TTL CDN。撤销事件触发 purge 只是加速，真正的上界由 token/TTL 决定。已经下载到离线设备的内容无法回收，应明确为产品和合规边界，并以审计记录证明撤销时间和后续访问。

继续追问：如果 CDN purge API 部分失败，你用什么指标证明暴露窗口没有超出承诺？

常见漏项：没有 token TTL、版本化 cache key 或现实的离线边界。

现场修复

列出 ACL、presigned URL、CDN cache 与本地下载四层的撤销能力和最大窗口。

证据
  • Dropbox · 3) How can you ensure file security?
