第 8 周讲义 · 主项目 Payment System · 迁移 Robinhood

支付与金融状态

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

定义金融真相

支付系统的真相是不可变 ledger entry，而不是某个可更新的 balance 字段。balance 是从分录导出的 projection，可重建；每笔业务事实必须能被审计和对账。

02 · 核心判断

划分状态机

payment_intent、authorization、capture、refund、provider_event 分开建模，避免把外部 provider 状态压成一个布尔值。每个状态转换都要携带 provider reference、请求幂等键和原因码。

03 · 核心判断

幂等是 API 合同

merchant 的 create/capture/refund 请求必须以 idempotency key 绑定业务意图和参数摘要。相同键返回同一结果；相同业务对象的不同键要由状态机拒绝或显式允许，不能靠客户端自觉。

04 · 核心判断

账本不变量

任何已记账业务都要满足借贷合计为零，并且金额、币种和精度不可在事后悄悄改写。余额投影可以异步，但结算、退款和额度判断必须从可验证的账本状态导出。

05 · 核心判断

处理外部不确定性

provider timeout、webhook 重复和乱序是常态。outbox/inbox 让本地状态与事件可靠衔接，reconciliation 定期以 provider report 发现未知结果；它们减少不确定性，但不能把第三方调用变成原子事务。

06 · 核心判断

把一致性放对位置

同一 payment 的状态推进和 ledger posting 需要强约束，报表、余额缓存和通知可最终一致。用小范围事务或条件写保护钱的状态，而不是让整个平台为全局串行化付费。

07 · 核心判断

迁移到 Robinhood

Robinhood 的 order、execution 和 portfolio 同样需要 append-only 事实与投影，但市场数据是外部高速流。订单状态必须可审计，portfolio 可短暂滞后；价格陈旧时 UI 要明确 timestamp 而不是伪造实时性。

08 · 核心判断

对账与运营

对账不是月底报表，而是持续发现本地账本、provider 事件和银行结算差异的控制回路。差异进入可重放的 investigation queue，修复以补偿分录而非改历史数据，才能留下可信审计。
