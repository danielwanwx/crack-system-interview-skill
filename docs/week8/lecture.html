Tinder

Tinder：从需求语义到故障修复

01

把候选视图与关系事实分开

Tinder 的候选栈是可重算的派生视图，Swipe 与 Match 才是不能随意丢失的关系事实。候选服务可以读取 geo index、偏好、活跃度和预计算分数，短暂返回旧排序通常只是体验问题；用户右滑后却必须立即写入权威 Swipe，并让后续候选过滤读到该事实。若双方都右滑，则以排序后的两个 user_id 作为 canonical match key，唯一约束保证只创建一次。这样缓存、预计算和搜索索引都可按性能独立演进，而不会把陈旧候选误当成可重复建立关系的许可。

能迁移的判断：feed 可以重算，swipe 与 match 必须有唯一、幂等的权威记录。
02

Swipe 写入要同时低延迟和可重试

客户端为一次 swipe 发送稳定 request_id，服务端以 (viewer_id,target_id) 作为业务唯一键执行 upsert，并保存方向、版本和时间。右滑事务写入后查询反向 swipe；若反向已为 like，则尝试插入 canonical match。两个方向同时到达时都可能判断满足条件，但 Match 唯一约束只允许一条，失败者读取已有结果。通知通过 outbox 异步发送并按 match_id 去重，不能把消息是否成功作为关系是否成立的依据。响应超时后客户端重试得到同一 swipe/match 状态，而不是再次触发配对。

能迁移的判断：唯一键解决重复事实，outbox 解决事实与通知之间的故障窗口。
03

候选生成采用预计算与在线过滤组合

完全在线查询能使用最新位置和偏好，却在高峰时把多维过滤、排序和排除集合都压到数据库；完全预计算读取快，但用户移动、修改偏好或被他人屏蔽后容易陈旧。更稳妥的方案是离线或异步为活跃用户生成较宽的候选池，在线读取时再按当前位置、年龄、隐私、block 和最新 swipe 做最终过滤，池低于阈值时补充。缓存保存候选 ID 和生成版本，不保存关系真相。监控 pool hit、filter rejection、refill latency 和空栈率，才能判断该增加预计算还是优化索引。

能迁移的判断：预计算降低成本，在线权威过滤守住新鲜度和安全边界。
04

已看集合是精确性与成本的选择

最直接的做法是在候选查询中排除所有历史 swipe，但集合增大后 join 或巨大 NOT IN 会拖慢请求。可把近期 swipe 放入按用户分区的 Redis Set，长期事实保留在数据库；候选服务批量检查缓存，miss 时查询数据库并回填。Bloom Filter 能用更少内存快速判定“肯定没看过”，但 false positive 会错误隐藏本可展示的人，因此只能在候选充足且产品接受少量漏展示时使用，不能拿它判断 match。过滤结果还要区分 swipe、block、report 和 unmatch，因为这些状态的保留期与安全要求不同。

能迁移的判断：近似结构只能优化候选筛选，不能决定用户关系或安全状态。
05

地理检索先取候选再精确判断

geo index 把用户位置映射到 cell 或空间树，查询当前及相邻区域可快速缩小候选范围，但 cell 覆盖的是近似形状，必须再计算精确距离。位置记录带 timestamp 和精度，过旧或精度不足时降低排名或不展示；最大距离、年龄和偏好作为在线过滤，而不是为每种组合建立爆炸式索引。高密度城市会形成热点 cell，可动态细分或复制只读候选；低密度区域逐级扩大半径。原始精确位置属于敏感数据，候选响应只暴露模糊距离，访问和保留都应最小化。

能迁移的判断：空间索引优化搜索范围，精确过滤和隐私规则决定是否真的可展示。
06

Senior 要主动覆盖删除、屏蔽与热点

真实系统不仅处理右滑，还要处理 block、report、账号删除、位置撤回和名人流量。block 必须在候选在线过滤和消息入口都立即生效，不能等待异步索引最终收敛；删除流程需要从候选池、缓存和搜索投影清除，同时按法规处理权威记录。热门 profile 会让 target 分区和反向查询过热，可按 viewer 分区保存 swipe、异步聚合 popularity，并对可见范围限流。指标应包含 swipe p99、duplicate request、match uniqueness conflict、candidate rejection reason、block leakage 和 empty-stack rate。运营还要定期抽样检查被屏蔽用户是否意外出现在候选、通知或推荐回填中。

能迁移的判断：Senior 深度体现在安全状态优先级、删除传播和热点访问模型。

Local Delivery Service

Local Delivery Service：从需求语义到故障修复

01

可配送性不是简单的距离查询

用户输入地址后，系统先用 geo index 找出覆盖半径内的配送中心，再结合营业状态、服务区域、当前容量和粗略 ETA 过滤。直接对所有仓库调用第三方交通服务会造成高延迟和成本，因此只对少量地理候选做精确 drive-time 估算，并缓存短期区域路况。返回结果要带估算时间和生成时刻，不能承诺为永久事实。地址解析失败或 ETA 服务降级时，可以回退到静态路网或距离阈值，同时扩大承诺时间；若无法安全判断则明确不可配送，而不是给出虚假快速承诺。

能迁移的判断：地理索引生成候选，交通、容量和服务边界共同决定可配送性。
02

浏览库存与结算库存是两种读

商品列表追求低延迟，可以从缓存或读副本读取仓库 SKU 的近似 available，并接受几秒陈旧；结算则会形成承诺，必须回权威主库重新验证。模型至少保存 on_hand、reserved、version，并让 available=on_hand-reserved 成为可检查不变量。缓存值应带 warehouse_id、SKU、version 与短 TTL，库存变更后主动失效，但即使消息丢失也只影响浏览展示，不会造成超卖。产品需允许“刚才显示有货、结算时售罄”的明确状态，并监控该比例决定是否缩短 TTL 或预留。

能迁移的判断：缓存回答可能有货，权威事务决定能否承诺。
03

优先选择单库事务解决库存争用

若一张订单由单个配送中心履约，可把 Order、OrderItem、Inventory 与 IdempotencyKey 放在同一 Postgres 分区或集群。事务先按稳定顺序锁定或条件更新所有 SKU，确保 available 足够，再创建订单和 reservation；任一 SKU 不足则整体回滚。相比跨两个数据库的分布式锁，这个基线更容易证明原子性、处理死锁并恢复。只有仓库数量、写吞吐或组织边界证明单库不可行时才按 warehouse_id 分区，而不是看到“本地配送”就主动引入复杂协调。上线后用锁等待、事务重试、死锁率和热 SKU 冲突证明是否真的需要拆分。

能迁移的判断：把共同变化的数据 colocate，通常比补偿跨库不一致更可靠。
04

幂等下单与支付要分阶段

客户端用稳定 checkout_id 提交，数据库唯一约束保证重试返回同一订单。库存 reservation 与订单 pending 可在事务内完成，支付调用在事务外通过 outbox 启动；支付成功推进 confirmed，失败或超时由持久化 timer 释放 reservation。不能在持有数据库锁时等待支付，也不能支付成功后才发现库存已卖光。若支付结果未知，订单保持 payment_pending 并查询支付方，释放动作必须检查 reservation version，防止旧 timer 清除已确认订单。每个阶段都有用户状态和可重试边界。

能迁移的判断：短事务锁库存，持久化工作流协调支付与超时。
05

缓存与读副本必须携带陈旧预算

availability 查询可以按 warehouse-SKU 缓存并批量读取，热门商品使用 request coalescing 和 jitter 避免过期回源。读副本承担商品详情和历史订单等容忍延迟的查询，但 checkout 读写必须路由主库或具备等价一致保证的 owner。系统在响应里可附 availability_version，结算请求提交看到的版本作为提示，不过最终仍由条件更新判断。指标应包含 cache age、replica lag、checkout stockout、stale-positive 和主库 fallback；若 lag 超标，停止从副本展示精确数量而不是继续假装新鲜。恢复后分批开放精确库存展示，避免缓存和副本同时回暖造成新的读风暴。

能迁移的判断：任何派生读都要声明最大陈旧、权威回源和超限降级。
06

按仓库分区时守住单一所有权

warehouse_id 是天然分区键，因为目录库存和大多数订单都在一个配送中心内完成，能把争用和故障隔离。路由目录把地址候选映射到 shard，库存写只由该 shard owner 接受；读缓存可以复制。仓库关闭或 shard 故障时，不能把原订单悄悄切到另一仓库，因为商品、价格、ETA 和 reservation 都可能不同，应取消旧 reservation 后重新报价并征得用户确认。再平衡要迁移版本和写所有权，旧 owner 通过 epoch 被 fencing。运营关注每仓 QPS、hot SKU、lock wait、stockout 与迁移冲突。

能迁移的判断：仓库分区降低跨节点事务，但故障转移仍是新的业务承诺。

Yelp

Yelp：从需求语义到故障修复

01

权威数据库与搜索索引各司其职

Business、Review、用户约束和审核状态应保存在支持事务与唯一约束的权威数据库；Elasticsearch 保存面向检索的反规范化 business document，包括名称、类别、坐标、rating 和部分展示字段。写入先提交权威状态和 outbox/CDC 位置，再由 consumer 幂等更新索引。搜索结果可能短暂旧，因此响应可携带 index_version 或更新时间；business 详情和写后读取必要时回权威库。索引丢失时能从数据库和变更日志全量重建，这一能力比声称双写“通常都成功”更重要。

能迁移的判断：数据库守住事实与约束，搜索引擎提供可重建的查询投影。
02

附近搜索是多阶段查询

用户查询“附近的川菜”包含文本、类别、地理范围、营业状态和排序。搜索引擎先用倒排匹配名称/类别，再用 geo shape 或 geo distance 过滤候选，最后按相关性、距离、rating 和质量信号排序；不能分别查数据库再在应用内对海量结果求交。空间索引返回近似候选后仍要精确距离判断，分页使用稳定 sort values/search_after，避免深 offset。位置或营业状态更新通过异步索引传播，产品明确最大 lag；紧急停业或安全下架需有快速 deny path。

能迁移的判断：复杂搜索把候选生成、精确过滤和稳定排序放在可索引的数据模型中。
03

一用户一商家一评论靠约束

Review 表使用 (user_id,business_id) 唯一键，create 采用条件插入，update 修改同一 review_id 并递增 version；客户端 request_id 处理响应丢失重试。若允许删除后重评，要明确软删除与 generation 语义，不能因 status=deleted 就绕过唯一性制造多条活跃记录。审核状态与正文分开，只有 approved 评论进入公开投影。并发编辑使用 optimistic concurrency，冲突让客户端基于最新版本重试。唯一约束比“先查询是否写过”可靠，因为两个请求可能同时通过查询。审计日志还要保留状态变化和审核理由，支持申诉、回滚与投影重建。

能迁移的判断：业务唯一键和版本控制让评论规则在并发下仍成立。
04

评分聚合必须幂等且可重算

搜索页不能每次扫描全部评论计算平均分。可维护每商家的 rating_count、rating_sum 和分布，评论创建、修改、删除生成带 event_id、old_rating、new_rating 的事件；聚合 consumer 记录已处理事件或使用版本条件，防止重复累加。聚合值更新后再投影到搜索索引和缓存，因此会有可度量 lag。夜间或按需 job 从 approved Review 重算并与增量结果对比，发现 drift 后修复。只存 average 无法正确处理删除和修改，也难以展示评分分布。发布重算结果时采用 generation 切换，并记录受影响商家和修正幅度供运营核验。

能迁移的判断：保存可逆统计量、事件幂等与重算通道，派生评分才值得信任。
05

城市和社区是版本化地理分类

“SoHo”或“湾区”不是简单字符串，它们有多边形边界、层级、别名、语言和版本。地址解析后得到坐标，location taxonomy 服务返回所属 city/neighborhood IDs；索引文档保存这些稳定 ID 和边界版本，查询按 ID filter。边界更新时通过后台 reindex 迁移受影响商家，期间可同时识别新旧版本并监控差异。用户输入别名先映射到 canonical location，再执行地理查询。不要把社区名直接写进每条记录后依赖模糊匹配，否则改名和重叠边界难以治理。

能迁移的判断：地名是有版本的领域数据，不是随手复制的展示字符串。
06

搜索质量、审核与故障降级

Yelp 的相关性会被刷评、重复商家、恶意关键词和热门地点流量影响。写入侧限制频率、验证业务唯一性并把可疑 Review 送审核；搜索侧只使用 approved 状态，紧急下架通过 deny list 立即覆盖陈旧索引。Elasticsearch 故障时可降级到热门类别缓存或按 business_id 详情查询，但不能返回未经权限和安全过滤的旧快照。指标包括 search p95、zero-result、index lag、rating drift、duplicate review conflict、moderation backlog 和 cache age。恢复时先重放 CDC，再抽样比对权威记录。索引恢复完成前保持安全过滤和降级标记，不能因流量压力提前解除保护。

能迁移的判断：搜索系统不仅追求召回和延迟，还要有安全覆盖、重建和质量证据。
