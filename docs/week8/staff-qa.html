第 8 周 Staff 追问与详细答案

Senior / Staff Drill

Tinder

Q1

两位用户同时右滑，如何保证只产生一个 Match？

为什么面试官会追：候选人先读反向 swipe 再直接 insert，没有并发保护。

详细回答

每次 swipe 先以 (viewer_id,target_id) 唯一键幂等写入。判断双方都 like 后，使用 min(userA,userB)、max(userA,userB) 组成 canonical match key，在数据库执行唯一插入；两个请求都到达也只有一个成功，另一个读取已存在 Match。关系事实和 notification outbox 在同一事务中写，后台至少一次发通知并以 match_id 去重。这样不依赖跨请求锁，也不会因客户端重试创建重复关系。若 unmatch 后允许再次配对，则需要显式 generation/version，而不是复用模糊布尔值。

继续追问：Match 已提交但响应丢失，客户端如何确认结果？

常见漏项：用“先查再写”假设消除竞争，忽略唯一约束和通知重复。

现场修复

并排画两个 swipe 事务，在 Match insert 处标出唯一键、winner 与 loser read。

证据

  • Dealing with Contention · Conditional Writes
Q2

候选栈如何在低延迟与最新位置之间取舍？

为什么面试官会追：候选人只选预计算或只选在线查询，没有陈旧边界。

详细回答

我会预计算一个较宽候选池，降低每次请求的多维检索成本；在线读取时再用最新位置、偏好、block 和 swipe 事实过滤并排序。候选池带 generated_at 与规则版本，位置移动或偏好变化可触发失效，但不要求每个事件立即重建全部列表。池低于阈值时异步补充，极端情况下回退到受限在线查询。产品明确候选位置允许多旧，并监控 filter rejection、pool age、refill latency 与 empty-stack。低延迟来自减少在线候选生成，不来自跳过最新安全和关系检查。

继续追问：用户从纽约飞到旧金山后，怎样避免继续展示旧城市候选？

常见漏项：把 cache hit 当成正确结果，没有在线验证。

现场修复

把候选生成拆成离线 pool、在线 filter、refill 三步并标出版本。

证据
  • Tinder · 2) How can we ensure low latency for feed/stack generation?
  • Caching · Cache Consistency
Q3

数亿次历史 swipe 怎样避免重复展示，又不让查询变慢？

为什么面试官会追：候选人把完整 ID 集合塞进单个请求或单个 Redis key。

详细回答

权威 Swipe 按 viewer_id 分区并对 target_id 建复合键，保证精确查询和幂等写。在线候选批量检查近期 Redis Set，长期集合可按时间分段或由数据库批量过滤；缓存丢失时仍能回权威数据。Bloom Filter 适合快速排除“可能看过”的候选，节省远程查询，但 false positive 会漏展示，因此只能作为优化，并在候选不足时绕过或重建，绝不能用于 Match 判断。还要给 block/report 单独精确集合，因为安全状态不接受近似误判。

继续追问：Bloom Filter 饱和后如何滚动升级而不突然重复展示？

常见漏项：忽略 false positive 的产品后果，或把所有用户历史放进一个热点集合。

现场修复

为近期、长期、block 三类集合分别写存储、精度和回源策略。

证据
  • Tinder · 3) How can the system avoid showing user profiles that the user has previously swiped on?
  • Data Structures for Big Data · Bloom Filter
Q4

为什么不把 Swipe 和 Match 都只放 Redis？

为什么面试官会追：候选人以低延迟为由让缓存成为唯一关系存储。

详细回答

Redis 的原子操作适合降低写延迟，但关系事实需要持久、可审计、可恢复，并支持删除和客服查询。若 Redis 是唯一来源，故障、淘汰或错误配置可能永久丢失 swipe；异步回写又会产生已通知 Match 却未落权威库的窗口。我会让数据库或具备持久一致保证的单分区存储保存 Swipe/Match，用 Redis 缓存近期 seen set 和候选池；若采用 Redis 原子脚本加速，成功后仍通过日志/outbox 持久化，并定义读取和重建边界。技术选择必须说明数据丢失预算。

继续追问：若数据库写延迟突然升高，产品如何降级而不丢 swipe？

常见漏项：只谈性能，不谈持久性、重建和用户关系风险。

现场修复

列出关系事实与派生缓存，模拟 Redis 全失效后的重建过程。

Q5

精确位置既用于匹配又高度敏感，系统边界如何设计？

为什么面试官会追：候选人只讨论 geohash 性能，没有隐私和删除。

详细回答

位置接入保存最小必要精度和短保留窗口，原始坐标加密并限制只有候选服务按目的访问；普通 API 只返回模糊距离，不暴露坐标或精确 cell。候选索引使用不可直接还原或粗粒度位置，并把 block、隐身和位置权限作为在线强制过滤。日志和分析事件避免携带原始坐标，调试使用受审计采样。用户撤回权限或删除账户时，权威位置、缓存、候选池和异步索引都有可追踪清除任务，监控 block leakage 与 deletion lag，而不是只删主表。

继续追问：为反欺诈需要保留位置时，如何限制用途扩张？

常见漏项：把 geohash 当匿名化，忽略其可反推区域和关联身份。

现场修复

画原始位置、索引位置、API 输出三种数据形态及访问主体。

证据
  • Database Indexing · Geospatial Indexes
  • Tinder · Senior

Senior / Staff Drill

Local Delivery Service

Q1

为什么不能从缓存直接判断库存并创建订单？

为什么面试官会追：候选人把低延迟 availability cache 当成库存 source of truth。

详细回答

缓存可能因 TTL、失效消息丢失或副本延迟而高估库存；多个请求还会同时读到同一剩余数量。浏览页可用缓存提供近似结果，但 checkout 必须在权威库存行上执行条件更新 available >= requested，或在事务内锁定并扣减 reservation。条件失败返回明确售罄或重新报价，不能仅靠缓存补偿。缓存记录版本和短 TTL，库存提交后失效，用 stale-positive 与 checkout stockout 衡量体验。这样陈旧只造成一次结算失败，不会形成不可兑现订单。

继续追问：热点 SKU 的权威行争用严重时，如何扩展而不超卖？

常见漏项：以为 cache decrement 的原子性自动覆盖数据库恢复和多区域问题。

现场修复

画浏览读与 checkout 写两条路径，圈出唯一承诺点。

证据
  • Local Delivery Service · 2) Make availability lookups fast and scalable
  • Caching · Cache Consistency
Q2

分布式锁和单个 Postgres 事务，为什么优先后者？

为什么面试官会追：候选人无条件引入 Redis lock 协调库存与订单。

详细回答

若订单与库存可以 colocate，Postgres 事务同时提供原子提交、崩溃恢复和约束检查，失败只需回滚；外部分布式锁还要处理 TTL、续租、旧持有者复活、锁成功但数据库失败以及 fencing。锁并不会自动让两个数据库形成事务。我的基线是在一个事务中按稳定顺序条件扣减 SKU、创建订单和 reservation，用 checkout_id 去重。只有数据必须跨独立 owner，且无法调整边界时才考虑 saga 或带 fencing 的协调，并明确中间状态和补偿。

继续追问：一张订单必须跨两个仓库履约时，状态机如何变化？

常见漏项：把“锁住了”当成持久性和原子性的证明。

现场修复

列出 lock 服务和 DB 各自在五个故障点的状态，再与单事务比较。

Q3

怎样把交通时间纳入可配送仓库选择，又不为每次请求调用所有仓库？

为什么面试官会追：候选人要么只算直线距离，要么对全量仓库同步请求 ETA。

详细回答

先用服务区域或 geospatial index 找出少量附近仓库，再根据营业、库存摘要和容量过滤，最后只对前 N 个候选调用或批量调用 ETA 服务。区域路况可按 origin cell、destination cell 和时间桶短期缓存，响应带生成时间；ETA 服务故障时回退静态路网或保守距离模型，并扩大承诺窗口。最终选择还要验证 checkout 时库存，不把 ETA 结果与商品承诺混为一谈。监控候选数、ETA 调用率、误差、fallback 和超时成本。

继续追问：交通突发变化导致已下订单延迟，是否自动换仓？

常见漏项：没有粗筛，或把昂贵外部 ETA 放在无界 fanout。

现场修复

画 geo 粗筛 → 库存摘要 → top-N ETA → checkout 四阶段漏斗。

证据

  • Database Indexing · Geospatial Indexes
Q4

读副本能否承担所有 availability 查询？

为什么面试官会追：候选人只说加 read replica，没有复制延迟语义。

详细回答

浏览和搜索可从读副本读取近似库存，只要产品接受陈旧并在响应中避免承诺精确数量；checkout 不能依赖可能落后的副本做扣减判断，必须路由到库存 owner。副本 lag 超过阈值时，可以隐藏“仅剩 1 件”、回退缓存摘要或把结果标为需结算确认，而不是继续显示精确值。失效消息也可能先于或晚于复制，因此缓存回填要记录版本。监控 replica lag、stale positive、主库 fallback 和 checkout rejection，用这些证据决定副本是否真正改善体验。

继续追问：用户刚取消订单后，如何让自己立即看到库存释放？

常见漏项：把读扩展与正确扣减混为同一问题。

现场修复

为 browse、cart、checkout 三类读写分别标注主库、副本、缓存和陈旧预算。

Q5

库存 shard 故障时，能否直接把订单切到另一个仓库？

为什么面试官会追：候选人把跨仓切换当作透明基础设施 failover。

详细回答

不能默认透明切换，因为另一仓库的商品、价格、税费、配送时间和真实库存都可能不同。对尚未承诺的搜索请求可重新选择候选；对已有 reservation 的订单，先确认旧 owner 是否提交，未知状态保持 pending 并恢复或对账，不能在新仓再次扣减。若明确未提交，可释放幂等意图、重新报价并让用户确认。每个仓库 shard 需要副本和明确 failover owner，epoch 防止旧主恢复后继续写。系统以 pending age、duplicate reservation、reprice rate 和 recovery lag 验证流程。

继续追问：旧主隔离但仍可写本地磁盘时，如何防止 split brain 库存？

常见漏项：只考虑可用性，没有重新承诺对用户的产品影响。

现场修复

区分 browse reroute、pre-order retry、committed order recovery 三种故障状态。

Yelp

Q1

评分平均值如何低延迟更新，又避免重复事件算两次？

为什么面试官会追：候选人只说异步聚合，没有幂等或重算。

详细回答

权威 Review 保存评分和状态，变更事务写 outbox。事件携带唯一 event_id、business_id、old_rating 与 new_rating；聚合器在同一事务记录已处理 event_id 并调整 rating_sum、rating_count 或分布，重复投递安全跳过。聚合结果再异步写搜索索引和缓存，允许短暂 lag。定期从 approved Review 全量或分区重算，与增量 projection 比较，发现 drift 后替换。这样低延迟来自增量计算，正确性来自幂等、可逆统计量和重算，而不是假设消息只到一次。

继续追问：评论从 5 星改成 2 星，再被删除，三个事件如何更新统计？

常见漏项：只存 average，无法正确处理修改和删除。

现场修复

用 count/sum 演算 create、update、delete，并重复投递其中一个事件。

Q2

如何保证一个用户对一家商家只有一条有效评论？

为什么面试官会追：候选人使用写前查询，没有数据库约束。

详细回答

在 Review 上建立 (user_id,business_id) 唯一约束，创建用 conditional insert；若已存在则返回冲突或按产品语义转为更新。编辑携带 version 做 optimistic concurrency，避免两个设备相互覆盖。软删除后是否允许重评要明确：可以复用同一记录并递增 generation，仍保持唯一活跃事实。request_id 防止响应丢失重试，审核状态不改变业务唯一键。缓存和搜索索引只保存投影，不能承担约束。这样即使并发请求同时到达，也由原子唯一性而不是时间窗口决定结果。

继续追问：用户删除账户后，评论匿名保留还是删除会怎样影响唯一键？

常见漏项：认为应用层先查就足够，忽略竞态与软删除语义。

现场修复

同时发两个 create 请求，验证唯一约束、返回状态和投影事件。

Q3

文本、距离、类别和评分组合查询如何避免全库扫描？

为什么面试官会追：候选人只说 Elasticsearch，却没有 mapping 和查询阶段。

详细回答

索引文档反规范化 business 名称、类别 IDs、geo_point、rating、营业状态和可搜索文本。查询先用倒排生成文本候选，同时应用 category/geo 过滤，再用 distance、relevance 与 quality signals 排序；需要精确距离时对有限候选复核。分页用 search_after 和稳定 tie-breaker，避免深 offset。字段 mapping、分析器和更新频率按查询设计，不能把所有字段设成全文。权威 business/review 仍在数据库，索引通过 CDC 更新并可重建，紧急下架由 deny filter 覆盖 lag。

继续追问：用户翻页期间评分变化，如何减少重复和漏项？

常见漏项：技术名词代替数据模型，忽略稳定分页和索引陈旧。

现场修复

写出索引 document、query/filter/sort 和 cursor 四部分。

Q4

怎样支持按城市或社区名搜索，边界变化时怎么办？

为什么面试官会追：候选人把地名当 business 表中的自由文本。

详细回答

建立 canonical location taxonomy：每个 city/neighborhood 有稳定 ID、多边形、层级、别名、语言和 version。商家地址 geocode 成坐标，再由 point-in-polygon 计算所属 location IDs；搜索索引保存 ID，而用户输入先解析别名到 canonical ID。边界或命名更新时生成新版本，后台重算受影响商家并 reindex，查询过渡期可识别两版，完成后收敛。监控 unresolved alias、边界重叠和 reindex lag。这样地名变化是可审计数据迁移，而不是批量改字符串。

继续追问：一个商家同时属于多个重叠社区时，排序和展示如何定义？

常见漏项：只用 geohash，无法表达行政或产品定义的多边形区域。

现场修复

设计 location、alias、boundary_version 三个实体并演示一次边界升级。

Q5

搜索索引落后时，用户刚提交的评论为什么没出现在搜索结果，如何处理？

为什么面试官会追：候选人承诺异步索引强一致，或让写请求同步等待 refresh。

详细回答

写请求在权威数据库提交后立即返回 review_id 和 pending/approved 状态；详情页对作者可读权威记录或合并本地 write overlay，实现 read-your-writes。搜索与公开评分通过 CDC 异步更新，产品给出可接受 index lag，并监控事件 age。普通请求不应同步等待全局 refresh，因为会把搜索集群故障扩散到写入。若紧急删除或安全封禁，使用小型强制 deny path 覆盖陈旧索引。恢复时从 checkpoint 重放，并抽样比较索引文档与数据库版本。

继续追问：CDC 消息永久丢失时，怎样发现并修复单个 business 文档？

常见漏项：没有区分作者写后体验、公共搜索和安全删除。

现场修复

分别写 create response、author detail、public search、moderation delete 的一致性策略。
