01

idempotency key 绑定什么？

参考回答
幂等键不能只是随机字符串；它必须绑定谁在对哪一笔 payment intent 做什么操作以及关键金额/币种。相同键不同参数要拒绝，避免调用方误把旧键复用到另一笔扣款。
常见失分
只说 Redis 去重。
继续追问
保留期为什么不能无限短？
02

为什么 balance 不是唯一真相？

参考回答
余额会因缓存、异步投影或 bug 变旧，但分录保留每次资金变化的来源和方向。以账本重建余额和做对账，才能解释任何差异；代价是读路径需要投影和快照。
常见失分
直接 update users.balance。
继续追问
余额投影落后时，哪些操作必须回账本确认？
03

double-entry 的不变量是什么？

参考回答
一笔 transaction 内所有 debit 与 credit 的代数和必须为零，并且同一货币单位按固定精度记录。这个约束让资金不会凭空创造或消失；跨币种则必须显式记录兑换腿和汇率。
常见失分
只说有两个表。
继续追问
手续费属于哪一条分录？
04

webhook 重复乱序延迟怎么处理？

参考回答
webhook 是提示和证据之一，不是可以盲目覆盖本地状态的命令。按 event id 去重、验证签名，并仅允许符合状态机的推进；未知或乱序事件存档后等待查询/后续事件补全。
常见失分
最后到达的事件覆盖前一状态。
继续追问
capture 先到、authorize 后到如何表现？
05

authorize 成功但 capture timeout 如何恢复？

参考回答
先把本地 intent 标成 pending/unknown，并用 provider reference 查询最终状态。确认未 capture 才可安全重试；若已 capture 则补记账和通知，绝不能凭超时再扣一次。
常见失分
超时即失败并重新收费。
继续追问
用户长时间看见 pending 怎么沟通？
06

退款和 chargeback 如何留痕？

参考回答
退款和拒付都应产生新的不可变业务事件与反向/调整分录，而不是修改原支付金额。这样可以追踪原因、操作者、provider reference 和争议生命周期；余额投影随新分录更新。
常见失分
直接把支付记录改为 refunded。
继续追问
部分退款如何表示？
07

outbox 解决什么，不解决什么？

参考回答
outbox 保证 payment 状态提交后，后续事件最终会被发布，即使进程在两步之间崩溃。它不能保证 provider 只执行一次，也不能消灭消费者重复，因此外部调用和消费端仍需要幂等。
常见失分
声称 outbox 实现全局 exactly-once。
继续追问
outbox 发布重复怎样处理？
08

对账发现差异怎么办？

参考回答
先将差异按金额、provider reference 和时间窗口归类，冻结高风险操作或标记 pending。确认真实状态后追加补偿分录、修正 projection 并记录根因；不能偷偷修改旧 ledger 行，否则审计链断裂。
常见失分
夜间脚本直接修 balance。
继续追问
哪些差异需要立即人工升级？
09

多币种精度放在哪里？

参考回答
金额以每种币种的最小可计量单位保存为整数，展示层才格式化。汇率、舍入模式和货币腿必须记录在 transaction 中；使用浮点数会让对账无法稳定复现。
常见失分
数据库 decimal 就够了。
继续追问
JPY 和 USD 的最小单位如何共存？
10

哪些部分需要强一致？

参考回答
同一 payment 的合法状态转换与对应分录必须在一个受约束的提交边界内，防止重复扣账。merchant dashboard、余额缓存、邮件和分析可以异步更新；这样把昂贵的一致性局限在钱的核心路径。
常见失分
整个系统都 serializable。
继续追问
额度检查为何常常也在强路径？
11

Robinhood 的 order、execution、portfolio 如何区分？

参考回答
用户下单只创建 order，撮合产生一个或多个 execution，持仓则由 executions 和 corporate actions 派生。三者分开才能处理部分成交、取消和市场数据延迟；把它们合成一张表会丢失审计语义。
常见失分
下单即更新持仓。
继续追问
部分成交时 UI 显示什么？
12

市场数据陈旧时 UI 怎么做？

参考回答
每个 quote 带来源和时间戳，前端在超过 freshness 阈值时明确标注延迟或暂不可交易。portfolio 估值可以使用最近价格，但执行决策必须走受控的市场数据路径；不能把旧缓存伪装成实时。
常见失分
只加更短 TTL。
继续追问
行情源故障时是否允许下单？
13

Donations receipt 如何可追溯不重复？

参考回答
receipt 应引用 donor、donation intent、最终 ledger transaction 和税务字段，并以版本化模板生成。发送失败可重发同一 receipt，不能重新创建 donation；这和支付幂等是同一个业务边界。
常见失分
支付成功就异步随机发邮件。
继续追问
捐款匿名与审计如何兼顾？
14

风控拒绝和技术失败的 API 语义？

参考回答
风控拒绝应返回明确且可审计的业务状态，不建议客户端无脑重试。网络超时或 provider 5xx 则是未知/可重试状态，服务端需保留 intent 并控制重试；两者混在一起会让商户重复扣款。
常见失分
所有非 200 都重试。
继续追问
哪些错误应暴露给终端用户？
15

如何隔离 merchant 重试风暴？

参考回答
单个商户的 bug 不能耗尽 provider 配额或工作池，因此在入口、队列和 provider adapter 分别做 per-merchant rate limit、并发上限和 circuit breaker。代价是该商户会被降级，但其他租户和资金正确性得到保护。
常见失分
只扩容队列。
继续追问
限流后怎样避免商户永远卡住？
16

核心 SLO 是什么？

参考回答
支付成功率和 P95 重要，但更关键的是账本不平衡必须为零、未知状态不能无限老化。监控 webhook 延迟、provider 对账差异和退款完成时间，才能及早发现看似成功但资金未收敛的问题。
常见失分
只看 API 错误率。
继续追问
哪项指标应触发停止新 capture？
