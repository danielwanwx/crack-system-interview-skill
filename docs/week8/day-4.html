今日唯一产出物

交付

候选仓库粗筛与 ETA 精排、库存事务、cache/read-replica 陈旧边界、分区策略和仓库故障 runbook。

验收标准

  • 附近仓库先按地理粗筛，再只对少量候选调用交通 ETA，避免 N 次昂贵外部请求。
  • 库存扣减用条件更新或单库事务证明不超卖，锁失败和事务重试有界。
  • 缓存、读副本和 partition 的陈旧窗口不会越过 checkout 正确性边界。
不通过怎么修

如果把 availability 当一个布尔值，分成 catalog visibility、cached estimate、authoritative stock 和 reserved quantity 四层，并写明每层过期与回源。

真实追问 1

为什么不能从缓存直接判断库存并创建订单？

触发：候选人把低延迟 availability cache 当成库存 source of truth。

先修：画浏览读与 checkout 写两条路径，圈出唯一承诺点。

真实追问 2

分布式锁和单个 Postgres 事务，为什么优先后者？

触发：候选人无条件引入 Redis lock 协调库存与订单。

先修：列出 lock 服务和 DB 各自在五个故障点的状态，再与单事务比较。

真实追问 3

怎样把交通时间纳入可配送仓库选择，又不为每次请求调用所有仓库？

触发：候选人要么只算直线距离，要么对全量仓库同步请求 ETA。

先修：画 geo 粗筛 → 库存摘要 → top-N ETA → checkout 四阶段漏斗。
