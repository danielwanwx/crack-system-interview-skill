Day 1 · Week 8

支付状态机

创建 PaymentIntent； 授权与扣款； 退款和拒付。
核心 invariant：同一 payment_id 的状态只能沿合法边迁移，terminal state 不可被旧事件回滚

参考原文 Payment · FR Payment · Entities 本日核心追问
需求边界
  • 先锁定 3 个核心动作：创建 PaymentIntent、授权与扣款、退款和拒付
  • 明确 out of scope，避免把后台运营、推荐、BI 一次性塞进主链路
  • 读写比例、峰值 QPS、对象规模、保留周期必须给数量级假设
  • 先定义用户可见语义，再讨论 Kafka、Redis 或数据库选型
正确性不变量
  • 同一 payment_id 的状态只能沿合法边迁移，terminal state 不可被旧事件回滚
  • 每个状态转换写 precondition；失败重试不能跳过中间状态
  • 数据库事实、缓存视图、异步派生数据必须标出 authoritative source
  • 重复、乱序、超时、部分成功四类故障逐一给答案
容量与 SLO
  • Week 8 Day 1：先估峰值写 QPS，再按分区键检查热点
  • 在线路径给 p50 / p99；后台 repair 给 freshness 与 backlog SLO
  • 吞吐扩容不能破坏 per-key 顺序、唯一性或余额守恒
  • 压测必须覆盖 skew：最热 1% key 承担 40% 流量
层次 面试中要落到的机制 必须暴露的指标 失败时动作
API 幂等键、分页游标、版本号、错误码 request_p99 / 409_rate 限流、返回可重试语义
Storage 主键、分区键、二级索引、唯一约束 lock_wait / hot_partition 迁移流量或拆热点
Async 事件 ID、顺序键、consumer checkpoint lag / redelivery / DLQ 重放并保持幂等
Derived watermark、版本、last_updated_at freshness / mismatch_rate 从 source of truth 重建
Repair 按 updated_at 扫描 stuck payment，向 PSP 查询后 CAS 修正 stuck_age_p99 / repair_count 先止血，再对账，再回填

算法 · Week 8 · Tries / Heap

Tries × Heap
前缀与 Top-K

Trie 用边表达字符、节点表达前缀；Heap 只维护当前最有希望的 K 个候选。重点是剪枝条件与堆顶语义。

模式 题目 要点
Trie Implement Trie insert/search/prefix 三套终止语义
Trie+DFS Word Search II 原地标记 + found 去重 + 剪空枝
Heap Kth Largest 大小 K 的 min-heap
Two Heaps Find Median lower max-heap 与 upper min-heap 平衡 
class TrieNode:
    children = {}
    terminal = False

def top_k(stream, k):
    heap = []
    for x in stream:
        heappush(heap, x)
        if len(heap) > k: heappop(heap)
    return heap  # invariant: heap contains largest min(k, seen)
