工作流与可靠执行

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

先定义执行承诺

Scheduler 应承诺 due job 至少被派发一次，并让业务 handler 以 idempotency 承担重复执行，而不是许诺不可实现的全局 exactly-once。schedule 定义、run、attempt 和 lease 必须可审计，才能解释任何漏跑或重跑。

02 · 核心判断

状态与时间

schedule 保存时区、cron、next_run_at 和版本；job_run 是某一逻辑时间点的一次责任；attempt 是实际执行尝试。把 logical run time 与 wall clock 分开，才能正确处理 DST、misfire 和 backfill。

03 · 核心判断

claim 与 lease

多个 scheduler 可以扫描 due runs，但只能用条件 claim 把一个 run 交给一个 lease owner。worker heartbeat 续租，超时后 reclaim；旧 worker 后续写回必须带 fence token，防止 lease overlap 破坏状态。

04 · 核心判断

重试与失败

每次重试创建新 attempt 并保留错误、开始/结束时间和 backoff 原因。可重试错误、永久错误和人工介入错误分流到不同策略；DLQ 是隔离和调查工具，不是永远不看的墓地。

05 · 核心判断

公平与隔离

队列必须按 tenant、优先级或资源池调度，避免一个客户的海量 job 饿死其他客户。公平不等于严格轮询，还要考虑长任务、并发上限和最低服务保证。

06 · 核心判断

控制面与数据面

control plane 负责创建 schedule、扩容和观察；worker data plane 在短暂控制面故障时可继续执行已 claim 的任务。两者失联时要限制新 claim，避免过期配置或重复 backfill。

07 · 核心判断

迁移到 GitHub Actions

Actions 把 job payload 变成不可信代码，因此 queue、runner、sandbox、artifact 和 secret 是一套完整生命周期。调度正确性仍适用，但安全隔离和资源治理成为主路径而非附加项。

08 · 核心判断

可观测性

最有价值的指标是 due-to-start delay、lease overlap、attempt success、retry age 和 tenant fairness。它们比 CPU 更接近用户承诺，也能区分 scheduler 漏派、worker 饱和和 handler 自身失败。
