News Aggregator

News Aggregator：从需求语义到故障修复

01

先建立可恢复的内容接入链

每个 publisher 有抓取策略、feed URL、last_success、next_due 和退避状态。scheduler 只把到期 source 放入持久化队列，worker 使用条件 lease 领取，下载后保存原始响应摘要并解析 Article。规范化 canonical URL、发布时间、标题和来源，再用 URL、内容 fingerprint 与来源 ID 去重。权威 Article 提交后写事件，索引、类别 feed、媒体处理分别消费。某个 consumer 失败只造成对应投影延迟，不应迫使重新抓取外站；checkpoint 与 replay 让故障恢复从已确认位置继续。每个阶段保存 attempt、错误类型和处理版本，便于重新解析、来源级熔断和端到端新鲜度归因。

能迁移的判断：抓取事实只提交一次，后续索引与 feed 都由可重放事件派生。
02

文章去重要保留来源与版本

同一新闻可能由转载、canonical URL 变化或标题轻微修改产生多条记录。只按 URL 去重会漏掉镜像，只按内容 hash 又可能合并不同版本。可以先规范化 URL 和 tracking 参数，再组合 publisher article ID、canonical tag、正文 fingerprint 与时间窗口生成候选簇，最终保留 canonical Article 和多个 SourceOccurrence。后续修订以 version/event 更新，不覆盖原始出处。去重属于可修正判断，错误合并要能拆分并重建 feed；指标包括 duplicate cluster size、merge reversal 和 source coverage。人工纠错必须产生新关联版本并触发受影响索引和 feed 重算，不能直接覆写历史证据。

能迁移的判断：去重生成可审计的关联，不应抹掉来源或历史版本。
03

稳定分页需要排序边界

offset 分页在新文章插入或排名变化时会重复和漏掉结果。cursor 应编码最后一项的稳定复合排序键，例如 published_at、score bucket 和 article_id，并携带 feed generation 或 snapshot watermark。下一页只读取严格落在边界之后的数据；若个性化模型改变，可选择继续旧 generation 或明确刷新 feed。cursor 必须签名、防篡改且有过期策略。对无限滚动，服务端限制页大小和最大历史窗口，客户端按 article_id 去重。监控 cursor error、duplicate exposure 和 page latency。删除或安全下架仍由读取时 deny 过滤覆盖旧快照，避免稳定分页成为撤销内容的后门。

能迁移的判断：cursor 是查询快照与排序规则的契约，不只是数据库主键。
04

低延迟 feed 来自分层物化与降级

基础全球或类别 feed 可以预计算并缓存，用户请求只做轻量过滤、个性化重排和分页；活跃用户可异步物化前几页，长尾用户在请求时合并类别候选。缓存 key 带 feed generation 和 locale，TTL 加 jitter，breaking news 通过事件主动刷新。个性化服务超时则回退地区/类别热门，不阻断基础内容。详情页从 Article 服务读取并重定向 publisher。端到端 200ms 预算应拆为 gateway、feed fetch、filter/rank 和 serialization，而不是笼统给 Redis。每个 fallback 都要在响应和指标中可见，防止系统长期降级却仍被误认为正常。

能迁移的判断：先准备可复用候选，再把个性化限制为有超时的可选步骤。
05

30 分钟新鲜度要拆成可测流水线

“文章 30 分钟内出现”包含 publisher 发现间隔、fetch queue age、解析、去重、索引、ranking 和 cache refresh。不同来源按更新频率和重要性动态调整 polling，支持 RSS/webhook 的来源优先用推送；失败采用带上限退避，不能因一个坏源拖住队列。每条 Article 保存 discovered_at、fetched_at、indexed_at、feed_visible_at，按来源计算分位数和漏抓率。积压时优先可信高价值来源，延后媒体转码或个性化，仍先让文本进入基础 feed。报警必须指出超时发生在哪一段、影响哪些来源和多少文章，并给出下一次追赶完成时间。

能迁移的判断：新鲜度 SLO 必须能定位到流水线阶段，才有可执行修复。
06

媒体和 breaking news 是独立压力源

图片与视频不应经过 feed API 同步转码。接入后把原始媒体写对象存储，异步做安全扫描、缩略图和格式转换，CDN 用内容版本 URL 分发；处理未完成时 feed 显示占位，不阻塞文章。breaking news 会同时造成 publisher 更新、用户读取和相同媒体热点，入口用队列吸收写峰值，边缘缓存热门 feed 与资源，保护权威库。若 CDN 或媒体管道故障，可退回文本卡片。指标分开观察 fetch burst、media backlog、origin egress 和 hot-key hit。紧急撤回媒体时使用短期 deny 与版本化 URL，让安全操作不依赖所有边缘缓存同时失效。

能迁移的判断：把媒体作为异步派生资产，文本 feed 才能在峰值下继续服务。

FB Post Search

FB Post Search：从需求语义到故障修复

01

Post 是事实，Index 是可重建视图

创建或编辑 Post 先提交权威数据库，保存 author、text、created_at、visibility、version 与删除状态，同时写 outbox/CDC 位置。indexer 幂等地生成搜索文档和 postings，记录 source_version，旧事件不能覆盖新状态。搜索结果返回 post_id 和摘要，展示前根据风险和成本做 visibility final check；详情始终回 Post 服务。索引损坏可从 checkpoint 重放或全量重建，双写失败不会永久丢帖。这个边界允许搜索为吞吐最终一致，同时不把索引当作权限和生命周期的唯一来源。抽样校验 source_version、可见性和删除状态，可以主动发现沉默的投影漂移。

能迁移的判断：写入事实一次，搜索投影可重复生成和校验。
02

倒排索引把扫描变成集合运算

文本经过与查询一致的 normalization、tokenization 和 analyzer，倒排表从 token 指向包含它的 doc IDs、词频和 position。单关键词读取 postings；多关键词做交集或 union；phrase 查询需要 position，或为常见短语建立 bigram/shingle，以更多存储和写放大换低延迟。结构化字段如 created_at、author、visibility class 和 like bucket 使用 filter/sort 结构，不能全部当全文。mapping 变更要版本化新索引并双读抽样，完成 backfill 后原子切 alias，避免线上原地改坏。上线前用真实语言、长短文本和高频词基准测试召回、索引大小、merge 成本与查询分位数。

能迁移的判断：查询能力来自具体 postings 与字段 mapping，不来自“用了搜索引擎”。
03

排序需要为访问模式准备投影

按 recency 可用 created_at + post_id 稳定排序；按 like count 若请求时从权威库 join 并排序会很慢。可以把 like_count 作为索引字段，但每个 like 都更新大文档会放大写入，因此先在流中按 post 聚合，按时间或变化阈值批量刷新搜索文档。排序值会变化，cursor 携带 sort values 和 generation，客户端仍按 post_id 去重。强时效热点可用小型 trending projection 覆盖，普通搜索接受秒级 lag。指标观察 ranking lag、refresh QPS 和结果漂移。删除和权限变化必须抢占排序刷新队列，避免低价值热度更新阻塞安全状态。

能迁移的判断：排序字段也是派生数据，需要批量、版本和陈旧预算。
04

高写入通过日志、批量和两阶段索引

Post 创建、编辑和 like 事件进入分区日志，indexer 按 post_id 保序并合并同一文档的短期变化。第一阶段快速写基础 searchable 文档，使新帖可发现；第二阶段补充 engagement、语言和质量字段。消费者 checkpoint 只在批量提交成功后推进，重复批次按 source_version 幂等。热点 post 的 likes 先分片计数再层级聚合，避免单 key 写瓶颈。队列积压时优先创建/删除/权限，延后 like 排序刷新，保证搜索安全与基本新鲜度。每类事件独立记录 oldest age 和失败样本，使降级选择与恢复顺序都有证据。

能迁移的判断：不同变更价值不同；让关键状态先索引，统计信号后收敛。
05

权限必须在缓存和索引之外再守一层

visibility 可能包含 public、friends、group、custom ACL、block 和地域规则，无法把所有用户关系展开进每份索引文档。索引先做粗粒度候选过滤，例如 public 或可访问 group IDs；结果返回前由权限服务批量 final check。热门 query cache 只能保存候选 IDs，并按 viewer scope/version 分隔，不能把 A 用户可见结果直接给 B。block、删除和权限收紧写入快速 deny store，在索引尚未刷新时立即过滤。监控 permission-denied-after-search、deny lag 和 cache scope mismatch。权限服务故障时私密候选 fail-closed，公共结果也要标记降级并限制缓存范围，不能为了延迟绕开检查。

能迁移的判断：搜索负责召回，权威权限过滤负责最后可见性。
06

存储优化不能破坏删除与重建

索引可只存搜索和排序所需字段，正文详情留在 Post 服务；旧 segment 压缩，删除先写 tombstone，再由 merge 回收。postings 使用压缩 doc ID 差值，常见词可按 shard 分散，冷索引迁到低成本层。优化前先量化 term vocabulary、documents、postings bytes、replica 和 refresh 成本。任何归档仍要支持隐私删除和 reindex：保留 source version、tombstone 和索引 generation，不能为了省空间删除唯一恢复依据。容量指标与删除 SLA 同时进入设计。定期演练 snapshot 加增量日志重建，测量实际恢复时间和期间新增变更是否完整追上。

能迁移的判断：搜索存储可以压缩和分层，但事实、版本与删除证据不能丢。

Web Crawler

Web Crawler：从需求语义到故障修复

01

Frontier 是有状态调度器

URL frontier 不只是 FIFO queue。每条 canonical URL 有 priority、depth、host、next_allowed_at、attempt、lease_owner 和 lease_expiry；按 host 分 bucket 后，scheduler 只释放符合 robots、crawl-delay 和并发上限的任务。worker 领取短 lease，成功提交 fetch metadata 与发现链接后 ack；崩溃则 lease 到期重新领取。优先级结合来源质量、页面变化率和业务目标，防止某个无限站点吞没资源。frontier oldest age、ready/blocked 数量、lease expiry 和 per-host QPS 是基本运营指标。调度状态必须持久化并带版本，旧 worker 或重复 timer 不能把已完成 URL 重新推回高优先队列。恢复演练还要确认分片转移后同一 host 的公平性和全局速率仍然成立。

能迁移的判断：frontier 同时保存进度、优先级和礼貌约束，不是无状态消息列表。
02

规范化与去重分两层

URL 进入 frontier 前先规范 scheme/host、默认端口、fragment、路径和可确认的 tracking 参数，并遵循 canonical tag；但过度规范可能合并实际不同页面，所以保留原始 URL 和规则版本。权威 seen store 用 canonical URL/hash 做条件插入，Bloom Filter 先挡掉大量明显重复查询，但 false positive 可能漏抓，关键种子或低覆盖域名应绕过并定期重建。抓取后再用内容 fingerprint 识别不同 URL 的重复页面，保存 occurrence 关系而不是删除来源。规则升级通过影子比较误合并和漏合并比例，再逐步切换并重新评估受影响 URL。

能迁移的判断：URL 去重节省抓取，内容去重节省存储；两者都要可审计和可修正。
03

Politeness 必须集中执行

worker 自己 sleep 无法在数千实例间保证同一 host 的全局并发。scheduler 维护 host state：robots policy/version、next_allowed_at、active leases、错误退避和 user-agent；只有拿到 host permit 才发布任务。robots.txt 按 TTL 缓存，抓取失败采用保守策略并记录决定；429/503 和 Retry-After 更新 host backoff。DNS/IP 共享、子域与重定向策略要明确，防止通过别名绕过限制。监控 robots deny、host request rate、429 和 politeness violation，并支持来源方投诉后的即时停抓。permit 的发放和归还都要幂等，lease 过期后由单一 owner 重新计算配额，避免短暂停顿造成突发请求。停抓操作必须快速传播到已排队任务和缓存策略。

能迁移的判断：礼貌是全局调度不变量，不能依赖分散 worker 的善意。
04

抓取步骤按至少一次设计

worker 可能下载成功后崩溃、内容保存成功但 ack 丢失，任务因此会重复。fetch 结果以 crawl_id 或 canonical_url+content_version 幂等写入，原始 body 放对象存储，metadata 条件提交；parser 以 content hash 去重，发现链接条件插入 frontier。只有所有必要提交成功才 ack lease。永久 404 与 robots deny 是终态，超时和 5xx 有上限退避，反复失败进入 dead-letter 并保留原因。恢复不追求不可能的网络 exactly once，而保证重复不会产生错误业务效果。定期对比 frontier、metadata 和对象索引，可发现 orphan body、悬空任务与长期未确认的抓取。

能迁移的判断：允许重复任务，用稳定 ID 和条件提交保证一次有效结果。
05

10B/5 天先做吞吐与带宽算术

最低平均吞吐约为 10B 除以 432000 秒，即两万多页每秒；再乘峰值系数、失败重试和 robots 限制，实际 fetch capacity 更高。若平均响应 100KB，入口带宽已达数 GB/s，还要估压缩原文、metadata、发现链接和索引写入。并行度由网络延迟和每 worker concurrency 推导，但 per-host politeness 限制意味着必须覆盖足够多域名。容量表应分别列 frontier DB、seen set、对象存储、parser CPU 和出口带宽，找出真正瓶颈后再扩 shard。还要把压缩率、保留期和重抓比例转成日增存储及成本，验证目标是否经济可持续。

能迁移的判断：规模数字连接 page/s、bytes/s、并行度和存储，而不是只证明需要很多机器。
06

Recrawl、毒页面与质量闭环

抓完一次不等于完成搜索资产。scheduler 根据历史变化率、HTTP cache headers、来源价值和抓取成本安排 recrawl；内容长期不变就降低频率，breaking source 提高。超大响应、无限日历、重定向环、压缩炸弹和解析器 bug 都会耗尽资源，因此设置 response/body/time/depth 限额，按错误指纹隔离毒页面并可回放到新 parser。质量指标包括 freshness、coverage、duplicate ratio、parse success、dead-letter 和 per-domain fairness。修复后从保存原文重跑解析，不必重新访问来源。抽样人工评估和域级覆盖报表能发现系统虽然吞吐达标、却长期只抓到低价值重复页面的问题。

能迁移的判断：爬虫优化的是长期覆盖与新鲜度，必须能隔离坏输入并重处理。
