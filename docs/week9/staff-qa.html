先独立作答，再展开下一个追问。

01

scheduler crash 如何不漏跑？

参考回答
先把应执行的 logical run 持久化，再由多个 scheduler 重复扫描和条件 claim。进程在 claim 前崩溃只会被下一轮扫描接管；在 claim 后崩溃则 lease 到期后重试，因此把重复交给 handler 幂等。
常见失分
靠内存 timer 恢复。
继续追问
scheduler 在生成 run 和更新 next_run 间崩溃呢？
02

多 scheduler 如何避免双重 claim？

参考回答
所有竞争者对同一 run 执行 compare-and-set，只有一个能从 queued 转为 claimed。成功者获得递增 fence token；后续 worker 的完成写必须带该 token，避免过期 owner 覆盖新 owner。
常见失分
先读是否空闲再写。
继续追问
数据库时钟漂移怎么影响 lease？
03

lease 到期但旧 worker 还在跑？

参考回答
网络分区下旧 worker 可能继续执行，系统必须假设 overlap 会发生。新 worker 可以接管，但旧 worker 以旧 fence token 提交结果会被拒绝；外部副作用仍靠业务幂等或补偿保护。
常见失分
认为 heartbeat 永远可靠。
继续追问
不能幂等的邮件发送如何设计？
04

retry 覆盖旧状态吗？

参考回答
attempt 是不可变执行记录，重试创建新的 attempt，而 run 汇总其当前责任和最终结果。这样能定位是哪个错误、在哪个 worker、用了多少次重试；覆盖旧状态会让 on-call 无法诊断。
常见失分
只在 jobs 表加 retry_count。
继续追问
retry budget 如何定义？
05

DST 和 misfire 如何定义？

参考回答
cron 语义必须绑定时区，并提前定义夏令时重复/缺失小时怎样映射。错过的 run 不应由实现偶然决定：产品选择 skip、立即补跑或合并成一次，并记录 logical time。
常见失分
全部转 UTC 后忽略业务时区。
继续追问
月底账单任务应该选哪种策略？
06

优先级如何不饿死低优先级？

参考回答
纯优先级队列会让低优先级在高峰永远不被执行。可保留最小 worker 配额、随等待时间提升优先级，并限制每租户并发；代价是紧急任务无法独占全部资源。
常见失分
只说 FIFO。
继续追问
紧急 on-call job 怎么绕过公平限制？
07

大 tenant 如何隔离？

参考回答
任务入队时带 tenant 和资源类别，调度器按 tenant token 或权重选择，而不是按总队列深度。单个租户超额时被限流或延后，但不会占满 runner；这是一种可解释的服务等级合同。
常见失分
给每个 tenant 一个物理集群。
继续追问
付费 tier 如何影响权重？
08

job 幂等谁负责？

参考回答
平台能保证稳定 run_id、重复检测和状态审计，但不知道业务副作用的等价关系。handler 必须用 run_id 或业务键让扣费、发信、写库安全重试；平台和业务各自负责一层，不能互相推卸。
常见失分
说 scheduler 可以保证一次执行。
继续追问
第三方 API 不支持幂等键怎么办？
09

progress event 要持久化吗？

参考回答
可恢复的阶段、checkpoint 和最终结果应持久化，因为重试和人工排障需要它们。每秒的进度条更新可作为可丢失事件聚合，否则会把状态库变成日志系统；取舍取决于恢复需求。
常见失分
所有日志都写主事务库。
继续追问
长任务从 checkpoint 如何安全 resume？
10

DLQ 如何重放？

参考回答
DLQ 条目要带原始 run、attempt、错误、handler 版本和重放次数，便于判断是否可重试。重放前先修复 schema、依赖或业务数据，并按租户限速；否则会把故障重放成雪崩。
常见失分
定时把 DLQ 全部塞回主队列。
继续追问
payload 含敏感数据如何保留？
11

GitHub runner 为什么要隔离？

参考回答
用户构建可以 fork bomb、挖矿、扫描内网或读取遗留文件，因此 runner 必须是短生命周期隔离环境。按 job 注入最小权限 secret、限制 egress 和资源，并在结束后销毁；单纯 Docker 命名空间不足以说明隔离边界。
常见失分
回答“用容器”就结束。
继续追问
如何防止 cache 成为跨租户泄露通道？
12

untrusted build 如何限制资源？

参考回答
为每次执行设置 CPU、内存、磁盘、进程数、运行时长和网络出口策略，并在调度层限制并发。超限要以可审计原因终止，清理工作区和临时凭证；这样资源治理和安全治理形成闭环。
常见失分
只设一个 timeout。
继续追问
需要访问依赖仓库时怎样最小授权？
13

artifact 生命周期怎么设计？

参考回答
artifact 是执行结果，不应散落在 runner 磁盘；上传后以 run、版本和权限登记到对象存储。保留期、下载签名和删除流程要明确，runner 回收不等于 artifact 已可用。
常见失分
任务完成后保留本地文件。
继续追问
取消 job 时半上传 artifact 怎么处理？
14

control plane outage 时 worker 继续跑吗？

参考回答
短暂控制面故障时，已获得有效 lease 的 worker 可以完成当前 attempt，避免无谓中断。新 claim、配置变更和大规模 backfill 应暂停或降级，因为系统失去协调和观察能力；恢复后再从持久状态收敛。
常见失分
所有 worker 立刻停止或无限制继续。
继续追问
如何撤销一个已经危险的 job？
15

如何 backfill 一个月？

参考回答
backfill 应作为有审计的批次，明确时间范围、时区、并发和幂等键，而不是临时改 cron。分段生成 logical runs、按租户/资源限速，并支持暂停和重跑；否则会压垮正常任务与下游。
常见失分
循环调用一次立即执行 API。
继续追问
历史数据已变化时，backfill 用哪个版本？
16

哪些指标触发 on-call？

参考回答
应监控 due job 的开始延迟和完成延迟，因为它们直接代表调度合同；再看异常 lease reclaim、重试率和 DLQ 增长。按 tenant 维度拆分能发现单个大客户挤压系统，而不是只看到总体平均值。
常见失分
只看 queue depth。
继续追问
哪个指标最早发现 scheduler 没有生成新 run？
