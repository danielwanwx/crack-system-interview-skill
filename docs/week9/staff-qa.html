Senior / Staff Drill

News Aggregator

Q1

新闻不断插入时，cursor 如何减少重复和漏项？

为什么面试官会追：候选人仍使用 page/offset，或 cursor 只保存 article_id。

详细回答

cursor 应绑定本次 feed 的排序规则和边界，例如 generation、snapshot watermark、last_score、published_at 与 article_id，并经过签名。后续查询只返回严格位于复合键之后的项；在同一 generation 内，新插入文章不会把已读项向后推。排名模型更新时可让旧 cursor 继续旧 generation，用户刷新才进入新版；若必须混合，则客户端按 article_id 去重并接受有限漏项。服务端限制 cursor 生命周期，监控重复曝光、空页和 cursor 失效，不能假装无限滚动具有数据库快照级稳定性。

继续追问：热门新闻分数在翻页过程中剧烈变化时，是固定快照还是允许重排？

常见漏项：只把 offset 编码成 Base64，实际漂移问题没有改变。

现场修复

用三篇新文章插入前后演算 offset 与复合 cursor 的结果差异。

证据
  • News Aggregator · 1) How can we improve pagination consistency and efficiency?
  • API Design · Pagination
Q2

如何把 feed p95 控制在 200ms，而不是只说加缓存？

为什么面试官会追：候选人没有候选准备、预算拆分或 fallback。

详细回答

先把地区/类别候选 feed 预计算并放在区域缓存，用户请求读取有限候选后做 block、locale 和轻量个性化。给 gateway、cache fetch、filter/rank、serialization 各自预算，所有远程个性化调用都有短 timeout；超时回退基础热门 feed。活跃用户可异步物化首屏，长尾用户按需合并。缓存 miss 使用 singleflight 并限制回源，响应带 generation。监控各阶段分位数、cache hit、fallback rate 和 candidate count，才能知道下一步优化是索引、网络还是 ranking，而不是继续堆缓存。

继续追问：缓存全失效时如何避免同时重建数百万用户 feed？

常见漏项：缓存作为唯一答案，没有 miss 和恢复风暴策略。

现场修复

画 200ms waterfall，并给每个远程调用 timeout 与 fallback。

证据

  • Scaling Reads · Add External Caching Layers
Q3

如何证明文章在发布后 30 分钟内进入 feed？

为什么面试官会追：候选人只提高 polling 频率，没有端到端时间线。

详细回答

我会为每篇文章记录 publisher_published_at、discovered_at、fetched_at、parsed_at、indexed_at 和 feed_visible_at，并按来源计算端到端与分段 p95/p99。scheduler 根据来源更新频率、可靠性和价值动态安排，下游队列按 deadline 与来源优先级处理；RSS/webhook 减少发现等待。若媒体或个性化积压，先发布可用文本到基础 feed。任何阶段超预算都能从 oldest queue age 和时间戳定位，失败源进入退避而不阻塞其他来源。30 分钟因此是可报警、可追责的 SLO，不是平均抓取周期。

继续追问：publisher 的发布时间不可信或时区错误时，SLO 起点如何定义？

常见漏项：只看最终 index lag，无法区分发现慢还是处理慢。

现场修复

为一篇文章填完整阶段时间戳并设计三条阶段报警。

证据
  • News Aggregator · 3) How do we ensure articles appear in feeds within 30 minutes of publication?
  • Managing Long Running Tasks · Managing Queue Backpressure
Q4

breaking news 时读写同时暴涨，先保护什么？

为什么面试官会追：候选人只说自动扩容，没有优先级和降级。

详细回答

先保护 canonical Article 提交和基础文本 feed，因为它们决定内容可见；媒体转码、深度个性化和低价值来源抓取可以延后。入口队列吸收抓取与索引峰值，按来源与 deadline 调度；热门 feed 和静态媒体由 CDN/区域缓存承载，miss 用 request coalescing 保护数据库。个性化超时回退类别热门，媒体缺失显示占位。系统监控 ingest oldest age、feed cache hit、origin QPS、media backlog 和 fallback rate，并对同一热点资源做单独容量隔离，避免一个事件拖垮所有类别。

继续追问：错误新闻被缓存到边缘后，如何快速撤回？

常见漏项：所有任务同等处理，峰值时关键路径被后台工作挤占。

现场修复

把流水线任务分 P0/P1/P2，并为每级写丢弃、延后和恢复规则。

证据

  • Caching · Hot Keys
Q5

项目页没有 Senior 等级小节，怎样仍给出 Senior 标准？

为什么面试官会追：候选人把缺少项目锚点当成无需覆盖等级预期。

详细回答

我会明确记录 source gap，不伪造项目 anchor；等级标尺引用 Hello Interview 通用 Senior 说明，再把 breadth、depth 和 proactiveness 映射到本题证据。breadth 是主动覆盖 ingestion、feed、media 与运营；depth 是对 cursor、新鲜度和热点至少一项讲清状态与故障；proactiveness 是在被问前指出 publisher 不可靠、索引可重建、个性化可降级。验收以本项目 deliverable、Q&A 和 metrics 为准，并保留来源边界，让学习者知道哪些是原项目内容、哪些是通用等级要求。

常见漏项：猜测一个不存在的 #senior fragment，破坏来源可审计性。

FB Post Search

Q1

热门关键词请求量很大，缓存搜索结果会不会泄漏私密帖子？

为什么面试官会追：候选人按 query string 全局缓存完整结果。

详细回答

全局缓存只能保存不含用户权限结论的 public candidate IDs 或公共片段，并带 index generation；个性化结果必须按 viewer scope/ACL version 分隔，成本通常过高。返回前权限服务批量检查 visibility、friend/group membership、block 和快速 deny list，即使索引与缓存陈旧也不泄漏。删除和权限收紧先更新权威状态与 deny path，再异步失效索引缓存。监控 search candidate 与 final allowed 的差异、deny lag 和 cache scope。性能不足时减少候选或缓存公共 postings，而不是跳过权限。

继续追问：权限服务故障时，public 与 private 结果分别怎样降级？

常见漏项：把搜索缓存视为最终响应，没有用户上下文和撤销路径。

现场修复

画 query cache、index candidate、permission batch check、response 四步。

证据
  • FB Post Search · 1) How can we handle the large volume of requests from users?
  • Scaling Reads · Add External Caching Layers
Q2

multi-keyword 与精确 phrase 查询的数据结构有什么不同？

为什么面试官会追：候选人只说对多个 postings 做 intersection。

详细回答

多关键词 AND 可以按 postings 长度从小到大做交集，OR 做 union；但仅有 doc IDs 不能证明词语相邻且顺序一致。精确 phrase 需要在 postings 保存 token positions，然后在候选文档内比较位置偏移；常见短语也可建立 bigram/shingle token，以更大索引和写放大换更快查询。analyzer 必须在写和读一致处理大小写、语言与停用词。短语过长或罕见时先用最稀有词缩小候选，再做 position 验证。要说明召回、空间和 latency 的取舍。

继续追问：中文分词和英文分词不同，索引 mapping 如何演进？

常见漏项：忽略 position，导致 phrase 退化为任意位置关键词共现。

现场修复

用“new york city”手算 unigram postings、positions 和 bigram 查询。

证据

  • Database Indexing · Inverted Indexes
Q3

点赞每秒海量变化，为什么不每次都更新 Elasticsearch 文档？

为什么面试官会追：候选人把每个 like 同步写进搜索索引。

详细回答

每次更新会产生 refresh、segment merge 和文档重写放大，还把搜索故障扩散到用户写路径。权威 like 事件先持久化并按 post_id 聚合，流处理器维护 count，按时间窗口或变化阈值批量刷新索引；热门榜单可另用更实时的小投影。搜索排序接受明确秒级 lag，用户自己的 like 状态从权威服务读。创建、删除和权限变更优先于 engagement 刷新。重复事件按 event_id 去重，定期从权威计数重算。监控 aggregation lag、index update QPS 和 ranking drift。

继续追问：一个病毒式帖子形成单 key 热点，聚合如何拆分？

常见漏项：只看到最终计数，没考虑索引写放大和故障隔离。

现场修复

拆出 like ingest、sharded counter、batch projection、search sort 四层。

  • Scaling Writes · Batching and Hierarchical Aggregation
Q4

怎样减少搜索存储而不失去可恢复性？

为什么面试官会追：候选人建议删除原文或只保留索引。

详细回答

索引只保存搜索、过滤和排序需要的字段，正文与权限事实留在权威 Post 服务；postings 使用压缩 doc IDs、segment merge 和冷 tier，副本数按可用性设置。索引文档保留 source_version 与 post_id，使详情和重建可关联。删除写 tombstone 并传播到 deny path，后台 merge 才物理回收；不能提前丢掉删除证据。权威日志/CDC 保留覆盖最大重建窗口，旧快照加增量可恢复当前 generation。容量优化要同时报告 index bytes/doc、replica、merge IO、rebuild time 和 deletion SLA。

继续追问：CDC 保留期短于全量 reindex 时间时，如何避免重建期间漏更新？

常见漏项：把索引当备份，或为了压缩删除 source version。

现场修复

写出 snapshot watermark + replay tail 的重建协议。

Q5

帖子已改为仅自己可见，但索引更新延迟一分钟，如何防止泄漏？

为什么面试官会追：候选人接受所有索引最终一致，没有安全例外。

详细回答

权限收紧先在权威 Post/ACL 服务提交，并写入低延迟 deny/change-version store；搜索索引和 cache 异步更新。每次返回候选前，权限服务按 viewer 批量检查当前 visibility/version，或至少对最近权限变更命中 deny store，因此旧索引只能增加无效候选，不能直接泄漏。权限服务不可用时 private 候选 fail-closed，public 可按风险继续。完成 reindex 后清理 deny 项。监控 permission mismatch、deny store age、final-check latency 和泄漏测试，安全撤销 SLO 独立于普通 index freshness。

继续追问：数十亿 block 边如何让 final check 保持低延迟？

常见漏项：认为缩短 refresh interval 就足够且永不失败。

现场修复

分别写普通内容更新和权限收紧的传播路径与 SLO。

  • Caching · Cache Consistency

Web Crawler

Q1

worker 下载页面后崩溃，如何既不丢进度又不重复污染数据？

为什么面试官会追：候选人说消息 ack，却没有 ack 前后的故障窗口。

详细回答

frontier 给任务短 lease，worker 下载后用稳定 crawl_id 或 canonical_url+content hash 幂等写对象与 metadata，parser 结果和 discovered URLs 也使用条件插入；全部提交后才 ack。若崩溃，lease 到期任务重做，但相同 ID 返回已有结果，不重复创建文档或 frontier 项。若 body 已保存而 metadata 未提交，后台 orphan scan 根据 crawl_id 修复或回收。checkpoint、lease expiry、duplicate write 和 orphan age 都要监控。可靠性来自可重试状态机，而不是希望每个网络步骤恰好执行一次。

继续追问：对象存储写成功却长时间不可见时，metadata commit 如何处理？

常见漏项：把 queue durability 等同于跨存储原子提交。

现场修复

列出 fetch、body put、metadata、discover、ack 五个崩溃点和重试结果。

Q2

数千 worker 如何共同遵守同一网站的 robots.txt 和 crawl-delay？

为什么面试官会追：候选人在每个 worker 内 sleep。

详细回答

按 host/domain 建集中或分片的 host scheduler，保存 robots policy/version、next_allowed_at、active lease count 和 backoff。worker 不直接从全局 URL 队列取任务，只消费 scheduler 已发 permit 的 host task；permit 通过条件更新保证跨实例并发上限。robots 按 TTL 缓存，刷新失败时按风险保守使用旧策略或暂停；429/Retry-After 立即延后 next_allowed_at。重定向到新 host 要重新获得许可。这样即使 worker 扩容，来源看到的总请求率仍受一个权威状态控制。

继续追问：同一组织拥有多个子域，限制按 host 还是 registrable domain？

常见漏项：局部限速无法约束全局并发，容易无意 DDoS。

现场修复

画 host bucket、permit、worker、feedback 四步，模拟两个 worker 竞争。

Q3

10B 页面五天抓完，需要多少吞吐，哪个资源最可能先触顶？

为什么面试官会追：候选人直接说横向扩展，没有数量级。

详细回答

最低平均吞吐约 10,000,000,000/432,000，约 23K pages/s；考虑峰值、失败与重试可按 2 倍预留。若平均 100KB，原始入口约 2.3GB/s，未含 TLS、重定向与发现链接；每天原文约 PB 级前应重新审视平均尺寸和保留策略。假设单 fetch 平均 500ms、每 worker 200 并发，可估实例数，但 per-host politeness 可能比机器更限制吞吐，因此需要足够域名并行。还要分别算 frontier/seen 写 QPS、parser CPU、对象存储和出口网络，依据监控确定瓶颈。

继续追问：多数页面集中在少数大型域名时，五天目标与 politeness 冲突怎么办？

常见漏项：只计算 worker 数，不计算带宽、存储和来源速率上限。

现场修复

完成 page/s、bytes/s、concurrency、frontier writes 四行容量表。

Q4

Bloom Filter 误判会漏抓页面，为什么仍值得用？

为什么面试官会追：候选人把 Bloom Filter 当精确 seen set。

详细回答

Bloom Filter 只能回答“可能存在”或“肯定不存在”。它适合在昂贵权威 seen-store 查询前快速放行肯定新 URL，减少远程读；命中仍可按来源价值决定查权威集合。若直接丢弃所有命中，false positive 会降低 coverage，所以高优先级 seed、低覆盖域名和审计样本应绕过，或使用可接受的小误判率并定期滚动重建。Filter 饱和、规则版本变化时创建新 generation，双查过渡。正确性仍由 canonical URL 条件插入保证，Bloom 只是成本优化。

继续追问：如何从容量和目标 false-positive rate 估位图大小与 hash 数？

常见漏项：认为 Bloom Filter 没有 false negative 就等于不会漏抓。

现场修复

写出命中/未命中各自路径，并标出权威 seen-store 的位置。

Q5

某个站点不断生成无限日历 URL，如何防止吞没 frontier？

为什么面试官会追：候选人只依赖 URL 去重，没有 crawl trap 防护。

详细回答

对每域名设置 frontier 配额、最大深度、路径模式和单位时间新 URL 增长阈值；canonicalization 去掉已知 session/tracking 参数，检测连续日期、重复路径段和低内容差异。scheduler 按域公平调度，单域爆发只占自己的 bucket，不挤压全局。疑似 trap 降低优先级并抽样抓取，内容 fingerprint 验证后可自动封禁模式，保留人工覆盖。任务仍记录发现来源和拒绝原因，便于修正规则。监控 unique growth、duplicate content、depth 与 per-domain frontier share。

继续追问：如何避免 trap 规则误伤真正的大型归档站点？

常见漏项：无限增加队列和存储，直到系统整体背压。

现场修复

为 domain bucket 写 quota、growth alarm、pattern quarantine 和人工解除流程。
