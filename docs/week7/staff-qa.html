第 7 周 Staff 追问与详细答案

先遮住答案口述；答案必须覆盖判断、机制、失败语义和可验证修复。

Senior / Staff Drill

Robinhood

Q1

WebSocket 断线后，如何证明用户没有错过关键行情更新？

为什么面试官会追：候选人只回答自动重连或心跳，没有说明数据缺口。

详细回答

心跳只能证明连接在某个时刻可达，不能证明报价连续。我会给每个顺序域分配单调 sequence，客户端记录最后已应用值；重连时携带 cursor，服务端从短期 replay log 返回缺口。若 cursor 已超出保留窗口，则先返回带基准 sequence 的完整 snapshot，再继续增量。中间报价可按产品语义合并，但客户端必须知道当前 quote age，并在过旧时显示 stale，而不是继续渲染成实时价格。服务端监控 sequence gap、replay miss 和每连接 lag，才能量化恢复是否有效。

继续追问：多个行情源的 sequence 互不兼容时，顺序域如何定义？

常见漏项：把连接恢复等同于数据恢复，忽略 snapshot 与陈旧标记。

现场修复

画 disconnect → reconnect(cursor) → replay 或 snapshot → resume 的时序，并标出 sequence ownership。

Q2

提交订单超时后，为什么不能直接重试到 broker？

为什么面试官会追：候选人把网络重试当成安全操作，没有业务幂等键。

详细回答

超时意味着结果未知，而不是失败；第一次请求可能已经被 broker 接受。客户端必须复用同一 client_order_id，API 先查本地唯一记录，存在就返回当前状态，不再创建第二笔意图。对外发送由 outbox 驱动，adapter 保存 client_order_id 与 broker_order_id 的映射；重发时查询或使用对方支持的幂等键。若仍无法确认，订单保持 pending confirmation 并由 reconciliation 查询外部系统。只有明确 rejection 才允许用户以新意图提交，否则盲目重试可能造成重复买入和真实资金损失。

继续追问：如果 broker 不支持幂等 key，也没有按 client ID 查询接口怎么办？

常见漏项：把 transport retry 和业务命令重试混为一谈。

现场修复

列出响应成功、明确失败、超时未知三种结果及各自允许的下一步。

Q3

成交回报与取消确认乱序到达，订单状态如何避免倒退？

为什么面试官会追：候选人用最后写入覆盖 status，或只依赖消息时间戳。

详细回答

我会把订单建模为受约束状态机，而不是 last-write-wins 字段。每条外部回报带稳定 execution_id 或 event_id，处理器先去重，再在事务中读取 version、filled_qty 和当前状态；只有合法边才更新。取消请求进入 cancel_requested，不能立即覆盖已成交事实；迟到 fill 仍按唯一 execution 累加，cancelled 只说明剩余数量不再成交。若外部序列可用就保存其 sequence，否则以事件类型和数量不变量拒绝非法倒退，并把冲突送入 reconciliation 队列人工或自动核对。

继续追问：如何从事件日志重建余额和持仓并验证结果？

常见漏项：认为“按时间排序”足以解决跨系统时钟和重复问题。

现场修复

写出订单状态图和 filled_qty ≤ requested_qty 等不变量，再模拟三种乱序。

Q4

为什么行情可以最终一致，而订单状态不能统一采用同样策略？

为什么面试官会追：候选人给整个系统贴上 AP 或 CP 标签。

详细回答

一致性是按业务不变量选择的。行情是持续变化的观察值，短暂陈旧通常只影响体验，因此可读本地 replica、合并更新并在断线后补齐；但下单、取消和成交决定资金与持仓，重复或倒退会产生不可接受的业务错误，需要单账户或单订单范围内的条件写、唯一约束和审计日志。即使订单读模型异步更新，权威状态转换仍必须串行化。回答时我会分别给出 quote age、order pending age 和 reconciliation mismatch 的 SLO，而不是笼统声称所有数据都“强一致”。

继续追问：portfolio 页面允许多旧，用户刚成交后的 read-your-writes 如何实现？

常见漏项：把数据库产品的 consistency level 当成业务正确性的完整答案。

现场修复

为 quote、order、execution、portfolio 四类数据写允许陈旧窗口与 source of truth。

Q5

热门股票暴涨时，怎样防止一个 symbol 拖垮整个行情系统？

为什么面试官会追：候选人只说横向扩容，没有识别 hot partition 与慢消费者。

详细回答

先用 per-symbol ingress rate、fanout subscriber count、partition lag 和 egress queue 找到瓶颈。分区层允许把热点 symbol 提升为独立分区或复制只读 fanout state；区域和进程内缓存减少重复编码与跨区传输。对行情事件可合并同一 symbol 的中间更新并限制每客户端刷新频率，但必须保留最新价格和 sequence；对订单相关事件绝不能用同样丢弃策略。若消费者持续落后，协议明确发送 snapshot-required 或 stale 标记，并优先保护订单入口和权威日志，而不是让无限队列耗尽内存。

继续追问：热点从一个 symbol 快速转移到另一个时，动态拆分如何避免重排序？

常见漏项：只增加 Kafka partition，却没有迁移顺序域或消费者状态。

现场修复

为单个热点 symbol 画 ingress、log、fanout、client 四级队列，并定义每级背压动作。

Senior / Staff Drill

Uber

Q1

司机位置多旧仍可用于派单，如何决定？

为什么面试官会追：候选人只说实时位置，没有 freshness budget 或用户后果。

详细回答

我会按 trip 阶段和速度定义 freshness SLO，而不是一个全局秒数。空闲静止司机可以较低频更新，正在接驾或高速移动时要求更短窗口。geo 查询返回 location_timestamp，matching 先过滤超过阈值的候选，并对少量最优司机发 fresh ping；无法及时确认就不建立 reservation。系统监控位置年龄分布、fresh ping 成功率、ETA 误差和因陈旧被过滤比例。过载时可以减少远端候选、扩大等待时间，却不能把未知位置伪装成准确位置。

继续追问：客户端时间不可信时，位置事件的时间戳如何处理？

常见漏项：只提高更新频率，忽略电量、带宽和入口过载。

现场修复

按 idle、searching、reserved、on_trip 四阶段写频率、freshness 和降级表。

Q2

两个 matching worker 同时选中同一司机，如何保证只发出一个有效邀请？

为什么面试官会追：候选人说“Redis lock”但没有 owner、TTL 或失效竞争。

详细回答

候选生成可以并发，但承诺点必须是权威 driver 状态上的条件写：仅当 status=available 且 version=v 时，写入 reserved、ride_id、reservation_id、expires_at 和 version=v+1。只有成功者发送邀请。若使用外部租约，也要把 fencing token 写进后续 accept 请求，防止旧 worker 在 TTL 过期后复活。拒绝、超时和取消只能由匹配 reservation_id/version 的操作释放，避免旧 timer 清除新租约。这样即使通知重复，司机端和服务端也只承认当前 token。

继续追问：条件写成功但通知发送失败，司机何时重新可用？

常见漏项：把互斥锁当成永不失效的魔法，忽略旧持有者与定时器。

现场修复

画两个 worker 竞争、winner 崩溃、TTL 到期、旧 worker 恢复的时序。

证据

Q3

高峰期把所有 ride request 放进队列就不会丢了吗？

为什么面试官会追：候选人把 durable queue 等同于完整派单语义。

详细回答

队列只能持久化任务，仍需定义幂等、deadline、顺序和用户状态。入口以 client_request_id 去重并创建 searching 记录，消息只携带 ride_id；消费者每次执行前读取当前状态，已取消或已匹配就安全跳过。任务有最大等待时间与区域优先级，失败使用有上限退避，超过 deadline 转为 no_driver_available 并通知用户。队列积压时显示预计等待、限制新请求或调整范围，不能无限隐藏延迟。必须监控 oldest message age、redelivery、match success 和取消率。

继续追问：同一区域如何在高价值订单与等待最久用户之间保持公平？

常见漏项：只讨论消息不丢，没有讨论业务任务是否还有效。

现场修复

为消息写出 dedupe key、deadline、retry limit、terminal state 和用户通知。

Q4

司机没有及时响应，为什么不能简单等 TTL 后再选下一位？

为什么面试官会追：候选人没有考虑延迟任务、重复 timer 和迟到接受。

详细回答

TTL 只是 reservation 的有效期，还需要可靠触发重新匹配。创建 reservation 时同时安排带 reservation_id/version 的 durable timer；到期 worker 条件检查仍为当前 reservation 且未接受，才释放司机并推进下一候选。司机迟到 accept 必须携带 token，若版本已变化就返回 expired，而不能重新占用。timer 至少一次执行，所以处理必须幂等；服务重启后可从到期索引或延迟队列恢复。产品还要限制每次等待和总尝试时间，让乘客看到 searching、retrying 或失败的明确状态。

继续追问：延迟队列不可用时，如何从数据库恢复所有到期 reservation？

常见漏项：依赖进程内 timer，崩溃后任务永久消失。

现场修复

模拟 timer 重复、worker 崩溃和 accept 迟到三种情况，验证 version 条件。

证据

Q5

按 geo cell 分片后，司机跨边界时如何避免消失或被重复匹配？

为什么面试官会追：候选人只说 geohash 分片，没有所有权迁移协议。

详细回答

位置索引可以在边界短暂双写以保证候选可见，但 driver reservation 必须有单一权威 owner。司机跨 cell 时，新 owner 以 driver_id 和递增 epoch 接管，旧 owner 看到更高 epoch 后停止写；查询相邻 cell 后按 driver_id 去重，并把 reservation 请求路由到当前 owner。迁移期间若无法确认 ownership，宁可暂不把司机作为候选。事件携带 epoch，迟到位置更新和旧 shard 的释放操作都会被拒绝。系统监控 ownership conflict、跨 cell 查询比例和迁移时延，热点区域再动态细分。

继续追问：整个 region 故障时，备用 region 如何安全接管尚未过期的 reservation？

常见漏项：把候选数据复制与写所有权混为一谈。

现场修复

画 old cell、新 cell、owner directory 和 reservation store 的 epoch 迁移时序。
