面试压力测试

先独立作答，再展开下一个追问。

01

库存的 source of truth 是什么？

参考回答
库存真相是带版本的 seat 状态和有效 hold，而不是 Redis、CDN 或前端 seat map。展示可以陈旧，但 hold/sale 必须在同一权威分区做条件更新；这把强一致集中在极小的写路径。
常见失分
把缓存命中当成座位可购买证据。
继续追问
seat map 显示可用但 hold 被拒绝时 UI 怎么处理？
02

hold 的原子条件怎么写？

参考回答
一次条件更新检查 seat 未售出且没有未过期 hold，然后写入新的 hold_id、owner、expiry 和递增 version。成功响应就是线性化点；失败返回当前状态，让客户端刷新，而不是靠先读后写。
常见失分
先 select 再 update。
继续追问
多个 seat 如何避免用户只拿到半套座位？
03

TTL 到期与付款成功同时发生怎么办？

参考回答
付款成功不能单独证明座位仍属于用户，checkout 必须带 hold_id/version 条件地把 held 转 sold。若 sweeper 已释放或版本已变，订单进入补偿路径并退款；宁可少卖一次，也不能双卖。
常见失分
相信 Redis TTL 的删除顺序。
继续追问
provider 成功但 sale 条件失败如何对账？
04

为什么不能只用 Redis lock？

参考回答
Redis lock 可以减少冲突，但不能替代 durable seat、hold 和 order 状态。锁过期、主从切换或进程崩溃时，系统仍需知道谁拥有哪个座位以及如何恢复；这必须落在权威数据模型。
常见失分
把 Redlock 当作事务。
继续追问
Redis 只能放在哪个辅助位置？
05

热门演出单分区过热如何处理？

参考回答
座位本身的串行性无法消除，因此目标是限制无效竞争而非假装均匀分片。为 event 设置 admission queue、用户配额和热点读缓存，并让独立的 payment、搜索和浏览服务不被抢购流量拖垮。
常见失分
建议把单个 seat 随机分片。
继续追问
如何防止排队 token 被转卖？
06

seat map 可以多陈旧？

参考回答
seat map 的容忍度由产品决定，通常数秒陈旧比全局强读便宜得多。关键是 UI 明确“正在确认”，并在 checkout 回到权威库存；陈旧展示会增加失败率，但不会破坏不超卖不变量。
常见失分
要求所有地图读线性一致。
继续追问
怎样用指标决定是否缩短缓存 TTL？
07

waiting room 能保证什么？

参考回答
等候室保护下游并给用户相对公平的进入顺序，但它不创建 hold，也不保证库存。只有用户真正进入购买流程并完成条件 hold 才获得资源；把两者混淆会造成错误承诺。
常见失分
说排到队首就必定买到。
继续追问
重连用户如何保持公平位置？
08

payment 成功但 order commit 超时如何恢复？

参考回答
超时意味着客户端不知道结果，不能再次扣款或直接释放座位。后台根据 provider reference 查询/处理 webhook，再条件完成 order；若 hold 已失效则退款并留下 audit trail。
常见失分
收到超时立即重新 charge。
继续追问
webhook 重复和乱序如何处理？
09

double click checkout 怎么幂等？

参考回答
checkout request 绑定稳定 idempotency key 和 hold_id，服务端把第一次处理结果持久化。后续相同请求返回原订单状态，不再创建新 charge；键的保留期要覆盖客户端与 provider 的重试窗口。
常见失分
只在前端禁用按钮。
继续追问
不同 key 却同一 hold 怎么处理？
10

网络分区时继续售票吗？

参考回答
对于唯一座位，错误出售的代价高于暂时不可用，因此不能在失去权威分区时离线售卖。可以继续提供浏览、队列和缓存信息，但 checkout 应明确失败或等待恢复。
常见失分
为了可用性让两边都卖再合并。
继续追问
哪些非关键功能可以继续服务？
11

如何防 bot 而不把风控当答案？

参考回答
设备信誉、速率限制、挑战和支付风险检查可以降低抢占，但它们不改变 seat hold 的原子不变量。面试中要先给出即使 bot 存在也不超卖的状态模型，再说明风控如何保护公平与容量。
常见失分
回答“加验证码”后结束。
继续追问
误杀正常用户怎样恢复？
12

退款如何影响库存？

参考回答
退款不必然释放座位，例如演出临近或转售规则可能不同。系统先记录 order/ledger 的退款事实，再根据产品政策触发受控的 inventory transition；不可把 payment refund 当作自动把 seat 改回 available。
常见失分
退款后直接删订单。
继续追问
取消与转售有什么不同状态？
13

拍卖为何允许后出价覆盖？

参考回答
拍卖资源不是一次 hold 后永久排他，而是在截止前接受满足最小增幅的更高 bid。系统保存每次 bid 和顺序/时间证据，close 时以确定性规则选赢家；Ticketmaster 则要在 hold 成功后阻止任何覆盖。
常见失分
把拍卖当作座位锁的同义词。
继续追问
相同时间戳的两次 bid 如何 tie-break？
14

hard close 如何定义公平？

参考回答
公平必须由权威服务端时间和单一排序规则定义，不能信客户端时钟。接近截止的请求可能因网络延迟失败，但规则一致且可审计；若做 anti-sniping 延时，延时规则也必须在事前固定。
常见失分
按客户端 timestamp 排序。
继续追问
跨 region 到达延迟怎么处理？
15

哪些写要 serializable？

参考回答
唯一资源分配和 close winner 计算需要隔离或等价的条件写，避免并发异常。seat 浏览、分析报表、通知和搜索投影可以异步；范围越小，强一致成本越可控。
常见失分
要么全系统强一致，要么全都最终一致。
继续追问
跨多个 seat 的事务范围如何控制？
16

哪些告警最早暴露故障？

参考回答
先监控不变量破坏和用户路径：同 seat 多个有效 owner、hold 到期清理延迟、checkout P95/失败率。再看 provider 回调滞后和对账差异；队列深度只是诊断信号，不能替代业务告警。
常见失分
只看服务器 CPU。
继续追问
哪项指标应触发停售开关？
每个节点答不出具体状态、错误语义或恢复路径，回到讲义对应段落修复。
