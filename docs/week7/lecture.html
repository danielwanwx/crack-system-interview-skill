Robinhood

Robinhood：从需求语义到故障修复

01

先把行情与订单分成两种语义

Robinhood 不是一条“实时数据流”就能解释的系统。行情是外部交易所产生的高吞吐读模型，允许毫秒到秒级陈旧，重点是 fanout、顺序、断线补洞和慢消费者；订单是用户资金相关的命令，必须幂等、可审计，并由经纪商或交易场所的回报推动状态机。白板应从一开始画成两条路径，只在 symbol、account 和 portfolio projection 处交汇。这样才能分别选择缓存、流、事务和降级策略，也能避免为了行情吞吐牺牲订单正确性，或为了订单强一致把报价链路做得昂贵而脆弱。

能迁移的判断：先按业务语义拆路径，再为每条路径选择一致性和延迟目标。
02

行情推送的可靠性来自序列与补读

WebSocket 只解决服务器如何更快把报价送到客户端，不保证消息永不丢失。行情接入层应按 symbol 或 shard 维护单调 sequence，归一化不同交易所格式后写入可重放日志；fanout 节点把最新值推给订阅者，并对慢连接合并中间更新。客户端重连时提交最后 sequence，服务端从短期 replay buffer 补洞，超出保留窗口则返回 snapshot 再继续增量。热门 ticker 可以复制 fanout 状态或在边缘聚合，但必须监控 lag、drop、replay miss 和连接队列长度，防止“连接还活着、数据已经旧了”。

能迁移的判断：实时连接负责低延迟，sequence、snapshot 与 replay 才负责恢复。
03

订单提交是一条幂等命令协议

客户端提交订单时生成稳定的 client_order_id；API 在账户范围建立唯一约束，把命令和 outbox 事件原子写入。若响应丢失，重试应返回同一订单，而不是再次发送到 broker。后台发送器可以至少一次投递，但外部 adapter 与回调处理器都要按 broker_order_id、execution_id 去重。订单状态只能沿允许的边单调推进，重复或迟到回报保留审计记录却不能倒退状态。这样系统无需承诺无法证明的“网络 exactly once”，仍能让用户、客服和 reconciliation job 对同一笔意图得到一致结论。

能迁移的判断：幂等键、唯一约束、outbox 和去重账本共同形成可证明的一次业务效果。
04

Partial fill 与取消是状态机竞争

市场订单可能分多次成交，取消请求也可能与最后一笔成交同时在途。不能用一个布尔 status 覆盖这类竞争，而要保存 requested_qty、filled_qty、remaining_qty、version 以及每个 execution 的唯一标识。取消是“请求取消”，不是立即成功；只有交易场所确认后才进入 cancelled，期间到达的合法 fill 仍要累加。数据库用条件更新或事务验证旧 version，失败者重新读取并应用事件。面试回答还应说明余额和持仓是订单事件的派生投影，出现差异时以外部成交记录和不可变 ledger 对账，而不是手工覆盖最终数字。

能迁移的判断：资金系统的正确性靠显式状态转换和对账，不靠乐观描述。
05

热点、慢消费者与背压

市场波动时少数 symbol 会形成极端热点，生产速率也会突然超过客户端和 fanout 节点的消费能力。对行情可合并同一 symbol 的中间值，只保留最新报价；对订单事件则不能丢，只能扩容消费者、限制入口或把非关键投影延后。分区键不能只看平均负载，应监控 top symbol 的单分区带宽和连接数，并允许热点 symbol 独立拆分。客户端协议还要给出最大队列、降采样和“数据已过期”标记，宁可明确告诉用户行情延迟，也不能静默展示看似实时的旧价格。

能迁移的判断：背压策略必须按事件价值区分可合并数据与不可丢业务事件。
06

Senior 主动讲外部故障与运营证据

Robinhood 的关键依赖在系统边界之外：行情供应商可能断流，经纪商可能接受订单却让响应超时，回调可能重复或乱序。Senior 答案要为每种故障定义用户状态，例如 quote 标记 stale、订单显示 pending confirmation、取消显示 requested；同时启动限流、重试和 reconciliation，而不是猜测成功或失败。运营指标至少包含 market-data age、sequence gap、fanout lag、order submit latency、pending age、duplicate execution、reconciliation mismatch 和 broker error rate。报警应绑定业务风险，修复后还要能重放日志重建投影并证明没有漏单。每次事故复盘还应记录不变量是否被破坏、哪些投影需要重算，以及用户补救是否已经完成。

能迁移的判断：外部系统不可控，内部状态、指标与对账必须让不确定性可见且可恢复。

Uber

Uber：从需求语义到故障修复

01

位置事实、可用性与行程状态必须分离

司机最近一次位置只是带时间戳的观察值，不代表此刻仍空闲；available、reserved、on_trip 才是派单权威状态。位置链路可以高吞吐异步写入内存 geo index，定期持久化轨迹；司机可用性与 ride 状态则需要条件写和可审计转换。附近搜索先从位置索引取候选，再过滤 freshness、车型、司机状态和产品边界。若把所有字段写进同一高频行，不仅造成写热点，还容易让陈旧位置覆盖新的 reservation。清楚拆分三类状态后，系统才能同时获得低延迟搜索和双重派单防护。

能迁移的判断：地理坐标是候选信息，司机 reservation 才是承诺边界。
02

高频位置上报需要自适应与分层

固定每秒上报会在司机静止或低密度区域浪费带宽，却可能在高速行驶和接驾阶段不够新。客户端可依据速度、方向变化、前后台状态和 trip 阶段调整频率；接入层验证时间戳、设备与跳变，批量写入区域内存索引，并把较低频轨迹异步落盘。服务端用 location age 过滤过期司机，必要时向少量候选请求一次 fresh ping。过载时降低非活跃司机频率、合并中间点并保护派单状态写；不能静默接受任意旧位置，否则低延迟查询只会更快返回错误候选。

能迁移的判断：位置质量由 freshness 与产品阶段驱动，不由固定更新频率驱动。
03

附近搜索是候选生成加精确过滤

geo index 通常把二维空间映射为 cell、geohash 或树结构，先找当前及相邻区域的司机，随后用真实距离和 ETA 服务做精排。单纯按经纬度两个 B-tree 范围查询会产生矩形候选且难以稳定扩展。索引选择还要处理边界：司机跨 cell 时短暂双写或带版本迁移，查询邻居 cell 并按 driver_id 去重。机场等热点可以细分 cell 或复制只读位置，但权威 reservation 仍归一个 owner。搜索半径应逐级扩大，并把最大等待时间、ETA 误差和位置年龄作为可观测质量指标。

能迁移的判断：空间索引减少候选，精确距离、ETA 与状态过滤保证结果可用。
04

派单靠短租约与 fencing 防双重承诺

matching 服务为 ride 选择候选后，必须原子地把 driver 从 available 改为 reserved，并写 reservation_id、ride_id、version 和 expires_at。只有条件更新成功的请求才能通知司机；失败者重新选人。TTL 防止 worker 崩溃后永久占用，但旧 worker 可能在租约过期后恢复，所以后续 accept 必须携带 reservation version，权威服务拒绝过期 token。司机接受后再把 ride 与 driver 状态一起推进，拒绝或超时由持久化计时器释放。锁不是一个组件名，而是一份有 owner、期限和 fencing 的协议。还要审计过期租约、拒绝旧 token，并按区域观察 reservation 冲突和泄漏率。

能迁移的判断：TTL 解决泄漏，fencing/version 解决过期持有者复活。
05

高峰派单需要有期限的持久化工作流

演唱会散场时请求到达速度会超过可用司机和 matching worker。入口先创建幂等 ride request 并返回 searching 状态，再写入可持久化队列；任务带区域、创建时间、服务等级和 deadline。消费者按区域供需与等待时间调度，失败使用有上限的退避，超过 deadline 明确通知用户而不是无限排队。司机超时后工作流记录尝试历史并选择下一候选，防止同一司机被频繁骚扰。队列长度、oldest age、match success、attempt count 和 cancel rate 共同决定扩容、限流与产品降级。

能迁移的判断：队列保存需求，但公平性、deadline 和用户反馈决定它是不是可用产品。
06

Geo-sharding 的难点是边界与热点

按城市或 geo cell 分区能把位置、匹配和通知留在低延迟区域，也便于独立扩容。但司机与乘客会跨边界，机场和市中心又会产生极端偏斜。设计应给每个实体一个 home region 和带版本的所有权转移协议；边界查询读取邻近 cell，候选去重后由权威 owner 执行 reservation。热点 cell 可以动态细分或给位置索引增加只读副本，但 ride 状态不能被多个 shard 同时写。跨区故障时宁可短暂扩大 ETA 或暂停新匹配，也不能以陈旧 ownership 造成双重派单。

能迁移的判断：地理分区降低延迟，所有权与边界协议守住正确性。
