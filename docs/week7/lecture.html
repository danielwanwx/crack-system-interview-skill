答题主线

01 · 核心判断

库存合同

座位销售的第一不变量是同一 seat 在同一场次最多进入一个有效 hold 或一个 sold order。不是每次读 seat map 都强一致，而是 hold 和 sale 的条件写必须有明确线性化点。

02 · 核心判断

状态模型

seat、hold、order、payment_attempt 各自有生命周期；seat 状态至少区分 available、held、sold 和 expired。hold 需要 owner、expiry 和 version，不能只有一个 Redis 锁，否则恢复和审计没有依据。

03 · 核心判断

读取与抢购分离

seat map 是可缓存的展示投影，允许数秒陈旧；真正下单时必须回权威库存执行 compare-and-set。等候室只平滑进入率和挡住机器人，不能保证排到前面的人一定买到某个座位。

04 · 核心判断

hold 的线性化

请求以 seat_id、expected_state 和当前时间执行条件更新，成功即创建带 TTL 的 hold。TTL 到期不是自动正确性保证，sweeper 与 checkout 都必须再次比较 hold 版本和 expiry，防止过期与付款并发双卖。

05 · 核心判断

支付与补偿

支付是外部副作用，不能包进库存事务。order 先绑定 hold，支付成功后条件地转 sold；任何超时都通过 provider 查询、幂等键和补偿任务恢复，而不是猜测结果。

06 · 核心判断

热点演出

一个热门演出会把所有写压到少数 seat/venue 分区，因此要按 event 隔离容量、控制入场、限制每用户并发，并对热点座位串行化。横向加节点不会消除同一 seat 的竞争，只能保护周边服务。

07 · 核心判断

故障决策

发生网络分区时，库存权威分区不可用就应停止出售，而不是为了可用性冒 oversell 风险。可继续展示缓存 seat map 和排队页面，但不能发出不可撤销的成功承诺。

08 · 核心判断

迁移到拍卖

拍卖允许同一 item 被更高报价覆盖，正确性是可审计的最高有效 bid 和确定的 close rule。Ticketmaster 的 hold 是排他资源锁；两者都要处理竞争，但状态转换与公平定义不同。
