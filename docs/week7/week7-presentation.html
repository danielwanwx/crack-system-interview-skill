Day 1 · Week 7

高并发库存模型

每座一票； hold 状态机； TTL 自动释放。
核心 invariant：任意 ticket_id 同时最多一个有效 hold；booked 不可逆（除退款补偿流）

参考原文 Ticketmaster · Entities Contention DDIA Ch8 本日核心追问
需求边界
  • 先锁定 3 个核心动作：每座一票、hold 状态机、TTL 自动释放
  • 明确 out of scope，避免把后台运营、推荐、BI 一次性塞进主链路
  • 读写比例、峰值 QPS、对象规模、保留周期必须给数量级假设
  • 先定义用户可见语义，再讨论 Kafka、Redis 或数据库选型
正确性不变量
  • 任意 ticket_id 同时最多一个有效 hold；booked 不可逆（除退款补偿流）
  • 每个状态转换写 precondition；失败重试不能跳过中间状态
  • 数据库事实、缓存视图、异步派生数据必须标出 authoritative source
  • 重复、乱序、超时、部分成功四类故障逐一给答案
容量与 SLO
  • Week 7 Day 1：先估峰值写 QPS，再按分区键检查热点
  • 在线路径给 p50 / p99；后台 repair 给 freshness 与 backlog SLO
  • 吞吐扩容不能破坏 per-key 顺序、唯一性或余额守恒
  • 压测必须覆盖 skew：最热 1% key 承担 40% 流量
层次 面试中要落到的机制 必须暴露的指标 失败时动作
API 幂等键、分页游标、版本号、错误码 request_p99 / 409_rate 限流、返回可重试语义
Storage 主键、分区键、二级索引、唯一约束 lock_wait / hot_partition 迁移流量或拆热点
Async 事件 ID、顺序键、consumer checkpoint lag / redelivery / DLQ 重放并保持幂等
Derived watermark、版本、last_updated_at freshness / mismatch_rate 从 source of truth 重建
Repair Redis TTL 为主释放；DB sweep 清理可读性 stale reserved stuck_age_p99 / repair_count 先止血，再对账，再回填 
Ticket { id, event_id, seat, status: AVAILABLE|BOOKED }
Booking { id, user_id, ticket_ids[], state: IN_PROGRESS|CONFIRMED }

Hold: SET ticket:{id} user:{uid} NX EX 600
State: AVAILABLE → (held in Redis) → BOOKED
脆弱答案：只讲 happy path
画完组件图就结束；没有唯一约束、状态机、重试边界，也没有说明超时后客户端该不该重试。
Senior 答案：不变量驱动
推荐
围绕「任意 ticket_id 同时最多一个有效 hold；booked 不可逆（除退款补偿流）」展开：写入原子点、异步传播、可观测性、自动修复与人工闸门形成闭环。
01
Scope

用 5 分钟确认 每座一票 与 hold 状态机 的成功语义，写出明确 out-of-scope。

02
Model

画实体、主键、状态与版本；指出哪张表或哪条日志是最终事实。

03
Path

先走一遍写路径，再走读路径；在跨服务边界标 idempotency key 和 timeout。

04
Failure

注入 consumer 宕机、重复消息和网络分区；用 Redis TTL 为主释放；DB sweep 清理可读性 stale reserved 收敛。

05
Scale

最后才做分片、缓存、批处理；逐项证明优化没有破坏正确性。

💡

路径 Max Path Sum 单边贡献返回父 
def max_path_sum(node):
    if not node: return 0
    left = max(0, max_path_sum(node.left))
    right = max(0, max_path_sum(node.right))
    global_best = max(global_best, left + right + node.val)
    return max(left, right) + node.val
