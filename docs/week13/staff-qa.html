01

Top K 的窗口必须先问什么？

参考回答
先问“过去一分钟、今天还是历史全量”，以及用户可接受多大延迟和误差。窗口决定 state 大小、是否需要 watermark 和能否用近似算法；不定义合同就无法合理选技术。
常见失分
上来就说 Redis ZSET。
继续追问
榜单每秒跳动是否是产品问题？
02

event time 与 processing time 差异？

参考回答
网络延迟、离线客户端和重试会让事件晚到，因此按 processing time 会把旧观看算进当前窗口。使用 event time 与 watermark 可得到更符合业务的窗口，但必须接受一段等待和迟到修正策略。
常见失分
认为消息到达即事实发生。
继续追问
客户端时钟不可信怎么办？
03

watermark 如何影响结果？

参考回答
watermark 不是保证所有事件到齐，而是系统基于延迟分布作出的推进判断。它越保守，榜单越晚稳定；越激进，迟到修正越多，产品需决定是否允许排名回跳。
常见失分
把 watermark 当成精确时间戳。
继续追问
重大网络故障后 watermark 卡住怎么办？
04

为什么不能总用精确 counter？

参考回答
全量精确计数在内容和用户规模大、窗口多且需要低延迟时，会产生巨大状态和跨分区合并成本。可对候选重热视频精确化、对长尾近似计数；取舍是可量化误差换取可用的资源预算。
常见失分
认为 counter 永远 O(1) 就无成本。
继续追问
哪些业务绝不能近似？
05

Count-Min Sketch 保证什么？

参考回答
CMS 以多个哈希桶累加，查询会受到其它 key 碰撞影响，因此通常给出上界估计而非精确值。它适合先找 heavy hitter，再对少量候选精确计数；不能用它直接承诺精确排名。
常见失分
说它随机上下误差。
继续追问
如何减少一个热门 key 被碰撞影响？
06

HyperLogLog 为什么不适合 Top K？

参考回答
HLL 用很小状态估算集合中不同元素数量，例如 unique viewers。Top K 需要每个内容的频次和相互比较，HLL 不保留这种映射；两者解决的是不同统计问题。
常见失分
把所有近似结构当互换。
继续追问
广告 UV 应用哪种结构？
07

重复 view/click 如何去重？

参考回答
生产者尽量生成稳定 event_id，流处理在业务允许的重复窗口内去重，并把去重状态设 TTL。TTL 过短会漏掉迟到重试，过长会膨胀状态；广告计费通常要求更严格的对账路径。
常见失分
依赖消息队列不重投。
继续追问
没有 event_id 的 legacy source 怎么办？
08

迟到事件怎么处理？

参考回答
在 watermark 前到达的事件更新窗口，超出 allowed lateness 的事件按产品选择丢到修正流、更新已发布值或留给离线报表。关键是用户能知道实时榜和最终报表可能不同，而不是悄悄改历史。
常见失分
所有迟到事件直接丢弃。
继续追问
怎样让榜单修正不造成用户困惑？
09

processor 重启如何恢复 state？

参考回答
processor 定期对状态和已处理 offset 做一致 checkpoint，故障后从最近 checkpoint 和输入 log 重放。输出写入也需幂等或事务性 sink；否则恢复会把同一窗口结果重复叠加。
常见失分
state 只放内存。
继续追问
checkpoint 太慢如何优化？
10

end-to-end exactly-once 的前提？

参考回答
框架可以让输入 offset、状态 checkpoint 和某些 sink 近似原子，但前提是各端都支持相同协议。邮件、第三方点击回调等外部副作用仍需 idempotency；不要把一个 connector 选项扩大成端到端承诺。
常见失分
开开关就所有业务 exactly-once。
继续追问
广告账单为何还要离线对账？
11

fraud 延迟判定怎么处理？

参考回答
实时聚合先给出 provisional 点击或收入，fraud 模型/人工审核随后产出调整事件。财务报表和结算使用可回放的最终化窗口，不能把实时 dashboard 直接当 billing truth。
常见失分
等 fraud 完成再显示任何数据。
继续追问
调整事件如何不重复扣减？
12

key skew 怎么处理？

参考回答
对超热 video/campaign 可在第一阶段加随机盐并行局部聚合，再按原 key 合并结果。这样分摊写入而不改变最终数值；盐的数量需按热点动态调节，否则普通 key 反而增加成本。
常见失分
只增加 partition 数。
继续追问
如何识别何时需要 salting？
13

游戏榜何时需强一致？

参考回答
实时展示允许数秒延迟和异步排名，但比赛截止、奖金结算或反作弊裁决需要一个确定的最终快照和 tie-break。把所有更新都做强一致会昂贵；把结算也最终一致会不可审计。
常见失分
所有排行榜都用同一 ZSET。
继续追问
并列分数怎么排序？
14

serving cache 如何避免跳动？

参考回答
榜单读服务可按短周期发布版本化 snapshot，客户端在同一请求/分页内固定版本，避免每次刷新看到无意义抖动。更高频的 provisional 更新可展示“实时中”标签；取舍是少量 freshness 换稳定感。
常见失分
每个事件直接改公开榜。
继续追问
突发热点时 refresh cadence 怎么调？
15

新算法如何 backfill？

参考回答
用保留的原始事件流在隔离集群重放新算法，和旧结果按窗口、内容和误差分布比较。验证后通过 serving version 切换并保留回滚；不能在线上直接重算并覆盖用户看到的榜。
常见失分
修改状态 schema 后原地继续。
继续追问
raw log 过期怎么办？
16

核心运营指标？

参考回答
watermark lag 和输出 freshness 表示实时性，state size/checkpoint 成功表示能否恢复，误差界和精确抽样对比表示结果质量。再看 hot key 和迟到比例，才能判断问题是流量、算法还是数据源。
常见失分
只看 consumer lag。
继续追问
哪项指标会暂停发布新榜？
