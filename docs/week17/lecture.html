答题主线

01 · 核心判断

分离会话与运行

conversation/message 是用户可见的 durable history，model run 是某次带 model、prompt、context、工具和输出版本的执行记录。二者分离才能支持流式取消、重试、模型迁移和审计，而不会把一次失败生成写成历史事实。

02 · 核心判断

流式响应合同

streaming 先声明 partial token 的可见性与完成边界：客户端断线不等于 run 必须取消，重新连接需能从 sequence/cursor 恢复或安全重取最终输出。取消必须向模型网关传播并标记计费/输出状态，避免后台无意义继续生成。

03 · 核心判断

model gateway

gateway 根据能力、延迟、成本、区域和租户策略选择模型，并记录版本与 routing 决策。fallback 不能只看 HTTP 成功：不同模型的工具/结构化输出能力、输出长度和安全策略必须在合同内兼容。

04 · 核心判断

检索摄取状态机

document 从 discovered、parsed、chunked、embedded、indexed 到 deleted 有版本化状态，每一步可重试和可回收。embedding 是对某个文档版本和模型版本的派生数据；原文与 ACL 才是权威。

05 · 核心判断

ACL-aware retrieval

权限过滤必须在候选召回时发生，确保模型永远不会获得无权 chunk。metadata filter、tenant partition 和必要的权威 ACL 检查共同构成边界；生成后遮盖引用已经太晚。

06 · 核心判断

context budget 与引用

检索不是把所有相似 chunk 塞进 prompt，而是在 token budget 内做权限过滤、重排、去重和截断。回答应记录引用的 document/chunk/version，并在无充分证据时拒答或降级，而不是用流畅性掩盖缺失证据。

07 · 核心判断

评估与发布

offline golden set 覆盖检索、权限、拒答和任务成功，online 同时看 groundedness、用户完成、延迟和成本。prompt、chunker、embedding、retriever 和模型必须版本化、灰度和可回滚；单看点赞无法发现权限泄露。

08 · 核心判断

数据与运营

PII、租户隔离、retention 和 audit 必须从 ingest 到 trace 一致执行。可观测性记录必要的请求/检索/模型元数据，不应把原始敏感内容无界复制进日志；故障时可回退到检索结果或明确不可用。
