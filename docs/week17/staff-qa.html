01

conversation state 与 model request 如何拆？

参考回答
会话保存用户可见的消息和访问控制，而一次 model run 保存模型、系统指令、检索上下文、stream 状态和输出版本。这样同一消息可因 retry 或新模型产生多个 run，最终展示哪个输出由显式策略决定。
常见失分
把所有 prompt/response 覆盖在一张 messages 表。
继续追问
编辑历史消息后如何处理旧 run？
02

stream 中断怎么恢复？

参考回答
服务端为流输出维护 sequence 或持久化最终响应，客户端重连时从最后确认位置续取或重新获取完成结果。网络断线不必自动取消 run；用户明确取消才向模型调用传播取消并记录部分输出状态。
常见失分
重连后重新发一次完整 prompt。
继续追问
重复 token 如何避免展示？
03

何时用同步请求、何时 durable run？

参考回答
低延迟聊天可用同步流式调用，但长检索、批量处理、工具调用或需要人工审批的任务应创建 durable run。后者能重试、观察和恢复，但会增加状态管理与用户等待交互。
常见失分
所有请求都丢进后台队列。
继续追问
同步调用超时后如何保留结果？
04

model fallback 如何不失控？

参考回答
fallback 只在替代模型满足同一输出/工具/安全合同且预算允许时执行，并把 routing 记录到 run。不能把任意模型当通用备份，否则结构化输出、引用格式或拒答行为变化会成为隐性产品 bug。
常见失分
429 就换最便宜模型。
继续追问
怎样做 fallback 的灰度验证？
05

文档更新/删除如何收敛？

参考回答
新版本文档生成新 chunk/embedding 并在索引切换后成为可召回版本；旧版本保留到切换安全完成。删除先从 retrieval filter 撤销，再清向量和缓存，防止正在异步清理时模型继续引用旧政策。
常见失分
只删除原始 PDF。
继续追问
embedding job 失败如何避免旧版本长期可见？
06

chunk size/overlap/filter 如何取舍？

参考回答
较小 chunk 提高定位精度但易丢上下文，较大 chunk 增加噪声和 token 成本；overlap 缓解边界但会重复。先用任务样本评估 chunker，再在候选阶段应用 tenant、权限、文档类型和时间 filter。
常见失分
固定 500 tokens 作为通用答案。
继续追问
表格和代码块怎么切？
07

为什么 ACL 必须进入 retrieval？

参考回答
一旦无权限 chunk 被放入模型上下文，即使最终 UI 不显示引用，也可能通过回答或日志泄露内容。因而权限是候选集约束，必要时再做权威校验；这是数据边界，不是显示格式问题。
常见失分
生成后再过滤敏感句子。
继续追问
群组成员变更如何快速生效？
08

context budget 满了怎么办？

参考回答
先按权限、相关性、freshness 和多样性重排，去掉重复 chunk，再在可解释的预算内选证据。若证据不足应请求澄清或拒答，而不是截断一半关键条款后自信回答；压缩也要保留可追溯来源。
常见失分
无限增大 context。
继续追问
长文档摘要如何验证不失真？
09

citation 如何可信？

参考回答
每个引用关联实际进入上下文且调用者有权限访问的文档片段与版本，点击可回到原文位置。模型生成的裸 URL 或未检索内容不能算 citation；引用覆盖率也应进入评估。
常见失分
让模型自行编参考链接。
继续追问
文档删除后旧会话引用怎么办？
10

何时应回答不知道？

参考回答
当检索不到足够、最新且有权限的证据，或来源相互冲突且风险高时，应明确说明范围并引导用户查原文/人工。拒答是系统质量的一部分；比起流畅幻觉，它保留信任和可修复路径。
常见失分
只要模型能写就回答。
继续追问
怎样区分需要澄清与直接拒答？
11

stale embedding 如何发现？

参考回答
每个 chunk 记录原文版本、embedding 模型和索引时间，文档更新会生成需要替换或撤销的任务。监控未处理版本差和删除 lag；否则系统会稳定地检索语义相似但已废弃的政策。
常见失分
只重嵌入新文档。
继续追问
模型升级如何双写向量？
12

golden set 应覆盖什么？

参考回答
golden set 不能只有容易问答，应包含应答、应拒答、跨租户拒绝、过期文档和多文档冲突。每例有期望证据、允许回答边界和评分标准，才能比较 prompt、retriever 或模型改动。
常见失分
只测 BLEU 或几个漂亮 demo。
继续追问
如何维护 case 不随产品变旧？
13

online 怎么同时看质量成本？

参考回答
在线记录任务是否完成、引用覆盖/用户纠错、拒答率、P95、token 和工具成本，并按模型/检索版本/租户切片。只优化满意度会掩盖昂贵或慢的路径，只压成本又会让检索质量下降。
常见失分
只看 token 数。
继续追问
什么情况下自动 rollback？
14

prompt/model 版本如何回滚？

参考回答
把 prompt template、model、retriever、chunker 和 policy 作为可寻址版本写进 run，发布用小流量灰度和对照集验证。出现质量/安全回归时切回上一配置，无需改写历史会话；版本组合比单个模型名更重要。
常见失分
在代码里直接替换 prompt。
继续追问
如何处理缓存中旧 prompt 输出？
15

tenant/PII/audit 放在哪里？

参考回答
tenant identity 与 ACL 从上传、chunk、vector filter 到 run trace 一直携带，敏感字段按最小必要保留和加密访问。审计记录谁访问了什么、用哪个版本处理，而不是无界保存完整 prompt/文档副本。
常见失分
只在 API gateway 校验 tenant。
继续追问
用户删除请求如何跨日志处理？
16

vector DB 为什么不是唯一真相？

参考回答
向量库擅长近邻召回，却不天然提供文档生命周期、权限、事务或事实校验。它应从权威文档系统派生并可重建；回答的证据和访问控制不能只依赖 embedding。
常见失分
所有数据直接写向量库。
