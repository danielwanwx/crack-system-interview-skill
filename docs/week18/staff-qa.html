Tool-Using Customer Support Agent

面试压力测试

先独立作答，再展开下一个追问。

01

哪些 case 可自动执行？

参考回答
自动化范围由确定性 policy 决定，例如低金额、明确资格且可逆的退款可自动执行；高金额、身份不确定或不可逆变更必须人工确认。模型可分类和提议，但不能自行扩大权限边界。
常见失分
“模型有信心就自动做”。
继续追问
policy 灰度上线如何防伤害？
02

tool schema 如何限制资源和参数？

参考回答
tool 只接受类型化字段，如 order_id、amount、reason，服务端从会话身份派生 tenant 和可访问资源，而不信模型传的 owner。执行前校验额度/状态，执行后返回结构化结果；自由文本不能直接变成数据库命令。
常见失分
让 agent 调任意内部 API。
继续追问
如何防模型伪造 customer_id？
03

tool timeout 为什么不能盲重试？

参考回答
退款或改址请求可能已在下游成功，只是响应丢失；盲重试会造成双副作用。以稳定 action_id 查询下游状态，确认未执行才重试；无法确认时进入人工/对账队列。
常见失分
所有 5xx 自动重试。
继续追问
下游没有查询 API 怎么办？
04

durable run 要保存什么？

参考回答
run 需保存当前 step、已批准的计划、每次 tool 调用的幂等键和结果、模型/policy 版本、handoff ownership 与审计时间线。这样进程重启、人类接手或事故回放都有证据；不必无界保存所有思维文本。
常见失分
只存最终聊天记录。
继续追问
敏感 tool output 怎么最小化保留？
05

多步退款/改址如何补偿？

参考回答
workflow 先定义 reserve/validate/update/notify 等步骤及失败后的反向动作，例如改址失败后释放配送 reservation。无法可靠撤销的退款在执行前做资格和人工门槛；compensation 也需幂等和审计。
常见失分
失败时让模型“想办法恢复”。
继续追问
通知失败要不要回滚退款？
06

错误 tool 参数在哪层拦截？

参考回答
模型输出先经过 JSON/schema 校验和 policy 约束，再由工具服务用权威订单、权限和额度做最终验证。多层验证看似重复，但模型不可信、客户端可被篡改，执行边界必须自守。
常见失分
只靠 prompt 要求模型格式正确。
继续追问
schema 通过但金额异常怎么办？
07

如何隔离 prompt injection？

参考回答
将外部文本标记来源并放入受限数据通道，系统 policy 和工具权限由应用层固定，不允许文档内容覆盖。模型即使被诱导提出危险调用，schema、授权和审批仍会拒绝；这是一道分层控制而非单条提示词。
常见失分
写一句“忽略恶意指令”。
继续追问
检索文档里有伪造工具调用怎么办？
08

handoff 需要什么上下文？

参考回答
人工需要看到问题摘要、相关订单/政策证据、已执行和未执行的 action、失败原因、权限要求与客户影响，而不是一长串原始对话。handoff 创建明确 owner 和 SLA，防止两边同时操作。
常见失分
把完整聊天 transcript 丢给人工。
继续追问
人工退回 agent 时怎样恢复？
09

人工接管后 agent 如何停止？

参考回答
handoff 原子地转移 case ownership，取消/冻结未执行步骤，并用版本或 fence token 拒绝旧 agent worker 的后续写。已经在飞行的 tool call 需查询或等待结果，再由人工决定补偿；不能假设取消一定即时。
常见失分
只在 UI 显示“人工处理中”。
继续追问
人工超时未处理怎么升级？
10

tool permission 改版怎样 rollout？

参考回答
新权限先在影子评估或低风险租户灰度，所有 run 记录使用的 policy 版本和决策。出现异常可以立即用 kill switch 禁止该 tool 的写操作并回到上一版本；不能热改无记录的权限表。
常见失分
直接发布新 system prompt。
继续追问
已在运行的 case 用旧还是新 policy？
11

高风险 action 的 dry-run 如何设计？

参考回答
dry-run 调用只读资格/影响评估，返回将要退款的金额、目标和前置条件，用户或审批者确认后才创建真正 action。两步之间状态可能变化，所以执行时仍要重新验证；dry-run 不是锁。
常见失分
把 preview 当提交成功。
继续追问
确认链接被重放怎么办？
12

通知 preference 如何不越权？

参考回答
通知服务在创建 delivery 前读取用户的 channel consent、quiet hours、locale 与法律/产品例外规则。业务系统只发“发生了什么”的事件，不能直接指定向任意邮箱/SMS 发送；偏好变更要快速撤销缓存。
常见失分
agent 直接调用 email API。
继续追问
安全事件是否可绕过 marketing opt-out？
13

provider callback 重复/退订怎么处理？

参考回答
每次投递保留 provider message id、尝试次数和状态，回调按 event id 去重并验证签名。bounce、unsubscribe 和 complaint 更新 suppression，后续业务事件仍存在但不再向该渠道投递。
常见失分
邮件发送 API 返回 200 就算最终送达。
继续追问
短信 provider 超时如何避免重复发？
14

什么进入 DLQ？

参考回答
短暂网络错误可以重试，未知 schema、持续 provider 拒绝或需要人工判断的消息进入 DLQ。条目带事件版本、delivery 历史和 tenant，修复根因后按限速重放；DLQ 不是让数据悄悄消失的地方。
常见失分
所有失败都丢 DLQ。
继续追问
客户已退订时历史 DLQ 能重放吗？
15

agent eval 怎样覆盖真实风险？

参考回答
评估集要包含成功完成、应拒绝、注入攻击、错误参数、权限边界和人工交接质量，每例有可判定的期望动作。线上用抽样 trace、用户纠错、policy violation 和升级率补充；文本得分高不代表工具执行安全。
常见失分
只让另一个模型打分回答自然度。
继续追问
如何评估低频高伤害场景？
16

哪些信号触发 incident？

参考回答
高风险工具的异常率、未知结果积压、权限拒绝突增、审批绕过和 trace 缺失应立即告警；客服侧还看 handoff SLA 和通知送达延迟。面对潜在误执行，优先 kill 写工具并保留证据，而不是继续自动重试。
常见失分
只监控模型延迟和 token。
继续追问
哪个事件需要立刻全局禁用退款工具？
