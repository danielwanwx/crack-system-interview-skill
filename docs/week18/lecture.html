第 18 周讲义 · 主项目 Tool-Using Customer Support Agent · 迁移 Notification System

Agent 工作流与通知

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

把 agent 视为工作流

客服 agent 不是一串 prompt，而是 case、run、step、tool_call、approval、handoff 的 durable 状态机。模型只提出下一步候选；真正的权限、执行、重试和审计由确定性 workflow 层控制。

02 · 核心判断

工具合同与权限

每个 tool 有强 schema、允许的资源范围、调用者身份和副作用等级，输入/输出在执行前后都验证。高风险动作如退款、改地址必须由 policy 和 approval gate 决定，不能让模型文本越权。

03 · 核心判断

幂等与补偿

tool call 带稳定 action_id 和业务幂等键，timeout 先查询真实结果再决定重试。多步动作以 saga 定义正向步骤和 compensation；不能补偿的动作需在执行前人工确认或 dry-run。

04 · 核心判断

不可信内容边界

用户消息、邮件、网页和检索片段都可能包含指令注入，因此只作为数据进入模型上下文，不能修改 system policy 或 tool 权限。工具执行只接受已验证的结构化参数，并按 least privilege 访问受限资源。

05 · 核心判断

人工交接

handoff 不是失败日志，而是一个带 SLA、上下文、已执行动作、待决权限和客户影响的队列。人工接管后 workflow 必须冻结或转 ownership，防止 agent 在后台继续退款、发信或改地址。

06 · 核心判断

通知作为独立系统

notification 负责 preference、template、channel、delivery_attempt、suppression 和 provider callback，不负责决定业务是否可执行。事件只在业务状态提交后通过 outbox 进入通知，渠道失败不能反向改写退款或 case 的真相。

07 · 核心判断

评估与追踪

agent eval 要覆盖 task success、policy violation、tool 参数正确性、handoff 质量和用户伤害风险，而非只评文本流畅。trace 记录 run/step/tool/approval 的因果链，支持回放、抽样审核和 incident 定位。

08 · 核心判断

安全运营

权限策略、tool schema 和模型/prompt 都需版本化、灰度与 kill switch。异常 tool error、approval bypass、escalation、notification failure 和 trace 缺失是第一类告警；对高风险能力先停写再调查。
