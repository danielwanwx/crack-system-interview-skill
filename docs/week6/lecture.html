Ticketmaster

Ticketmaster：从需求语义到故障修复

01

活动目录、搜索结果与票务库存不是同一种真相

Ticketmaster 的活动名称、场馆、演出时间适合由目录数据库保存，并异步投影到搜索索引；某场次某座位是否还能卖，则必须由库存事实表裁决。座位图缓存可以把 AVAILABLE、HELD、SOLD 渲染得很快，却只是一张带 inventoryVersion 的快照，不能凭缓存命中直接承诺购票。高层设计应让浏览请求走缓存或索引，让最终确认回到 eventId 与 seatId 定位的权威库存。这样即使索引延迟、缓存失效或搜索集群降级，用户可能短暂看到旧颜色，但两个订单不会同时拥有同一座位，也能明确说明哪些页面可降级为仅浏览。

能迁移的判断：目录负责发现活动，缓存负责快速展示，库存条件写才有资格承诺座位。
02

用带版本的临时保留把抢票竞争变成一次条件转换

选择座位时创建 holdId，并在单个原子操作中验证座位仍为 AVAILABLE、seatVersion 等于客户端读到的版本，再写成 HELD、记录 ownerId、expiresAt 与新版本。重复请求携带同一 idempotencyKey，只返回原 hold；竞争者得到明确的已被占用，而不是在应用层先查再改。保留期限由服务端决定，清理任务只能对 holdId 与版本仍匹配的记录执行 HELD 到 AVAILABLE 的条件转换。支付完成同样要求当前 hold 归属正确且未被更高世代替代，随后变为 SOLD。这个模型把锁定范围压到单座位，也使超时后的查询和安全重试有确定依据。

能迁移的判断：保留、释放与售出都验证 holdId 和 seatVersion，扫描器只负责触发而不越权。
03

支付确认与保留过期相撞时先承认结果未知

付款请求离开本系统后，网络超时既不能证明扣款失败，也不能证明扣款成功。PaymentAttempt 要保存 providerRequestId 与本地 holdId；回调、主动查询和客户端重试最终都汇入同一状态机。若支付成功事件到达时座位仍由该 hold 持有，事务内把座位改为 SOLD，并写出票 outbox；若保留已被合法释放或转售，则进入人工可解释的退款或替代席位流程，绝不能悄悄覆盖新持有人。过期清理与支付确认同时发生时，双方都以条件写竞争同一版本，只有一个提交。接口返回 PROCESSING 并提供查询地址，比把超时谎报为失败更符合财务事实。

能迁移的判断：用统一 PaymentAttempt 和库存版本化解过期竞态，对未知结果提供查询而非盲目重扣。
04

等待室控制准入速率而不是伪造无限库存吞吐

热门开售的危险不是总日活，而是数百万人在同一分钟刷新同一 eventId。入口先发不可篡改的排队凭证，其中包含 eventId、随机化后的顺序、issuedAt、admissionWindow 与签名；等待室只按库存服务可持续的条件写吞吐发放短期 admission token。token 绑定用户和场次并限制并发请求，刷新页面不能插队，多设备也不能复制资格。库存变慢时收紧放行速率，队列页仍可用静态资源和轻量轮询展示估计等待时间。公平性需要监测顺序跳跃、凭证重放和等待分位数，并公开“获得入场资格不等于保证有票”的产品语义。

能迁移的判断：等待室的容量单位是后端可承受的尝试速率，公平由签名顺序凭证和可审计放行保证。
05

搜索游标与热门查询缓存不能泄漏成售票承诺

活动搜索按城市、日期、类别和热度建立倒排索引，深翻页使用包含排序值与 eventId 的 search_after 游标，避免 offset 在索引变化时既昂贵又重复。查询缓存的键应覆盖规范化过滤条件、地区、语言和索引版本，设置短时有效期，并用请求合并防止热门键同时过期造成击穿。活动取消或时间变更通过变更事件刷新索引和失效相关缓存；若更新尚未收敛，详情页仍回源目录复核。座位可用数量可以作为近实时提示，但结算前始终重新读取库存，因此扩大搜索副本和缓存不会复制出售权，也不会让陈旧索引决定成交。

能迁移的判断：稳定游标服务发现，缓存保护检索；进入预订时重新校验目录与库存事实。
06

抢票演练要同时观察超卖、公平和恢复水位

容量测试以单场开售峰值建模：估算每个候选用户的排队轮询、座位图读取、保留条件写和支付回调放大倍数，再为单 eventId 预留节点故障后的吞吐余量。关键指标包括有效 hold 冲突数、SOLD 唯一约束违规、hold 创建成功率、过期释放滞后、等待室放行与完成率、支付成功到出票的 p99，以及目录、索引、座位图之间的版本差。演练依次注入缓存击穿、库存主节点切换、支付回调重复和清理任务延迟；恢复后用 seatId 对账订单、有效 hold 与库存状态。任何超卖必须立即阻断发布，而不是用平均延迟正常来掩盖正确性事故。

能迁移的判断：验收以零超卖和可解释排队为先，再用版本水位与尾延迟证明系统已恢复。

Online Auction

Online Auction：从需求语义到故障修复

01

Auction 聚合保存当前裁决，Bid 日志保存每次证据

在线拍卖把 Auction 视为单个竞争域：记录 auctionId、sellerId、状态、serverDeadline、currentPrice、winnerId、version 与 lastBidSeq；每次被接受或拒绝的出价另写不可变 Bid 记录，包含 bidderId、amount、requestId、receivedAt 和裁决原因。Auction 行回答“此刻谁领先”，Bid 日志回答“系统为何得出这个结论”，两者在一次事务里推进。浏览缓存和推送消息只是从已提交 bidSeq 派生，不得反向改价。这个模型既避免每次展示扫描所有出价，又让争议复盘、故障重放和关闭后审计都有完整证据，而不是只剩一个无法解释的最高价数字。

能迁移的判断：聚合行承担串行裁决，不可变出价日志承担审计与恢复，两者共享 bidSeq。
02

接受出价必须把金额、状态、期限和版本放进同一原子判定

提交 Bid 时先用 requestId 去重，再由 Auction 所在分片执行条件事务：状态必须为 OPEN，服务端接收时刻未超过 deadline，amount 满足最小加价规则，version 等于读取值；成功后同时递增 version 与 bidSeq、更新最高价和领先者、插入 Bid 与 outbox。两个并发出价只能有一个基于旧版本成功，失败者重新读取后决定是否仍满足规则。不能把“校验当前价”和“写新价格”拆到两个服务，也不能让 Redis 自增值成为唯一事实。对单个热门 auction 的协调是业务必需，但它不应阻塞其他 auctionId 的出价。

能迁移的判断：把 OPEN、截止时间、加价规则与 version 合成一次条件提交，冲突后重读而非覆盖。
03

拍卖截止由服务器裁决点定义，而不是相信客户端倒计时

客户端显示的倒计时只是体验信息，真正有效性由接收出价的权威 Auction owner 根据 serverDeadline 判断。请求在截止前离开客户端却在截止后到达，默认按服务器收到时间拒绝；若产品支持反狙击延时，则“最后若干秒的有效出价延长截止”也必须与该出价在同一事务内更新 deadline 和 version。关闭任务携带预期版本，只能把仍为 OPEN 且 deadline 已过的聚合改为 CLOSED；与边界出价竞争时，两者只有一个先提交，另一方重读新状态。所有响应返回 serverNow、最终 deadline 与 bidSeq，让客户端修正展示并能解释边界判定。

能迁移的判断：用服务器接收时刻和版本化关闭解决边界竞争，产品若延时必须写成明确原子规则。
04

分片所有权世代防止故障切换后出现两个拍卖裁判

Auction 按 auctionId 分区，每个分区在某一 ownerEpoch 只有一个可接受写入的 leader；路由目录可以缓存位置，但每次提交都带世代并由存储拒绝旧 owner。事务日志至少跨故障域复制后才确认，Bid 与 outbox 同事务落盘，推送消费者可重复读取但按 auctionId、bidSeq 去重。leader 在确认后崩溃时，新 owner 从已提交日志恢复 currentPrice 与 lastBidSeq；若客户端没收到响应，则用 requestId 查询原裁决，而不是再生成一笔出价。定期从 Bid 日志重算聚合并比较哈希，可以发现静默丢失或错误程序写坏当前价。

能迁移的判断：写确认绑定复制提交和 ownerEpoch，切主后按 requestId 查询与日志重算恢复。
05

实时最高价推送携带 bidSeq，断线后先补洞再接直播

提交事务产生 BidAccepted 事件，实时服务按 auctionId 订阅并向已连接观察者推送 currentPrice、winnerMask、bidSeq 与 deadline。连接落在哪台网关由一致性路由管理，但网关不保存权威最高价。客户端只接受比 lastSeenBidSeq 更大的事件；发现跳号或重连时，先调用 catch-up API 读取从 lastSeenBidSeq 之后的已提交变化，若保留窗口已过则取当前 Auction 快照，再继续订阅。慢客户端限制发送缓冲，超过预算就断开并要求补读，不能让一个浏览器拖垮整个房间。重复事件可忽略，顺序恢复以持久序列为准。

能迁移的判断：WebSocket 只负责低延迟通知，bidSeq 与补读接口才保证重连后的连续视图。
06

千万场普通拍卖可分片，单场超级热点要用排队和明确背压

十百万并发拍卖大多能按 auctionId 均匀分片，但明星藏品可能把所有写压到一个不可再拆的串行聚合。先把浏览流量交给缓存与实时扇出层，把不同拍卖隔离到独立分区；对热点出价入口设置有界队列、每用户速率限制和基于 serverReceivedAt 的确定顺序，队列接近截止却无法在延迟预算内裁决时必须拒绝，而不是接受后丢失。可批量持久化日志和合并推送，但不能批量挑选“本批最高价”而跳过业务逐项裁决。容量验收观察单 auction 队列年龄、版本冲突、裁决 p99、拒绝原因和推送水位差。

能迁移的判断：横向分片解决拍卖数量，单拍卖热点靠有界排队与背压，不能放松确定顺序。

Payment System

Payment System：从需求语义到故障修复

01

PaymentIntent 与 PaymentAttempt 分开表达用户意图和每次通道尝试

商户创建 PaymentIntent 时提供 merchantId、orderId、amount、currency 和 idempotencyKey；唯一约束保证同一业务订单不会因重试生成第二笔付款。Intent 保存总体状态，Attempt 则记录每次向收单方发出的 providerRequestId、通道、当前阶段和最后证据。接口不会把一次网络超时压成 FAILED，而是返回 PROCESSING 与可查询的 paymentId。状态迁移只允许从 CREATED 到 PROCESSING，再依据有证据的授权、捕获或失败结果推进；晚到事件携带版本，不能把已退款的付款倒退成成功。分层状态让路由重试、换通道和用户查询都保持同一笔业务意图。

能迁移的判断：Intent 保持商户级唯一语义，Attempt 保存外部交互证据，未知结果必须显式存在。
02

外部网络超时后的核心动作是查询和对账，不是再次扣款

向支付网络提交后如果连接断开，本系统只知道结果未知。后台协调器用原 providerRequestId 查询通道，回调与主动查询都转换为带 providerEventId 的统一事件，并按 PaymentAttempt 当前版本幂等应用。短期无法确认时保持 PROCESSING，达到业务时限后转为需要调查，而不是凭本地计时宣布失败。定时对账导入通道结算文件或流水，与 Attempt、订单和账本逐笔匹配，找出已扣未记、已记未扣及金额币种不符。任何修复都追加新事件和更正分录，不篡改原证据，这样才能在异步网络中保持财务完整性。

能迁移的判断：同一个通道请求号贯穿提交、查询、回调和对账，重试只重查状态而不重做扣款。
03

不可变复式账本把资金守恒变成数据库可验证约束

支付状态说明流程走到哪里，LedgerEntry 才说明钱从哪个账户流向哪个账户。每个 ledgerTransactionId 在单一事务中写入至少两条不可变分录，所有 debit 与 credit 按币种求和必须为零，并关联 paymentId、attemptId、businessEventId 和创建版本。已入账记录禁止 UPDATE 或 DELETE；退款、手续费调整和冲正通过新的反向分录表达。业务表更新、账本事务和 outbox 应处于同一本地事务边界，避免已经向下游宣告成功却没有资金证据。每日按商户、通道和币种做余额滚算，再与外部结算对账，差异可追溯到原事件。

能迁移的判断：余额是不可变复式分录的派生值，修正靠反向分录，不能改写历史。
04

Webhook 投递是一个可重放产品协议，不是随手发出的 HTTP 请求

Payment 状态提交时同事务写入 webhook outbox，事件包含稳定 eventId、paymentId、eventType、resourceVersion 与 occurredAt。投递器按商户端点读取，使用签名和时间戳防伪，有限指数退避；同一 eventId 可多次投递，因此接收方要幂等，事件版本帮助其拒绝状态倒退。成功以约定的 2xx 为准，永久 4xx、持续超时和签名配置错误进入可见的失败队列，并给商户控制台提供重放与端点停用。不同付款之间不承诺全局顺序，同一 paymentId 则按版本检查。绝不能在核心支付事务里同步等待商户服务器，否则一个慢端点会拖垮收款。

能迁移的判断：outbox 保证事件不丢，eventId 与版本承担重复和乱序语义，商户故障由独立队列隔离。
05

令牌化缩小敏感数据边界，授权与审计必须覆盖每次访问

卡号应由受控采集页面直接交给合规 tokenization 服务，核心 Payment API 只接收不可逆用途受限的 paymentMethodToken；日志、指标、消息和客服界面都禁止出现完整 PAN 或安全码。商户凭证按环境隔离，密钥放专用管理系统并定期轮换，服务间请求使用短期身份和最小权限。Webhook 使用按商户独立密钥签名并校验时间窗口，所有敏感读取和管理动作写入不可变审计记录。风险引擎可以拒绝或挑战高风险付款，但风控超时时要按金额、地区和商户策略决定 fail closed，不能用一个全局默认悄悄扩大欺诈面。

能迁移的判断：不持有原始卡数据是最有效的边界；令牌、最小权限、密钥轮换与审计共同缩小爆炸半径。
06

一万 TPS 的扩展必须保留单笔付款顺序和商户级隔离

入口按 merchantId 与 paymentId 稳定路由，创建 Intent 的唯一键落在同一分片；不同付款可并行，同一 paymentId 的事件必须按 resourceVersion 串行应用。外部通道调用进入分商户或分通道的有界队列，突发流量由配额和背压吸收，队列年龄超过支付时效就拒绝新请求而不是无限堆积。账本分区仍要保证单个 ledgerTransaction 的平衡分录同事务提交，跨分区汇总走异步报表。热门大商户使用独立分区和限额，避免拖垮长尾。容量看创建 TPS、通道延迟、未知结果年龄、outbox 积压、账本提交 p99 与对账差异，而非只看 API QPS。

能迁移的判断：按付款并行、按单笔版本有序，队列和租户隔离吸收突发但不放松资金不变量。
