先独立作答，再展开下一个追问。

01

为什么大文件不经过 API server？

参考回答
应用服务器若代理字节流，会把带宽、连接数和重试成本集中到最难横向扩展的一层。让客户端直传对象存储后，控制面仍以 commit 决定可见性；这保留了权限和版本语义，也把数据面扩容交给对象存储。
常见失分
只说对象存储更便宜，没有说明发布权威仍在 metadata commit。
继续追问
直传成功而 commit 超时，客户端怎样重试？
02

multipart 重试如何不产生脏 asset？

参考回答
分片只属于一个 upload_session，服务端在 commit 时验证完整 manifest 与最终 hash。相同 idempotency key 重试返回同一 asset 结果；未被提交引用的对象延迟 GC，避免删除仍在网络中的合法上传。
常见失分
要求对象存储和数据库做分布式事务。
继续追问
GC 如何避免误删刚上传完成的对象？
03

怎样保证用户看不到半成品？

参考回答
post 在元数据上只能引用 ready rendition，发布动作是状态转换而不是文件上传完成事件。这样转码、审核或缩略图失败只会停留在 processing，不会由 CDN 或 feed 把半成品暴露出去。
常见失分
让前端根据转码进度自行决定是否展示。
继续追问
审核必须异步时，发布前和发布后各有什么代价？
04

转码重复消费会怎样？

参考回答
重复执行本身可以接受，但同一 rendition 不能被两次写成不同版本或触发两次发布。worker 先用稳定键检查已有成功结果，再以条件写入完成状态；副作用如通知由 outbox 只发一次。
常见失分
把 Kafka exactly-once 当成业务幂等的替代。
继续追问
转码输出已写入但状态回写失败怎么恢复？
05

bitrate ladder 谁决定？

参考回答
平台根据时长、分辨率、内容类型和成本选择要生成的 bitrate ladder，播放器再按实时吞吐量选择其中一个 rendition。不能让客户端任意指定 profile，否则容量和体验不可预测；新 ladder 通过版本化配置逐步回填。
常见失分
只列 360p、720p、1080p，没有说明选择和版本。
继续追问
低端网络用户为何不应强制下载最高码率？
06

4K 长视频造成 starvation 怎么办？

参考回答
长视频会长期占用 GPU/CPU，不能与短视频共享无界 FIFO 队列。按媒体类别和租户做队列、并发配额与优先级，并把单个转码拆成可恢复阶段；代价是调度更复杂，但能保护正常上传的完成时间。
常见失分
只说增加 worker。
继续追问
付费创作者的优先级怎样不饿死免费用户？
07

asset metadata 和对象存储如何处理原子性？

参考回答
不跨两个系统做二阶段提交，而是把 metadata commit 作为用户可见的线性化点。blob 成功但 commit 失败只留下孤儿对象，可重试或回收；commit 成功后读路径永远从 metadata 找到被引用的版本。
常见失分
声称上传完成即天然原子。
继续追问
commit 成功但 event publish 失败怎么补发？
08

CDN 为什么不是 source of truth？

参考回答
CDN 是可丢失的缓存副本，命中并不证明内容仍被允许访问。它只能缓存版本化、已授权的响应；删除和 ACL 修改要通过签名、TTL 和 purge 收敛，权威判断仍回到控制面。
常见失分
把 CDN 当成存储数据库。
继续追问
公开视频改私密后如何缩短泄露窗口？
09

删除视频如何全链路收敛？

参考回答
先将 asset/post 标为不可见并拒绝新签名请求，确保新读立即失败。随后以带重试的删除工作流清理搜索、feed 引用、对象副本和 CDN；这是最终收敛，必须有审计和 backlog 指标。
常见失分
只删除对象存储文件。
继续追问
已下载到用户设备的副本属于什么边界？
10

热门视频 cache miss 风暴如何防？

参考回答
热点首次失效时，边缘节点不能同时回源数万次。使用 request coalescing、origin shield、分层缓存和有限 stale-while-revalidate，把回源收敛到少数请求；代价是短暂返回旧 manifest。
常见失分
只说多加 CDN 节点。
继续追问
新版本发布时怎样避免所有边缘同时 miss？
11

转码区域故障如何恢复？

参考回答
任务必须在 durable queue 和状态表中，而不是存在 worker 内存。区域失效后，调度器在其它区域重新 claim 未完成任务；已完成 rendition 用版本键检查，避免重复发布，用户看到的是处理延迟而不是数据丢失。
常见失分
依赖 worker 自动重启。
继续追问
跨区对象尚未复制时怎么办？
12

直播和点播的根本区别？

参考回答
点播在发布前可以等待完整文件和所有 rendition，直播必须持续切片、近实时转码并容忍某些 profile 延后。直播的状态是不断增长的 session 和 segment window，因此延迟预算比存储吞吐更关键。
常见失分
认为两者只是不同文件大小。
继续追问
直播断流后播放器如何恢复？
13

内容审核放在发布前还是后？

参考回答
高风险内容应在发布前同步或快速异步阻断，代价是发布延迟和误杀。低风险模型可先发布后复核，但必须能撤回、通知和清理缓存；选择取决于风险级别而不是统一策略。
常见失分
回答“都用 AI 审核”。
继续追问
模型置信度边界如何进入状态机？
14

自适应播放如何工作？

参考回答
服务端生成多个版本和 manifest，客户端按当前吞吐、buffer 余量和设备能力选择下一个分片的码率。切换在分片边界发生，优先避免 rebuffer；代价是低带宽用户看到较低清晰度。
常见失分
让服务端为每个用户持续转码。
继续追问
怎样量化 rebuffer 与画质的取舍？
15

你会监控哪些用户语义指标？

参考回答
先看上传提交成功率和 upload-to-publish P95，因为它们对应创作者是否能发出内容。播放侧看 startup time、rebuffer 和 CDN/origin 错误；删除侧看撤销到各副本收敛的延迟，不能只看 CPU 和 QPS。
常见失分
只报 worker CPU 与队列深度。
继续追问
哪个指标最早发现某个 region 的转码堆积？
16

Kafka exactly-once 是不是必须？

参考回答
媒体发布真正需要的是同一 asset version 不重复可见，不是消息系统的抽象承诺。使用 at-least-once 消费、稳定任务键和条件状态转换即可实现；exactly-once 仍不能覆盖对象存储、通知和外部审核的全部副作用。
常见失分
把技术名词当作正确性证明。
继续追问
哪些副作用最需要单独做幂等？
每个节点答不出具体状态、错误语义或恢复路径，回到讲义对应段落修复。
