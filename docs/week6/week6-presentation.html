Day 1 · Week 6

文件元数据模型

content-addressed chunk； path namespace； device cursor。
核心 invariant：chunk 由 hash 唯一标识；同内容只存一份；path 指向当前 version_id

参考原文 Dropbox · Entities Data Modeling Large Blobs 本日核心追问
需求边界
  • 先锁定 3 个核心动作：content-addressed chunk、path namespace、device cursor
  • 明确 out of scope，避免把后台运营、推荐、BI 一次性塞进主链路
  • 读写比例、峰值 QPS、对象规模、保留周期必须给数量级假设
  • 先定义用户可见语义，再讨论 Kafka、Redis 或数据库选型
正确性不变量
  • chunk 由 hash 唯一标识；同内容只存一份；path 指向当前 version_id
  • 每个状态转换写 precondition；失败重试不能跳过中间状态
  • 数据库事实、缓存视图、异步派生数据必须标出 authoritative source
  • 重复、乱序、超时、部分成功四类故障逐一给答案
容量与 SLO
  • Week 6 Day 1：先估峰值写 QPS，再按分区键检查热点
  • 在线路径给 p50 / p99；后台 repair 给 freshness 与 backlog SLO
  • 吞吐扩容不能破坏 per-key 顺序、唯一性或余额守恒
  • 压测必须覆盖 skew：最热 1% key 承担 40% 流量
层次 面试中要落到的机制 必须暴露的指标 失败时动作
API 幂等键、分页游标、版本号、错误码 request_p99 / 409_rate 限流、返回可重试语义
Storage 主键、分区键、二级索引、唯一约束 lock_wait / hot_partition 迁移流量或拆热点
Async 事件 ID、顺序键、consumer checkpoint lag / redelivery / DLQ 重放并保持幂等
Derived watermark、版本、last_updated_at freshness / mismatch_rate 从 source of truth 重建
Repair chunk 引用计数归零才 GC；GC 前 verify 无 file_version 引用 stuck_age_p99 / repair_count 先止血，再对账，再回填 
File { path, owner, current_version_id }
FileVersion { version_id, chunk_hashes[], mtime }
Chunk { hash, size, ref_count }

device_cursor { user, device, last_change_seq }
脆弱答案：只讲 happy path
画完组件图就结束；没有唯一约束、状态机、重试边界，也没有说明超时后客户端该不该重试。
Senior 答案：不变量驱动
推荐
围绕「chunk 由 hash 唯一标识；同内容只存一份；path 指向当前 version_id」展开：写入原子点、异步传播、可观测性、自动修复与人工闸门形成闭环。
01
Scope

用 5 分钟确认 content-addressed chunk 与 path namespace 的成功语义，写出明确 out-of-scope。

02
Model

画实体、主键、状态与版本；指出哪张表或哪条日志是最终事实。

03
Path

先走一遍写路径，再走读路径；在跨服务边界标 idempotency key 和 timeout。

04
Failure

注入 consumer 宕机、重复消息和网络分区；用 chunk 引用计数归零才 GC；GC 前 verify 无 file_version 引用 收敛。

05
Scale

最后才做分片、缓存、批处理；逐项证明优化没有破坏正确性。

def reverse(head):
    prev = None
    while head:
        nxt = head.next
        head.next = prev
        prev, head = head, nxt
    return prev
