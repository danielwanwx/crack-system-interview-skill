2 分钟中文讲译预听

第 3 周导览：实时不等于不落盘

不自动生成

本周从 WhatsApp、FB Live Comments 和 Online Chess 练三种实时系统。共同点是用户期待马上看到变化，不同点是消息要保证可达、直播评论允许在超级热点时采样、棋局则必须维持严格轮次与公平时钟。通读日先完整画出发送、广播、落子三条主路径，并标记事实在哪里提交：消息表和设备确认水位、按直播排序的评论日志、按对局顺序接受的棋步。WebSocket、SSE、pub/sub 和连接目录都只是传输与路由工具；连接断开时，真正决定能否恢复的是持久日志、单调游标和幂等确认。深挖日要主动回答连接落在哪台服务器、服务器扩缩容时如何重新路由、通知丢失后怎样补读、慢客户端怎样背压。还要识别热点形态：十亿用户是连接规模，超级直播是单对象广播规模，五十万棋局是大量独立单写者。不要用一个万能队列抹平差异。mock 会故意制造 Redis 通知丢失、SSE 断线、游戏服务器双主和时钟偏差；你的答案必须说明用户看到什么、事实是否受损、如何从 last sequence 恢复。周末复盘时，用同一张表比较三题的顺序范围、可丢数据、重连协议与核心指标，直到你能清楚解释“实时体验”和“可靠事实”为什么是两条不同但相连的路径。使用时先暂停音频，凭记忆画出事实状态、提交点和故障分支，再继续核对。每天结束前必须留下可复查的图、容量计算和一条 repair 记录；如果答案只能说出组件名，就回到原文对应标题，把请求时序与用户可见结果补全。周末的综合 mock 不追求背诵固定架构，而要在约束变化后仍能守住不变量、说明取舍，并用指标决定降级、扩展和回滚。
