01

什么定义一个 time series？

参考回答
同名 metric 只要 label 组合不同，就是不同的 time series；时间点上再写入 sample。这个定义直接决定索引、分区和基数预算，因此 label 不是随意附加的维度。
常见失分
认为每个 metric name 只有一条序列。
继续追问
instance label 为什么通常可接受？
02

为什么 user_id label 会毁掉系统？

参考回答
user_id 几乎无上限，会让每个用户生成永久/长期 series，写入索引和内存远超样本值本身。用户级分析应进 logs/traces 或专门事件系统；metric label 应是低基数、可枚举维度。
常见失分
只加一个更大集群。
继续追问
如何在入口阻止高基数？
03

写路径为什么需要 WAL？

参考回答
WAL 在接收成功后先以顺序写保存样本，使节点崩溃时可恢复尚未刷盘的内存数据。它也让批量 compact 与索引更新异步化；但 WAL 满了必须背压或降级，而不是无界增长。
常见失分
只把数据放 memory buffer。
继续追问
WAL 损坏如何恢复？
04

按时间还是按 series 分区？

参考回答
时间 bucket 有利于保留、压缩和范围扫描，series hash 有利于把不同序列分散到节点。实践常组合两者，并为热点 series 加副本或额外切分；选择必须从查询和写入形态推出。
常见失分
说一致性哈希而不谈时间。
继续追问
全局查询过去一年如何限制 fanout？
05

跨 shard 查询怎样保护系统？

参考回答
coordinator 将查询拆给相关 shard，各 shard 先做 downsample/aggregation，再合并少量结果。入口为时间范围、匹配 series 和返回点数设置预算；否则一个 wildcard dashboard 能同时压垮所有历史分片。
常见失分
让 coordinator 拉回全部 raw samples。
继续追问
慢 shard 如何处理部分结果？
06

rollup 对告警有什么影响？

参考回答
五分钟平均可能抹掉十秒错误尖峰，也可能把噪声平滑成安全，因此告警规则必须声明读原始还是 rollup。长期趋势图可用低分辨率节省成本，实时 SLO 告警通常不能盲用同一层。
常见失分
认为 rollup 只是压缩。
继续追问
p99 指标能否简单平均？
07

out-of-order sample 怎么办？

参考回答
采集端重试和网络延迟会让旧 timestamp 晚到，存储可在开放 time block 内重排/合并。超过窗口的样本可拒绝、旁路或写修正表；策略要与告警和查询一致，不能静默任意覆盖。
常见失分
假设所有样本按序到达。
继续追问
同 timestamp 不同值如何裁决？
08

metrics 为什么不适合事件原文？

参考回答
metrics 设计为低维度聚合和快速时间序列查询，无法高效保存每次请求的完整上下文。请求级诊断应走日志或 trace，通过 exemplar/ID 关联；硬塞进 label 会同时失去两边的优势。
常见失分
给 metric 加 payload label。
继续追问
如何从告警跳到具体 trace？
09

retention 删除如何不伤查询？

参考回答
数据按时间段组织后，可在保留期到达时标记/删除整段并让 compaction 回收，避免逐点删除。查询 planner 知道哪些 raw/rollup 段存在，并对超出保留期的请求返回明确边界。
常见失分
在线逐行 delete。
继续追问
合规删除某租户数据怎么办？
10

dashboard refresh 如何扩展？

参考回答
大量仪表盘往往重复查询相同窗口，应以 query fingerprint 缓存短期结果并合并并发请求。客户端 refresh 加 jitter，避免整点 thundering herd；告警计算与人类 dashboard 还应走不同资源池。
常见失分
给每个用户单独跑查询。
继续追问
缓存会让告警延迟吗？
11

何时转 logs/traces？

参考回答
当问题需要知道“哪一个用户、哪一个请求、哪段调用链”时，应记录 log/trace，而不是增加 label。metrics 用来快速发现趋势和触发告警，三种信号以 trace_id 或 exemplar 连接。
常见失分
所有观测都存一种系统。
继续追问
错误样本如何抽样不丢关键请求？
12

Strava 如何拆建模？

参考回答
一次骑行/跑步保存路线、时间、隐私和统计等活动事实，随后异步构建好友 feed、区段榜和个人统计。这样删除或隐私变更可以重新派生投影；不要把一条活动压成 time-series label 后无法表达权限。
常见失分
所有数据写进 TSDB。
继续追问
活动 route 数据如何做隐私裁剪？
13

隐私如何影响 activity 查询？

参考回答
活动和路线有 owner、可见范围与地理隐私规则，feed/leaderboard 查询必须带调用者上下文。隐私变更通过事件撤销投影和缓存；宁可短暂少展示，也不能暴露已设私密的路线。
常见失分
只在前端隐藏活动。
继续追问
公开活动改私密后怎样清 cache？
14

backfill 如何不压在线？

参考回答
回填使用低优先级或独立资源池，按时间/租户分段并记录 checkpoint，实时 ingest 始终优先。写入放大、compaction backlog 和 query P99 必须作为闸门；否则修历史会破坏当前监控。
常见失分
夜间全量写入。
继续追问
怎样验证 backfill 没有重复样本？
15

alert evaluation lag 为什么独立监控？

参考回答
样本能被存储但 rule engine 堵塞时，用户会以为系统健康却收不到告警。因此 evaluation lag、规则失败和通知投递都应有单独 SLO；它们不是 TSDB QPS 的副产品。
常见失分
只监控 ingest success。
继续追问
告警引擎落后时如何降级？
16

核心 SLO 是什么？

参考回答
要同时看接收是否丢样本、是否因高基数拒绝、查询是否在预算内返回，以及最新 sample/告警评估是否陈旧。按 tenant 和 metric family 分解这些信号，才能区分容量问题与某个坏 label。
常见失分
只看集群磁盘。
继续追问
哪项指标触发写入保护模式？
