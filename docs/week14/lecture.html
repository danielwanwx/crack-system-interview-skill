答题主线

01 · 核心判断

定义 series 预算

一个 metric name、完整 label set 和时间点共同定义 time series sample；label cardinality 决定存储和查询爆炸风险。把 user_id、request_id 放进 metric label 会把监控系统变成不可控的事件库。

02 · 核心判断

写路径

agent 批量采样后经 gateway 认证、限流和压缩，写入 WAL/queue 再落 TSDB。WAL 让短暂存储/节点故障可恢复，buffer 吸收尖峰；但每一层都要有背压，不能无限堆积。

03 · 核心判断

分区与存储

写入常按 time bucket 与 series hash 组合分区，使顺序写和查询并行成为可能。选择只按时间会让高热 series 集中，选择只按 series 会让时间范围查询昂贵，因此需要权衡和副本。

04 · 核心判断

rollup 与保留

原始高分辨率数据保留较短，长期查询使用预聚合 rollup；保留期是数据合同的一部分。rollup 会丢失峰值细节，告警评估需要明确读取原始还是聚合层，不能事后发现误报。

05 · 核心判断

查询控制

dashboard 查询常跨 shard，需要并行 fanout、局部聚合和合并，同时限制时间范围、series 数和结果点数。查询预算保护多租户与存储，不能让一个 wildcard 面板拖垮整个集群。

06 · 核心判断

乱序与回填

网络抖动会产生 out-of-order sample，存储需设迟到窗口和合并规则。历史 backfill 应走隔离/限速路径，避免把正常的实时写和 compaction 一起压垮。

07 · 核心判断

迁移到 Strava

Strava activity ingestion 是带 route/隐私的事件事实，feed、统计和 leaderboard 是投影。它可复用高写入、时间窗口与派生视图思想，但活动本身不应被压缩成只有 label 的 metric。

08 · 核心判断

运营边界

监控平台必须监控自身：ingestion loss、query P99、compaction backlog、cardinality 拒绝和 alert evaluation lag。否则最需要它时，系统可能悄悄丢掉正告警的序列。
