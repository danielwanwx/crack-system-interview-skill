Week 1｜Bitly 与 Dropbox 两分钟预听

不自动生成

这一周用两道看似简单、实际边界完全不同的题建立系统设计的基本动作。Bitly 是极端读多写少：创建短链时要保证 short code 唯一，跳转时则要在一百毫秒内从短码找到原始 URL。先从 API 和访问路径反推数据模型，再判断 Postgres 的唯一索引是否已经足够；只有读流量和热点证明确实需要时，才加入 cache、replica 或 CDN。Dropbox 的困难不是把文件放进对象存储，而是把大字节内容、metadata、权限和多设备同步拆成可恢复的协议。客户端用预签名 URL 分片上传，完成后再以 idempotency key 提交 metadata；对象已经上传但提交失败时，保留可重试状态并延迟回收孤儿 blob。同步正确性来自 change log、cursor 和 pull repair，WebSocket 只负责更快通知。两题放在一起训练一个核心判断：组件名不是答案，必须说清权威状态在哪里、每条请求如何改变状态、失败后用户看到什么，以及什么时候才值得增加复杂度。试听后请先关掉讲义，用一张白板分别重建两题的权威状态和失败路径；只有能解释缓存过期、孤儿对象与权限撤销的修复上界，才算真正完成这一周。
