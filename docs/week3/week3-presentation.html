参考原文 Metrics · 功能需求 TSDB · 数据模型 TSDB · Rollup DDIA Ch12 · 流处理
指标类型 — 面试必会三种
Counter 计数器
只增不减（可 reset）。适合：请求总数、错误次数、点击数。

查询时用 rate() / increase()，不要直接画原始累计值。
Gauge 仪表盘
可上可下。适合：CPU%、内存用量、队列深度、当前连接数。
Histogram 直方图
分布统计。适合：延迟分位数 p50/p95/p99。

客户端按 bucket 上报，服务端聚合分位数。
一条样本长什么样
// Prometheus 风格 — metric identity = name + labels
{
  "metric": "http_request_duration_seconds",
  "labels": {
    "service": "checkout",
    "method": "POST",
    "status": "200"
  },
  "value": 0.042,
  "timestamp": 1720886400
}
Tag Cardinality 基数爆炸
  • 每个 user_id 当一个 label → 百万级时间序列
  • 存储、索引、查询成本指数上升
  • 解法：cardinality budget、采样、高基数 label 放日志而非 metric
Raw → Rollup 两层存储
  • Raw：10s 粒度，保留 7 天
  • Rollup：1min / 1h / 1d 预聚合，保留更久
  • 查询远时间范围 → 走 rollup，不走 raw scan

Day 3

Ad Click
Aggregator

经典数据管道题：10k 点击/秒峰值，广告主按 1 分钟粒度查报表。
核心不是「存 click」，而是 ingest → 去重 → 窗口聚合 → 低延迟查询。

参考原文 Requirements Dedup Reconciliation Kafka · 架构 DDIA Ch12 · 窗口
Click Event Schema
ClickEvent {
  event_id      UUID       -- 可选，用于幂等
  impression_id UUID       -- 去重键：每次展示唯一
  ad_id         string
  timestamp     epoch_ms   -- event time，不是 processing time
}
完整数据流
flowchart LR
    U[User Click] --> CP[Click Processor]
    CP -->|验证 HMAC| DEDUP{Impression
已见过?}
    DEDUP -->|否| K[(Kafka)]
    DEDUP -->|是| DROP[丢弃]
    K --> F[Flink 窗口聚合]
    F --> OLAP[(OLAP 预聚合表)]
    OLAP --> ADV[Advertiser 查询]
    K --> S3[Raw → Data Lake]
    S3 --> REC[每日 Reconciliation]
    REC --> OLAP
    style DEDUP fill:#fff4cc,stroke:#ff9f0a,color:#7c3d00
    style OLAP fill:#d1f4e0,stroke:#34c759,color:#1a6335
Batch Only
延迟高
每 5 分钟 Spark 批处理。实现简单，但广告主总看到几分钟前的数据；流量尖峰时 batch 积压更严重。
Stream + Batch 校正
推荐
Flink 实时聚合（速度层）+ 每日 Spark 对账（批处理层）。Lambda 架构：快且最终正确。Reconciliation 修正流处理误差。
At-Least-Once 下如何避免重复计数
1
Impression ID 每次广告展示生成唯一 ID，签名后随点击上报
2
写入流之前去重 Redis 查 impression_id；跨分钟窗口去重必须在 ingest 侧完成
3
先写 Kafka 再写缓存 缓存失败可容忍少量重复；丢 click 不可接受
4
Reconciliation Raw events 落 S3，日批重算作为 source of truth
⚡

Event Time vs Processing Time：迟到事件应归入实际发生的分钟桶，不是到达时刻。Flink 用 watermark 判断窗口何时可以关闭；允许少量 lateness 缓冲。

三种方案对比
方案思路延迟成本误差
Heap + 全量计数内存维护 Top K 堆低内存 O(视频数)精确
分片 Counter按 video_id 分片并行计数中中高精确
Count-Min Sketch概率数据结构估频率极低百 MB 级可控误差
Bloom Filter
判断「一定不存在 / 可能存在」。适合去重预判，有误判无漏判。
Count-Min Sketch
估计元素频率。Top K 候选集过滤，内存远小于全量 hash。
HyperLogLog
估计唯一访客数（UV），不用于排序 Top K。
📖

回填路径：在线聚合只保最近窗口；历史 Top K 用批处理 backfill（MapReduce / Spark）从原始事件重算。面试要说清「在线近实时 + 离线校正」的分工。

三层保留策略
Hot · 原始 10s
SSD，保留 7 天。告警和近期 dashboard 用。

消费语义
  • At-most-once：先 commit offset 再处理 → 可能丢
  • At-least-once：先处理再 commit → 可能重复
  • Exactly-once：需要幂等 + 事务，成本高
Consumer Lag
生产速度 > 消费速度时 lag 堆积。告警阈值 + 自动扩容 consumer。Hot partition 是常见根因。

预热卡 — 四个必答点
Partition Key 选型
Ad Click 用 ad_id 保序；Metrics 用 host_id。错误 key → 乱序或热点。
Watermark
「事件时间已推进到 X，X 之前的窗口可关闭。」允许 max lateness 缓冲迟到事件。
Checkpoint 代价
状态快照到 S3；恢复时从 checkpoint replay。窗口越小，丢状态代价越低。
何时不需要 Flink
分钟级延迟可接受 → Spark 批处理可能更简单。先问「要多实时？」

Stack
栈模式

本周算法 tag：Stack。核心模式：匹配消除、单调栈、表达式求值、状态模拟。
识别信号：括号匹配、Next Greater、直方图面积、碰撞消除。

题目链接 Valid Parentheses Daily Temperatures Largest Rectangle Generate Parentheses
两种栈思维
匹配 / 消除栈
遇左括号入栈，遇右括号配对弹出。不匹配则失败。
stack = []
for c in s:
    if c in '({[': stack.append(c)
    elif not stack or not match(stack.pop(), c):
        return False
单调栈
维护递增/递减序列。每个元素入栈一次出栈一次 → O(n)。
# Daily Temperatures: 找右侧第一个更高温
while stack and T[i] > T[stack[-1]]:
    j = stack.pop()
    ans[j] = i - j
stack.append(i)
