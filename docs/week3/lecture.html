文件同步与协同编辑

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

范围与一致性边界

先把一致性边界说窄：一次文件提交必须让 metadata 版本原子前进；跨设备何时看到新版本可以异步。这样既不把所有读做成强一致，也不会让两个版本被静默覆盖。Google Docs 的边界更细，提交单位是编辑 operation，不是整个文件。

02 · 核心判断

权威状态与数据模型

blob 与 metadata 必须分家。对象存储擅长保存不可变大对象；metadata store 才负责 path、ACL、当前 version、cursor 与 change log。把对象存储 list 当同步真相，会失去 rename、delete、权限和一致性语义。

03 · 核心判断

上传与提交协议

上传是两阶段协议：客户端先以预签名 URL 直传对象存储，得到 hash 后才请求 metadata commit。commit 带 idempotency key 与 base version；只有 commit 成功文件才对其他设备可见。孤儿 blob 不用分布式事务处理，延迟 GC 即可。

04 · 核心判断

Delta Sync

同步正确性依赖 cursor pull，而不是通知。每个 namespace 的 change log 有单调 sequence；设备只拉取 cursor 之后的变更，并且本地事务成功后才推进 cursor。push 只是加速器，丢了最多变慢，不会丢文件。

05 · 核心判断

冲突与离线编辑

冲突的关键不是避免并发，而是定义并发的用户语义。客户端提交 base version；若服务端版本已前进，通用文件生成 conflict copy 或要求合并。last-write-wins 看似简单，却会把真实用户数据悄悄删掉。

06 · 核心判断

扩展与分区

按 namespace 分区把 listing、ACL、change log 和大多数写入局部化。代价是超大共享目录会变成热分区，因此必须单独限流、缓存和拆分 change fanout；不能用平均 QPS 掩盖热点。

07 · 核心判断

故障、修复与可观测性

故障时只保护正确性路径：metadata 主库不可用则停止产生新版本；对已有版本可在权限允许下提供可能陈旧的只读。对象已上传但尚未 commit 的用户得到可重试状态，而不是假装文件已保存。

08 · 核心判断

迁移：Google Docs

迁移到 Google Docs 时，复用身份、ACL、通知和持久化基础设施；替换文件版本冲突语义为 operation ordering、补洞与会话级收敛。presence 是 TTL soft state，不应写入 metadata 主库。
