WhatsApp

WhatsApp：从需求语义到故障修复

01

消息先落 Inbox，再追求实时到达

发送接口为每条消息生成稳定 messageId，并记录 chatId、senderId、serverReceivedAt、内容引用和聊天内排序字段。对于最多一百人的群聊，可以在写消息后为每个接收设备写 Inbox 指针；消息正文只存一份，Inbox 保存 messageId、deviceId 与 delivered/acked 状态，避免复制大附件。服务端只有在消息及离线投递记录持久化后才向发送者确认 accepted，随后 Redis Pub/Sub 只是唤醒在线 Chat Server 的低延迟通道。这样订阅者暂时缺席不会让事实丢失，断线设备可在三十天窗口内按游标补取。附件上传走对象存储和签名 URL，聊天记录只引用 blob 元数据，发送链路不转发整段字节。

能迁移的判断：实时推送可以丢通知，消息与每设备待投记录不能只存在 Pub/Sub 中。
02

两亿长连接需要连接目录与稳定路由

十亿用户中若两亿同时在线，即使单台 Chat Server 维持一百万到两百万连接，也需要数百台主机，并额外预留发布和故障容量。WebSocket 入口先鉴权，再把 connectionId、userId、deviceId、serverId 和租约登记到连接目录。发送方不应广播给所有服务器：可按 userId 做一致性哈希，把同一用户路由到固定服务组；或由消息总线按用户主题把通知送到当前连接所在服务器。前者减少订阅数但再平衡复杂，后者更灵活却增加 broker 压力。服务器退出时先停止接收新连接，再让客户端带抖动重连，防止一台机器故障引发同时握手风暴。容量要计算文件描述符、每连接缓冲区、心跳流量和 TLS CPU，而不只是业务消息 QPS。

能迁移的判断：连接本身是需要分片、续租、排空和容量预算的状态。
03

多设备把“已送达”从用户级拆成客户端级

同一账号的手机在线、笔记本离线时，用户级 delivered 位无法回答笔记本还缺哪些消息。Clients 表按 userId 列出有效 deviceId、公钥、createdAt 与 lastActive，且应限制例如每账号三台设备；Inbox 也改为按 deviceId 维护游标或未确认项。新消息写入每个活跃设备的收件记录，在线设备立即推送，离线设备恢复后从自己的 cursor 同步。注销设备需要一个 membershipVersion，之后产生的消息不再写给旧设备；正在发送的批次若看到版本变化，必须按明确边界完成或重算。已读回执也要区分“某设备看到”和“用户任一设备看到”，再决定 UI 汇总语义。端到端加密场景还会要求为每台设备封装会话密钥，放大与设备数成正比。

能迁移的判断：设备是投递与加密的独立终点，不能用一个用户级 Inbox 假装所有客户端同步。
04

WebSocket 心跳、ACK 与补同步各解决一类故障

TCP keepalive 可能几分钟才发现半开连接，聊天应用可每十秒发送轻量 heartbeat，五秒未回应就判定失联；两亿在线用户意味着约两千万次 ping/pong 每秒，必须把这笔固定流量计入预算。服务器推送 messageId 后等待客户端 ACK，五百到两千毫秒内未确认可有限重发，因此 UI 要按 messageId 去重。连接关闭后客户端指数退避重连，并携带各聊天或全局 inbox cursor 拉取缺口。ACK 证明某设备收到，不证明用户阅读；heartbeat 证明近期通路可用，也不能替代持久 Inbox。切换网络时旧连接和新连接可能短暂共存，连接 epoch 决定哪条通道接受后续确认，避免旧连接晚到的 ACK 推进新会话游标。

能迁移的判断：心跳检测死链，ACK 缩短丢包恢复，Inbox 同步才提供最终补漏。
05

顺序目标是所有客户端可解释，而非还原物理发送瞬间

不同手机经不同 Chat Server 发消息，网络本就会乱序；要求全球绝对顺序会引入昂贵协调，而且用户通常更在意低延迟。可由接收消息的服务器写 serverReceivedAt，并以 (timestamp,messageId) 稳定排序；NTP 只保证大致接近，晚到消息偶尔插到上方是可接受产品行为。若需要可靠发现漏消息，可为每个 chat 分配单调 sequence，客户端看到 43 后直接收到 45，就按 44 的缺口请求 Inbox；安静聊天仍靠周期同步或心跳携带最高水位兜底。群聊迁移分片时，要让旧、新 owner 在一个 epoch 内交接 sequence，不能各自发号。客户端缓冲很短窗口即可平滑显示，无需等待所有可能迟到消息。

能迁移的判断：显示顺序、缺口检测和业务因果要分别定义，不能一句“按时间戳排序”包办。
06

last seen 应从连接事件派生，而不是每次心跳写库

若两亿用户每十秒心跳一次并都更新数据库，将产生每秒两千万次几乎没有业务价值的写，而且刚写完就可能过时。在线状态优先查询 Chat Server 的活跃连接目录；只有连接关闭、心跳判死或服务器排空时，才把 disconnectAt 写入 DynamoDB。一个用户有多设备时，只要任一有效连接存在就报告 ONLINE；最后一条连接消失才用条件表达式写入较大的时间戳，防止旧服务器的迟到事件把 last seen 倒退。服务器整机崩溃无法逐连接写断开时间，可按 server lease 到期批量推断，并把结果标记为近似。查询还必须经过隐私规则，例如只向联系人显示分钟粒度，避免把基础设施精度直接暴露成跟踪能力。

能迁移的判断：把高频在线信号留在内存，把低频断开事实有条件地落库。

FB Live Comments

FB Live Comments：从需求语义到故障修复

01

评论历史与实时事件要共享同一个可恢复游标

POST /comments/:liveVideoId 先验证 JWT 中的用户身份，把 commentId、liveVideoId、authorId、createdAt 和正文写入数据库。DynamoDB 可用 liveVideoId 作分区键、可排序 commentId 作排序键，历史读取采用 cursor 而非 offset；新评论不断插入时，offset 会让翻页重复或漏项。实时层发送的 event id 直接使用 commentId，用户加入直播先建立推送水位，再读取该水位之前的最近历史，避免“先查历史、后连流”之间出现空洞。评论库是补播依据，SSE 与 Pub/Sub 是低延迟派生通道。若允许最终一致，UI 可乐观显示本人刚提交的评论，但服务器拒绝后必须撤回，且客户端按 commentId 合并持久结果与实时副本。

能迁移的判断：用 commentId 贯穿写库、历史游标、实时事件和断线补播。
02

直播评论的单向流量更适合 SSE

绝大多数观众只接收评论，发言仍可走普通 HTTP POST，因此为每人维护双向 WebSocket 的能力并没有产生对应价值。SSE 建立长时间 HTTP 响应，服务器用 text/event-stream 推送 id 与 data，浏览器原生支持自动重连和 Last-Event-ID；代理层必须禁用响应缓冲，并设置足够长的空闲超时。每个域名的浏览器连接限制、移动端后台行为和负载均衡的流式支持都要验证。SSE 不是无成本：服务器仍保存每条连接的发送队列、订阅 videoId 和最近进度。若产品后来加入高频双向互动或游戏控制，再重新评估 WebSocket，而不是因为“实时”两个字默认选它。

能迁移的判断：协议选择来自读写方向、代理能力和重连语义，不来自流行度。
03

水平扩展的核心是只把评论送到真正有观众的服务器

轮询让每个观众持续打数据库；朴素 Pub/Sub 又让每台 Realtime Server 处理所有直播的所有评论，即使本机没有对应观众。更好的方法是按 hash(liveVideoId)%N 划分有限频道，并让 L7 负载均衡按 liveVideoId 把同一直播的观众尽量共置，使服务器只订阅本地需要的频道。另一种是 Dispatcher 维护 videoId→serverIds 的心跳目录，评论到来时直接转发到相关节点；映射陈旧会漏发，所以数据库历史仍要兜底。服务扩缩容时先迁移连接或双发短窗口，再更新订阅，不能在路由切换与频道切换之间留下空白。容量分别计算长连接数、每条评论跨服务器复制数和最终向客户端的扇出字节。

能迁移的判断：观众共置和有限频道降低无效处理，持久历史吸收目录短暂不准。
04

Mega-stream 应改变产品语义，而不是执着投递每一条

若一场直播每秒五千条评论，屏幕只显示二十条，那么每条平均停留约四毫秒，用户根本读不完。此时保证每人收到全部评论既浪费带宽也没有体验价值。第一层可按速度自适应采样，例如一百条每秒传一半、五千条每秒传百分之一，并提高关注用户或高质量评论的权重。更极端时服务器维护最近一两百条的 ring buffer，每秒生成快照放到 CDN，客户端轮询边缘并平滑动画；延迟从两百毫秒放宽到一两秒，却把数百万相同响应交给 CDN。切换阈值要有迟滞，例如高于十万观众进入、低于七万才退出，防止模式反复抖动；本人评论在本地先显示以保留读己之写。

能迁移的判断：在不可读的流速下，代表性与稳定体验比逐条可靠扇出更重要。
05

Last-Event-ID 把短断线变成有界范围查询

每个 SSE event 带唯一 commentId，浏览器重连时在 Last-Event-ID 中报告最后收到的值。Realtime Server 先查询该 liveVideoId 上大于游标的评论，按顺序补完后再接入当前流；为防止补播与实时并发造成重复，客户端按 commentId 去重，服务端在同一高水位处切换。短断线可从内存 ring buffer 读取，较长断线查 DynamoDB 游标，超过保留或缺口过大则返回最新快照并明确跳过旧内容。重连必须指数退避加抖动，一场赛事网络恢复时不能让百万观众同时查询历史。Last-Event-ID 只说明客户端最后确认的事件，不保证后面的所有内容已永久保存，所以补源仍是评论数据库。

能迁移的判断：断线恢复由事件游标、可查询历史和流切换水位共同完成。
06

两百毫秒预算下要控制慢观众与审核分支

端到端两百毫秒可拆成写入与鉴权五十毫秒、事件发布二十毫秒、服务器路由三十毫秒、连接排队与网络一百毫秒。某个移动客户端读得慢时，不能让它的发送缓冲阻塞同直播其他观众；每连接设置字节上限，超过后丢弃较旧的非关键评论并发送 resync-required，客户端改拉快照。评论审核若同步调用重模型会占满预算，可先做廉价规则与封禁检查，高风险内容延迟发布，后续审核删除则发送 tombstone 事件。观测按 liveVideoId 分桶：broadcast p99、每评论扇出数、SSE 队列字节、重连率、采样比例和从 Last-Event-ID 补回的数量。热门直播必须单独看，全球平均会掩盖真正的扇出热点。

能迁移的判断：慢连接要局部丢弃和重同步，不能把背压沿整棵广播树传播。

Online Chess

Online Chess：从需求语义到故障修复

01

一盘棋由单一权威状态机裁决

Game 记录 gameId、两名 playerId、初始规则、currentPosition、sideToMove、moveNumber、剩余时钟、status 和 version；Move 作为追加事件保存 SAN/UCI、客户端 requestId、接收时刻与结果。客户端发 move(gameId,expectedVersion,from,to,promotion)，Game Server 验证轮到谁、棋子移动合法、王未被将死，再以版本条件提交新局面。动画、音效和本地棋盘只是预测视图，服务器拒绝时必须回滚。对局内所有命令交给同一 actor 或同一分区顺序处理，比跨多个无锁服务拼接更易守住“一回合只接受一步”。定期快照 FEN，加上之后的 Move 日志，可在进程重启时重建，而无需每次重放整局。

能迁移的判断：服务器版本化棋局并裁决合法性；客户端预测不能成为最终棋盘。
02

公平匹配是等待时间与分差的动态交换

进入队列时保存 ticketId、playerId、rating、region、模式、timeControl 和 enqueuedAt。匹配器先在同一区域、相同模式和很窄分差内寻找；等待每增加几秒，允许的 rating 窗口逐步扩大，但跨地域前仍检查预计 RTT，避免为了快而生成明显不公平的对局。Redis sorted set 可按 rating 排序，另用时间或模式分桶；领取两张 ticket 必须原子删除，并以 ticketId 防止同一玩家同时匹配两场。大量玩家不应过早按 rating 细碎分片，否则相邻分片看不到彼此；可按大区和模式分池，再让协调者查询相邻 rating 段。取消与匹配并发时，以 match 状态条件写决定唯一结果。

能迁移的判断：搜索半径随等待扩大，评分、公平、延迟和重复匹配都要有可测规则。
03

五十万并发棋局按 gameId 分片并保留可接管证据

五十万局意味着约一百万 WebSocket 连接，Game Server 既维护棋局 actor，也运行两个计时器。入口通过 gameId 路由到 owner，连接目录记录 serverId 与 lease epoch；同一局的两名玩家必须到同一权威 actor，避免双方落子跨节点竞争。每步提交后追加 Move 日志，并每若干步或每分钟写快照；服务器崩溃时新 owner 读取最近 FEN、重放后续事件、恢复剩余时间，再让客户端以 lastMoveNumber 重连。正常迁移先停止接收新命令、刷新日志、转移 epoch，旧 owner 的晚写因 epoch 或 version 过期被拒绝。容量按连接、每秒落子、合法性计算和快照 IO 分开估算，因为大多数连接静默，不能用连接数等同于命令 QPS。

能迁移的判断：单局单 owner 保顺序，事件与快照让 owner 更换后能恢复。
04

棋钟以服务器单调时间结算，不按客户端倒计时扣款

服务器无需每毫秒持久化两个时钟。棋局保存 activeSide、remainingMs 和 turnStartedMonotonic；收到合法落子时计算 elapsed=nowMonotonic-turnStarted，扣除当前方时间、加 increment，再切换 activeSide。墙上时间可写日志但不参与间隔计算，避免 NTP 回拨。网络延迟不对称时，完全按到达时间会惩罚高 RTT 玩家；可采用客户端显示预测、服务器周期校准，并只补偿经过认证的有限传输估计，例如取近期 RTT 的一半且设置上限，绝不能信任客户端自报思考时间。断线不会暂停权威棋钟，除非产品规则明确提供断线宽限。每次响应带 serverRemainingMs 与 moveNumber，客户端平滑校正而非突然跳动。

能迁移的判断：权威扣时发生在落子边界，单调时钟与有上限的延迟补偿共同维持公平。
05

并发落子用 moveNumber 或 version 拒绝旧命令

玩家双击、手机与网页同时在线或网络重试，都会让同一方提交重复 move。请求携带 clientMoveId 与 expectedMoveNumber；actor 若已处理该 clientMoveId 就返回原结果，若 expectedMoveNumber 不等于当前值则返回最新局面，只有版本匹配且合法才追加下一步。两名玩家几乎同时发送时，轮到的一方先被接受，另一方会因 sideToMove 或版本失败，不需要给整盘棋加跨服务分布式锁。服务器故障后重放日志得到相同 moveNumber，重试仍可去重。WebSocket 只负责低延迟传输，不保证命令恰好一次；每条接受结果都要等持久事件提交后再广播给双方。

能迁移的判断：expectedMoveNumber 是棋局级乐观并发条件，clientMoveId 收敛网络重试。
06

排行榜是赛果的派生视图，评分更新才是事实

Game 结束时生成唯一 resultId，评分服务读取双方当前 rating，以同一 resultId 计算 Elo 或其他算法，并在事务中写两人的新 rating 与 ratingHistory；重复结束事件命中唯一约束后返回原版本，不能二次加减。Redis sorted set 用 score 保存当前 rating，支持 Top N 和按分数附近查询，但它是可重建视图：数据库事务提交后经 outbox 更新 ZSET，失败可重放。查询“我的名次”要说明并列规则，十 million 玩家直接 ZREVRANK 很快，但强一致地跨分片维护全局顺序很昂贵；可按赛季和模式分榜，允许数秒延迟。分页用 (score,playerId) 游标，不能用频繁变动的 offset。赛果撤销则生成补偿版本并重算相关历史。

能迁移的判断：resultId 保证评分只结算一次，Redis 提供快速排名而不取代可审计评分账本。
