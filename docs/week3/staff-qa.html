面试压力测试

先独立作答，再展开下一个追问。

01

为什么上传大文件不通过应用服务器？

参考回答
让客户端直传对象存储。应用服务只签发短期 upload session，并在 commit 时校验 hash、大小和 ACL；这样应用不承担大文件带宽，也不会因为一次重试写出两个版本。
常见失分
只说 CDN 或对象存储快，没有说明 metadata commit 才决定文件是否对其他设备可见。
继续追问
对象上传完成、metadata commit 超时时，客户端怎样重试而不产生两个文件版本？
02

blob 与 metadata 的原子性如何处理？

参考回答
我不做跨存储分布式事务。先写不可变 blob，再以 content hash、idempotency key 和 base version 原子提交 metadata；未被引用的 blob 延迟回收，commit 可安全重试。
常见失分
声称跨对象存储和数据库做分布式事务。
继续追问
GC 如何避免删除一个刚上传但网络延迟中的合法 blob？
03

客户端如何知道要同步哪些变化？

参考回答
每个 namespace 有单调 change sequence，设备保存 cursor 并拉取增量。通知只负责唤醒；通知丢失后，周期 pull 和 cursor replay 会收敛。
常见失分
把 WebSocket 通知当作唯一同步协议。
继续追问
用户一年未上线，cursor 过期或日志压缩后怎么办？
04

两个离线设备同时改同一文件如何解决？

参考回答
提交携带 base version。若当前版本变化，二进制文件生成 conflict copy 或交给用户合并；绝不静默 last-write-wins，因为那会制造不可察觉的数据丢失。
常见失分
简单说 last-write-wins，忽略数据丢失。
继续追问
哪些类型的文件可以自动 merge，哪些不可以？
05

rename、delete 和 restore 如何进入同步协议？

参考回答
rename、delete、restore 都是 namespace metadata event。delete 写 tombstone 而非立即物理删除，因此旧设备能够在 cursor 流中看到删除并安全收敛。
常见失分
只建 content 表，没有 path/namespace 版本。
继续追问
同一路径被 rename 与新建并发占用时如何裁决？
06

复制延迟会造成什么用户可见问题？

参考回答
commit 后的确认从权威 metadata 路径读；副本可短暂陈旧。权限撤销、当前版本确认等安全敏感读必须走强制校验，普通 listing 可以接受短暂 lag。
常见失分
要求所有读全局线性一致，未说明成本与范围。
继续追问
共享文件的 ACL 收回时，哪条读路径必须立即强制？
07

为什么按 namespace 分区？

参考回答
namespace 是目录、ACL、listing 和 delta sync 的自然事务边界。共享目录极热时，我会分离 read cache、fanout 和写配额，并为该 namespace 做子分区方案。
常见失分
按 file_id 随机分片，导致目录列举和 cursor 聚合昂贵。
继续追问
一个百万成员共享目录如何避免单分片过热？
08

客户端重连时如何避免重复应用变更？

参考回答
每个变更有稳定 ID 或 sequence，客户端只有在本地应用成功后推进 cursor。网络 ACK 丢失只会重放已见事件，应用逻辑按 version 或 ID 幂等。
常见失分
只依赖 exactly-once 推送。
继续追问
如果 cursor 已推进但本地落盘失败，下一次怎么修复？
09

权限变更如何保证旧设备不继续看到文件？

参考回答
ACL 是 metadata 权威字段。访问 blob 前检查 ACL 或短期签名；撤销时使 token/cache 失效并在 change log 传播。已下载到离线设备的内容不能魔法撤回，要明确产品边界。
常见失分
只在 UI 隐藏文件。
继续追问
离线客户端下载过的内容能否被收回？如何向面试官界定边界？
10

multipart upload 怎样校验完整性？

参考回答
multipart 记录每片与最终 content hash；commit 时验证最终对象。相同 hash 可以做存储去重，但不能绕过引用该 blob 的 ACL、审计和版本推进。
常见失分
只说使用 checksum，没有放到 commit 边界。
继续追问
hash 相同但用户无权限引用已有 blob 时怎么办？
11

通知服务丢消息是否会导致文件不同步？

参考回答
不会丢同步正确性，只会增加 freshness。因为 cursor pull 是 repair path；通知可以合并、去重、限速，移动端还可按电量控制 pull 周期。
常见失分
承诺通知 exactly-once。
继续追问
移动端长时间休眠时如何平衡电量与 freshness？
12

如何处理 hot shared folder？

参考回答
先识别 namespace 热点，再对 change fanout、metadata reads 和写提交分别限速。必要时拆子分区，但要维护目录与 ACL 的协调，而不是简单随机散列 file_id。
常见失分
笼统说加机器。
继续追问
子分区后全目录 listing 和 cursor 如何组织？
13

Dropbox 与 Google Docs 的冲突语义有何根本不同？

参考回答
Dropbox 的原子单位是文件版本，冲突可保留两个副本；Docs 的原子单位是 operation，必须让并发编辑收敛到同一文档状态。WebSocket 只是连接手段，不是差异本身。
常见失分
因为 Docs 用 WebSocket，所以它不同。
继续追问
什么时候中心化 sequencer 足够，什么时候考虑 OT 或 CRDT？
14

Google Docs 的 presence 为什么不能是数据库强一致记录？

参考回答
presence 是可失效的连接提示，用 heartbeat + TTL 即可。把光标和在线状态写进主库会产生高频写放大，却不能增加文档正确性。
常见失分
将每个光标位置同步写进主数据库。
继续追问
presence 更新风暴如何限流和聚合？
15

metadata 主库不可用时如何降级？

参考回答
metadata 主库不可用时拒绝会产生新版本的写；已有版本可受 ACL 约束地降级只读。上传已完成但未 commit 的用户看到 pending/retry，恢复后用同一 idempotency key 完成。
常见失分
把写请求排在内存队列并承诺永不丢失。
继续追问
对象上传已经完成的用户在故障窗口应看到什么？
16

你会监控哪些 SLO？

参考回答
我看 upload-to-commit completion、cursor lag、sync freshness、conflict rate、notification-to-pull delay 和 orphan blob rate。这些比 CPU/QPS 更能告诉我文件是否真的在收敛。
常见失分
只报 CPU、QPS 和错误率。
继续追问
哪个指标最早暴露某个 region 的 change log 卡住？
每个节点答不出具体状态、错误语义或恢复路径，回到讲义对应段落修复。
