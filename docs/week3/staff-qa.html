WhatsApp

Q1

Redis Pub/Sub 在发布瞬间丢了一条消息，为什么在线用户最终仍能收到？

为什么面试官会追：候选人把 Redis 频道当消息事实库，或无法解释订阅者断开时的数据去向。

详细回答

发送链路先把 messageId 与正文持久化，并为每个目标设备写入 Inbox，确认这些记录成功后才 publish。Redis Pub/Sub 只通知当前 Chat Server 立即推送，因此 at-most-once 的通知丢失不会抹掉 Inbox。客户端跟踪聊天 sequence 或 inbox cursor；收到 45 却没见 44 时立即发起 gap sync，心跳也可携带服务器端最高水位，安静聊天再以较低频率周期同步兜底。同步结果和实时推送可能重复，客户端按 messageId 去重。若持久化未成功则发送接口不返回 accepted，绝不能先 publish 再补写，否则订阅者会看到一条之后无法恢复的幽灵消息。

继续追问：两亿客户端每三十秒轮询会产生约七百万 QPS，你怎样把补漏频率压下来？

常见漏项：用 Redis 副本解释 Pub/Sub 耐久性，却忽略没有订阅者时消息本来就不会保存。

现场修复

把发送时序改成 Inbox commit→publish，并演练 publish 丢失、重复和聊天沉默三条恢复路径。

  • WhatsApp · 4) What happens if Redis fails to deliver a message?
  • Redis · Redis for Pub/Sub
Q2

同一用户有手机、平板和电脑，其中电脑离线一周；你的投递状态怎样建模？

为什么面试官会追：候选人仍使用 userId 级单一已送达标记，无法说明某一设备补消息。

详细回答

Clients 表维护 userId 到最多三台有效 deviceId 的映射，Inbox 或同步游标必须按 deviceId 保存。新消息正文只写一次，但为当时 membershipVersion 中的每台设备建立待投引用；手机和平板各自 ACK 后只推进自己的 cursor，电脑一周后仍能在三十天保留期内补取。注销电脑时记录 deactivatedAt 和新 membershipVersion，之后的消息不再为它放大；正在进行的发送按捕获到的版本完成，避免半批次随意变化。UI 若展示用户级 delivered，可以定义成“所有活跃设备收到”或“任一设备收到”，但底层证据不能合并丢失。加密密钥也按设备封装，使撤销设备具有可执行边界。

继续追问：新增第四台设备时，它应该回放全部三十天消息还是从注册时刻开始？

常见漏项：只让发送服务向多条连接广播，没有为离线设备保留独立游标和生命周期。

现场修复

为三个 deviceId 各画 Inbox 水位，再用注销和新增设备验证 membershipVersion。

证据
  • WhatsApp · 2) What do we do to handle multiple clients for a given user?
Q3

一台 Chat Server 维持一百万连接时突然宕机，怎样避免重连把剩余集群再次打垮？

为什么面试官会追：候选人只说负载均衡会把客户端分到其他服务器，没有重连控制与容量余量。

详细回答

正常运行不能把每台机器用满，至少预留失去一个故障域后的连接、内存和心跳容量。客户端检测断线后采用指数退避加随机抖动，例如从一到三十秒分散尝试，并复用认证令牌，避免一百万次 TLS 与鉴权同时发生。入口按 userId 一致性哈希选择健康服务组，过载实例返回带 Retry-After 的明确拒绝，而不是接入后立即断开。新连接携带 deviceId 与上次 cursor，先完成轻量注册再限速补同步；历史消息拉取与实时心跳使用不同资源池，防止补数据挤掉所有在线连接。运维看每秒握手、连接建立 p99、认证依赖饱和、同步队列年龄和每机文件描述符水位。

继续追问：一致性哈希把故障节点的用户集中映射到一个后继节点时，如何避免新的热点？

常见漏项：容量只按平均连接数配置，没有 N+1 余量，也没有客户端侧抖动和补同步限流。

现场修复

做一次百万连接断开的故障演算，分别限制握手、鉴权、注册和历史回放速率。

Q4

消息 45 先于 44 到达，你会在服务器等多久再交付，还是让客户端处理？

为什么面试官会追：候选人宣称网络消息天然有序，或为了绝对顺序无限缓冲。

详细回答

聊天产品通常选择尽快展示而非等待未知时长。服务器给同一 chat 的消息分配 sequence 时，客户端收到 45 且当前水位是 43，先暂存 45 并立即请求 44；在很短的重排窗口内补到就顺序展示，超时则显示已有消息并在 44 到来时插入。若系统只用 serverReceivedAt，也要加 messageId 形成稳定次序，并接受迟到内容偶尔上移。sequence 的主要价值是发现缺口，不是证明发送者物理点击顺序。分片迁移时旧 owner 停止发号、新 owner 从持久化高水位继续；重复 44 由 messageId 去除。窗口大小应由移动网络延迟分布和 UI 容忍度决定。

继续追问：如果 44 永远没到且聊天停止，客户端靠什么确认自己仍有缺口？

常见漏项：把 NTP 当成精确全序，或为极少数乱序牺牲所有正常消息的端到端延迟。

现场修复

写清 display order、gap detection、buffer timeout 和静默聊天补查四个独立规则。

Q5

为什么不在每个 heartbeat 都更新 last seen？多设备竞态怎样避免时间倒退？

为什么面试官会追：候选人用高频数据库写换取看似精确的在线状态，没有算放大。

详细回答

两亿连接每十秒一次心跳就是约两千万次每秒，全部写库既昂贵又仍然瞬间陈旧。在线状态直接由带租约的活跃连接目录回答；只有连接关闭或被 heartbeat 判死时才记录 disconnectAt。一个用户可能在手机断开的同时电脑仍在线，所以聚合器先检查是否还有有效 device lease；只有最后一条连接消失才更新。多个服务器迟到上报时，用 DynamoDB 条件表达式“仅当新 timestamp 更大”防止较早断开覆盖较晚值。整机失联则以 server lease 过期作为近似断点，并允许显示分钟级而非虚假的秒级精度。隐私层还可隐藏或模糊该字段。

继续追问：服务器网络分区但客户端实际仍在线，连接目录与数据库分别会报告什么？

常见漏项：只做写入节流，没有处理任一设备在线、迟到断开事件和服务器整机消失。

现场修复

用手机先断、电脑后断、旧服务器迟到上报的顺序测试条件更新。

Q1

为什么直播评论选 SSE 而不是 WebSocket？这个选择会在哪些基础设施上失败？

为什么面试官会追：候选人看到实时场景就条件反射使用 WebSocket，没有考虑单向读多写少。

详细回答

观众持续接收、偶尔才通过 HTTP POST 发言，SSE 的服务器到客户端单向流正好匹配，且基于标准 HTTP、浏览器可自动重连并携带 Last-Event-ID；WebSocket 的双向通道在这里没有明显收益。SSE 仍需验证代理和负载均衡是否缓冲流式响应、空闲超时是否会切断连接、浏览器每域连接限制以及移动端后台行为。服务器要定期发送注释 heartbeat 防止中间设备误判空闲，并给每连接发送队列设上限。若未来需求变成高频双向互动，或客户端环境不支持 EventSource，再以真实流量重新选择协议，而不是把 SSE 当永久教条。

继续追问：HTTP/2 多路复用会改善浏览器连接限制，但负载均衡粘性和流式超时如何配置？

常见漏项：只说 SSE 更轻量，未指出代理缓冲、连接上限、断线游标和发送队列。

现场修复

列一张 POST 与下行流量比例表，并在测试环境主动经过真实 CDN、代理和移动网络。

Q2

同一直播的观众散在两百台 Realtime Server 上，新评论怎样只送到相关机器而不是全局广播？

为什么面试官会追：候选人让每台服务器订阅所有评论主题，计算量随全站直播数增长。

详细回答

可以把评论流按 hash(liveVideoId)%N 放入有限频道，同时让 L7 入口按 liveVideoId 一致性路由，使相同直播的连接尽量集中；服务器只订阅本机观众涉及的频道，并在最后一个观众离开时取消。若共置不够灵活，Dispatcher 根据服务器心跳维护 videoId 到 serverIds 的集合，把 commentId 只转发给目标节点。扩容迁移时先让新节点建立订阅，再切部分连接，短时间允许双发，客户端按 commentId 去重；顺序反过来会产生漏发。映射短暂陈旧仍可由持久评论库与 Last-Event-ID 修补。要监控每评论跨节点复制数和空订阅命中率。

继续追问：一个频道包含一千个 videoId，其中只有一个在本机有观众，你怎样降低频道级无效流量？

常见漏项：创建每直播一个 Kafka topic，忽略百万直播带来的 topic 与动态订阅管理成本。

现场修复

用三个直播、四台服务器画订阅集合，再演示增加节点时双发与去重。

Q3

世界杯直播每秒五千条评论、数百万观众，你仍承诺每人两百毫秒内收到全部内容吗？

为什么面试官会追：候选人机械扩大广播集群，没有质疑用户是否能消费如此密集的信息。

详细回答

不会，因为二十条可见槽位意味着每条只停留约四毫秒，逐条全量投递既不可读又把扇出成本放大到天文数字。我会先按评论速率做自适应采样，让每位观众稳定看到可读的几十条，并提升关注者或高质量内容权重。超过例如十万观众或五百条每秒后，切到每秒生成一次最近评论快照并交给 CDN，客户端轮询边缘、按时间戳平滑播放；延迟放宽到一两秒。本人的评论本地乐观插入。进入和退出阈值分开，防止模式抖动。产品 SLO 从“全部送达”改为“代表性、可读且持续更新”，并公开采样率。

继续追问：采样后两个观众看到不同内容是否可以接受，审核与高质量排序怎样影响公平？

常见漏项：把更多 Pub/Sub 分区当唯一答案，没看到相同内容的 CDN 复用和需求降级空间。

现场修复

先算屏幕消费速度，再写出 SSE、采样、CDN 快照三档切换阈值与迟滞。

Q4

观众断网五秒后带 Last-Event-ID 重连，怎样保证补历史和新实时流之间没有空洞？

为什么面试官会追：候选人只说重连后查数据库，却没有定义查询水位与切换竞态。

详细回答

服务器接收 lastId=120 后，先在该 videoId 的事件序列上捕获当前高水位 H，例如 135；查询 (120,135] 并按 commentId 返回，再从 135 之后订阅实时流。订阅建立与读取高水位必须有可验证的交接协议，也可以先订阅缓冲、再查历史，最后按 id 合并。若补播期间 136 已到缓冲区，历史结束后继续发送，不会漏；重复则由客户端 commentId 集合去除。短缺口优先读内存 ring buffer，较长缺口查 DynamoDB 游标；超过最大补播条数就返回 snapshot-required。百万客户端同时恢复时加入重连抖动并限制历史查询并发，避免恢复路径压垮评论库。

继续追问：commentId 不是严格连续整数时，服务端如何表达高水位并查询范围？

常见漏项：把“先查历史再订阅”当成原子动作，遗漏两者之间刚发布的评论。

现场修复

用编号 120 到 136 逐条演算 subscribe-first 与 snapshot-first 两种交接方案。

Q5

一个移动观众读取很慢，SSE 发送队列不断增长；你是背压全直播还是丢它的数据？

为什么面试官会追：候选人没有每连接内存上限，或让最慢消费者决定广播速度。

详细回答

直播评论优先可用与实时感，不能让一条慢连接阻塞其他数百万观众。每个连接设置最大排队字节和最大事件年龄；超过阈值时先合并或丢弃旧的普通评论，发送包含最新 commentId 的 resync-required，然后关闭或让客户端拉最新快照。高优先级的系统公告可保留独立小队列。Realtime Server 的 event loop 只做非阻塞写，慢消费者隔离到自身连接，不把 backpressure 推回 Comment Management Service。观测 slow-consumer 数、丢弃事件量、队列驻留时间和重同步成功率；若某网络区域普遍变慢，则降低该区域采样率，而非不断扩充单连接缓冲导致内存耗尽。

继续追问：用户刚发布的本人评论被采样或丢弃时，如何维持读己之写体验？

常见漏项：无限缓存以追求不丢评论，最终因为一个慢客户端耗尽服务器内存。

现场修复

为连接队列写出字节、时间两种上限，并测试超过上限后的 resync 流程。

Online Chess

Q1

白方手机和网页几乎同时提交两步不同的棋，你怎样保证最多接受一手？

为什么面试官会追：候选人把并发落子错误地引用到 matchmaking，或只说 WebSocket 有序。

详细回答

两端请求都携带同一 expectedMoveNumber=26 和各自 clientMoveId，且按 gameId 路由到唯一棋局 actor。actor 串行检查 sideToMove、规则合法性和版本；第一条被接受后追加 Move 27 并把 version 改为 27，第二条随后因 expectedMoveNumber 过期被拒绝并收到最新 FEN。若同一请求因超时重发，clientMoveId 索引返回第一次的裁决，不再次走棋。即使 actor 模型用数据库落地，UPDATE game ... WHERE version=26 的乐观条件也只允许一个成功。广播必须在 Move 事件持久化后进行，防止双方看到后来无法恢复的局面。

继续追问：actor 在写事件成功、尚未回客户端前崩溃，重连请求具体得到什么？

常见漏项：把连接内帧顺序等同于跨设备业务顺序，没有棋局版本和幂等请求标识。

现场修复

用两个 expectedMoveNumber=26 的请求画条件提交，明确第二个返回 conflict 与最新状态。

Q2

两个玩家 RTT 分别为二十毫秒和三百毫秒，服务器怎样扣钟才不让客户端作弊？

为什么面试官会追：候选人信任客户端上报剩余时间，或完全忽略不对称网络造成的体验差异。

详细回答

权威值只在服务器：保存 activeSide、remainingMs 和 turnStarted 的单调时刻，合法落子抵达时扣除服务器观测到的 elapsed。客户端每秒动画只是预测，响应中的 serverRemainingMs 定期校准。若产品允许延迟补偿，可根据近期经服务端 ping/pong 测得的 RTT 估算单程时间，补偿值取有限窗口统计并设置严格上限；不能采用客户端声称的“我思考了多少毫秒”。网络极差时宁愿显示连接质量并让玩家选择更慢时限，也不能无限返还时间。NTP 调整只影响审计时间戳，计算间隔使用 monotonic clock，服务器迁移则把剩余值和最后持久化落子一起恢复。

继续追问：服务器在某回合中途崩溃，重启耗时五秒，这五秒算谁的？

常见漏项：每秒把时钟写数据库，既放大 IO 又仍不能解决客户端延迟与服务停顿语义。

现场修复

用一回合时间线计算无补偿、有限补偿和恶意客户端三种结果。

Q3

承载一万盘棋的 Game Server 宕机后，如何恢复局面、时钟和连接而不产生两个 owner？

为什么面试官会追：候选人只说客户端重连到另一台机器，没有持久状态与所有权 epoch。

详细回答

连接目录中的 server lease 到期后，协调层为受影响 gameId 分配更高 epoch 的新 owner；旧进程即使恢复也不能接受低 epoch 写。新服务器读取最近 FEN 快照，按 moveNumber 重放之后的 Move 事件，得到 currentPosition、sideToMove 和最后持久化的 remainingMs，再按预先定义的故障暂停或继续计时策略校正。玩家以 gameId、device token 与 lastMoveNumber 重连，新 owner 返回缺失落子和权威时钟。为了不让一万局同时重放打爆存储，快照间隔要限制每局重放长度，并按比赛剩余时间和连接到达分批恢复。所有客户端命令仍用 expectedMoveNumber 去重。

继续追问：旧服务器只是网络分区并未真正死亡，它怎样收到或被迫遵守新 epoch？

常见漏项：把 WebSocket 重连当成恢复全部，漏掉棋局日志、计时规则和僵尸 owner fencing。

现场修复

故障演练必须包含 lease 过期、新 epoch、快照重放、客户端补步和旧 owner 晚写。

Q4

匹配等待越久就放宽分差听起来简单，怎样防止重复匹配和跨地域高延迟对局？

为什么面试官会追：候选人只说 Redis sorted set 最近邻，没有原子领取、取消竞态或区域约束。

详细回答

ticket 包含唯一 ticketId、playerId、rating、region、mode、timeControl 与 enqueuedAt。匹配器先在同 region 和模式内查 rating 邻域，允许窗口随等待秒数扩大；只有等待超过阈值且双方预计 RTT 仍低于上限时才跨区。选中两张 ticket 后用 Lua 或条件事务同时确认两者仍为 WAITING、原子改为 MATCHED 并创建 gameId，任何一个已取消就整体失败重选。playerId 还有 active-ticket 唯一约束，阻止多设备重复排队。分池不宜按细 rating 段硬切，否则边界玩家互相看不见；应按区域、模式粗分并查询相邻分数范围。质量看等待 p95、rating 差分布、跨区 RTT 和取消竞态率。

继续追问：Redis 匹配池节点故障时，已经从队列移除但尚未创建 Game 的两名玩家如何回收？

常见漏项：只优化最快配对，没有把公平、网络质量和一次性领取写成原子条件。

现场修复

用匹配、取消同时发生的时序验证 ticket 状态机和 gameId 幂等创建。

Q5

一场比赛结束事件重复送达三次，如何让双方评分和千万玩家排行榜仍然正确？

为什么面试官会追：候选人直接对 Redis ZSET 做 ZINCRBY，没有可审计赛果和重复保护。

详细回答

Game 完成时生成稳定 resultId，评分服务在数据库事务中检查 resultId 唯一约束，读取双方同一版本 rating，计算并写两人的新值与 history；第二、三次事件命中已处理记录后返回原评分，不再增减。事务同时写 outbox，异步消费者用 ratingVersion 更新 Redis sorted set，重复更新是覆盖同版本，乱序旧版本被拒绝。排行榜因此可有几秒延迟却不会决定事实；ZSET 丢失时从评分表重建。我的名次用确定的并列规则和 (score,playerId) 游标查询。若赛果被撤销，不删除旧账，而是写补偿 result 并产生更高评分版本。

继续追问：两场属于同一玩家的比赛几乎同时结束，评分计算应按什么顺序串行？

常见漏项：把 Redis 原子自增当成跨两名玩家的一次评分事务，无法处理重复和审计。

现场修复

画出 result ledger、双人评分事务、outbox 和 ZSET 投影，重放同一 resultId 三次。
