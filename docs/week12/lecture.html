ChatGPT

ChatGPT：从需求语义到故障修复

01

先把 LLM 当成昂贵的黑盒依赖

这道题的范围是 ChatGPT 产品 serving，不是训练 transformer。核心实体至少有 conversation、message、generation run、model deployment 和 usage record。用户提交 prompt 时，服务先用 client_message_id 幂等写入 user message，再创建 run；只有消息持久化成功才能让历史对话可恢复。run 记录 queued、running、completed、failed、cancelled 等状态，并引用不可变的输入 conversation version。这样 API 超时重试不会重复生成或重复计费，模型 worker 也不需要承担业务真相。assistant message 只在 run 完成后成为正式版本；失败尝试保留 attempt、错误和 usage，便于恢复、对账与判断是否应向用户收费。这个边界也让模型部署替换不改变对话事实。

能迁移的判断：消息是可恢复事实，run 是昂贵且可取消的执行；两者不能混成一个请求。
02

流式输出优化的是 Time to First Token

用户更在意按下发送后多久看到第一个 token，而不是完整答案何时结束。API gateway 建立 SSE，worker 生成小批 token 后经过 stream service 推送；每个 chunk 带 run_id 和递增序号，便于检测缺口。服务必须处理 backpressure：慢客户端不能无限占用内存或 GPU，可设置缓冲上限、暂停读取或取消 run。断线后是继续后台生成、允许短窗口 resume，还是立即取消，要由产品和成本共同决定；无论哪种都要有显式状态，而不是依赖 TCP 连接是否存在。

能迁移的判断：SSE 只是传输，run 状态、序号、背压和取消才决定流是否可靠。
03

GPU 调度按成本而不是请求数

两个请求都算一次 QPS，但 500 token 上下文与 100k token 上下文的 prefill、显存和执行时间完全不同。admission controller 先按模型、上下文长度、预计输出、租户余额和服务层级估算 token-seconds，再进入分层队列。scheduler 可在同模型与相近长度间 continuous batching，减少 padding 浪费；同时用 weighted fair queue 或每租户并发上限避免重用户霸占 GPU。模型加载和 KV cache 占用也影响路由，不能只按 worker CPU 或连接数做 round-robin。调度结果必须携带租约与 deadline；worker 丢失时由 scheduler 判定重排，而不是让两个 attempt 同时输出并重复计费。

能迁移的判断：稀缺资源是 GPU memory 与 token-seconds，公平和容量都必须用它们表达。
04

过载时先保护边界，再谈扩容

GPU 不可能像无状态 Web 服务一样瞬间扩容。队列等待超过 SLO 时，系统应在 admission 层拒绝低优先级请求、限制 max output、提示用户重试或降级到更小模型；已经运行的付费请求不应被无序挤掉。每个阶段都有 timeout：排队、prefill、decode、stream idle。worker 心跳丢失后 run 是否安全重试取决于客户端是否已经看到部分 token；通常要创建新的 attempt 并保留旧 attempt 的 trace，避免把两段输出静默拼接。降级响应要明确模型变化和质量边界，并通过 queue age、拒绝率、取消延迟和回收显存时间证明保护动作确实生效。

能迁移的判断：过载策略必须明确谁被拒绝、谁能降级、何时重试以及是否重复计费。
05

长上下文是一条质量与成本曲线

每轮都把全部历史传给模型会让 prefill 延迟和推理成本随对话增长。最简单策略是保留最近窗口并截断旧消息，但可能忘掉关键承诺；摘要能压缩历史，却会引入信息损失和摘要漂移，因此必须保存原消息、记录 summary_version，并在用户编辑或模型升级时失效。稳定 system prompt 和历史前缀可做 prompt/prefix cache，命中要求模型版本、tokenizer、权限与内容 hash 一致。Senior 答案要说明哪种对话允许哪种损失，并用长对话 eval 验证，而不是只说“做 RAG”。

能迁移的判断：上下文策略同时是产品记忆语义、缓存一致性和成本控制问题。
06

Agent 延伸必须留在可信运行时

只有在用户任务需要模型动态选择工具和多步决策时才引入 agent loop；确定性的检索、分类或工作流优先用普通编排。工具 schema、参数校验、OAuth/secret、租户授权、费用上限和高风险确认都属于可信应用运行时，不能靠模型承诺。每一步记录 trace、tool input/output、policy decision 和最终结果，离线用可重放 eval 验证成功率、越权、循环和成本。这个延伸只服务 ChatGPT/Agent 项目，不应反向污染 Bitly、Dropbox 等通用系统设计路线。所有有副作用的工具还要具备幂等键、审批状态和人工可撤销的补偿路径。

能迁移的判断：模型提出动作，可信运行时决定动作是否允许并留下可审计证据。
