今日唯一产出物

交付

SSE 流、GPU admission/scheduling、按 token 成本公平限流、长上下文策略、取消与 overload failure matrix。

验收标准

  • 能解释 TTFT 与总生成时长不同，并处理 backpressure、断线、恢复和取消。
  • 调度键包含模型、上下文长度、预计输出和服务层级，不用普通 FIFO 掩盖 GPU 成本。
  • 摘要、截断与 prefix/KV cache 的质量、版本、隐私和失效风险都有 repair path。
不通过怎么修

把一次 generation run 从 admission 到 GPU 回收画成状态机；每个转移写 timeout、cancel、retry、metric 和用户状态。

真实追问 1

为什么要先持久化 user message，再创建 generation run？

触发：候选人让 gateway 直接调用模型，生成完才一次性保存对话。

先修：画 message 与 run 两个状态机，并标出 client_message_id 和 attempt_id。

真实追问 2

客户端关掉页面后，GPU 上的 generation 应该继续吗？

触发：候选人把 TCP 断开自动等同于任务取消，或永远继续。

先修：写 queued/running/cancelling/cancelled/completed 竞争条件表。

真实追问 3

为什么普通 FIFO 队列会让 GPU 利用率和公平性都变差？

触发：候选人只按请求到达时间调度，不看模型和 token 成本。

先修：用四个不同模型/长度/租户请求手算一次队列选择。
