Senior / Staff Drill

ChatGPT

Q1

为什么要先持久化 user message，再创建 generation run？

为什么面试官会追：候选人让 gateway 直接调用模型，生成完才一次性保存对话。

详细回答

先持久化消息能让客户端超时、stream 断开或 worker 崩溃后仍恢复用户意图。写入使用 client_message_id 幂等，随后创建引用该 message/conversation version 的 run；重试查询同一 run，而不是再次生成。assistant message 可以按 chunk 暂存，完成后原子标记 final；失败则保留 run 状态和可见错误。若生成后才保存，模型已经消耗 GPU 但服务可能没有任何事实记录，既无法恢复，也容易重复计费和产生两份回答。

继续追问：用户连续快速发送两条消息时，两个 run 的 conversation version 如何定义？

常见漏项：只谈数据库可靠性，没有幂等、状态和计费。

现场修复

画 message 与 run 两个状态机，并标出 client_message_id 和 attempt_id。

证据
  • ChatGPT · 2) Users should be able to view past chats and resume a conversation with context carried across turns
  • Data Modeling · Entities, Keys & Relationships
Q2

客户端关掉页面后，GPU 上的 generation 应该继续吗？

为什么面试官会追：候选人把 TCP 断开自动等同于任务取消，或永远继续。

详细回答

产品需要显式策略。若答案会保存到对话，短暂断线可继续并允许客户端按 run_id/sequence resume；若用户明确点 stop，或 stream 在 grace period 后无人消费，应发送 cancel，worker 在安全 token boundary 停止并释放 KV cache。取消是幂等的，race 到 completed 时返回最终状态。还要设置总 token、wall-clock 和 idle timeout，防止孤儿 run 长期消耗 GPU。指标应区分 user cancel、disconnect cancel、timeout 和 worker abort，以便判断体验还是容量问题。scheduler 只有观察到租约释放或 worker 被隔离后才重新分配显存预算，避免账面回收与真实 GPU 占用脱节。

继续追问：cancel 已发出但 worker 不响应，scheduler 什么时候能回收资源？

常见漏项：没有 grace period、resume 或资源回收证据。

现场修复

写 queued/running/cancelling/cancelled/completed 竞争条件表。

证据
  • ChatGPT · Cancelling a run and reclaiming the GPU
  • Real-time Updates · "How do you handle connection failures and reconnection?"
Q3

为什么普通 FIFO 队列会让 GPU 利用率和公平性都变差？

为什么面试官会追：候选人只按请求到达时间调度，不看模型和 token 成本。

详细回答

FIFO 会让一个超长上下文请求阻塞大量短请求，造成 head-of-line blocking；不同模型无法在同一 worker 上自由切换，长度差异过大的 batch 还会产生 padding 和显存浪费。我会先按 model/deployment 分池，再按 context bucket 组织可合批请求，同时在租户和付费层级间做 weighted fair scheduling。admission 使用预计 input/output token 和并发限制，防止重用户垄断。调度目标同时看 TTFT、tokens/s、GPU utilization、queue age 和 starvation，不能只优化利用率。

继续追问：高优先级请求持续涌入时，如何保证免费队列不会永久饥饿？

常见漏项：回答加更多 GPU，没有调度策略或成本单位。

现场修复

用四个不同模型/长度/租户请求手算一次队列选择。

证据
  • ChatGPT · 2) How do we route and schedule generation requests across GPU workers?
  • ChatGPT · 3) How do we keep heavy users from monopolizing GPUs while giving paid tiers a better experience?
Q4

长对话为什么不能简单地永远做摘要？

为什么面试官会追：候选人把 summary 当成无损压缩，未讨论版本与质量。

详细回答

摘要会选择性丢失细节，也可能把模型误解固化到后续每轮。应保留原始 messages 作为 source of truth，summary 只是带覆盖范围、模型和 prompt 版本的派生数据；新消息、用户编辑、模型升级或质量告警可触发重算。不同对话类型的损失容忍度不同：闲聊可更激进压缩，代码或法律上下文需要保留关键原文。要用长对话 golden set 评估事实保持、指令保持、TTFT 和成本，摘要失败时回退到更长原文窗口。

继续追问：两个并发 run 都想更新同一个 conversation summary，如何避免覆盖？

常见漏项：没有原始历史、summary version、失效或 eval。

现场修复

为 summary 写 source range、version、model、invalidated_at 字段并列出触发器。

证据
  • ChatGPT · 4) As conversations get longer, how do we control inference cost without making the assistant feel forgetful?
  • Caching · Cache Consistency
Q5

加入 tool calling 后，模型能否直接执行支付或删除动作？

为什么面试官会追：候选人认为模型输出了合法 JSON 就可以执行。

详细回答

不能。模型输出是不可信提议，可信 runtime 必须验证 schema、业务不变量、用户/租户授权、幂等键、费用和风险级别。支付、删除、外发消息等高影响动作需要明确确认或预先授权策略；secret 不进入 prompt，工具只拿最小权限短期凭证。每个 tool call 记录 trace 和结果，失败重试要区分只读与有副作用操作，避免重复执行。还要设置最大步骤、预算和循环检测，并用官方框架建议的 eval/guardrail 思路做回归。

继续追问：工具调用成功但结果回传模型前进程崩溃，重试如何避免重复副作用？

常见漏项：把 JSON schema 验证等同于权限和业务安全。

现场修复

为一个有副作用工具写完整 trust boundary、approval 与 idempotency contract。

证据
  • Building agents · Augmenting your agents with tools
  • Building effective agents · When (and when not) to use agents
