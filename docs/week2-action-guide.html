2 分钟中文讲译预听

第 2 周导览：先把高频状态说清楚

不自动生成

这一周的三道题看似分散，其实都在问同一个问题：大量并发请求到来时，哪一小块状态必须被正确推进。Rate Limiter 里是客户端和规则对应的令牌桶；Distributed Cache 里是键的版本、过期时间与当前所有者；Job Scheduler 里是某个计划时间对应的唯一执行实例。第一天不要急着说 Redis、哈希环或队列。先沿官方页面从问题理解读到高层设计，把实体、接口、同步承诺点和用户可见结果画成一条时序。第二天再进入深挖：令牌扣减为什么需要单键原子操作，缓存节点切换怎样阻止旧主继续写，调度租约过期后旧 worker 为什么必须被 fencing。你要反复区分事实状态和派生状态：响应头、缓存副本、队列通知、执行统计都能重建，不能拿来代替唯一事实。容量题也不要只报总 QPS，要算一次逻辑动作放大成多少读写、哪个键会先热、故障时还剩多少容量。mock 时我会追问超时后的结果未知、重复消息、时钟漂移和热门键；复盘必须写出不变量、故障注入、指标与修复，而不是追加更多组件。完成本周后，你应当能把高频基础设施题讲成可验证的状态机，并知道何时用一致性换正确，何时用近似和降级保护整体。使用时先暂停音频，凭记忆画出事实状态、提交点和故障分支，再继续核对。每天结束前必须留下可复查的图、容量计算和一条 repair 记录；如果答案只能说出组件名，就回到原文对应标题，把请求时序与用户可见结果补全。周末的综合 mock 不追求背诵固定架构，而要在约束变化后仍能守住不变量、说明取舍，并用指标决定降级、扩展和回滚。
