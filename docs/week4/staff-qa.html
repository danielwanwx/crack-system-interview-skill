16 个真实追问节点Senior / Staff

面试压力测试

先独立作答，再展开下一个追问。

01

为什么只保证 conversation 内顺序？

参考回答
用户感知的因果主要在同一会话；全局总序需要跨分区协调且无业务价值；每个 conversation 可独立分区和扩缩
常见失分
把 Kafka partition order 直接等同于所有用户的全局顺序。
继续追问
同一用户在两个群聊中的消息顺序需要关联吗？
02

发送成功的 ACK 代表什么？

参考回答
至少表示消息已在服务端 durable log 持久化；与设备投递、已读 receipt 分离；ACK 丢失时 client_message_id 支撑幂等重试
常见失分
把发送 ACK 说成对方已经收到。
继续追问
服务端持久化后 fanout 失败，用户界面如何展示？
03

断线重连如何不丢消息也不重复？

参考回答
客户端持有已应用 sequence/offset；按缺口 pull；客户端应用与 receipt 更新必须对同一 message_id 幂等
常见失分
依赖 WebSocket 自动重连。
继续追问
客户端已收到但 ACK 未送达，服务端会怎样？
04

多设备用户的 delivery completion 如何定义？

参考回答
每个设备独立维护 delivery state；产品可定义任一活跃设备或全部有效设备的展示语义；不要把设备级 receipt 覆盖成一个不可审计布尔值
常见失分
只保存 user_id 到 message_id 的一条已送达记录。
继续追问
设备被移除后，历史未投递任务如何处理？
05

小群和超大群的 fanout 怎么不同？

参考回答
小群 fanout-on-write 换取低读延迟；超大群会产生巨大写放大；可使用在线推送、分层 fanout、离线拉取或读时聚合
常见失分
声称一个策略适用于所有群规模。
继续追问
如何定义从哪一规模开始切换策略？
06

presence 为什么不能作为强一致状态？

参考回答
连接和心跳自然丢失；状态短命并允许近似；应设置 TTL 并让客户端容忍陈旧
常见失分
每次心跳都写事务数据库。
继续追问
用户多个设备同时在线时 presence 怎样聚合？
07

相同会话如何跨地域路由？

参考回答
使用稳定 conversation ownership；写入路由到同一逻辑 leader/partition 保证序号；failover 后要防止双 leader 和 sequence 重用
常见失分
随机选最近 region 写入，再异步合并。
继续追问
home region 故障时，用户发消息是阻塞还是降级？
08

发送者重试会造成什么问题？

参考回答
网络超时不代表服务器未持久化；client_message_id 作为幂等键；重复请求应返回原 message/sequence，不再追加
常见失分
仅依赖 HTTP retry count。
继续追问
idempotency key 的保留期怎么选择？
09

端到端加密会改变哪些设计？

参考回答
服务端无法基于正文做搜索、审核或去重；仍可保存密文、envelope metadata 和投递状态；密钥/设备变更会影响多设备投递与历史访问
常见失分
忽略加密后服务端能力受限。
继续追问
群成员变更后密钥和历史消息如何界定？
10

如何防止恶意用户导致群聊写入雪崩？

参考回答
按 sender、conversation、tenant 做配额；对超大群设置 fanout 和 buffer 上限；限流应在持久化和推送前尽早生效
常见失分
只说增加 Kafka partition。
继续追问
限流拒绝时如何让客户端安全重试？
11

为什么在线消息也要持久化？

参考回答
接收端可能在 ACK 前掉线；重连和多设备需要可回放的历史；durable log 才能定义 delivery 语义
常见失分
直接由 socket server 转发后丢弃。
继续追问
保留期和用户删除如何影响 log？
12

慢消费者如何处理？

参考回答
连接 buffer 必须有限；超过阈值可丢实时推送并令客户端从 durable log pull；不能让一个慢设备阻塞会话全体
常见失分
无限制缓存消息。
继续追问
怎样选择丢弃阈值并向客户端传达？
13

Online Chess 与聊天消息的状态约束差异？

参考回答
棋局移动必须按规则从当前权威状态验证；每个 room 可有 single writer/sequencer；冲突 move 不能简单保留两个分支给用户选择
常见失分
直接复用聊天的 eventual ordering。
继续追问
玩家断线重连后怎样恢复 board 和 move number？
14

棋局超时如何裁决？

参考回答
由权威服务器时钟判断；客户端时钟只用于展示；需要记录 move deadline 和 timeout event 以便审计/重放
常见失分
相信客户端倒计时。
继续追问
网络分区时双方都认为自己落子成功怎么办？
15

什么时候需要 Kafka，什么时候不该把它当读库？

参考回答
Kafka 可承载持久事件、异步 fanout 和回放；会话读/随机历史查询仍需要面向访问模式的存储；consumer lag 与 retention 不是用户 mailbox 查询语义
常见失分
把所有 API 读都改成扫描 Kafka。
继续追问
如何从消息事件构建可查询的 inbox projection？
16

哪些指标证明消息系统真的健康？

参考回答
durable accept latency；delivery lag、duplicate rate、reconnect recovery time；presence update rate、socket buffer saturation、consumer lag
常见失分
只列服务器 CPU 与连接数。
继续追问
某 region 的 delivery lag 升高但 accept latency 正常，先查哪里？
每个节点答不出具体状态、错误语义或恢复路径，回到讲义对应段落修复。
