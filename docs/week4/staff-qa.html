Week 04 · Staff Q&A

第 4 周 Staff 追问与详细答案

先遮住答案口述；答案必须覆盖判断、机制、失败语义和可验证修复。

Senior / Staff Drill

FB News Feed

Q1

什么时候把作者从 fanout-on-write 切换为 fanout-on-read？

为什么面试官会追：候选人用一个固定粉丝数拍板，未考虑发帖频率和读写成本。

详细回答

我会估算作者每篇帖子的 follower 写放大、每日发帖次数，以及这些粉丝实际打开 feed 的比例。预扇出的成本近似 followerCount × postRate，读时合并的成本则受 activeFollowers、名人关注数和每次候选窗口影响。阈值由队列陈旧预算和 Feed Service 的 p99 共同校准，超过阈值先在小比例作者上切换；读端必须能同时合并预计算列表与名人帖子，才能在阈值变化时不中断用户视图。

继续追问：某位作者突然爆红时，如何在已有扇出任务尚未完成的情况下切换？

常见漏项：只谈粉丝数，没有双写过渡、去重规则或读侧归并。

现场修复

为一位粉丝数快速增长的作者画出旧任务排空、标记切换和读侧去重时序。

证据
  • FB News Feed · 1) How do we handle users who are following a large number of users?
  • FB News Feed · 2) How do we handle users with a large number of followers?
Q2

Post 已提交，但 fanout worker 在写第一批 feed 后崩溃，怎样证明不会永久漏帖？

为什么面试官会追：候选人认为队列会自动解决一切，没有事件与批次的持久边界。

详细回答

Post 事务同时写 outbox，发布器可重复发送同一 post-created 事件；扇出任务再按 follower 范围生成稳定 batchId。每个 feed 更新以 viewerId、postId 幂等 upsert，批次只有覆盖完整范围后才记录完成，所以崩溃后重跑不会重复展示。监控以事件最老年龄和未完成范围为准，超预算时对受影响 viewer 启用读时合并。最后抽样从 Follow 与 Post 重新计算期望窗口，验证物化视图确实收敛。

继续追问：如果 follower 关系在扇出过程中改变，批次使用哪个关注快照？

常见漏项：用“至少一次投递”替代幂等键、批次水位与对账。

现场修复

补出 outbox、batchId、feed 唯一键和期望窗口巡检四个证据。

证据
  • FB News Feed · 2) How do we handle users with a large number of followers?
  • 第十二章：流处理 · 维护物化视图
Q3

为什么 feed 翻页不能用 offset，cursor 应包含什么？

为什么面试官会追：候选人的下一页在新帖插入或删除后出现重复、跳项。

详细回答

倒序时间线持续插入，offset=20 的含义会随着列表头新增内容而移动，第二页可能重新读到第一页末尾。cursor 至少包含上页最后一条的 createdAt 与 postId，并用“严格小于这个二元组”查询；首请求还可以记录 upperBound，保证整次浏览不会吸入后来发布的帖子。同一时间戳由 postId 提供全序，过滤删除项时多取候选但仍沿原排序游标前进。这个 cursor 与底层复合索引一致，查询成本不会随页数线性增长。

继续追问：多个来源在读时归并后，如何生成一个可恢复的统一 cursor？

常见漏项：只把最后 postId 放进游标，却无法表达多个作者流各自读到哪里。

现场修复

用三位作者、同时间戳和一次删除手算两页结果，证明无重复无遗漏。

证据
  • FB News Feed · 4. Users should be able to page through their feed.
  • Data Modeling · Indexing for Access Patterns
Q4

一个 postId 占到缓存集群 30% 的读流量，扩容为什么可能没用？

为什么面试官会追：候选人只增加分片数，却没有改变单键仍映射到一个节点的事实。

详细回答

普通一致性哈希只会把这个 postId 放到其中一个分片，新节点无法共同承担读流量。若内容近乎不可变，我会在应用层创建若干带版本和 salt 的副本键，读请求随机选择；每个节点再做 request coalescing，避免并发 miss 同时回源。编辑或删除发布新版本，旧副本短 TTL 退场，并保留事实库的删除检查。要观察单键 QPS、缓存节点带宽、回源并发和旧版本命中，确认复制没有把一致性风险藏起来。

继续追问：如果帖子包含经常变化的点赞数，你还会把整个对象一起复制吗？

常见漏项：把热键与缓存总容量混为一谈，忽略网络带宽和回源风暴。

现场修复

拆开不可变正文与易变计数，为两者分别给缓存键和失效策略。

证据
  • FB News Feed · 3) How can we handle uneven reads of Posts?
  • Redis · Hot Key Issues
Q5

用户关注十万账户时，如何把一次 feed 读取限制在可预测成本内？

为什么面试官会追：候选人打算每次向十万作者逐一查询并全量排序。

详细回答

这类高出度 viewer 应拥有有限长度的预计算 feed，发布侧持续把普通作者的新 postId 写入窗口；名人作者仍走少量读时合并。读取只取固定数量的预计算候选和每个例外来源的有界最近窗口，再用堆做 k 路归并。窗口之外的极深历史可以明确不保证，或低频回退到事实表生成。成本上限由候选数而不是关注总数决定，同时以 feed 新鲜度和回退率监控是否需要扩大窗口。

继续追问：预计算窗口只有 200 条，用户读到末尾时产品应该怎样定义行为？

常见漏项：为了理论上的无限翻页，给在线路径引入无界 fan-in。

现场修复

写出候选上限、归并复杂度和深历史产品边界。

证据
  • FB News Feed · 1) How do we handle users who are following a large number of users?
  • FB News Feed · 3. Users should be able to view a feed of posts from people they follow.

Senior / Staff Drill

Instagram

Q1

如何防止帖子已经对粉丝可见，但它引用的视频仍在转码？

为什么面试官会追：候选人把上传成功直接等同于发布成功。

详细回答

我会让 MediaVersion 拥有显式处理状态，并定义一组发布所必需的 rendition。处理任务输出不可变对象并回写校验结果，协调器确认必需集合齐全后才把版本从 PROCESSING 条件更新为 READY。CreatePost 事务必须验证 owner 与 READY 状态，成功后才产生 feed 事件。若客户端重复提交同一 publishKey，返回原 postId；任何失败版本只对作者显示诊断，不会进入粉丝 feed。

继续追问：处理配置升级时，怎样切换到新 rendition 又不让旧帖子短暂破图？

常见漏项：依靠队列顺序猜测媒体已经完成，没有可查询状态和原子发布条件。

现场修复

画 UploadSession、MediaVersion 与 Post 三个状态机，标出唯一可见性切换点。

证据
  • Instagram · 1) Users should be able to create posts featuring photos, videos, and a simple caption
  • Managing Long Running Tasks · Workers
Q2

四 GB 上传在 95% 时断线，续传协议需要保存哪些证据？

为什么面试官会追：候选人只说 multipart，却无法判断哪些 part 可以信任。

详细回答

UploadSession 持久化 uploadId、owner、对象键、partSize、已确认 partNumber 与各自 checksum；对象存储回执是服务端接受 part 的证据。重连后客户端查询清单，只补缺失或校验不符的 part，finalize 根据有序清单计算整体校验并以 uploadId 幂等。凭证限制对象前缀、part 范围和有效期，session 过期后不能继续写；清理器仅删除未被 READY MediaVersion 引用且超过宽限期的对象。所有恢复判断都以服务端与对象存储的回执为准。

继续追问：客户端声称上传完成，但某一 part 在对象存储中缺失时返回什么状态？

常见漏项：只在客户端记进度，换设备或应用重装后无法可信恢复。

现场修复

写出 create、uploadPart、listParts、complete 四个接口及幂等键。

证据
  • Instagram · 2) The system should render photos and videos instantly, supporting photos up to 8mb and videos up to 4GB
  • Managing Long Running Tasks · Workers
Q3

为什么 Feed API 不应该代理图片和视频字节？

为什么面试官会追：候选人让应用服务承担媒体上传和下载，导致带宽耦合。

详细回答

Feed Service 的职责是权限、排序和元数据，若它代理大对象，少数慢连接会占住工作线程与出站带宽，使元数据请求尾延迟跟媒体大小一起恶化。服务端应返回带版本的 manifest 和短期授权，客户端直接从 CDN 或对象存储取媒体；图片先取适配屏幕的变体，视频先取封面或首段。对象键不可变便于长缓存，删除权限通过短 token 与元数据检查控制。这样元数据扩容和媒体带宽扩容互不牵连。

继续追问：私有账户的媒体怎样使用 CDN，又不把永久公开 URL 泄露出去？

常见漏项：认为使用 CDN 就自动解决授权、版本和删除传播。

现场修复

把 Feed JSON、授权 token、CDN cache key 与对象键分别画出。

证据
  • Instagram · 2) The system should render photos and videos instantly, supporting photos up to 8mb and videos up to 4GB
  • Scaling Reads · CDN and Edge Caching
Q4

名人发帖时，如何避免向数千万 feed 写入又保持首屏低延迟？

为什么面试官会追：候选人只会全量扇出或每次遍历全部关注者。

详细回答

对超大粉丝作者保留 post 事实，不进入普通 fanout 队列；Feed Service 读取 viewer 的预计算窗口后，再批量拉取其关注的少量名人近期帖子，按 createdAt、postId 归并。普通作者继续 fanout-on-write，冷用户可在回流时重建。名人集合、每个来源候选数和归并总数都有上限，超时就返回预计算部分并允许刷新。阈值用写放大、名人关注数、读端 p99 与 feed 陈旧度共同调节。

继续追问：一个普通作者突然爆红时，切换期间如何防止同一 postId 出现两次？

常见漏项：没有统一去重与排序键，混合两条路径后结果不稳定。

现场修复

用一位刚跨阈值的作者演示双路径归并和 postId 去重。

证据
  • Instagram · 1) The system should deliver feed content with low latency (< 500ms )
  • Instagram · 3) The system should be scalable to support 500M DAU
Q5

怎样证明媒体管线恢复后没有留下公开破图？

为什么面试官会追：候选人只关注 worker 成功率，没有从帖子引用反查对象。

详细回答

我会周期抽样所有 PUBLISHED Post，解析其 manifestVersion，验证必需 rendition 的对象存在、checksum 匹配且授权元数据可用；这是面向用户结果的破图率。管线还记录每个 mediaVersion 的状态年龄、任务尝试与最后错误，超过预算进入可重放死信。恢复时只重跑缺失输出，成功后重新校验整个 manifest，再允许切换。孤儿对象扫描必须从 Post 和 READY MediaVersion 的引用集合反向确认，不能按年龄直接删掉仍在使用的内容。

继续追问：如果校验发现某个热门帖的一个 rendition 丢失，如何降级而不下架整个帖子？

常见漏项：把队列清空当作恢复完成，未检查用户可见引用。

现场修复

定义 manifest 完整性巡检、破图告警和单 rendition 回填步骤。

证据
  • Instagram · 3) The system should be scalable to support 500M DAU
  • 第十二章：流处理 · 失败后重建状态

Senior / Staff Drill

YouTube

Q1

completeUpload 响应丢失后，客户端怎样避免生成第二个视频？

为什么面试官会追：候选人把 HTTP 重试当成新的上传完成动作。

详细回答

客户端必须复用 uploadId 与同一个 completeKey。服务端在唯一约束下把 uploadId 映射到 videoId，并先核对对象存储中的 part 清单、长度和整体 checksum；映射已存在时直接返回原 videoId，不再次触发新管线。若部分 part 未确认，返回仍可查询的 INCOMPLETE 和缺失列表。处理事件由持久 outbox 产生，即使响应丢失也会继续，客户端查询 video 状态即可判断结果。

继续追问：对象存储 complete 成功、本地映射事务失败时怎样恢复？

常见漏项：只给每个 part 幂等，却让最终 complete 重复创建业务实体。

现场修复

为 uploadId 到 videoId 建唯一映射，并演练 complete 的三个崩溃点。

证据
  • YouTube · 2) How do we support resumable uploads?
  • YouTube · 1) Users can upload videos
Q2

怎样把一个两小时视频转成多码率，又避免失败后全量重做？

为什么面试官会追：候选人只有一个笼统 transcoder 服务，没有任务边界。

详细回答

先 probe 原片并固定 pipelineVersion，再按时间范围切 segment；每个 rendition、segmentRange 是独立任务，输出键带 inputHash 和配置版本。协调器维护 DAG 依赖，音频与不同分辨率可并行，失败仅重试缺失节点。最低可播放集合完成后可以进入 READY_PREVIEW，正式发布仍需 manifest 所要求的全套输出校验。死信保存原始参数和错误分类，修复 worker 后可精确重放，不覆盖已验证结果。

继续追问：不同码率的 segment 边界为什么必须对齐？

常见漏项：把并行转码说成多个重复整片任务，成本和恢复粒度都不可控。

现场修复

画 probe、segment、encode、package、publish 五层 DAG，并标出幂等输出键。

证据
  • YouTube · 1) How can we handle processing a video to support adaptive bitrate streaming?
  • Managing Long Running Tasks · Workers
Q3

为什么不能边转码边修改同一个播放 manifest？

为什么面试官会追：候选人让播放器可能读到半成品清单。

详细回答

原地更新会让缓存和播放器在不同时间看到不完整集合：清单已经列出 segment，但对象尚未可读，或不同码率的时间片未对齐。我会为每次管线结果生成 immutable manifestVersion，先验证它引用的必需对象，再用一次条件写更新 Video.publishedManifestVersion。播放器一次会话固定读取该版本，清单短 TTL，segment 长 TTL。新版本异常时回退指针即可，旧对象保留到会话和缓存窗口结束。

继续追问：跨区域复制尚未完成时，published 指针何时可以切换？

常见漏项：认为对象存储单对象原子写就足以保证多个 segment 的集合原子性。

现场修复

列出 v16 到 v17 的准备、验证、切换和回滚四步。

证据
  • YouTube · 1) How can we handle processing a video to support adaptive bitrate streaming?
  • 第十一章：批处理 · 工作流调度
Q4

一个视频突然获得一亿播放，最先保护哪条路径？

为什么面试官会追：候选人只说增加 CDN，却未处理并发 miss 与元数据热点。

详细回答

首先把不可变 segment 放到边缘缓存，并在区域设置 origin shield；同一 segment 的并发 miss 在 shield 做请求折叠，避免所有边缘直打对象存储。manifest 和视频元数据是较小但更易热的键，可做多副本缓存，键中带 publishedManifestVersion。只预热首批高概率播放片段，避免为长视频搬运大量无人观看尾段。监控 edge/shield 命中、单 segment 回源并发、对象存储带宽和首帧，超过阈值再扩区域缓存容量。

继续追问：视频被设为私有后，已缓存 segment 如何阻止继续访问？

常见漏项：把 CDN 当作无限容量，忽略 origin shield、授权和冷热片段差异。

现场修复

画 edge miss 到 shield 再到 origin 的时序，并给每层写限流与命中指标。

证据
  • YouTube · 3) How do we scale to a large number of videos uploaded / watched a day?
  • Scaling Reads · CDN and Edge Caching
Q5

转码队列积压时，如何决定降级而不是无限扩容？

为什么面试官会追：候选人没有任务优先级、成本或用户可见目标。

详细回答

我会按 upload-to-playable SLO 排序，而不是要求所有 rendition 同时完成。先保证音频、封面和一组基础码率，暂停昂贵的高分辨率增强；长视频、批量上传与重试流量使用独立配额，避免占满交互上传。看最老任务年龄、每分钟视频的算力成本、各节点失败类型和可播放完成率。扩容前先确认瓶颈是 CPU 编码、对象读写还是队列分区；降级结束后补跑被延迟的 rendition，并重新验证 manifest。

继续追问：如何避免一个大客户的批量上传饿死普通创作者？

常见漏项：用队列长度单指标扩容，未区分任务大小和最低可播放目标。

现场修复

为短视频、长视频、高分辨率和重试任务设计四个配额队列。

证据
  • YouTube · 3) How do we scale to a large number of videos uploaded / watched a day?
