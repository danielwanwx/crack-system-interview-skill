Day 1 · Week 4

实时连接模型

WebSocket 全双工； SSE 单向推送； 连接注册表与心跳。
核心 invariant：每个 user+device 在任意时刻最多一个活跃 gateway 连接负责投递；断线后重连必须带 last_event_id 补拉

参考原文 Real-time Pattern Networking Real-time DD 本日核心追问
需求边界
  • 先锁定 3 个核心动作：WebSocket 全双工、SSE 单向推送、连接注册表与心跳
  • 明确 out of scope，避免把后台运营、推荐、BI 一次性塞进主链路
  • 读写比例、峰值 QPS、对象规模、保留周期必须给数量级假设
  • 先定义用户可见语义，再讨论 Kafka、Redis 或数据库选型
正确性不变量
  • 每个 user+device 在任意时刻最多一个活跃 gateway 连接负责投递；断线后重连必须带 last_event_id 补拉
  • 每个状态转换写 precondition；失败重试不能跳过中间状态
  • 数据库事实、缓存视图、异步派生数据必须标出 authoritative source
  • 重复、乱序、超时、部分成功四类故障逐一给答案
容量与 SLO
  • Week 4 Day 1：先估峰值写 QPS，再按分区键检查热点
  • 在线路径给 p50 / p99；后台 repair 给 freshness 与 backlog SLO
  • 吞吐扩容不能破坏 per-key 顺序、唯一性或余额守恒
  • 压测必须覆盖 skew：最热 1% key 承担 40% 流量
层次 面试中要落到的机制 必须暴露的指标 失败时动作
API 幂等键、分页游标、版本号、错误码 request_p99 / 409_rate 限流、返回可重试语义
Storage 主键、分区键、二级索引、唯一约束 lock_wait / hot_partition 迁移流量或拆热点
Async 事件 ID、顺序键、consumer checkpoint lag / redelivery / DLQ 重放并保持幂等
Derived watermark、版本、last_updated_at freshness / mismatch_rate 从 source of truth 重建
Repair 扫描 zombie session：heartbeat 超时清理 registry；重连风暴时限流 + jitter backoff stuck_age_p99 / repair_count 先止血，再对账，再回填 
POST /v1/connect { device_id, auth_token }
→ 201 { connection_id, heartbeat_interval_sec: 30 }

WS /v1/stream?connection_id=...
Client heartbeat every 30s → server SET presence:conn:{id} EX 90
On disconnect: DEL subscriptions; mark offline after 2× interval
脆弱答案：只讲 happy path
画完组件图就结束；没有唯一约束、状态机、重试边界，也没有说明超时后客户端该不该重试。
Senior 答案：不变量驱动
推荐
围绕「每个 user+device 在任意时刻最多一个活跃 gateway 连接负责投递；断线后重连必须带 last_event_id 补拉」展开：写入原子点、异步传播、可观测性、自动修复与人工闸门形成闭环。

窗口题先定 invariant：维护的区间 [left,right) 满足什么性质。可变窗口在违反约束时收缩 left。

def longest_no_repeat(s):
    left, seen, best = 0, {}, 0
    for right, ch in enumerate(s):
        if ch in seen and seen[ch] >= left:
            left = seen[ch] + 1
        seen[ch] = right
        best = max(best, right - left + 1)
    return best
