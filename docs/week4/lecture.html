FB News Feed

FB News Feed：从需求语义到故障修复

01

先定义一条可分页的时间线契约

News Feed 的基础契约不是“推荐最相关内容”，而是把所关注作者的帖子按倒序稳定返回。Post 以 postId 保存正文与 createdAt，Follow 记录 viewerId 到 authorId 的边；第一页请求还要冻结一个 upperBound，后续 cursor 编码最后一条的 createdAt 与 postId。这样新帖插入列表头部时不会把旧内容推到下一页造成重复，同一毫秒的帖子也能靠 postId 决胜。删除或不可见帖子可以在读取时过滤并多取候选，但不能改用 offset 后让每次插入都移动页边界。

能迁移的判断：先把排序键和 cursor 的稳定性讲清，再讨论 feed 怎样预计算。
02

普通作者走 fanout-on-write

普通用户发布帖子后，Post Service 先只提交帖子事实和一条 outbox 事件，不能等待成百上千个 feed 写完才响应。扇出 worker 按 followerId 范围切块，每个任务把 postId 幂等地 prepend 到对应 PrecomputedFeed，并把列表截断在固定窗口。队列至少一次投递意味着同一块可能被重复处理，因此 feed 项应以 viewerId、postId 去重；worker 中途崩溃只需从未完成块继续。这个路径把昂贵工作移出请求，但必须给出“发帖成功到粉丝可见”的陈旧预算。批次水位还要能指出哪段粉丝范围尚未完成，便于定向重放。

能迁移的判断：帖子提交与 feed 物化分开，用 outbox、切块和幂等写保证最终可见。
03

名人账户改用混合扇出

给九千万粉丝逐个写 feed 会制造不可接受的写放大，所以高粉丝作者不进入普通预扇出。Feed Service 读取 viewer 的预计算列表，同时查询其关注的名人最近帖子，将两组有序流按 createdAt、postId 做 k 路归并，再截取一页。阈值不能写成永久常量，应由 follower 数、发帖频率、扇出队列时延和读放大共同决定；一个高粉丝但很少发帖的作者与持续高频发布者成本不同。为了控制尾延迟，读侧可以并行拉取并限制名人候选窗口，超时时返回预计算部分并标记可继续刷新。

能迁移的判断：按作者成本选择写时或读时扇出，在请求端做有界归并。
04

预计算 feed 只是可重建视图

PrecomputedFeed 不是帖子真相，它可能因 worker 重试出现重复，也可能在区域故障后暂时落后。恢复时从 Post 与 Follow 的事实水位重新生成某个 viewer 的窗口，比手工修补未知缺口更可靠。每条 feed 项记录来源 postId 和 materializedAt，巡检任务抽样比较“应出现的关注帖子”与实际窗口，发现缺失后投递定向修复。若总队列积压超过一分钟陈旧预算，系统可以暂时对受影响用户回退到按关注作者查询，而不是继续返回一个看似成功但长期缺项的列表。

能迁移的判断：为派生 feed 保留重建、抽样对账和读时降级三条恢复路径。
05

病毒帖要解决单键热读

一条爆款帖子会让大量 Feed 请求同时读取同一个 postId，即使整体数据库 QPS 不高，也可能压垮承载该键的分区。应用缓存之外还需请求合并：某个缓存节点只允许一个 miss 回源，其余等待同一结果；特别热的不可变帖子可以复制成带盐的若干缓存键，由读方随机选择。帖子编辑或删除时用版本化 key 发布新版本，并让旧版本短 TTL 退场，避免全网同步失效形成惊群。缓存只保存内容，权限和删除判定仍要走可更新的元数据路径。

能迁移的判断：热点治理针对单个 postId 的负载形状，而不是笼统扩大缓存集群。
06

用三类水位判断 feed 是否健康

只看 Feed API 的 p99 会漏掉“很快地返回旧数据”。用户体验指标应同时包含请求尾延迟、首屏空缺率和发帖到可见的分位数；管线指标看 outbox 未发布数、扇出队列最老年龄、每个 worker 的 follower 写速率；正确性指标则抽样计算重复 postId、关注帖子缺失和 cursor 倒退。名人阈值调整要做小流量对照，观察读合并成本是否抵消写放大的收益。事故恢复完成的标准不是队列清零，而是可见延迟恢复并且抽样对账差异回到基线。

能迁移的判断：延迟、积压与内容正确性要同屏观察，缺一项都可能误判。

Instagram

Instagram：从需求语义到故障修复

01

发布帖子是媒体状态机的最后一步

Instagram 的 Post 不能在客户端刚拿到上传地址时就变成可见。UploadSession 先记录 owner、允许的对象前缀、分片清单和过期时间；对象完成后校验大小、类型与 checksum，MediaVersion 再经历 SCANNING、PROCESSING、READY 或 FAILED。创建 Post 时用条件写确认引用的 mediaVersion 属于当前用户且处于 READY，随后才发布 postId。这样处理 worker 重试、用户重复点发布或孤儿对象清理都不会产生公开破图。发布服务还要记录 mediaVersion 的处理年龄和最后错误，使作者能看到失败原因，运营也能找出长期卡住的版本。

能迁移的判断：把上传完成、媒体就绪和帖子发布分成三个可查询的承诺。
02

大视频直传而不穿过应用服务器

四 GB 视频若经 Post Service 中转，会同时占满其连接、内存和东西向带宽。更合适的协议是服务端创建 uploadId 并签发限时、限定对象键与 part 范围的上传凭证，客户端把分片直接送入对象存储；完成请求携带 part 编号与校验值，服务端核对清单后 finalize。断线时客户端先查询已持久化的 part 集合，只补缺失片段。凭证的对象前缀绑定 userId，finalize 也必须幂等，防止凭证被拿去覆盖别人的媒体。上传进度以服务端清单为准，不采信客户端自报。

能迁移的判断：控制面负责授权和状态，字节面直接进入对象存储。
03

同一原片派生多种交付版本

上传完成后，媒体管线按 mediaVersion 生成缩略图、不同分辨率图片和视频 rendition。每个任务的输出键包含输入 checksum、处理配置版本与 rendition 名称，因此同一任务重复消费只会命中相同不可变结果。协调器维护必需输出集合，只有全部校验通过才把版本标为 READY；非关键增强可以稍后补齐。若处理规则升级，新版本并行生成，Post 仍指向旧的完整集合，待新 manifest 验证后一次切换，避免用户在滚动时读到一半新一半旧。

能迁移的判断：输出内容寻址、任务幂等、manifest 原子切换让媒体处理可安全重放。
04

feed 返回元数据，媒体由边缘单独交付

Feed API 应返回 postId、作者、caption、媒体 manifest、尺寸与短期授权信息，而不是把照片或视频字节塞进 JSON。客户端先根据尺寸保留布局，优先请求低分辨率预览，再按屏幕与网络加载更高质量版本；视频可先拿封面和短首段。对象 URL 含不可变版本，适合长 TTL 的 CDN 缓存，帖子删除则通过授权元数据和短期 token 快速阻断新请求。这个分离让 Feed Service 的 p99 不受四 GB 视频传输影响，也能独立扩展元数据与边缘流量。

能迁移的判断：元数据决定展示，CDN 交付不可变媒体，两条链路各自预算延迟。
05

关注流仍要面对名人与冷用户

普通作者可在发帖后异步把 postId 写入粉丝的有限 feed 窗口，读取时只需批量取回帖子元数据；超大粉丝作者若照做会制造巨量写放大，应在请求时合并其最近帖子。长期不活跃用户没必要持续维护昂贵 feed，可以在回流时用 Follow 与近期 Post 重建窗口。混合策略的去重键是 postId，排序键是 createdAt 加 postId；阈值同时参考粉丝数、发帖频率、粉丝活跃率和扇出积压，不因“社交产品都预计算”而固定一种方案。

能迁移的判断：对普通、名人和冷用户采用不同物化策略，但保持同一排序契约。
06

用首帧与破图率定位 500M DAU 的瓶颈

Instagram 的容量问题不能只统计 Feed QPS。上传侧要看分片失败率、finalize 重试、对象存储写带宽和处理队列最老年龄；浏览侧看 Feed p99、图片首字节、视频首帧、CDN 命中与回源并发；正确性侧持续抽样 PUBLISHED Post 是否都能解析到 READY 媒体。热门帖子回源异常时先启用 origin shield 和请求合并，处理积压时降低非关键 rendition 优先级。任何扩容结论都要能指出是哪条资源曲线先饱和，而不是笼统归因于用户总数。区域流量失衡还要单独观察回源带宽与处理槽位。

能迁移的判断：将上传、处理、元数据读取和媒体交付分别观测，才能找到真实限速环节。

YouTube

YouTube：从需求语义到故障修复

01

上传会话让大文件可恢复

YouTube 上传从 createUpload 开始，服务端返回 uploadId、推荐 partSize 和受限对象前缀。客户端并行发送分片，每个 part 带序号与 checksum；服务端只在对象存储确认后记录完成。completeUpload 根据有序 part 清单验证总长度与内容哈希，同一 uploadId 重复 complete 只能返回同一 videoId。原始文件保持不可变，后续处理都引用其 contentHash。这个协议让浏览器断线、设备切网或服务重启后都能查询已完成分片，而不是从零开始传输数十 GB。过期会话进入宽限清理，只有确认没有 Video 引用后才删除原片。

能迁移的判断：持久 part 清单、校验和与幂等 complete 是断点续传的核心。
02

转码不是一个任务，而是一张可重放 DAG

原片进入处理管线后，先探测编码、时长和音轨，再切分时间段并生成多种分辨率与码率；音频、字幕和缩略图可形成并行分支。每个节点以 inputHash、pipelineVersion、segmentRange 和 rendition 组成输出键，重试不会产生不同结果。协调器记录依赖完成水位，单个 4K rendition 失败不会迫使已成功的 720p 全部重做。高优先级路径先产出最低可播放集合，让创作者尽早预览，完整质量随后补齐。任务运行时还应记录编码器版本，防止同一输出被不同二进制悄然改写。

能迁移的判断：显式依赖、内容寻址输出和局部重试把转码从黑盒变成可运营工作流。
03

播放清单必须一次发布一个完整版本

播放器不能拿到引用尚未存在分片的 manifest。处理协调器先在私有前缀生成全部必需 segment，逐一校验大小和 checksum，然后写 immutable manifest-v17；最后只更新 Video 元数据中的 publishedManifestVersion。读请求始终解析一个确定版本，即使新处理仍在进行也继续播放 v16。若 v17 监测到错误，只需把指针回退而不用失效数百万分片。清单短 TTL，媒体 segment 因带版本和内容哈希可长时间缓存，二者采用不同缓存策略。发布日志保存旧指针与验证摘要，便于事故后解释为何回退。

能迁移的判断：先构造完整不可变版本，再原子切换短小指针。
04

自适应播放依赖可切换的同时间片

各码率 rendition 必须使用对齐的 segment 边界和关键帧，播放器才能在片段之间无缝切换。开始播放时先获取较保守码率的首段，根据近期下载吞吐、缓冲区长度和设备能力选择下一片，而不是等待测速完成。服务端 manifest 描述每个 rendition 的带宽、分辨率与 segment URL；播放器遇到网络变差先降码率保住连续性，网络恢复再逐步升高。首帧时间、rebuffer ratio 和平均清晰度要一起看，单纯追求最高码率会恶化真实体验。

能迁移的判断：ABR 的目标是连续播放与画质的动态平衡，不是固定选择最高分辨率。
05

爆款视频需要 origin shield 和分层缓存

热门视频的同一批 segment 会在全球被重复请求，边缘 CDN 命中可消除大部分源站流量；但多个边缘同时 miss 仍可能冲垮对象存储。区域 origin shield 先合并回源，单 key 再做请求折叠，只让一个下载穿透。新发布的预计热门内容可预热首批 segment，却不必复制完整长视频。缓存键包含 manifestVersion 与内容哈希，因此不同版本不会串读；删除或权限变化由短期清单和授权层阻断，媒体对象可在后台按保留策略清理。

能迁移的判断：边缘、区域 shield、对象存储形成三级读路径，并用版本化 key 避免失效风暴。
06

处理公平性与播放 SLO 要分开运营

长视频会占据大量转码槽，若队列只按到达顺序，大文件可能拖慢所有短视频。调度器应按租户、视频长度、目标 rendition 和优先级分队列，给每类设置并发配额；重试任务进入独立预算，持续失败的输入送死信而非无限抢资源。处理侧看 upload-to-playable、各 DAG 节点 p99、最老任务和单位分钟算力；播放侧看 manifest 错误、首帧、缓冲、CDN 命中和源站带宽。两组指标分开，才能判断是后台制作慢还是前台交付差。

能迁移的判断：资源调度保证处理公平，播放指标直接反映观众体验。
