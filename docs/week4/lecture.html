消息、状态与实时协作

主线 45 分钟白板 30 分钟追问 30 分钟

答题主线

01 · 核心判断

语义边界

消息系统应保证会话内有序、至少一次持久投递和可恢复；不承诺全局总序。presence 是可失效的软状态，不能与消息持久化语义混淆。

02 · 核心判断

身份与会话数据

conversation、message、device_delivery、read_receipt、presence 分离建模。message_id 支撑去重；conversation sequence 或服务端 log offset 支撑会话内顺序。

03 · 核心判断

发送主路径

客户端带 client_message_id 发送；服务端在会话分区持久化并分配 sequence，之后异步 fanout 到接收方设备。发送确认至少代表服务端持久化，不应伪装成对方已读。

04 · 核心判断

离线、重连与多设备

设备以最后确认的 sequence/offset 拉取缺口；ACK 可丢失，因此投递和客户端应用必须幂等。多设备 delivery 是多个 device receipt，不是一个模糊的用户状态。

05 · 核心判断

群组与扇出

小群可 fanout-on-write；超大群要控制每条消息产生的写放大，可采用分层 fanout、在线推送加离线拉取或混合读路径。

06 · 核心判断

路由与多地域

同一 conversation 的写应稳定路由到一个逻辑 leader/partition，以简化顺序。跨地域复制可以异步，但 failover 必须明确消息序号和重复投递的恢复方式。

07 · 核心判断

压力与失败

WebSocket gateway 负责连接管理，不承担 durable queue。慢消费者必须有 buffer 上限、丢弃或拉取恢复策略；重试风暴要受发送者、会话、租户配额约束。

08 · 核心判断

迁移：Online Chess

棋局移动需要 authoritative game state 和规则校验，通常由单房间 writer 线性化；它比聊天消息更不能接受乱序或两个并存状态。
