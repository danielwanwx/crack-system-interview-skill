16 个真实追问节点Senior / Staff

面试压力测试

先独立作答，再展开下一个追问。

01

Live Comments 的顺序保证在什么范围内？

参考回答
按 room_id 的局部 sequence/offset；不承诺跨直播间全局顺序；客户端可容忍有限延迟但重连必须能按 offset 补洞
常见失分
要求全局 total order 或完全忽略排序。
继续追问
同一房间跨 region 写入时如何避免两个 sequence owner？
02

late joiner 如何既看到历史又不漏实时评论？

参考回答
先取 recent window 与 high-water mark；从该 offset 建立订阅或补拉；客户端按 sequence 去重并处理历史与实时重叠
常见失分
先拉最后 100 条再打开 WebSocket，未处理两步之间的空洞。
继续追问
历史窗口很大时如何分页且不阻塞实时流？
03

为什么 WebSocket gateway 不应保存唯一真相？

参考回答
连接节点会断开和扩缩；durable room log 才能支撑重连和审计；gateway 只维护订阅和有界 buffer
常见失分
把所有已发评论留在 socket server 内存。
继续追问
gateway 崩溃时客户端如何迁移并恢复位置？
04

一个百万观众直播间的最大瓶颈是什么？

参考回答
通常是 fanout egress、订阅连接和慢消费者；可分层 fanout 到 edge/gateway；需要限速、采样或降级策略而非简单增加单 room 写分片
常见失分
只增加数据库写节点。
继续追问
评论写入量和观看连接量各自如何估算？
05

慢消费者怎么处理？

参考回答
per-connection buffer 上限；超过阈值可丢实时推送并要求从 room log pull；不能让慢客户端反压整个 room
常见失分
无限缓存或给每个用户专属队列。
继续追问
客户端怎么知道自己需要切换到补拉模式？
06

评论被审核删除后，已经看到的用户如何收敛？

参考回答
delete/moderation event 进入同一 room event stream；客户端按 comment_id 应用 tombstone；搜索、缓存和 replay 窗口也要处理删除
常见失分
只从主数据库删除。
继续追问
审核决定晚到时，怎样审计先前已展示的内容？
07

重连后 offset 从哪里恢复？

参考回答
客户端保存最后已应用的 room sequence；服务端保留可查询/可回放的 room log；若 offset 已过 retention，明确降级到 recent snapshot
常见失分
依赖 socket session ID。
继续追问
retention 缩短会如何影响产品契约？
08

如何抵御水军和突发 spam？

参考回答
按 sender、room、IP/tenant 的早期限流；审核/反滥用在 fanout 前尽量过滤；对热门 room 可降低单用户速率并做负载隔离
常见失分
只在数据库写失败后再限流。
继续追问
被限流的评论应该丢弃、排队还是给用户可重试错误？
09

room 分区热点如何处理？

参考回答
room 是排序边界，不能随意把一个 room 的写拆成无协调分区；可用单 writer 加分层读取/扇出；极端情况下要声明降级评论速率或放宽顺序窗口
常见失分
用随机 shard key 后再希望顺序自然出现。
继续追问
什么业务指标触发对热门房间的降级？
10

跨地域故障时如何恢复直播评论？

参考回答
明确 room owner/leader 与复制方式；failover 依据持久 offset 恢复且防双写；客户端重连到新 gateway 后按 sequence 补洞
常见失分
DNS 切换后假设状态自动一致。
继续追问
主 region 网络隔离时是继续接收评论还是只读？
11

为什么 Live Comments 不等同于 News Feed？

参考回答
评论是 room 内实时广播和短窗口回放；Feed 是按用户的候选、排序与读路径问题；Feed 的一致性与延迟目标可与实时房间不同
常见失分
两者都说成 fanout。
继续追问
什么数据模型可以在两者间复用？
12

News Feed 的 fanout-on-write 和 fanout-on-read 如何取舍？

参考回答
普通用户关注者规模有限时可写扩散换低读延迟；celebrity 写扩散会爆炸，宜读时合并或混合；排名和 freshness 会影响缓存策略
常见失分
只根据读写比例选择，忽略 follower 分布长尾。
继续追问
celebrity post 如何进入用户已经物化的 feed？
13

ranking 特征变更后如何处理缓存？

参考回答
feed item 与 ranking result 分层；使用短 TTL、版本化特征或异步重算；不应为了实时特征变化全量失效所有 feed
常见失分
简单清空 Redis。
继续追问
如何让 A/B ranker 结果可回滚和可比较？
14

复制延迟会让用户在 Feed 上看到什么？

参考回答
刚发帖后部分读路径可能暂时看不到或顺序不同；可为作者 read-your-writes 走权威/粘性路径；对关注者需明确 freshness SLO 而非假设即时一致
常见失分
承诺所有 follower 立刻看到。
继续追问
删除帖子时为什么比发布更需要快速收敛？
15

哪些内容需要持久化，哪些可以短暂保存？

参考回答
评论、删除事件、审核状态需 durable；socket buffer、订阅连接、短期 presence 可 ephemeral；recent window 可是可重建 cache，但来源必须可回放
常见失分
把所有状态都写数据库或都留内存。
继续追问
如果只保留 24 小时 comment log，如何支持举报审计？
16

你会用哪些指标发现系统已失去用户语义？

参考回答
end-to-end comment latency；fanout drop/reconnect replay、buffer saturation；moderation propagation delay、room sequence gap rate
常见失分
只看 QPS 和机器利用率。
继续追问
某直播间 drop rate 上升但写入正常，优先检查哪条链路？
每个节点答不出具体状态、错误语义或恢复路径，回到讲义对应段落修复。
