Senior / Staff Drill

Google Docs

Q1

两个人在同一位置同时插入字符，系统如何保证最终文本一致？

为什么面试官会追：候选人只说 WebSocket 广播，未定义并发操作的合并语义。

详细回答

两条操作都携带相同 baseServerVersion。协作 owner 先接受其中一条并分配下一 serverVersion；处理另一条时依据已接受操作做 transform，或用 CRDT 的稳定位置标识决定顺序。同位置冲突必须使用确定 tie-break，例如 clientId，而不能依赖各客户端本地到达顺序。服务端广播规范化后的操作和版本，所有副本按同一序列应用。测试至少覆盖插入/插入、插入/删除与删除/删除，并比较最终内容哈希。

继续追问：合并算法升级时，旧客户端产生的操作如何保持兼容？

常见漏项：把传输顺序当作并发语义，无法解释离线编辑或跨服务器重放。

现场修复

手算 baseVersion 相同的三组冲突，列出 transform 前后的位置。

证据
  • Google Docs · 2) Multiple users should be able to edit the same document concurrently.
  • Dealing with Contention · Optimistic Concurrency Control
Q2

编辑已保存并广播，但原客户端没收到 ack，重连后怎么办？

为什么面试官会追：候选人让客户端生成新操作编号再次提交，导致文字重复。

详细回答

客户端必须保留原 clientId、clientSeq 和操作体。重连时先发送 lastAppliedServerVersion 与未确认序号；服务端的去重索引若已有该 clientSeq，就返回原 serverVersion，不重新执行。随后按 serverVersion 补齐缺失广播，客户端忽略已应用版本并在发现跳号时请求区间。只有服务端明确未见该序号才重新接收原操作。这个协议同时处理 ack 丢失、广播重复和进程重启，不依赖连接恰好一次。

继续追问：去重记录可以在什么时候安全清理？

常见漏项：把连接恢复写成重新加载全文，却没有未确认本地操作的去向。

现场修复

画 persist、broadcast、ack 丢失、reconnect、dedupe 五步时序。

证据
  • Google Docs · 3) Users should be able to view each other's changes in real-time.
  • 第十二章：流处理 · 幂等性
Q3

客户端离线十分钟后带着一批旧编辑回来，如何重新接轨？

为什么面试官会追：候选人直接把旧文本覆盖服务端，或丢弃全部离线工作。

详细回答

客户端上传的是带连续 clientSeq 和共同 baseServerVersion 的操作批，而不是完整正文。服务端先返回 base 之后的远端操作；客户端或 owner 按选定合并算法把离线操作逐条 rebase，并在每步保留确定顺序。若 base 已早于日志保留水位，则下载最新快照，再借助客户端保留的意图重放；无法自动合并的结构变化应显式提示冲突副本。成功后每个操作仍按原 clientSeq 去重，并获得新的 serverVersion。

继续追问：若离线期间对应段落被另一用户全部删除，插入意图落在哪里？

常见漏项：把离线同步简化为 last-write-wins，导致大段协作成果丢失。

现场修复

构造删除段落与离线插入的冲突样例，写出产品可接受的结果。

证据
  • Google Docs · 3) Users should be able to view each other's changes in real-time.
  • Real-time Updates · "How do you handle connection failures and reconnection?"
Q4

怎样扩到百万 WebSocket，同时避免同一文档出现两个写 owner？

为什么面试官会追：候选人把负载均衡器轮询到任意服务器，没有连接目录和切换协议。

详细回答

入口按 documentId 查询目录并路由到当前协作分片，分片可持有很多文档连接；热门文档单独隔离。所有权记录含 ownerId、epoch 和日志水位。切换时新 owner 从快照与日志追到水位，再条件提升 epoch；旧 owner 的后续写因 fencing token 过期被事实存储拒绝，客户端收到重定向后带 lastServerVersion 重连。容量按连接内存、心跳、编辑率和每文档扇出估算，而不是只报机器连接上限。

继续追问：目录短暂返回旧 owner 时，客户端会看到什么？

常见漏项：只会增加 socket server，忽略状态共置与双主写入。

现场修复

补一张 owner epoch 切换图，并模拟旧连接迟到提交。

证据
  • Google Docs · 1) How do we scale to millions of websocket connections?
  • Real-time Updates · Server-Side Push/pull
Q5

什么时候可以删除被新快照覆盖的操作日志？

为什么面试官会追：候选人生成快照后立刻清日志，忽略离线客户端和副本。

详细回答

新快照要标明 coveredVersion，并在独立位置完成内容哈希校验后才原子设为 current。日志删除水位取多个约束的最小值：所有耐久副本已拥有快照、允许的离线恢复窗口已越过、审计保留满足、没有正在从更早版本重建的任务。客户端 base 太旧时必须下载新快照，不能请求已删除区间。压缩失败或校验不一致时继续使用旧指针，旧日志保持可重放，避免一次后台任务破坏文档历史。

继续追问：文档很大时，分块快照如何保证目录与各块属于同一版本？

常见漏项：只按时间 TTL 删除操作，不记录覆盖和消费者水位。

现场修复

列出 snapshotVersion、replicaWatermark、offlineFloor 和 deleteFloor 的关系。

证据

  • 第十二章：流处理 · 失败后重建状态

LeetCode

Q1

用户代码死循环、fork 炸弹或尝试访问网络时，哪一层负责阻止？

为什么面试官会追：候选人只说使用容器，没有列出具体隔离与资源策略。

详细回答

宿主调度层创建一次性 sandbox：非特权用户、只读镜像、受限系统调用、独立 PID/网络命名空间、无外网、临时磁盘配额，以及 CPU、内存、进程数和输出字节硬限制。watchdog 依据墙钟与 CPU 时间终止任务，秘密测试通过受控输入注入。容器退出后销毁文件系统，节点达到运行次数或出现异常就回收。安全违规、资源超限和普通 WA 分开记录，便于发现攻击而不污染题目统计。

继续追问：语言运行时本身需要子进程时，怎样设置进程限制而不误杀合法代码？

常见漏项：把 Docker 名称当作安全证明，未讨论网络、系统调用和宿主逃逸面。

现场修复

列出 sandbox 能访问的 CPU、内存、文件、网络和系统调用白名单。

证据

  • LeetCode · 3) Users should be able to submit their solution and get instant feedback
Q2

一份 submission 有 200 个测试，怎样并行又保证 verdict 可复现？

为什么面试官会追：候选人让 worker 各自直接覆盖最终状态。

详细回答

协调器用 submissionId、testCaseId、testSuiteVersion 生成稳定任务，worker 只提交该 case 的不可变结果和 attempt 元数据。Aggregator 按题目规则等待必需水位：普通题可在确定失败后取消未开始任务，计分题则汇总全部分数；最终结果通过版本条件写一次进入 FINAL。重跑相同 case 使用同一环境版本，输出可用 hash 对比。迟到结果保留审计但不能倒退 finalVerdict，从而兼顾并行速度与确定性。

继续追问：取消信号到达时某些 worker 已在运行，如何处理其迟到结果？

常见漏项：把测试并发等同于多个服务同时写 Submission.status。

现场修复

画 TestExecution 表与 Aggregator 水位，模拟一条重复和一条迟到结果。

证据

  • Managing Long Running Tasks · Workers
Q3

worker 写完所有测试后崩溃，为什么用户仍能得到最终 verdict？

为什么面试官会追：候选人把汇总动作只放在执行进程内存中。

详细回答

每个测试结果先持久化并带唯一任务键，完成写会产生可重复消费的 result event。Verdict Aggregator 从 TestExecution 表或事件水位判断 required cases 是否齐全，条件写 finalVerdict；若它崩溃，下一实例重新读取相同集合会计算同一结果。submissionId 的 FINAL 状态不可被普通更新覆盖，重复汇总返回已存在 verdict。监控长期 RUNNING 但测试齐全的记录并重新触发聚合，避免静默卡住。

继续追问：测试规则升级后，需要重判旧 submission 时怎样保留两个结果？

常见漏项：依赖队列消费者恰好一次，未给聚合结果版本和修复扫描。

现场修复

补充 evaluationVersion、requiredCount 和 stuck-finalizer 巡检。

Q4

实时榜单丢失一个 Redis 分片后，如何恢复且不改错名次？

为什么面试官会追：候选人把 sorted set 当成唯一计分数据库。

详细回答

Redis 只保存按 contestId 分片的排名投影。每个 FINAL verdict 产生不可变 ScoreEvent，eventId 去重，排序字段明确总分、罚时与 tie-break。恢复时加载该比赛最近 checkpoint 的用户累计分，再按事件序号重放；期间可以返回带 asOf 水位的旧榜或暂时降级。重建完成后抽样与事实事件全量聚合比较，比赛封榜必须以可审计事件重算结果，而不是信任事故前的内存排名。

继续追问：跨分片 top 100 如何合并，游标怎样保持稳定？

常见漏项：只讨论 Redis 主从切换，无法从业务事实证明榜单正确。

现场修复

定义 ScoreEvent schema、checkpoint 水位和最终封榜对账。

Q5

十万人在最后一分钟提交，系统应该拒绝、排队还是扩容？

为什么面试官会追：候选人只报自动扩容，没有反馈时限和公平策略。

详细回答

先快速持久化带比赛截止时间和 clientSubmitKey 的 Submission，接受与否以服务端收到时间为准；执行可以在赛后继续。队列按 contestId、语言和资源级别分流，每用户限制并发，预热池吸收预测峰值，并给非竞赛流量保留配额。若 queue-to-start 超过目标，暂停非关键分析、显示预计等待并限制重复提交。扩容依据最老任务、各语言槽位和 sandbox 启动耗时，不能让 API 成功率掩盖判题反馈已严重落后。

继续追问：截止前收到但截止后才执行的提交，排名时间取哪个时刻？

常见漏项：把请求处理时间当比赛资格，或允许重试改变提交时间。

现场修复

明确 acceptedAt、executionAt 与 leaderboard penalty 的关系。
