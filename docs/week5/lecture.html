Google Docs

Google Docs：从需求语义到故障修复

01

编辑协议传的是操作，不是整份文档

Google Docs 客户端不能每敲一个字就上传整篇正文，否则两个用户会用各自旧副本互相覆盖。编辑消息应包含 documentId、clientId、clientSeq、baseServerVersion 以及 insert/delete 的位置和内容；服务端接受后分配单调 serverVersion，并把规范化操作追加到文档日志。clientSeq 在单客户端内唯一，重发同一操作只返回原确认。Document 快照代表某个 coveredVersion，加载时先取快照，再补该版本之后的操作。操作日志还要保存协议版本，旧客户端字段经规范化后再进入统一序列，避免协议升级把同一份历史文档悄悄分叉。

能迁移的判断：用可去重的细粒度操作和服务端版本构成协作事实。
02

并发编辑必须有确定性收敛规则

Alice 与 Bob 都在 version 40 的同一位置插入字符时，服务端不能按网络到达顺序让不同副本各自解释。采用 OT 时，后到操作要相对于已接受操作做 transform，并以 clientId 等稳定规则处理同位置；采用 CRDT 时，字符位置标识和并发顺序也必须确定。关键不是背算法名字，而是证明所有客户端应用同一规范操作序列后得到相同文本。删除与插入重叠、两个删除交叉、离线批量回放都应写成测试向量，升级合并逻辑还要保留 algorithmVersion。

能迁移的判断：选择一种合并模型，并用具体冲突向量证明所有副本收敛。
03

确认丢失时靠 clientSeq 去重

服务端可能已持久化 edit 并广播，但给发起者的 ack 在网络中丢失。客户端重连后会再次发送 clientId、clientSeq；协作分片先查询去重记录，若已接受就返回对应 serverVersion，而不是重复插入文字。客户端维护 lastAppliedServerVersion，收到广播若版本已有则忽略，若出现缺口先暂停实时应用并请求缺失区间。只有在连续补齐后才恢复 live stream，避免界面看似在线却永久少一段编辑。去重记录必须与文档日志同一事务提交，不能只放在会随连接重启消失的内存里。

能迁移的判断：发送序号解决重复，服务端序号发现缺口，两者职责不同。
04

documentId 决定协作分片与连接归属

同一文档的活跃编辑应汇聚到一个带 ownershipEpoch 的协作分片，才能顺序化操作并及时广播。连接入口查询目录，把 documentId 路由到当前 owner；扩容时新 owner 先载入快照与日志水位，再提升 epoch，旧 owner 即使仍持有 WebSocket 也不能提交操作。百万连接的容量要按每连接内存、心跳频率和每文档消息扇出计算，热门文档可独占分片，但不能拆成两个同时接受写入的 owner。目录缓存必须采用短时有效期，并由写存储复核所有权世代，才能挡住网络分区后迟到的旧连接。

能迁移的判断：连接可以分散，单文档写入所有权必须带 epoch 转移。
05

快照压缩要保留可验证的覆盖水位

操作日志无限增长会拖慢打开文档和灾难恢复。压缩 worker 读取 snapshot@1000 与之后连续操作，生成 snapshot@5000，验证内容哈希后原子发布新指针；只有当所有需要的离线窗口、审计策略和副本都越过 5000，旧日志才可删除。压缩任务按 targetVersion 幂等，失败不影响当前快照。若新快照校验不一致，保留原日志并报警，而不是把错误文本设为基线。大文档可分块快照，但块目录也必须对应一个一致版本。

能迁移的判断：先证明新快照完整覆盖，再推进删除水位。
06

presence 是软状态，正文不是

光标、选区和在线头像允许短暂丢失，应通过心跳和 TTL 存在内存或快速存储中；客户端每隔一段时间上报 cursor 相对当前 serverVersion 的位置，其他人若版本不同可暂缓或变换显示。协作服务重启后 presence 自然过期并由在线客户端重报，不需要从正文日志恢复。运营时分别看编辑确认 p99、远端可见延迟、serverVersion 缺口与收敛哈希；presence 则看心跳超时、幽灵用户和广播丢弃。把两类指标混在一起会掩盖真正的数据分叉。

能迁移的判断：可重建的在线感知走 TTL，文档内容走耐久版本日志。

LeetCode

LeetCode：从需求语义到故障修复

01

Submission 必须冻结题目与运行环境

LeetCode 用户点击提交时，系统创建不可变 submissionId，记录 sourceObject、languageRuntime、problemVersion、testSuiteVersion、比赛身份与 clientSubmitKey。题目后来修正测试或运行镜像升级，都不能悄悄改变已经给出的 verdict；重判应产生新的 evaluationVersion 并保留原因。源码体放对象存储，Submission 表只保存哈希和元数据。客户端重复点击使用同一 submitKey，返回原 submissionId，避免一个意图在队列里复制成多份昂贵执行。题目版本与运行镜像摘要共同写入审计记录，客服和出题人才能完整复现一次有争议的判题。

能迁移的判断：冻结代码、题目和环境版本，判题结果才可复现。
02

不可信代码运行在一次性隔离舱

执行 worker 不应直接在宿主进程运行用户代码。每个 TestExecution 启动只读基础镜像，挂载限额临时目录，使用非特权身份，关闭外网并限制系统调用、CPU 时间、内存、进程数和输出字节；超限由宿主 watchdog 终止。测试输入通过受控管道注入，标准输出截断后归档，秘密测试不暴露给容器。执行节点定期回收而非长期复用未知状态，镜像只从签名白名单加载。隔离违规和资源超限是不同 verdict，也要分别统计。

能迁移的判断：安全边界由宿主强制资源与能力限制，不能相信用户进程自觉退出。
03

测试用例可以并行，最终判决只能汇总一次

协调器把 submissionId 与 testCaseId 组成稳定任务键，按测试组并行派发；worker 回写 duration、memory、outputHash 和 result，重复尝试以 attempt 记录但不覆盖已确认结果。某些题可遇到首个失败就取消剩余任务，比赛评分题则可能需要全部分数。Verdict Aggregator 等待 required test 水位，按 problemVersion 的确定规则生成 finalVerdict，并用条件写从 RUNNING 推进到 FINAL。迟到 worker 只能补审计，不得把最终 WA 改回 AC。聚合器还要保存已消费测试用例集合的摘要与完成水位，以便及时发现测试遗漏、重复执行或版本混用。

能迁移的判断：执行并行化提高速度，submission 状态机守住唯一最终结果。
04

队列按语言和资源画像隔离

不同语言启动成本、内存和运行时长差异显著；把所有任务放进一个 FIFO，会让一批慢测试阻塞轻量提交。入口根据 languageRuntime、预计资源级别与比赛优先级选择队列，执行池预热对应镜像并设置并发配额。重试任务使用单独预算，基础设施错误可换节点重跑，确定性的编译错误不重试。调度指标包括 queue-to-start p95、各语言槽位利用、超时分布和最老任务，扩容要看瓶颈是冷启动、CPU、内存还是镜像拉取。

能迁移的判断：异构执行需要分类排队和配额，而不是一个无差别 worker 池。
05

Leaderboard 是可重建的排名投影

最终 verdict 生成 ScoreEvent，包含 contestId、userId、problemId、score、penalty 和 acceptedAt。实时榜可用 Redis sorted set 或内存分片维护，但耐久事实仍是不可变 ScoreEvent；重复 eventId 去重，用户总分变化通过确定的增量更新。排序规则必须编码总分、罚时和稳定 tie-break，不能只存一个浮点 score。比赛封榜时从事实事件重算最终名次，与实时投影对账；若 Redis 丢失，先恢复最近 checkpoint，再重放后续事件。实时榜单投影携带评分事件水位，客户端才能区分展示陈旧与名次真实变化。

能迁移的判断：实时排名追求快，最终排名必须能从计分事实重算。
06

十万人开赛要保护执行面而非只扩 API

比赛开始和临近结束会形成提交尖峰，真正昂贵的是编译与测试，不是创建一行 Submission。API 可快速持久化请求并返回 QUEUED，队列通过每用户并发限制和 contest 配额防止单人刷提交；执行池按预测提前预热，但保留全站普通练习的资源份额。若积压超过反馈目标，界面显示诚实等待时间，优先停止非关键分析任务。运营同时看接收成功率、判题最老年龄、每提交成本、sandbox 故障和榜单滞后，避免“API 还很快”掩盖反馈已经失效。

能迁移的判断：竞赛容量由安全执行槽决定，入口必须背压并展示真实进度。
