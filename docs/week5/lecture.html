答题主线

01 · 核心判断

服务合同与顺序范围

Live Comments 需要直播间内低延迟、有界乱序或单调序列；不需要全球顺序。系统应明确定义 late joiner 看最近多少、删除何时生效、慢消费者何时转拉取。

02 · 核心判断

状态与分区

comment 是 durable event；room sequence/offset 允许重连和补洞；socket connection 与短期 buffer 是 ephemeral state。以 room_id 分区便于局部排序，但热门房间需要分层 fanout。

03 · 核心判断

写入与扇出路径

评论先经鉴权和限流后持久化到 room log，再由 fanout service 推给 room subscribers。WebSocket gateway 处理连接，不应成为排序或消息持久化的唯一节点。

04 · 核心判断

加入、重连与慢消费者

late joiner 先获取 recent window 和当前 high-water mark，再订阅后续事件；断线后从最后确认 offset 补洞。慢连接的内存 buffer 有上限，超限时让客户端重新 pull。

05 · 核心判断

审核、删除与反滥用

审核决策和 delete/tombstone 必须进入同一房间事件流，确保已推送客户端能收敛。限流/反滥用优先保护 room partition 和 fanout，而不是只保护 API。

06 · 核心判断

容量与多地域

热门直播的瓶颈通常是订阅连接和 fanout egress，而非 comment 写入本身。room ownership、regional edge gateways、backpressure 与跨区域复制必须有明确边界。

07 · 核心判断

迁移：FB News Feed

News Feed 的问题从实时广播转为按用户生成、排序和读取。普通用户可 fanout-on-write；celebrity 需要 fanout-on-read 或混合策略，以避免巨量写放大。

08 · 核心判断

可观测性与恢复

端到端 comment latency、fanout drop、buffer saturation、reconnect replay、moderation propagation 是用户语义指标。region 切换后以 room offset 和可重放日志恢复，不用依赖连接状态。
